# 出版审查清单与盲区分析

> 本文从【专业技术图书出版】的完整审查维度出发,对照本项目现有审查体系,系统梳理【已覆盖】和【漏掉/薄弱】的检查项。
>
> **核心结论**:项目在"文风/AI 味"和"内容技术准确性(agent 多视角核查)"两块做得远超业界平均,但在【技术正确性的持续回归】和【成书/合规(Production)阶段】有大片盲区——其中部分是**活体硬伤**,5 轮文风飞轮根本碰不到(文风审查不查技术一致性和出版合规)。
>
> 整理自三个独立 agent 的交叉研究:技术图书编辑视角(十年出版经验)、架构师视角(技术正确性)、业界出版标准调研(O'Reilly / Manning / Pragmatic / 国家三审三校)。2026-06。

---

## 0. 一句话结论

项目的审查火力 **~90% 集中在文风/AI 味**(`docs/去ai味道自查方法.md`)。文风擦得再亮,也补不了三类盲区:

1. **技术一致性的活体硬伤**——CAP 归类全书不自洽(ch9 没对齐 ch7 的谨慎立场)。
2. **从未做过的文字校对**——错别字/语病通读(出版社三校的核心,本项目零覆盖)。
3. **整个缺失的"成书/合规"阶段**——AI 辅助披露、书后索引、商标声明、参考文献著录格式。

这些是出版要求的**底线**,不是锦上添花。

## 1. 专业技术书的完整审查维度(业界标准)

按编辑阶段分六组。来源见 §6。

### 1.1 Developmental Editing(发展性/结构编辑)— 管"这本书值不值得出、能不能立住"

| 维度 | 审什么 |
|------|--------|
| 选题定位与市场必要性 | 解决什么问题、市面有无同类、读者凭什么买这本 |
| 目标读者画像与难度曲线 | 读者真实水平、前置知识、难度是否单调递进 |
| 整体结构与逻辑递进 | 章节顺序构成逻辑链、有无前置依赖倒置(用了后面才定义的概念) |
| 篇幅均衡与节奏曲线 | 各章字数比例、有无 saggy middle、章内密度起伏 |
| 钩子与导读兑现 | 每章导读能否抓人、是否承接上章、是否兑现 |
| 小结与"一句话收获" | 章末是否给出可带走的一句话、是否呼应导读闭环 |
| 教学法与可学习性 | 先动机后定义、抽象层逐步搭建 |
| 例证与真实场景密度 | 抽象论点有无具体例子/数字/报错支撑 |
| 横向对比的平衡与深度 | 三家笔墨均衡、对比讲清根因、口径一致 |
| 承上启下与伏笔回收 | 章间衔接、前面埋的概念后面用上 |
| 跨章重复与冗余 | 同一论点/措辞不同章是否变相复述 |
| 作者声音与判断一致性 | 全书语气统一、同一取舍前后说法一致 |
| 标题体系 | 书名/章/节标题是路标还是目录条目、层级 ≤3 |
| 金句密度与质量 | 金句稀缺(每章 ≤2)、可独立引用 |

### 1.2 Technical Review(技术审校)— 技术书出版社最不同于大众书的环节

**业界标准做法**(本项目用 agent 部分替代,但有同源盲区):
- **外部独立技术审校人**:Manning 派多名外部 reviewer、O'Reilly 作者自找 ~5 位外部专家、三审制复审引入同行外审。
- **分阶段多轮**:Manning 明确分 1P(完成 1/3)、2P(2/3)、3P(全书)三轮外审。
- **代码/命令/参数可复现性**:reviewer 实际跑一遍。
- **机制/算法对源码核对**(不是查文档,是查实现)。
- reviewer 在致谢署名。

| 维度 | 审什么 |
|------|--------|
| 机制描述准确性 | 单点机制、流程时序、算法、边界条件与源码/官方一致 |
| 代码/命令/SQL 可运行性 | 语法、版本存在性、默认行为、可复现上下文 |
| 版本与默认值 | 参数名、默认值、版本引入/移除表述、无越界参数 |
| 数字与 benchmark 可核实 | 有出处、有前提、不过时 |
| 图表正确性 | 编号一致、**语义**与正文一致、数值正确 |
| 横向对比公平性 | 三家均衡、不厚此薄彼、同口径、无过时信息 |
| 因果与 tradeoff 论证 | 因果链成立、两面都说、定性有支撑 |

### 1.3 Line / Substantive Editing(行文编辑)
段落论点句、过渡、节奏、长句/碎句平衡、作者声音一致性(不磨平)、比喻密度与去留。

### 1.4 Copyediting(文字编辑)
术语全书一致 + 缩写首次全称、中英文标点(全/半角)与空格、专有名词大小写、数字与单位间空格、二进制单位(KB/KiB)统一、标题层级体例、引用著录格式统一、**交叉引用逐条核验**、图表编号三重一致。

### 1.5 Proofreading(校对)— 三校一读
付印样最终通读;找 typo、漏字、排版错位、孤行寡行、图表错位;出版后勘误(errata)渠道。

### 1.6 Production / 成书制作(业界必做,本项目几乎全空白)
| 维度 | 业界做法 |
|------|---------|
| 版权与许可 | 引用第三方代码/图/表的授权、开源协议合规、商标规范 |
| 书后索引 | 专业 indexer 制作,技术书当工具书用的关键 |
| 术语表 | 独立成附录的面向读者的术语对照 |
| 版权页 | CIP、ISBN、版次、印次 |
| **AI 辅助披露** | Springer/Elsevier/IEEE 2023 起强制 |
| 作者简介/致谢/勘误 | 出版惯例 |

## 3. 🔴 活体硬伤(已发现、未修)

### 3.1 CAP / 一致性归类全书不自洽(P0)

ch7 已修成**谨慎立场**(line 86/190/232:"本书不作绝对的 AP/CP 一刀切归类""社区对此归类有争议""分区时少数派停止写入偏 CP"),但 **ch9 没对齐**:

| 位置 | 现状 | 问题 |
|------|------|------|
| `ch9:101` | "**Redis 本质上是 AP**" | 与 ch7"不作绝对归类"直接打架 |
| `ch9:59` | "这是 **AP**(高可用优先)的取向" | 用了 ch7 已避开的扁平定性 |
| `ch9:143` | "这和 Redis 一样是 **AP 选择**"(说 MySQL 异步复制) | 与 ch7"**MySQL 走 CP**"口径冲突 |
| `ch7:7`(导读) | "Redis 走 AP" | 导读用简写,后文才往回收 |

**修法**:确立全书统一口径(Redis Cluster = 异步复制可丢写/AP 倾向,但分区时少数派停写/CP 行为,社区有争议,本书不作绝对归类),把 ch9 三处 + ch7 导读对齐到这个口径。`docs/verify/` 2026-06-23 就标红了这个 P0,但修正没落地——见 §5 流程盲区。

### 3.2 量化数字无出处(P0)

抽查发现 **26 处裸数字 vs 7 处带源**,失衡严重。技术书的数字一旦被读者验证为"拍脑袋"或过时,公信力崩塌。重点裸数字:

| 位置 | 数字 | 问题 |
|------|------|------|
| `ch4:32` | "Redis fork 开销:约 1 ms/GB 实例内存" | 无出处,且依赖页表/内核/大页 |
| `ch6:24,143` | "Redis 单核十万级 QPS" ×2 | 无出处,7.x 多线程 IO 下"单核"已不准 |
| `ch7:150` | "单分区约 10MB/s 或约 1 万条/秒" | 无消息大小前提 |
| `ch8:46` | "1GB 内存实例 ≈ 300-500MB 快照" | 无出处 |
| `ch4:33` | "命中率在数据量低于缓冲池 10% 时断崖下跌" | "10% 断崖"无出处 |

**修法**:写确定性脚本扫所有 `\d+\s*(GB|MB|万|亿|TPS|QPS|ms|秒|条/秒)` 数字,排除已带括号注源的;产出裸数字清单交 agent,每条要么补出处、要么补前提、要么删。

### 3.3 命令/SQL 在基线版本的可运行性(P0)

全书散布大量 `SET/GET/SELECT/INSERT/redis-cli/kafka-topics` 片段,读者照敲。零覆盖。抽查疑点:
- `ch9:101` `min-replicas-to-write` + `min-replicas-max-lag` 在 7.x 的命名/别名。
- `ch6:41` ACL `SETUSER alice on >pwd ~keys:* +get +set` 在 7.x 的 access selector 语法。
- `ch9:151` 半同步变量是否混用了新旧名。

**修法**:扩展 `source-contributor` agent,把全书 backtick 内命令提取成清单,逐条对照官方手册该基线版本。

## 4. 系统盲区优先级清单(补法)

### 🔴 P0 — 必须做,直接影响"对不对、能不能出版"
1. **CAP 归类对齐**(§3.1)— ch9→ch7 立场,具体有界。
2. **AI 辅助披露**(合规底线)— 前言或后记加一段:AI 用于初稿/文风审查/图表,技术内容由作者把关;声明 AI 不署名。改动小。
3. **错别字/语病通读**(三校核心,从未做)— 性价比最高。脚本初筛常见错别字表 + 人工通读。
4. **verify 断言回归机制**(§5,治本)— 把 verify/ 的 ❌ 转成 grep 断言,改稿后回归。

### 🟠 P1 — 系统盲区
5. **裸数字溯源脚本**(§3.2)— 扫裸数字 → 补出处/前提/删。
6. **SVG 语义审查 agent** — svg_audit 治了几何,语义(箭头/流程/数值)完全空白。每图 agent 读源码对照正文。
7. **交叉引用脚本** — 248 处"第 N 章/节",解析 + grep 验证指向。脚本可 80% 自动化。
8. **商标声明** — 前言加一行:Redis®/MySQL/Kafka 商标归属。
9. **读者术语表附录** — 把 book-bible §7 扩成面向读者的 glossary(中英 + 一句话定义)。

### 🟡 P2 — 锦上添花 / 工具设施
10. **书后索引** — 脚本提候选词 + 人工精选 200-400 条,附章节锚点(HTML 可交互)。
11. **参考文献格式** — 补字段(作者/出处/年份/URL/DOI),统一 GB/T 7714 或 CMOS。
12. **全书前置依赖逻辑链** — 抽取"概念首次定义章"表,检查正文引用都在定义之后。
13. **小结一句话收获兑现** — 逐章 N.7 强制提取核心论点。
14. **勘误渠道** — errata.md 或 README 加"报告错误"入口。
15. **外部真人技术审校** — 找 1-2 位 Redis/MySQL/Kafka 有生产经验的人通读,弥补 agent 同源盲区(这是技术书生命线,agent 不能完全替代)。

## 5. 流程盲区:核查闭环没接上(最大的元问题)

`docs/verify/` 在 2026-06-23 做了一轮高质量核查(587 题,标 20 ❌ + 133 ⚠️)。但 §3.1 的 CAP 矛盾证明:**verify 是一次性快照,没有"修正 → 回归"闭环**。

- 后续任何改稿都可能把已修的弄回错的状态(CAP 就是:ch7 修了,ch9 没接)。
- verify 之后再也没复跑过,新增的错误查不到。

**治本**:把 verify/ 的所有 ❌ 和高优先级 ⚠️ 转成一条条确定性"断言"(如"ch09 不得出现'Redis 本质上是 AP'且无 CP 补充"),写进一个 assertions 清单;每次改稿后 grep 一遍。没有这个机制,前面所有核查都是一次性消耗品。
