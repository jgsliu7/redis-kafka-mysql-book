# 出版前审查工具包（给另一本书的 AI 用）

> 这是一套在《架构之道》（Redis/MySQL/Kafka 架构共性）成书过程中跑通、验收过的"发行前审查"方法论 + 工具 + Agent 人设 + 实战样本。把它整个拷给你的项目，按下面的说明套用即可。
>
> **核心信念**：不要相信任何一个 Agent。每个问题必须 ≥2 个 Agent 独立一致才算定论；冲突项查字节 / 查官方文档裁定。这一条规矩在本项目五轮审查里**每一轮都拦下了会引入错误修复的单 Agent 误报**。

---

## 一、整套方法在做什么

一本技术书发行前要过六道关（详见 `01-方法论/出版审查清单与盲区.md`）：

1. **发展性编辑**（选题/结构/难度曲线/导读兑现）
2. **技术审校**（机制准确性/命令可运行/数字可核实/图表正确）← 技术书最不同于大众书
3. **行文编辑**（段落/过渡/节奏/声音）
4. **文字编辑**（术语统一/标点/交叉引用/图表编号）
5. **校对**（三校：错别字/语病/排版）
6. **成书制作**（版权页/AI 披露/商标/参考文献著录/索引）← 最容易全空白

本项目的方法是：用 **专项 Agent** 部分替代外部审校人，但用 **多视角并行 + 交叉印证 + 回归断言** 补上"单 Agent 不可靠"的短板。

### `02-审查Agent人设/`（15 个现成审查员，复制即用）
每个 `.md` 是一个 Agent 的 system prompt。按职责分：

- **文风/AI 味**：`ai-writing-detector`（统计指纹）、`tech-editor`（阅读体验）
- **文字校对**：`typo-grammar`（错别字/语病/标点，三校）、`term-consistency`（术语统一）
- **技术准确性**：`architect-reviewer`（架构师）、`source-contributor`（源码贡献者）、`fact-consistency`（跨章自洽）、`number-provenance`（数字溯源）、`command-runnable`（命令可运行）
- **图表**：`svg-semantics`（图语义，非几何）
- **交叉引用**：`crossref-validity`
- **结构教学**：`structure-pedagogy`、`interviewee-reader`（2-3年读者视角）
- **定位**：`ddia-reader`（竞品差异化）
- **出版合规**：`compliance`（AI披露/商标/参考文献格式/版权页）

> 用法：在你的 Agent 工具里注册这些 system prompt，或直接把内容贴给 AI 当"审查员人设"。

## 三、推荐工作流（给另一本书的 AI）

```
第 0 步：搭底盘
  - 读 AGENTS.md + 出版审查清单与盲区.md，建立六维度心智模型
  - 把 15 个 Agent 人设注册进你的环境
  - 仿 pub_assertions.py 建一个空的回归断言脚本

第 1 步：多轮分域审查（至少 5 轮，每轮一个域）
  每轮派 ≥2 个独立 Agent 扫同一范围：
    轮1 去 AI 味：ai-writing-detector + tech-editor
    轮2 文字校对/术语：typo-grammar + term-consistency + 独立第三票
    轮3 技术事实：fact-consistency + number-provenance + command-runnable
    轮4 引用/图表：crossref-validity + svg-semantics（+ 独立复核）
    轮5 合规/结构：compliance + structure-pedagogy（+ 独立复核）

第 2 步：交叉印证（关键）
  - 两 Agent 都命中 → CONFIRMED
  - 只一个命中 → 派独立验证员复核（读原文/查字节/WebSearch 官方文档）
  - 冲突 → 查字节/官方文档裁定，绝不靠肉眼/记忆

第 3 步：写每轮报告（CONFIRMED + 已驳回 + 优化动作）

第 4 步：落地修复 + 验收
  - 改完跑回归断言 + 几何审计 + 重建
  - 把新修的正确事实补成断言，锁死防回弹
```

## 四、本项目踩过的坑（直接拿走，别再踩）

1. **单 Agent 既会假阳性也会假阴性**：本项目五轮各拦下过——"半角分号 7 处"（全假）、"短暂双写改脑裂写"（改了反而错）、"RDB version 0010 是二进制错"（其实是 ASCII 正确）、"商标声明齐全"（实际漏 ZooKeeper/InnoDB）。**所以必须 ≥2 Agent。**
2. **字符类问题必须查字节**：引号是弯是直、分号是全角半角，肉眼会看错，用 `python3 -c "ord()"` 或 grep 字节码裁定。
3. **版本/参数类必须查官方文档**：命令默认值、参数改名、版本引入，凭记忆会错（本项目抓出 `allow.everyone` 张冠李戴、LDAP 插件漏前缀、日志加密版本错）。WebSearch 官方文档/源码。
4. **CAP 这类有争议的归类要全书一个口径**：定一次"不作绝对归类 + 说清行为边界"，全书照抄，并写成断言防回弹。否则改一稿弹回一稿。
5. **核查闭环不接上是最大元问题**：一次性审查的成果会被后续改稿改回去。必须把已核实的事实编码成 grep 断言（pub_assertions.py），让一次性核查变持续回归。
6. **成书/合规阶段最容易全空白**：版权页、致谢、作者简介、勘误渠道、AI 披露、商标声明、参考文献 GB/T 7714 著录——这些是"能不能出版"的底线，却最容易被技术审查忽略。
7. **本环境无法看图**：插图 QA 靠 svg_audit.py（几何）+ svg-semantics agent（语义）+ 源码坐标推理，没有"肉眼看图"这条路。
