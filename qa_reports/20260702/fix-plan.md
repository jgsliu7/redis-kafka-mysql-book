# 🟠🟡 批量修复计划

> 总计 ~116 处。按病理类型分 5 批，每批独立执行、验证、提交。
> 原则：修同一类问题不改其他东西，出问题回滚只丢一批。

## 批次规划

| 批 | 病理 | 数量 | 风险 | 预估 |
|----|------|------|------|------|
| 1 | 说教腔 + 导师式交付 | ~46 | 中 | 主要挑战：需要读上下文判断改法 |
| 2 | 自述结构 N2 | ~20 | 低 | 机械替换为主 |
| 3 | 破立句式残留 | ~14 | 中 | 需要改句式结构 |
| 4 | 广告腔 N5 + 绝对化断言 | ~17 | 低 | 词汇级替换 |
| 5 | 其余（N1/N3/N4/N6/N7/N8 + 零散） | ~19 | 低 | 杂项收尾 |

## 每批工作流

```
1. READ    → agent 读所有涉及文件，列出精确 old→new
2. FIX     → 逐条 Edit
3. BUILD   → build_html.py + build_pages.py
4. VERIFY  → agent 通读修改处，确认无衔接断裂/新问题
5. COMMIT  → git commit 单批
```

## 回滚策略

每批独立 commit。如果验证不通过，`git reset --hard HEAD~1` 回到上一批，重新设计改法。

## 执行顺序

1 → 2 → 3 → 4 → 5，串行。上批验证通过才开下批。
