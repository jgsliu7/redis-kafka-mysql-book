# F06: 版本保鲜期审查报告

审查日期: 2026-07-03
审查员: version-shelf-life agent
基准版本: Redis 7.x / MySQL 8.0.x / Kafka 3.x (Tiered Storage 以 3.9+ 为准)

---

## 摘要

全书在版本标注纪律上表现良好——大部分断言有版本号标签，架构规律与实现细节的区分基本清晰。但最大问题是**版本基线本身已严重滞后**。

当前（2026-07）各软件最新版本：

| 软件 | 书中基线 | 当前最新 | 基线状态 |
|------|---------|---------|---------|
| Redis | 7.x | 8.8.0 | 7.x LTS 仍在维护，但 8.x 是主流 |
| MySQL | 8.0.x | 9.7.1 LTS | **8.0 本月（2026-07）EOL** |
| Kafka | 3.x | 4.3.1 | **3.x 已 EOL** |

MySQL 8.0.x 和 Kafka 3.x 均已或即将（本月）到达生命周期终点。书中以这两个版本线为锚点的细节性断言面临最大过时风险。

以下按严重度组织发现。

---

## 过时风险高（版本变了会变成错误或误导）

### F06-R1: 全书版本基线已严重滞后
- **文件**: chapters/01-introduction/chapter.md:210
- **原文**: "Redis 7.x、MySQL 8.0.x、Kafka 3.x"
- **当前情况**:
  - MySQL 8.0.x 标准支持截止于 2026-07-31（本月）。书中所有以 8.0.x 为锚的描述，读者若不迁移到 8.4 或 9.7，会认为自己的版本仍在主流支持期内。
  - Kafka 3.x 已完全 EOL。Kafka 4.0（2025-03）已移除 ZooKeeper。书中多处对 ZooKeeper/KRaft 的双轨描述现在已经是纯历史记载。
  - Redis 7.x 仍在 LTS 支持中（7.4.9），但 8.8.0 是当前最新。
- **风险**: 读者不知基线版本已 EOL，可能会继续在 EOL 版本上使用书中的配置建议。
- **建议**: 在再版时更新基线声明，或至少在前言增加"本书撰写时的版本基线"与"当前版本状态"的对照。

### F06-R2: Kafka "3.x 依赖 ZooKeeper" 的双轨描述已成为历史
- **文件**: chapters/03-lifecycle/chapter.md:142
- **原文**: "KRaft 模式自 3.3 起对新集群进入生产可用（KIP-833），3.5 起 ZooKeeper（ZK）模式被标记弃用...3.9 是最后一个支持 ZK 模式的 bridge release，4.0 彻底移除 ZooKeeper"
- **当前情况**: Kafka 4.0 已发布（2025-03），确实彻底移除了 ZooKeeper。3.x 已 EOL。书中对 ZK 模式的详细描述现在只对遗留系统有参考价值。
- **风险**: 读者在 2028 年阅读时，可能从未见过 ZK 模式。书中大量"KRaft vs ZK"的对比将成为需要额外解释的历史背景。
- **建议**: 把 KRaft/ZK 双轨描述改写为"KRaft is the only mode since 4.0; ZK mode described here was the pre-4.0 approach"，作为架构演化的案例保留。

### F06-R3: Kafka "x 版本起默认"的默认值基准随时间漂移
- **文件**: chapters/07-cluster/chapter.md:162
- **原文**: "acks=all（自 Kafka 3.0 起成为默认，对应 KIP-679）"
- **当前情况**: 在 Kafka 3.0 中 acks=all 是默认。在 Kafka 4.x 中这个默认值仍然成立。不是错误，但"自 3.0 起"的说法在 4.x 语境下需要读者自己翻译当前版本的适用性。
- **建议**: 可以补一句"该默认值在 4.x 中未改"。

## 架构规律与版本细节未清晰区分

### F06-O1: MySQL 缓冲池相关默认值跨版本稳定但未标注
- **文件**: chapters/04-memory-disk/chapter.md:137
- **原文**: "innodb_old_blocks_time（默认 1000 毫秒）"
- **分析**: 这个参数名和默认值自 MySQL 5.x 以来未变过，在 8.0、8.4、9.7 中沿用。上下文中是 InnoDB 通用的设计描述，没有标注"在 8.0 中"。因为该值长期稳定，风险较低。
- **建议**: 可补"（该默认值自 MySQL 5.x 以来未变，在 8.4、9.7 中沿用）"

### F06-O2: innodb_redo_log_capacity 默认值已标版本（做得好）
- **文件**: chapters/08-storage-format/chapter.md:140
- **原文**: "8.0.30 起改为由 innodb_redo_log_capacity 统一管控的动态容量（默认 100MB）"
- **分析**: 明确标了版本（8.0.30），做得很好。该参数在 8.4/9.7 中继续存在，默认值未变。
- **建议**: 维持现状。可视为版本标注的正面范例。

### F06-O3: Kafka acks=1 作为"中间档"的举例依赖历史变迁知识
- **文件**: chapters/04-memory-disk/chapter.md:297
- **原文**: "Kafka 的 acks=1（3.0 之前的默认；3.0 起默认已升至 acks=all，这里指主动下调到中间档的场景）"
- **分析**: 准确但依赖于读者了解 3.0 变迁史。4.x 用户不知道"3.0 之前的默认"的含义。
- **建议**: 可将"中间档"的表述改为"主动下调的档位"，减少对历史默认值的依赖。

## 长期有效（架构规律，不受版本影响）

- **状态机复制模型**（ch09:30）：分布式系统基础理论。

- **存储单元要对齐访问单元**（ch08:270）：格式设计的第一原则。
- **存储格式是访问模式的镜像**（ch08:260）：格式设计的根本规律。

## 保鲜期总评

| 项目 | 评分 |
|------|------|
| 版本标注完整性 | A-。大部分参数和版本特定行为有版本号标签。做得比绝大多数技术书好。 |
| 架构规律 vs 版本细节区分 | A-。书中的"架构启示"章节通常独立于版本细节，迁移性好。 |
| 时间敏感措辞 | B。全书避免了"目前""最新版本"这类无锚措辞，但未对基线的 EOL 状态预警。 |
| 参数默认值的"为什么" | A。几乎所有参数在给出默认值的同时给出了设计推理。 |

### 全书估计保鲜期

- **架构规律部分**（约 40%）：**10 年以上**。第 5 章分层规律、第 7 章 CAP 取舍、第 8 章格式设计原则、第 10 章五条共性规律——与版本无关。
- **实现机制部分**（约 45%）：**5-7 年**。各章对机制的原理性描述（B+ 树、Raft、WAL、LRU）在主要版本更迭中仍正确。
- **参数/默认值/具体行为部分**（约 15%）：**2-3 年**。过时风险最高的内容。

### 最易过时的 3 个断言

1. **"MySQL 8.0.x"作为全书的 MySQL 版本基线**（ch1:210）—— MySQL 8.0 本月 EOL。即使 InnoDB 机制几乎全部延续到 8.4/9.7，读者需要知道这一点才能正确判断书中内容的适用范围。
2. **Kafka 依赖 ZooKeeper 的描述**（各章多处）—— Kafka 4.0 已彻底移除 ZooKeeper。书中"KRaft vs ZK"的双轨对比已成为纯历史话题。
3. **Kafka 3.x 默认值的声明**（如 ch7:162 "自 Kafka 3.0 起默认"）—— 这些默认值虽大多在 4.x 中未变，但"3.x"的标签会把读者限制在已 EOL 的版本上思考。

### 最耐久的 3 个规律

1. **"先定真相之源，其余皆为派生"**（ch10:43）—— 有状态系统设计的最高原则。
2. **"把随机变成顺序，把一次变成一批"**（ch10:57）—— 硬件物理规律不变，这个优化原则就永远有效。
3. **"用'层'把可变性关进笼子"**（ch10:51）—— 软件工程的永恒主题。
