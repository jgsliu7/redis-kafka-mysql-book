## 🟡 轻微不一致

### 1. ch07 表 7-1：MySQL "CAP 立场"单元格措辞不对称

**位置 A：** chapters/07-cluster/chapter.md:190（表 7-1 第三行）
**原文：**
```
| CAP 立场 | 偏 AP（异步复制可丢写，但分区时少数派停止写入偏 CP；社区有 AP/CP 之争，详见 7.2.3） | CP（少数派分区停服） | 3.x 默认偏安全（acks=all），可由 acks / unclean.leader.election 在 AP↔CP 间滑动 |
```

**对比位置 B：** chapters/07-cluster/chapter.md:232（第 7 章小结）及 chapters/09-data-sync/chapter.md:143（正文展开）
**原文：**
- ch07:232："MySQL 走 CP 路线，用 Paxos 多数派确认保住强一致"
- ch09:143："就这一默认形态而言，它和 Redis 一样是异步可丢写的偏 AP 选择...但要注意，这只是 MySQL 复制的默认形态，不是它的产品定位：MySQL 通过半同步和组复制把一致性一路推到强一致...整体定位偏 CP"

**矛盾点：** 表 7-1 将 MySQL 的 CAP 立场简写为"CP"（纯大写、无"偏"字），而 Redis 的单元格则带有完整的"偏 AP + 分区时偏 CP + 社区有争议 + 详见交叉引用"四重限定。MySQL 单元格与 Redis 单元格在措辞繁琐度上严重不对称。正文中 MySQL 的表述始终是"产品定位偏 CP"或"整体定位偏 CP"，从未使用无"偏"字的纯"CP"标签。读者若只扫表 7-1，可能得出"MySQL = CP（绝对定位），Redis = 偏AP（含争议）"的不对称印象。

**严重度：** 🟡 中。不构成事实错误（全文已说清 MySQL "默认复制偏 AP + 产品定位偏 CP"），但表格有误导风险。

**建议统一口径：** 将表 7-1 MySQL 单元格改为：
```
偏 CP（产品定位；少数派分区停服，详见第 7 章正文）
```
或至少加"偏"字：`偏 CP（少数派分区停服）`。与 Redis 单元格的限定级别对齐。
