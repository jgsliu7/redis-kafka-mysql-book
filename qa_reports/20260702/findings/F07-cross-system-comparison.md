### 发现 #1 (🔴) — Ch4 L189 磁盘顺序写吞吐 vs 内存随机读吞吐

**文件**: `chapters/04-memory-disk/chapter.md:189`
**严重度**: 🔴 口径不一致导致错误结论

**对比主题**: 磁盘顺序写吞吐 vs 内存随机读吞吐

**口径问题**: 书中将 SATA SSD 顺序写（300-500MB/s）/ NVMe 顺序写（2-14GB/s）与内存随机读（单线程几百 MB/s、多通道 GB/s 量级）放在同一句直接比较，并得出结论"磁盘顺序写在吞吐上能追平甚至超过内存随机访问"。这里同时存在两个口径错位：操作类型不一致（写 vs 读）和访问模式不一致（顺序 vs 随机）。

原文：
> SATA SSD 顺序写吞吐约 300 到 500MB/s，NVMe SSD 可达 2–14GB/s...而内存随机读的吞吐在单线程条件下约几百 MB/s、多通道下可达 GB/s 量级。磁盘顺序写在吞吐上能追平甚至超过内存随机访问。

**三家实际口径**:
- Redis: 把 100% 数据放内存，延迟亚毫秒级
- MySQL: 数据在磁盘，Buffer Pool 在内存
- Kafka: 数据在磁盘（顺序写），PageCache 在内存

**为什么危险**: 作者想论证"Kafka 用磁盘也能达到高吞吐是合理的"，但因为比较的是"磁盘顺序写"和"内存随机读"，读者可能会误解为"磁盘在大多数情况能和内存一样快"。正确的是比较"磁盘顺序写 vs 内存顺序写"或"磁盘随机读 vs 内存随机读"。

**建议**: 改为"磁盘顺序写的吞吐量级（NVMe 可达 GB/s）已经接近内存的带宽水平（多通道 DRAM 可达 GB/s 量级），差异主要在延迟而非吞吐。"或者直接比较同类操作：顺序写吞吐（磁盘 vs 内存）。

> 注：F02 已发现此问题，本审查确认其完全成立。

### 发现 #2 (🟠) — Ch7 跨章节 CAP 口径不一致：Table 7-1 说 CP，正文说异步

**文件**: `chapters/07-cluster/chapter.md:190`（Table 7-1）与 `chapters/07-cluster/chapter.md:7`（正文导读）
**严重度**: 🟠 口径不一致但结论大致正确

**对比主题**: MySQL CAP 立场

**口径问题**: Ch7 Table 7-1 的"CAP 立场"行直接把 MySQL 写为"CP（少数派分区停服）"，但同章导读 L7 明确写着：

> MySQL 默认复制形态偏 AP（异步）、产品定位偏 CP（靠半同步/MGR 才走到宁可停服也不丢一笔）

这是一个严重的"各自为政的论断放在一起构成误导"案例。MySQL 的默认复制（异步）在 CAP 术语里明确偏 AP：主节点写完不等从节点、故障切换时已提交事务可能丢失。Table 7-1 直接用 CP 简化会导致读者误以为 MySQL 在任何配置下都是 CP。Ch9 Table 9-1 则处理得更好，把 MySQL 的一致性模型描述为"异步/半同步/多数派强一致（连续谱）"。

**三家实际口径**:
- Redis: 书里各处都注明"偏AP（归类有争议）"
- MySQL: Table 7-1 写 CP，同章正文写默认异步（偏 AP）
- Kafka: "3.x 默认偏安全，可由 acks/unclean.leader.election 在 AP↔CP 间滑动"

**影响分析**: 这一对比在跨章间自相矛盾。Ch7 同一章内正文和表格打架，Ch9 又用了不同的口径（"连续谱"）。读者读 Ch7 Table 7-1 时认为 MySQL 是 CP，读 Ch9 Table 9-1 时则发现 MySQL 是从异步到强一致的谱。两表口径不一致，且 Ch7 Table 更误导。

**建议**: Ch7 Table 7-1 的 MySQL 格应改为"异步/半同步/MGR 在 AP↔CP 间可调（默认异步偏 AP）"，与 Ch9 Table 9-1 保持一致。

### 发现 #3 (🟠) — Ch1 Table 1-1 "性能取向"行列示不同类指标

**文件**: `chapters/01-introduction/chapter.md:117`
**严重度**: 🟠 口径不一致但结论大致正确

**对比主题**: 三款软件的性能取向

**口径问题**: Table 1-1 的"性能取向"一栏：
- Redis: "低延迟"
- MySQL: "稳定事务语义"
- Kafka: "高吞吐"

"低延迟"和"高吞吐"是同类指标（都是性能维度），但"稳定事务语义"是正确性/行为维度，不是性能指标。在同一张"性能取向"的表格行里混入非性能指标，口径不一致。

**三家实际口径**:
- Redis 页面：性能取向确实是低延迟
- MySQL 本身：性能取向通常描述为"稳定的事务吞吐"而非"稳定事务语义"
- Kafka 页面：性能取向确实是高吞吐

**为什么危险**: 读者会疑惑"MySQL 的性能取向到底指什么"——事务语义不直接对应可测量的性能指标。如果这一行是"性能取向"，则三栏都应该是可测量的性能维度（延迟/吞吐/QPS/IOPS）；如果是"设计取向"，应该统一表述为"延迟优先/正确性优先/吞吐优先"。

**建议**: 将该栏名改为"核心设计取向"或"首要优化目标"，并将 MySQL 改为"事务一致性"或"正确性优先"，统一表述层级。

### 发现 #4 (🟠) — Ch1 Table 1-1 "关键取舍/可舍弃"行列示不同层面

**文件**: `chapters/01-introduction/chapter.md:116`
**严重度**: 🟠 口径不一致但结论大致正确

**对比主题**: 各系统的关键取舍

**口径问题**: Table 1-1 的"关键取舍/可舍弃"一栏：
- Redis: "内存容量与复杂聚合"
- MySQL: "极致写性能"
- Kafka: "低延迟与灵活查询"

Redis 的"可舍弃"是"内存容量与复杂聚合"——这是容量和功能层面的取舍。MySQL 的"极致写性能"是性能层面的取舍。Kafka 的"低延迟与灵活查询"混合了性能（低延迟）和功能（灵活查询）两个层面。三个系统的可舍弃属性不在同一条坐标轴上。

**三家实际口径**:
- Redis 的确在容量和复杂聚合上受限
- MySQL 的确写性能受事务开销拖累
- Kafka 的确延迟高且不支持灵活查询

**为什么危险**: 读者无法直接横向对比——Redis 让的是"容量和功能"，MySQL 让的是"写速度"，Kafka 让的是"延迟和查询能力"。三者不是同一个维度上的取舍。

**建议**: 统一到"数据模型能力 / 性能 / 扩展性"三个维度中的 1-2 个维度。或者拆成两行："可让出的性能指标"和"可让出的功能能力"。

### 发现 #5 (🟡) — Ch1 L41 vs Ch7 L41: Kafka 吞吐口径粒度不一致（跨章）

**文件**: `chapters/01-introduction/chapter.md:41` 与 `chapters/07-cluster/chapter.md:41`
**严重度**: 🟡 笔墨不均

**对比主题**: Kafka 吞吐数字粒度

**口径问题**: 
- Ch1 L41（引言概览）: "LinkedIn 公开基准里 3 台廉价机曾测到约 200 万次写入/秒（TPS）"

这些数字本身并不矛盾（一个是集群 3 台机器，一个是单分区），但书中未明确说明粒度差异。Ch1 的"200 万 TPS"是 2014 年 Kafka 0.8.1 的基准，F04 已标记为过期。而"几十万到百万"与"1-5 万/分区"之间的关系也未做桥接——读者自然想知道"200 万 TPS 大概需要多少个分区、多少台 Broker"。

Redis, MySQL 没有类似的多粒度数字对比问题，所以 Kafka 自己出现了跨章粒度不一致。

### 发现 #6 (🟡) — Ch4 Table 4-3 "崩溃恢复代价"定性与定量混排

**文件**: `chapters/04-memory-disk/chapter.md:255`
**严重度**: 🟡 笔墨不均

**对比主题**: 崩溃恢复代价

**口径问题**: Table 4-3 "崩溃恢复代价"行：
- Redis: "加载 RDB / 回放 AOF（大实例数分钟）" ——有具体时间
- MySQL: "重放重做日志 + 恢复脏页" ——只有机制没有时间
- Kafka: "几乎无（PageCache 热度取决于 OS）" ——定性描述

Redis 给了具体时间数（"数分钟"），MySQL 只给了机制描述，Kafka 是定性描述。三者在同一行的信息密度不均衡，且给出的信息类型不同（时间 vs 机制 vs 定性）。

**三家实际口径**:
- Redis 大实例恢复的确可能是数分钟（取决于数据量）
- MySQL 恢复时间变化极大（取决于 checkpoint 频率、脏页量、未提交事务量）
- Kafka 恢复确实快（因为数据在磁盘上、不需要重放）

**为什么不是大问题**: 这种不均衡部分源于三家软件的特性差异——MySQL 恢复时间高度依赖负载，不适合给一个固定数字。但读者在表中看到 Redis 有明确时间、MySQL 没有，会下意识认为"MySQL 恢复比 Redis 快"——这个假设在大事务场景下可能是错的。

**建议**: 将 Redis 的时间描述也改为定性，统一口径。如 MySQL 改为"取决于脏页量与 checkpoint 频率，通常快于 Redis 全量加载"，Kafka 改为"几乎无需恢复（日志即数据）"。或统一加具体时间谱。

### 发现 #7 (🟡) — Ch3 Table 3-2 "关闭速度与可预测性"混用不同指标

**文件**: `chapters/03-lifecycle/chapter.md:199-206`（Table 3-2）
**严重度**: 🟡 笔墨不均

**对比主题**: 关闭速度与可预测性

**口径问题**: Table 3-2 "关闭速度与可预测性"行：
- Redis: "快（仅刷脏）" ——只刷脏数据
- MySQL: "慢且不可预测（事务回滚 + 脏页）" ——事务复杂
- Kafka: "中等（取决于受控关闭时 Leader 迁移耗时）" ——网络协调

这一行的口径本身是一致的（都在说关闭速度），但隐含的评估尺度不统一。Redis 的"快"是指纯单机刷盘，Kafka 的"中等"是指集群协调，MySQL 的"慢"是指事务清理。这三个"速度"的"计费项目"不同：Redis 计费的是"刷脏数据的时间"，Kafka 计费的是"等网络确认的时间"，MySQL 计费的是"回滚长事务的时间"。读者容易误以为三者比的是"同一个时钟下谁快谁慢"，实则各系统的"关闭完成"定义不同。

**建议**: 在该行下方添加注解，说明各系统的关闭完成定义不同：Redis 是"数据保证落盘"，MySQL 是"事务全部清理"，Kafka 是"Leader 已迁移"。或者将"关闭速度"拆为"单机关闭耗时"和"集群影响关闭耗时"两行。

## 总体评价

本书是跨系统对比的架构书，横向对比表格（N.5 节）在全书中占核心位置。总体而言，大多数对比表的维度设计是合理的——每行描述同一个维度，三栏填入各系统的做法。

主要问题集中在：
1. **发现 #1（Ch4 L189）**：这是唯一一个🔴问题，且 F02 已先发现。比较时混合了操作类型和访问模式。
2. **发现 #2（Ch7 Table 7-1 vs 正文）**：Ch7 Table 7-1 的 MySQL CAP 行与同章正文矛盾，且与 Ch9 Table 9-1 口径不一致。这是最需要修复的问题。
3. **发现 #3-4（Ch1 Table 1-1）**：引言总表的"性能取向"和"关键取舍"两行混合了性能、功能和正确性三个不同维度。虽然不导致错误结论，但给全书的口径一致性定了一个不那么严谨的基调。

其余 #5-#7 是程度较轻的小瑕疵，可以按优先级修复。
