## 新发现

### 1. 连环设问堆叠 - 三分法变体
**chapters/01-introduction/chapter.md L171 严重度：中**

原文：
```
**第一，先写日志还是先改数据？** 可靠存储系统的常见做法是先写日志（WAL）...
**第二，日志落盘的粒度多细？** 是每条命令落盘、每秒落盘，还是交给操作系统决定何时刷盘？...
**第三，日志记的到底是什么？** 这三款软件在这里分歧最大...
```

病理分析：已知变体只列了"连续 3+ 个设问句悬空不答"。这里的三问不仅连续，而且每一问的结构完全一致（加粗序号 + 问题 + 自答），形成"设问排比"叠加效应。三个问题并列关系，读起来像背 checklist，缺乏自然推进感。

建议：去掉序号"第一/第二/第三"，改成一气呵成的自然句式。

### 2. 破立句式新变体 - 统计对比型
**chapters/01-introduction/chapter.md L7 严重度：中**

原文：
```
能说出"避免锁竞争"的人占大多数，但能进一步说出"瓶颈不在 CPU 而在内存和网络"的，每个面试季最多一两个。
```

病理分析：已知破立句式的新变体——不是"不是A而是B"，也不是"关键不在X在Y"，而是"大多数人能说出A，但极少数能说出B"。核心语义结构完全相同（先贬低表面认识，再推崇深层认识），但措辞完全不同（"大多数...但...最多一两个"），grep 必然漏。

建议：去掉"大多数"和"最多一两个"的数据暗示，保留对比实质。

### 3. 破立句式新变体 - "同样是X，后果却不同"型
**chapters/03-lifecycle/chapter.md L7 严重度：中**

原文：
```
同样是"杀进程"，后果却天差地别。同样是 `kill -9`，对 Redis 通常只是丢掉最近一小段时间的数据...对 MySQL 可能要做崩溃恢复...对 Kafka 则可能触发分区的 Leader 重选...
```

病理分析：隐藏的破立句式变体。表面是"同样的操作，不同结果"的客观对比，但语义骨架是"你以为这个操作是简单的/统一的，但其实后果完全不同"——先否定隐含预期，再揭示更深真相。"天差地别"是情绪化修饰。

建议：直接用客观对比，去掉"天差地别"。

### 4. 连环设问堆叠 - 嵌入式
**chapters/03-lifecycle/chapter.md L9 严重度：低-中**

原文：
```
一个进程从可执行文件到"可对外服务"中间必须完成哪些事、为什么不能直接断电、"优雅（graceful）"二字到底优雅在哪里，下面逐一讲清楚。
```

病理分析：三个问题嵌在同一句话里（"哪些事", "为什么不能", "到底优雅在哪里"），用顿号连接。虽不占独立行，但语义上仍是三个结构相同的设问连发。

建议：压缩成两个问题或直接陈述。

### 5. 结构公式化 - 练习模板
**chapters/10-summary/chapter.md L120-128 严重度：中**

三个"练习"结构完全一致（场景 → 常见思路 → 框架推导 → 借鉴建议），句式、长度、结构如出一辙。这种"整齐到不自然"的并列结构，是典型的结构公式化现象。

建议：去掉序号合并叙述，或让每个练习的长度和结构略有差异。

### 6. 不可验证的绝对化断言
**chapters/04-memory-disk/chapter.md L13 严重度：低**

原文：
```
任何一个存储软件都活在同一组物理约束之下。
```

病理分析："任何一个...都"是全称量词。虽然从物理层面这句话是对的，但"任何"加上"存储软件"这个宽泛范围仍构成无法实际验证的全称断言。

建议：缩小范围或去掉全称量词。

### 7. 破立句式 - 作者个人成长型
**chapters/10-summary/chapter.md L7 严重度：低**

原文：
```
我回头看自己的职业生涯，最大的认知转折点不是学会了哪门新技术，而是有一天我突然不再问"这个系统快不快"，开始问"它在什么约束下做了哪些取舍，这些取舍今天还成不成立"。
```

病理分析：标准的"不是A而是B"破立句式。因是作者第一人称自我总结，严重度不高，但结构仍在。

建议：去掉"不是...而是..."，改用自然递进。

### 8. 连环设问 - 嵌套变体
**chapters/01-introduction/chapter.md L135-136 严重度：低**

"使用者关心的是...架构师追问的是..."的对比结构连续抛出四个问题，加上"使用者vs架构师"的破立句式，命中两种病理。

建议：将架构师的问题减少到2-3个，或改为叙述句式。
