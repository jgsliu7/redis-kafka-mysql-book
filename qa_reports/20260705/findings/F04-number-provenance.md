# F04 量化数字溯源审查报告

## 发现汇总

- 总裸数字数：约 80+
- 有出处的：约 15（含版本号、配置默认值、有明确来源的数字）
- 需补前提的：约 8
- 应改定性的：约 3
- WebSearch核实：对 8 组关键可疑数字进行交叉验证

## 逐条发现

### 01-introduction/chapter.md

**L41 `约 200 万次写入/秒（TPS）`**
- 上下文：Kafka 吞吐基准，"3 台廉价机曾测到约 200 万次写入/秒"
- 判定：**有出处**。标注了来源"2014 年 LinkedIn 工程博客，Kafka 0.8.1, 100 字节消息，作为历史参考"
- 评估：全书中少有的附带完整来源、硬件、消息大小场景的数字。合格。

### 02-data-structures-protocols/chapter.md

**L71 `zstd，压缩比取决于消息内容相似度与压缩级别；基准测试中常见 2-6 倍`**
- 判定：**有出处**（KIP-110 和社区基准），且附带了前提。合格。

### 03-lifecycle/chapter.md

**L33 `fork 拷贝页表约 10-20 ms/GB（4KB 页表，x86_64）`**
- 判定：**需补前提**。实际 fork 时间差异极大——物理机约 9-13ms/GB，KVM 虚拟机约 23ms/GB，EC2 旧 Xen 实例可达 239-424ms（来源：Redis latency 文档、Chen et al. VLDB 2023）。建议补来源或加"裸金属物理机"前提。

**L33 `数十毫秒` / `数百毫秒` / `数秒至数十秒`**
- 判定：**需补前提**。未说明实例规模阈值、内存配置、透明大页状态的对应关系。

### 04-memory-disk/chapter.md

**Table 4-1: 存储介质访问延迟**
- `L1 缓存 ~1ns` / `DRAM ~100ns` / `SSD 随机读 ~100μs` / `机械盘随机寻道 ~10ms`
- 判定：**常识量级**，表格标注"真实值会随 CPU、磁盘、文件系统、队列深度与负载模型变化"，前提充足。合格。

**Table 4-1: `SSD 顺序写 ~300-500MB/s (SATA) ~2-14GB/s (NVMe)`**
- 判定：**需补前提**。虽附带代际递增说明，但没有标注 NVMe 基准来源。建议补"基准来源"。

**L33 `Redis fork 开销：随实例内存增长`**
- 判定：同 Ch3 的 fork 数字，**需补前提**。

**L34 `生产典型 8-128GB`（InnoDB 缓冲池）**
- 判定：**需补前提**。云上 256GB-512GB 缓冲池也不罕见。建议改为"中大型生产使用 8-128GB，云上可达 512GB"。

**L76 ``maxmemory-samples` 控制，默认 5`**
- 判定：**有出处**（Redis 配置默认值）。合格。

**L140 `midpoint...新老分区占约 5/8、3/8`**
- 判定：**需补前提**。`innodb_old_blocks_pct` 默认 37%（约 3/8），说 3/8 可接受但差了一点（37% vs 37.5%）。建议补出处。

**L186-188 `vm.dirty_background_ratio=10%、vm.dirty_ratio=20%`**
- 判定：**有出处**（Linux 内核默认值）。建议加"Linux 内核默认值"的来源声明。

**L191 `SATA SSD 顺序写吞吐约 300 到 500MB/s`**
- 判定：同 Table 4-1 分析，**需补前提**。

### 06-security/chapter.md

**L26 `单核十万级 QPS（小包 GET/SET 场景）`**
- 判定：**需补前提**。"十万级"准确，但"小包"未量化（1 字节 vs 64 字节 vs 512 字节差距数倍）。建议补充"64 字节 payload、50 并发"等条件。

**L147 `开启 TLS，单线程吞吐相比无加密约降至六到八成`**
- 判定：**需补前提**。"六到八成"范围太宽。Redis TLS 的 QPS 下降约 36%（长连接），短连接可达 40-60%。建议区分长连接/短连接场景。

**L165 `RSA 证书方案吞吐降幅约两到三成`**
- 判定：**需补前提**。Kafka 的 TLS 开销随副本数量、消息大小变化，该数字过于笼统。建议补充来源或改为定性描述。

### 07-cluster/chapter.md

**L42 `Redis Cluster：16384 个槽，Gossip bitmap 仅 2KB`**
- 判定：**有出处**（Redis Cluster spec，16384/8 = 2048 字节 = 2KB）。合格。

**`实际节点数上限约 500-1,000`**
- 判定：**需补前提**。建议补 Redis Cluster spec 链接和社区实践出处。

**`跨 AZ 部署时每笔事务要多约 5-10ms`**
- 判定：**需补前提**。MGR 的 Paxos（XCom）需 1.5-2.5 RTT 完成一轮共识。建议补"基于 XCom 协议的 1.5-2.5 轮往返估算"。

**L44 `Kafka 单分区吞吐：约 10-50MB/s`**
- 判定：**需补前提**。范围极宽（1-50MB/s 差异达 50 倍）。建议补 Confluent 官方容量规划指南。

**`KRaft 模式下分区数可达百万级`**
- 判定：有前提但**需补出处**。建议补社区实践出处。

### 08-storage-format/chapter.md

**L46 `Kafka RecordBatch V2：每批元数据约 61 字节`**
- 判定：**有出处**（Kafka 协议规范 V2）。合格。

**L46-47 `V0 单条约 14 字节、V1 约 22 字节`**
- 判定：**有出处**（Kafka 消息格式规范）。合格。

**L47 `Batch 级压缩在同类消息上可达 2-6 倍（zstd）`**
- 判定：**需补前提**。前提好但缺具体基准测试或论文出处。建议加 KIP-110 或 Facebook zstd 基准链接。

**L48 `Redis RDB 压缩比：1GB 内存实例 ≈ 300-500MB 快照`**
- 判定：**需补前提**。不同类型压缩比差异大（整数集合可达 0.2，小字符串仅 0.8-1.0）。建议补社区来源。

**L188 `索引体积小到能常驻内存：一个 TB 的 Topic 索引可能只有几 GB`**
- 判定：可演算（1TB/4KB * 8B ≈ 2GB），属可接受估算范围。合格。

### 09-data-sync/chapter.md

**L37 `Redis PSYNC backlog：默认 1MB`**
- 判定：**有出处**（Redis 配置默认值）。合格。

**`1 万次写/秒、每条约 100 字节的负载下，约覆盖 1 秒的断线窗口`**
- 判定：**需补前提**。最好说明这仅为示例假设。

## WebSearch 核实结果

| 书中数字 | 核实结论 | 业界真实范围 | 来源 |
|---------|--------|-------------|------|
| fork 时间 10-20 ms/GB | 仅限裸金属物理机 | 物理 9-13ms/GB, KVM ~23ms/GB, 旧 Xen 239-424ms/GB | Redis latency docs, VLDB 2023 |
| Redis 单核 QPS "十万级" | 准确但需量化"小包" | GET 13-21万 QPS (64B, 50并发) | Alibaba Cloud white paper |
| Redis TLS "约六到八成" | 范围合理但缺场景区分 | 长连接保留 ~64%, 短连接保留 ~40-60% | Redis GitHub issue #7595 |
| Kafka V2 Batch 头 61 字节 | 准确 | 61 字节（含 CRC） | Kafka 协议文档 |
| Kafka zstd 压缩 2-6x | 合理范围 | zstd -1 约 2.887x, 文本数据可达 4-8x | Facebook zstd 基准, KIP-110 |
| Kafka ZK 分区上限 20万 | 公认实践值 | Cloudera 建议上限 200K | Cloudera Guide, Karafka docs |
| MGR 上限 9 节点 | 官方规定 | MySQL 官方 MGR 硬上限 9 节点 | MySQL Group Replication docs |
| MGR Paxos RTT 2-3 倍 | 约 1.5-2.5 RTT 合理 | XCom 需要 1.5-2.5 轮共识往返 | 阿里云 MGR 工程博客 |

## 需优先处理的裸数字

### 应补出处的
1. Ch3: `fork 拷贝页表约 10-20 ms/GB` — 补"物理机条件下的 Redis 官方延迟文档"并标注虚拟化差异
2. Ch4 Table 4-1: NVMe SSD 顺序写吞吐 — 补 Gen3/Gen4/Gen5 的 PCIe 标准文档来源
3. Ch7: `Kafka 单分区吞吐约 10-50MB/s` — 加 Confluent 官方容量规划链接
4. Ch7: `MGR 跨 AZ 延迟多约 5-10ms` — 补 MySQL 手册或测试报告出处

### 建议改为定性表述的
1. Ch6: `开启 TLS 单线程吞吐约降至六到八成` — 数字太宽，改"下降显著，实测约 30-60%"
2. Ch6: `RSA 证书降幅约两到三成` — 无可靠来源，建议改为"有明显降幅，以实测为准"

### 代际漂移风险
NVMe SSD 顺序写速度 Gen3 → Gen6 约每代翻倍。书中对每一代给出了范围，在 2026 年仍属准确，但 Gen6（PCIe 6.0, 2025-2026 起）的顺序写速度可达 ~28GB/s，书中没有包含。建议添加"Gen6 可达 ~28GB/s"或改为"各代 NVMe 协议速度上限"的表述。

## 结论

全书参考文献列表结构良好，但绝大多数具体量化数字在正文中没有指向这些参考文献。最大的系统性问题不是"没有参考文献"，而是"正文数字缺乏跳转到参考文献的交叉引用"。

三点改进建议：
1. 能引用官方文档的数字优先引用
2. 性能数字必须附前提 + 最低要求加引用
3. 经验估算性数字建议改为范围或量级表述
