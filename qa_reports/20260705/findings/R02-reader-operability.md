# R02 读者可操作性审查报告

## 总体评价

本书是一本架构分析书，不是操作教程或配置手册。前言已声明"不讲最新的调优技巧，不讲基础知识，也不逐行贴源码"，前置要求写明读者"至少实际用过三者之一，知道客户端怎么连、命令或 SQL 怎么写"。因此全书大部分命令/参数以**示例说明**而非**可执行指令**形式出现，读者不会被要求逐条敲命令验证。

但书中有若干处涉及可执行命令（`ACL SETUSER`、`redis-benchmark`）、协议级命令（`PSYNC ? -1`、`FLUSH PRIVILEGES`）、以及第10章主动建议读者"把旋钮往反方向拧一次"的动手练习。在这些点上，读者会遇到操作性问题。

## 发现汇总

- 执行上下文不清：4 处
- 前置条件缺失：3 处
- 参数无解释：2 处
- 流程不完整：2 处
- 缺少错误处理提示：1 处
- 实验可复现性不足：整体

## 逐条发现

---

### 发现 1 | 章3 L43 | 执行上下文不清: 命令 `FLUSH PRIVILEGES`

**原文**: "改完权限之后要执行 `FLUSH PRIVILEGES`。内存里的权限快照和磁盘表对不上，需要重新加载。"

**问题**: 没有说在哪个环境中执行这个命令。读者如果是 MySQL 新手，可能不知道这是在 mysql 命令行客户端执行，还是在某个管理工具中执行。没有给出连接 MySQL 的先决步骤（`mysql -u root -p`）。

**严重程度**: 低。MySQL 常规操作，前置要求已假设读者会用。

---

### 发现 2 | 章6 L42-L44 | 执行上下文不清 + 前置条件缺失: 命令 `ACL SETUSER alice on >pwd ~keys:* +get +set`

**问题**:
1. 没有说这条命令在哪里执行。读者需要推导出这是在 `redis-cli` 里敲的。
2. 没有前置条件：要执行 `ACL SETUSER`，Redis 版本必须 >= 6.0（ACL 是 6.0 引入的），且当前用户必须有 `@admin` 权限（通常是 default 用户）。
3. 命令语法中的 `>pwd` 容易被误解——`>` 在 shell 中是重定向符，在 redis-cli 中却是密码标记。如果读者错误地在 bash 中执行此命令，会产生意外行为。
4. 没有提示：如果 Redis 版本低于 6.0，或者当前用户没有 ACL 权限，会报什么错。

**严重程度**: 中。这是全书最"像指令"的代码块，但缺少上下文。读者如果照着敲，有可能踩坑。

**建议**: 加一句"在 redis-cli 中执行以下命令，假设当前为 default 用户且有 admin 权限"。提醒密码标记 `>` 是 ACL 语法的一部分，不是 shell 重定向。

---

### 发现 3 | 章9 L67 | 执行上下文不清 + 前置条件缺失: 命令 `PSYNC ? -1`

**原文**: "主节点收到副本的 `PSYNC ? -1`（表示'我什么都没有'），触发 `BGSAVE`"

**问题**: `PSYNC ? -1` 是 Redis 副本（replica）在建立初始连接时自动发送的**协议内部命令**，不是用户在 redis-cli 里敲的命令。如果读者试图在 redis-cli 里输入 `PSYNC ? -1`，会得到错误 `-ERR unknown command 'PSYNC'`（PSYNC 不是 redis-cli 暴露给用户的命令，它是复制协议内部的）。读者读到这一段可能会尝试敲这个命令，然后遇到报错。

**严重程度**: 中。协议内部命令出现在正文上下文中易被误解为可执行命令。

**建议**: 明确标注这是"由 Redis 副本进程自动发送的协议内部命令，不能在 redis-cli 中手动执行"。

---

### 发现 4 | 章2 L75 | 前置条件缺失: 操作 `nc` 连接 Redis

**原文**: "RESP 是纯文本协议，用 `nc` 连上 6379 端口直接打字就能交互。"

**问题**: 假设读者本地已有 `nc` 工具和运行中的 Redis 实例（端口 6379）。没有提示如果读者机器上没有 `nc`（某些 Linux 发行版默认没有安装），或者 Redis 设了 `requirepass`/`protected-mode`，`nc` 连接上去后打命令会返回 `-NOAUTH` 或 `-DENIED` 错误。没有给出最小可验证环境（如 `docker run redis`）。

**严重程度**: 低。前置要求已假设读者会用 Redis，这些是基础。

---

### 发现 5 | 章4 L76 | 参数值解释不充分: `maxmemory-samples` 默认值 5

**原文**: "淘汰时从全部键里随机采样 N 个（由 `maxmemory-samples` 控制，默认 5），从中淘汰最久未用的那一个。采样数越大越接近精确 LRU，CPU 开销也越高。"

**问题**: 说了"采样数越大越接近精确 LRU，CPU 开销也越高"，但没有解释**为什么默认值是 5**。5 这个数字是否来自某种理论分析（比如 5 个样本就能以>95%概率接近真实 LRU 的效果）还是来自 antirez 的经验值？读者会记住"默认值是 5"但不知道为什么是 5。

**严重程度**: 低。架构书不需要深究每个默认值的来源，但作为读者可操作性审查，参数值如果有解释依据会更完整。

---

### 发现 6 | 章4 L139 | 参数值解释不充分: `innodb_old_blocks_time` 默认值 1000

**原文**: "...冷区停留超过 `innodb_old_blocks_time`（默认 1000 毫秒）之后再次被访问"

**问题**: 没有解释为什么默认值是 1000 毫秒（1 秒）。为什么不是 500ms 或 2000ms？这个值是通过什么基准测试确定的？读者会记住"默认 1000ms"但不知道这个时间窗口的含义依据。

**严重程度**: 低。

---

### 发现 7 | 章10 L176-L179 | 流程不完整 + 实验可复现性不足: 动手实验建议

**原文**: "把 Redis 的 `appendfsync` 从 `everysec` 改成 `always`，用 `redis-benchmark` 看吞吐怎么跌。把 MySQL 的 `innodb_flush_log_at_trx_commit` 从 1 改成 0，强杀进程看丢多少。把 Kafka 的 `acks` 从 all 改成 1，断一个 Broker 看消息丢不丢。"

**问题**: 这是全书最强烈的动手建议，但：
1. 没有给出任何一步操作的具体步骤。读者需要自己知道如何修改 Redis 配置（用 `CONFIG SET` 还是改 `redis.conf` 并重启？）、如何运行 `redis-benchmark`（默认参数够不够？需要哪些 flags？）。
2. "强杀进程看丢多少"——如何精确测量丢失了多少数据？没有给出测试方案或数据验证方法。
3. "断一个 Broker 看消息丢不丢"——没有说明如何控制变量（需要几个 Broker？topic 副本数？产生多少消息？如何知道丢了哪些？）。
4. 这些实验需要在专用测试环境执行，但书中没有提供任何 docker-compose.yml、配置文件片段或脚本模板。
5. 没有给出预期结果的范围："吞吐大约从多少降到多少"、"大约丢了多少条/字节"——读者做完了无法自我验证是否正确。

**严重程度**: 高。这是全书最明确的"读者应该动手"的部分，但每一步都不完整。如果读者真的去测试，很可能因为方法不对得出错误的结论。

**建议**: 为每个实验提供最小可复现步骤，或者至少给出一个可参考的已知基线值让读者对照。

---

### 发现 8 | 章8 L82 | 执行上下文不清: 命令 `redis-check-aof --fix`

**原文**: "AOF 损坏可以人工编辑修复（纯 AOF 模式下编辑器打开删掉坏掉的那条即可；混合持久化模式下 AOF 文件前段为二进制 RDB 数据，需先用 `redis-check-aof --fix` 工具修复）"

**问题**: 只提到工具名，没有说明该工具从哪里获得（Redis 发行版自带，在 `src/` 目录或 PATH 中）、需要什么参数、以及在什么状态下可以安全执行（需要用 `redis-cli SHUTDOWN` 先停掉 Redis 实例吗？）。混合持久化模式下原文说"先用 `redis-check-aof --fix` 工具修复"，但没说修复之后还需要做什么。

**严重程度**: 低。这属于运维工具箱的内含知识，前置要求已覆盖。

---

### 发现 9 | 章3 L191 | 流程不完整: K8s 实践清单

**原文**: "preStop hook 主动检查条件而非 sleep；grace period = 刷盘/迁移耗时 + 30 秒余量；关闭期间 readiness probe 返回 false，避免 K8s 反复重启。"

**问题**: 给出的是三条原则性建议，但没有给出一份可复制的 YAML 配置片段或脚本模板。读者需要自己将这些建议转化为 Kubernetes Deployment 的 `lifecycle.preStop` 配置、`terminationGracePeriodSeconds` 等实际配置。虽然这符合全书"讲原理不讲操作"的定位，但对于想验证的读者，门槛较高。

**严重程度**: 低。

---

### 各章隐含的版本假设

全书在第 1 章 1.4.3 节已显式标注版本基线为 **Redis 7.x、MySQL 8.0.x、Kafka 3.x**，并给出了版本差异的说明。这一点做得非常清楚。后续各章在涉及版本差异的地方也做了显式说明（如 "7.0 的多部 AOF 改造"、"8.0.20 起双写缓冲独立"、"3.3 起 KRaft 生产可用"）。**版本假设方面没有发现问题**。

但有一个细节：章6 中提到 `ACL SETUSER` 命令，标注了"6.0 引入 ACL"，但命令所在段落没有再次标注版本号——读者如果从 ch06 直接跳到 ACL 部分，没有读之前的版本说明，可能不知道他们的 Redis 版本是否支持。

---

### 实验可复现性：整体评级

**评级: C**

全书没有任何一处为读者提供可直接复现的最小实验环境。具体缺什么：

1. 没有提供任何 `docker-compose.yml` 或 Docker 运行命令模板。读者要自己知道 `docker run redis:7`、`docker run mysql:8.0`、`docker run apache/kafka:3.x` 这样的基础命令。
2. 没有提供任何配置文件模板（`redis.conf` 片段、`my.cnf` 片段、`server.properties` 片段）。
3. 没有提供任何测试数据集或压测脚本。
4. 第 10 章的动手实验建议缺乏可操作步骤，读者靠自己摸索很难得到有意义的结论。

这是本书定位决定的——它是架构书，不是实验手册。但后记中明确建议读者动手做实验，如果"动手"这条建议是认真的，则这些缺失是实质性的。

## 总结

因其定位（架构分析书）不要求读者跟着敲命令，因此大部分章节不存在"跑不通"的问题。所有命令都以**示例/说明**形式出现，而不是操作指令。主要风险集中在：
1. **章6 `ACL SETUSER`**：全文最像"操作指令"的代码块，但没有执行环境提示。
2. **章9 `PSYNC ? -1`**：协议内部命令容易被读者误操作或产生困惑。
3. **章10 + 后记动手实验建议**：全书最鼓励读者动手的部分，但没有任何可操作步骤，读者无法自行验证。
