### 教学损伤评估

**严重度：高。** 损伤了全书的"教学契约"。

具体损伤机制：

1. **框架冲突**。序言用"三层意思"作为全书纲领，但 ch10 用完全不同的两套框架（五条规律 + 五个权衡维度 + 四层认知台阶）。读者大脑里挂着三套框架（三层意思 / 五条规律 / 四层台阶），彼此没有任何显式映射。认知负荷增加。

## 🔴 问题二：ch02 偏离全书模板

### 位置
`chapters/02-data-structures-protocols/chapter.md`

### 教学损伤评估

**严重度：中高。**

损伤机制：

1. **读者节奏中断**。读者从 ch01（特例模板）进来，看到 ch02 又是一种新模板（按"数据结构/协议"维度而非按软件），到 ch03 才回到标准模板（N.2/N.3/N.4）。前三章三套模板，读者需要三次调整阅读预期。教学法上：模板一致性降低认知开销。不一致性读起来"累"（认知解译成本）。

2. **但 ch02 的主题式编排并非错误选择**——数据结构天然适合横向对比。三家放在同一个维度下讲，比"先讲完 Redis 所有数据结构再讲 MySQL 所有数据结构"更加高效。问题在于读者缺乏对"为什么这章不一样"的预期管理。

## 🔴 问题三：后记库存故事重复

### 位置
`chapters/10-epilogue.md:15-22`

### 教学损伤评估

**严重度：高。** 损伤了"后记作为全书升华"的教学功能。

具体损伤：

1. **"后记"的格式塔期待**：读者到后记时已经读完九章，期待的是"新高度"或"全书收束"，而不是"又读了一遍 ch07/ch10 已经见过的案例"。重复阅读同一案例，读者会产生"作者是不是没什么story可讲"的隐式判断。

2. **库存事故故事在全书出现三次**：
   - ch07 导读（"用户下了单，仓库说没货"）
   - ch10 练习一（秒杀库存系统选型）
   - 后记（"一个真实的故事"）
   
   三次都聚焦"网络分区导致库存多卖"这个单一细节。读者读到第三次时，感官已经钝化。

3. **理想中的后记故事应该承载的功能**：① 展示"学完思想后如何落地"；② 是全书的剖面——一句"这就是全书的缩影"；③ 让读者带着行动感离开。库存故事做到了 ①，但重复使用减损了 ② 的新鲜度。

### 修复分类：DEFER（需要作者级判断）

**原因**：这个库存故事质量很高——具体、真实、有戏剧张力（"Redis 没 bug 但出事了"）。问题是放错了位置/用了太多次。需要作者决定：

**建议方向 A（推荐）**：保留后记的故事，删除 ch07 导读和 ch10 练习一中的版本。后记是全书升华的最终位置，故事放这里冲击力最大。理由：ch07 导读有替代故事（Redis Cluster 扩容常见事故），ch10 练习可有替代案例。

**建议方向 B（最小改）**：保留三处，但让后记版本明确标注"这个案例之前在练习中讲过，今天我们换一个角度"——如：库存问题之前用做选型练习，今天我们从'数据流回放'的角度再看一遍，注意它体现了本章的哪几条规律。"这样做了区分，读者不会觉得重复。

### 复杂度
涉 3 个文件（ch07/ch10/epilogue），跨章决策，作者必须参与。

**可选：不修复**。ch02 的"教科书式"风格对"数据结构"这个相对理论化的主题可能反而是优势。保留 ch02 的中性声音，只要其他章节有足够的个人声音平衡即可。

### 5c："架构启示"模板化

**位置**：ch02 2.5 / ch09 9.6 / ch05 5.6 等

**教学损伤评估**

**严重度：低中。** 模板化不致命，但降低"金句感"。

具体分析：

每章的"架构启示"都遵循同一格式：`**启示X：[陈述句]**。`后接一段解释+例子。这种模式在全书中重复 7~9 章（第 2、5、6、7、8、9 章都有），到第 6 章以后读者已经能预判前 5 个字"启示一："的内容。

教学损伤：**预告削弱**。读者看到"启示一"三个字就知道"又要来一条规律了"，失去意外感和惊喜感。而"架构启示"是本书的核心资产（"读完后能带走什么"），应该每次都让读者觉得"这条真值钱，笔记要做"，而不是"哦，又一章启示来了"。

差异化开启句式可以有效打破模板感，而不改变内容。

**修复分类：Quick Fix（改句式前缀）**

建议将 ch02、ch05、ch09 的启示开头"**启示X：**"替换为多样化句式。示例：

**ch02（当前）** → 改为：
```
**启示一：数据结构服务于核心目标。**
改为 → **能让你走多远的路，看你选什么数据结构。** Redis 要快，所以...（后续内容不变）

**启示二：协议是数据结构的外部映射。**
改为 → **怎么对话，取决于数据结构的长什么样。** Redis 操作简单...
```

**ch09（当前）** → 改为：
```
**启示一：数据同步的本质是状态机复制。**
改为 → **同步不是同步状态，是同步导致状态的操作。** 它们的同步都在做同一件事...

**启示二：进度坐标是同步的通用货币。**
改为 → **多份副本要对齐，需要一把共同的尺子。** 这三款软件用了形态各异的进度坐标...
```

注意：以上只是"句首"微调，不改变启示正文。关键原则是：**每条启示的"句首样式"在全书中不重复超过 2 次**。可以预先分配几种变体：
- "能让你..."式（能力导向）
- "不是...而是..."式（纠偏导向）
- "xx，就需要yy"式（条件导向）
- 反问句开式
- 数据/对比开式

### 复杂度
2-3 章，每章改 4-6 个句首前缀。低风险（不改内容只改表述）。
