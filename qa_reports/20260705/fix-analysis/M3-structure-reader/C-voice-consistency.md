## 二、A01 报告的 5 个 AI 味问题：判断 + 处理建议

### 1. ch05: "一句话"+加粗金句同章复发 4 次
**文件**: chapters/05-layered-architecture/chapter.md
**位置**: L72, L88, L142, L248
**判断**: 🟡 **真 AI 味，需修**

4 次"一句话"引出的加粗金句在同一章内复发，即使金句本身有内容价值，表达方式的高度同质化就是 AI 味。读者会感觉到"这段是作者说的，这段是模板生成的"。

**建议保留 1-2 处，其余改为自然收束**。

应保留：
- L88 `一句话：**性能敏感处，分层要薄，跨层要直接**` — 这是 Redis 节的总收束，结构合理。

应修改：

**a. L72 原文**:
```
这张图的解读只有一句话：**逻辑层只看 type，存储层只看 encoding**。
```
**修改为** (去掉 "一句话" 引子，直接陈述):
```
这张图的解读很直接：**逻辑层只看 type，存储层只看 encoding**。
```

**b. L142 原文**:
```
这套结构的取舍一句话：**标准化接口让生态能长出来，代价是每次行操作都过一次虚函数**。
```
**修改为** (去掉 "一句话"，改成判断句式):
```
这套结构的取舍很清晰：**标准化接口让生态能长出来，代价是每次行操作都过一次虚函数**。
```

**c. L248 原文**:
```
说到底就是一句话：存储介质的多样性，决定了引擎抽象该不该做。
```
**修改为** (去掉 "说到底就是一句话"):
```
存储介质的多样性，决定了引擎抽象该不该做。
```

### 2. ch09 L7: "这就是"+"本质上就是"+"就是一件事"三重叠放
**文件**: chapters/09-data-sync/chapter.md
**位置**: L7
**判断**: 🔴 **真 AI 味，需修**

三重"就是"在同一句内叠放，语义冗余。读者读到这里会感受到明显的"在凑字数"。

**原文**:
```
这就是"数据同步"。本质上就是一件事：把一处状态变化可靠地传给其他副本，而且延迟有上限。
```

**修改为**:
```
数据同步要解决一个核心问题：把一处状态变化可靠地传给其他副本，而且延迟有上限。
```

### 3. ch04 L6-L8: 三句镜像排比 + "说到底"收束
**文件**: chapters/04-memory-disk/chapter.md
**位置**: L6-9
**判断**: 🟢 **作者风格，不修**

三句镜像排比（"Redis 把数据直接放内存...MySQL 的内存只是磁盘页的副本...Kafka 把加速交给操作系统..."）是本书的核心比较方法——每一章都在做三家对比。这不叫"AI 味排比"，这叫人设成立后的结构性修辞。

但 L9 的 **"说到底"** 可以视为轻微 AI 味残留。要不要改取决于严格程度。

**可选修改 (轻微)**:
```
说到底就是一个选择：数据放在哪里。
→ 归根结底是一个选择：数据放在哪里。
```
或
```
→ 这就是那个核心选择：数据放在哪里。
```
**建议：可改可不改，非必要。**

### 4. ch04 L55-59: 三段范式同构命名
**文件**: chapters/04-memory-disk/chapter.md
**位置**: L55-59
**判断**: 🟢 **作者风格，不修**

这是作者自己建立的三种范式分类法，属于本书的原创框架。范式命名当然要同构，否则框架不成立。AI 不会发明这种分析框架。

### 5. ch05 L86-L88: "正是"+"一句话"连打收束
**文件**: chapters/05-layered-architecture/chapter.md
**位置**: L86-88
**判断**: 🟡 **轻微 AI 味，可修可不修**

"正是"作为强调词单独出现不算问题，但配合 L88 的"一句话"金句形成了连打。这里的问题是"一句话"复发（见 A01-1 的分析），"正是"本身不那么严重。

**原文**:
```
这种"零抽象"的层间通信，正是 Redis 能把延迟压到亚毫秒的原因。
```

**修改为** (去掉"正是")：
```
这种"零抽象"的层间通信，是 Redis 能把延迟压到亚毫秒的原因。
```

## 三、A02 病理兜底发现的 6 个新问题：判断

### 1. ch01 L171: 连环设问堆叠（"第一/第二/第三"三分法）
**文件**: chapters/01-introduction/chapter.md
**位置**: L170-173
**判断**: 🟢 **作者风格，不修**

"第一，先写日志还是先改数据？...第二，日志落盘的粒度多细？...第三，日志记的到底是什么？" 这是作者在展示架构视角的三层追问方法——这是本书的核心方法论（架构视角的三把钥匙：约束/演化/抽象），不是 AI 模板。三个问题是递进且独立的，每个都有实质内容。

### 2. ch01 L7: 破立句式新变体
**文件**: chapters/01-introduction/chapter.md
**位置**: L7
**判断**: 🟢 **作者风格，不修**

"大多数...但最多一两个"是来自作者真实面试经验的观察，有具体数字支撑（"面试过数百个""每个面试季最多一两个"），和作者的"我"绑定。这不是空洞的"大多数人认为...但其实"模板，而是亲身经历的判断。

### 3. ch03 L7: 破立句式新变体
**文件**: chapters/03-lifecycle/chapter.md
**位置**: L7
**判断**: 🟢 **作者风格，不修**

"同样是'杀进程'，后果却天差地别。同样是 kill -9，对 Redis...对 MySQL...对 Kafka..." 这是本书的标准比较句式——三款软件面对同一场景的不同表现。这种"同样是X，后果却不同"是本书的写作方法论，不是 AI 套话。且其后跟了具体的机制解释，不是空洞断言。

### 4. ch03 L9: 连环设问嵌入式
**文件**: chapters/03-lifecycle/chapter.md
**位置**: L9
**判断**: 🟢 **作者风格，不修**

"一个进程从可执行文件到'可对外服务'中间必须完成哪些事、为什么不能直接断电、'优雅'二字到底优雅在哪里"——这是章节导读的承诺性问句，属于传统的"本章将回答哪些问题"修辞。每个问句有独立内容，不堆砌。

### 5. ch10 L120-128: 结构公式化练习模板
**文件**: chapters/10-summary/chapter.md
**位置**: L120-128
**判断**: 🟡 **有模板感，可修可不修**

三道练习题结构完全相同："**练习N：...？** 常见的思路是...但...用规律N的判据：...所以...借鉴的是..."。作为教材的练习题，同构模板可以理解（便于读者快速掌握框架用法）。但三道连在一起读确实有"填空模板"的感觉。

**建议方案**：保留练习一和练习二的模板结构（让读者建立习惯），练习三改为不同的表述方式，例如去掉"思路—判据—借鉴"三段式，改成"这个问题看似A实则B，原因正是规律N揭示的..."。

### 6. ch04 L13: "任何一个存储软件都活在同一组物理约束之下"
**文件**: chapters/04-memory-disk/chapter.md
**位置**: L13
**判断**: 🟡 **轻微 AI 味，建议修**

"任何一个X都Y"是典型的绝对化断言句式。虽然内容本身正确（存储软件确实面对同样的物理约束），但这句话没有与个人经验绑定，读起来像教科书开头。

**原文**:
```
任何存储软件都活在同一组物理约束之下。
```

**修改为** (绑定个人观察):
```
做存储软件，说到底都逃不开同一组物理约束。
```
或（更激进，但增加声音）:
```
不管你写的是数据库、缓存还是消息队列，都必须回答同一组物理问题。
```

**是否强制修**：不强制。ch02 的"声音弱"符合内容特征——它是一本工具书里的"零件手册"。

## 八、整体判断

### 这本书读起来像一个人写的吗？
**像**。全书的声音分布虽然不均衡（ch02/ch04 偏弱、ch01/ch10/后记偏强），但这是合理的——技术说明章节天然比立意章节更需要客观陈述。序言→引言→技术章节→总结→后记有一条完整的声音弧线：作者在开头和结尾"露面讲自己的故事"，中间章节"退后讲技术"。

唯一可能让读者感到"声音断层"的地方是 ch02 和 ch04——这两章几乎没有作者在场感。但这不致命，因为 ch02 是本"参考手册"，ch04 结尾的选型争论故事（"我参与过一次选型争论"）已经给了一个有力的作者时刻。
