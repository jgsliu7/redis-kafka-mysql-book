```

Kafka 从一开始就得面对**天生分布式**。它的层与层之间本来就隔着网络。Broker 与 Broker 之间、Producer 与 Broker 之间，全是 TCP。这意味着 Kafka 的分层是为了水平扩展：每一层都要能在多台机器上并行跑起来。
```

## 全局诊断

这 11 个问题可以归为两个族群：

### 族群 A：三件套平行比较的同构病（问题 1-6）

这是本书最核心的写作方法——六个问题都发生在对三款软件做横向对比时。作者习惯用"Redis...MySQL...Kafka..."的平行句来展示对比。一次两次是有效的排比修辞；全书十几次出现时，读者建立起"我看到平行句就知道下一句说什么"的预测，注意力开始逃逸。

**系统层面建议**: 全书的"三件套平行句"应该有变化策略——有些地方用平行结构，有些地方用总分结构，有些地方用递进结构。同一个章节内，三次以上的"软件1→软件2→软件3"句式结构不得超过两种。

### 族群 B：说教腔（问题 8-11）

四个问题分别出现在序言、引言、正文、后记——几乎贯穿全书。说教腔的本质是作者替读者定位角色：作者是引路人/老师，读者是学生。破局方法不是把措辞变得更委婉，而是把角色从"教"转为"分享"：所有关于读者应该做什么、应该注意什么的表述，都转为"实践中常见的情况"或"值得讨论的问题"。
