## 上下文分析

完整段落（L83-L89）：

```
支撑 MySQL 设计哲学的，是一整套为正确性服务的机制。InnoDB 提供完整的 ACID……
这些机制不是图快，它们的目的只有一个：让数据在并发和故障面前都对得上。

代价当然存在。为保证持久性，事务提交要做日志落盘（fsync）；为保证一致性，写操作伴随 redo 与 binlog 的两阶段提交开销；为支持 MVCC，旧版本数据要靠 undo 链维护，长事务会拖累性能；调优门槛高，缓冲池、刷新策略、隔离级别都需要根据负载仔细配置。此外，单机扩展有天花板，纵向加内存加 CPU 容易，横向分库分表则代价不菲。MySQL 由此付出的代价是。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。

典型场景因此集中在核心业务：交易系统、订单与账户……
一句话定位：**MySQL 是事务关系型数据库领域应用最广泛的存储方案。**
```

## 问题根因诊断

`MySQL 由此付出的代价是。` 在句中充当"总括句"：前面已列出 MySQL 在持久性（fsync）、一致性（两阶段提交）、MVCC（undo 链）、调优、扩展性各方面的具体代价，此句本应点明"MySQL 的整体代价是什么"，但写到"是"后就断了。

## 上下文语义推断

从下文"它的优先级一直是'对'排在'快'前面：可以慢，但不能错"可以明确判定：MySQL 的代价就是**用性能/速度换正确性**。这一判断与全书语境完全一致：

| 位置 | 原文 | 与本文一致 |
|------|------|-----------|
| ch01 L43 | "MySQL 一直把'对'排在'快'前面，宁可慢也不丢不错" | 一致 |

| ch09 L188 | "参数选择就是延迟和安全之间的取舍" | 一致 |
| ch08 L252 | "代价是写放大" | 具体代价，非矛盾 |

**无跨章矛盾。** 各章对 MySQL"正确性优先于性能"这一核心取舍的描述完全一致。

## 修复方案

### 方案A（推荐）：直接删除不完整句

**操作**: 删除 `MySQL 由此付出的代价是。`（含紧邻的句号）。
**结果**:

```
……横向分库分表则代价不菲。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。
```

**理由**:
1. 前面已用 5 个分句详尽列举了各项具体代价，不需要再逐字重复"代价是……"。
2. 下一句"它的优先级一直是……"本身就是总括性解释——它说明了"代价"的根本原因，与 Redis 段（"这些短板的成因是明确的"）和 Kafka 段（"这些约束在 Kafka 中被视为合理代价"）的段末总括句职能对等。
3. 三段（Redis L71、MySQL L85、Kafka L99）的结构对比：

| 软件 | 代价列举 | 段末总括（sum up） |
|------|---------|-------------------|
| Redis | "这种选择的代价：数据量受物理内存约束……" | "这些短板的成因是明确的：这些场景该交给别的系统" |
| MySQL | "代价当然存在。fsync……两阶段提交……undo 链……调优……扩展……" | **【删除后】** "它的优先级一直是'对'排在'快'前面" |
| Kafka | "代价同样明确。只追加不修改……批量与顺序优先……" | "这些约束在 Kafka 中被视为合理代价，它优先保证吞吐与可扩展" |

删除后 MySQL 段落结构与 Redis/Kafka 完全一致。
4. **不引入新词，零跨章矛盾风险。** 删除操作只移除语法碎片，不改变语义。

### 方案B（备选）：补全并合并

**操作**: 将 `MySQL 由此付出的代价是。` 改为 `MySQL 由此付出的代价是速度上的让步——`（破折号连接下句）。
**结果**:

```
……横向分库分表则代价不菲。MySQL 由此付出的代价是速度上的让步——它的优先级一直是"对"排在"快"前面：可以慢，但不能错。
```

**理由**:
- 保留了"由此付出的代价是"这一句式，与 ch09 L61（"Redis 由此付出的代价是，用'丢失少量写'换来……"）句式一致。
- 破折号连接了"代价"与"为什么是这个代价"，逻辑更紧密。

**缺点**: "速度上的让步"是新加的词，虽然语义正确且与全书一致，但修改幅度略大于方案A。

### 方案C（备选，不推荐）：单纯补宾语后接句号

**不推荐理由**: 若写成 `MySQL 由此付出的代价是速度。`，其后另起一句"它的优先级一直是……"，两者变成松散并列，不如合并流畅。
