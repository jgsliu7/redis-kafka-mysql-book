# G5-ch01-sentence: 句子成分残缺 "代价是。"

## 定位

- **文件**: `chapters/01-introduction/chapter.md`
- **行号**: 85
- **原文**: `MySQL 由此付出的代价是。`

## 上下文（原文第 83-87 行）

```
83  支撑 MySQL 设计哲学的，是一整套为正确性服务的机制。……
84
85  代价当然存在。为保证持久性，事务提交要做日志落盘（fsync）；为保证一致性，写操作伴随
    redo 与 binlog 的两阶段提交开销；为支持 MVCC，旧版本数据要靠 undo 链维护，长事务会拖
```

## 文法分析

"MySQL 由此付出的代价是。" — 这是一个完整的句子，以句号结尾。

主语：**MySQL 由此付出的代价**（偏正短语，中心词"代价"）
系词/谓语：**是**

问题：系词"是"在古代/书面语中偶尔可作句末判断词使用（如"其为人也是"），但在现代汉语中，"是"作为判断动词必须带表语（宾语）才能构成完整谓语句。句末只有一个孤立的"是"，缺少必要的表语成分。读者读到"代价是"之后期待一个具体说明（"代价是 XXX"），却遇到句号终结，语义链条断裂。

## 作者本意推断

前文列举了 MySQL 的一系列具体代价（fsync 落盘、两阶段提交、undo 链、调优门槛、扩展天花板等），这些具体代价最终指向一个**总纲式的概括**：MySQL 为了"正确性"牺牲了"速度"。紧接着的下一句"它的优先级一直是'对'排在'快'前面"恰恰就是这个总纲。

因此，"MySQL 由此付出的代价是"后面应当衔接的内容就是"它的优先级一直是'对'排在'快'前面"——即"性能上的让步"。

## 修复方案

### 方案 A（推荐，最小改动）

把句号改为冒号，将两句合并为判断句：

> MySQL 由此付出的代价是：它的优先级一直是"对"排在"快"前面：可以慢，但不能错。

**原理**：冒号引出表语内容，文法完整。两处冒号分别对应两个层级——第一个冒号解释"代价是 X"，第二个冒号展开"X 的具体表现为……"。这种嵌套冒号在现代汉语说明性文体中常见且合法。

**改动范围**：只改一个字符（句号 → 冒号），不增不减不重写。

**old_string**: `MySQL 由此付出的代价是。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。`

**new_string**: `MySQL 由此付出的代价是：它的优先级一直是"对"排在"快"前面：可以慢，但不能错。`

---

### 方案 B（消解嵌套冒号，用逗号）

> MySQL 由此付出的代价是，它的优先级一直是"对"排在"快"前面：可以慢，但不能错。

**原理**：用逗号替代第一层冒号，降低嵌套感。逗号在此处承担"提示下文"的功能，虽然力度弱于冒号，但文法正确。

**old_string**: `MySQL 由此付出的代价是。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。`

**new_string**: `MySQL 由此付出的代价是，它的优先级一直是"对"排在"快"前面：可以慢，但不能错。`

---

### 方案 C（补全表语，显式写出"代价是什么"，再衔接原后续句）

> MySQL 由此付出的代价，是在性能上的让步。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。

**原理**：补出"在性能上的让步"作为表语，让"代价是"完整收束，然后另起一句。优点是最明确无歧义，缺点是改动略大（改动了两个句子）。

**old_string**: `MySQL 由此付出的代价是。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。`

**new_string**: `MySQL 由此付出的代价，是在性能上的让步。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。`

---

## 推荐结论

**方案 A（把句号改冒号）** 是改动最小、语义最准确的修复。保留原文措辞，只更正语法结构，且嵌套冒号在这个上下文中的表述层级自然合理。
