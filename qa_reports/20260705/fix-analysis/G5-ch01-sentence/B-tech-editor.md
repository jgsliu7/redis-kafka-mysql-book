# G5-ch01-sentence 技术编辑审读意见

## 问题说明

章节文件：`/Users/liu/dev/demos/redis-kafka-books/chapters/01-introduction/chapter.md` 第 85 行

原文片段：

> 此外，单机扩展有天花板，纵向加内存加 CPU 容易，横向分库分表则代价不菲。MySQL 由此付出的代价是。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。

句子"MySQL 由此付出的代价是"以系动词"是"结尾，缺少表语，是一个语法上未完成的句子。句号之后读者会停顿、回溯，猜测这句话到底想说什么，然后发现下文句首用了"它"，语义上与前句紧密绑定。这不是一个"作者想说的意思不对"的问题，而是**成句层面出现了编辑遗漏**。

## 上下文分析（完整段落）

段落的逻辑链条：

1. 引子："代价当然存在。"（一句独立的短句，宣告话题）
2. 列举四个具体代价（分号串接）：日志落盘、两阶段提交、undo 链拖累性能、调优门槛高
3. 第五个代价（句号结尾）：横向扩展困难
4. **[断句]** "MySQL 由此付出的代价是。" —— 到这里，读者以为要总结了，结果句子断了
5. 实际总结：MySQL 的优先级是"对"排在"快"前面

可见第 4 句是第 5 句的**冗余引导**——作者想写一个类似"MySQL 由此付出的代价是：快不起来"的总括句，但写了一半改变了写法，直接跳到第 5 句的核心表述，而断句残留没有清除干净。

## 修复方向判断

有两条修复路径：

### 路径 A：删除断句（推荐）

删除"MySQL 由此付出的代价是。"，让第 3 句（横向扩展代价）直接连到第 5 句（优先级总结）。

改写后：

> ...此外，单机扩展有天花板，纵向加内存加 CPU 容易，横向分库分表则代价不菲。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。

**理由**：
- 段落第 1 句"代价当然存在"已经完成了"宣布代价"的功能。后面不需要再重复"付出的代价是"。
- 删除后两句直接相邻："代价不菲"的句号读者还在消化横向扩展这个点，紧接着读"它的优先级一直是……"，节奏上形成"列举具体代价 → 揭示底层逻辑"的递进，有力度。
- 无信息损失。

### 路径 B：补全断句

补上表语，把断句与下一句合并：

> ...横向分库分表则代价不菲。MySQL 由此付出的代价是慢：它的优先级一直是"对"排在"快"前面——可以慢，但不能错。

**问题**：这句补完后的信息冗余。"由此付出的代价是慢"其实已经被第 5 句的"可以慢，但不能错"覆盖了。读者会感觉到重复。

### 我的判断

路径 A 更好。这个断句不是"少了一部分"的残缺，而是**写作过程中旁逸斜出的冗余引导**。作者的真实意图已经充分表达在后续句子里了。删除是去赘，不是减内容。

## 节奏影响

当前段落的行文节奏：

| 部分 | 长度 | 节奏感受 |
|------|-----|---------|
| "代价当然存在。" | 4 字短句 | 宣告，顿挫有力 |
| 四个代价（分号串） | 约 70 字长句 | 密集列举，没有喘息 |
| 横向扩展代价 | 约 25 字 | 略有松弛 |
| **[断句] "MySQL 由此付出的代价是。"** | 9 字 | **卡顿——读者停在这里等解释** |
| 优先级总结 | 约 20 字 | 揭示本质 |

删除断句后，读者从"横向分库分表则代价不菲"直接读到"它的优先级一直是……"，节奏变成：

列举齐整 → 最后一个代价句号收尾 → 紧接哲学层面的总结

这里有一种"从事实上升为原则"的感觉，干脆利落。断句的去除反而加强了段落末句的收束力。

## 修复建议（精确文本）

将第 85 行：

```
此外，单机扩展有天花板，纵向加内存加 CPU 容易，横向分库分表则代价不菲。MySQL 由此付出的代价是。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。
```

改为：

```
此外，单机扩展有天花板，纵向加内存加 CPU 容易，横向分库分表则代价不菲。它的优先级一直是"对"排在"快"前面：可以慢，但不能错。
```

仅删除"MySQL 由此付出的代价是。"共 9 个字符（含句号）和其后的一个空格。其他不动。

## 后续检查建议

修改后建议做一次小范围的**读校**（reading proof），确认删除后前后句的衔接在语义和韵律上都自然。我的预读判断是自然的，但敏感编辑可能会发现"代价不菲""它的优先级"之间缺少一个过渡——如果觉得跳跃，可以在"代价不菲"后加一个冒号或破折号，不过我不认为有这个必要。当前写法足够直接。
