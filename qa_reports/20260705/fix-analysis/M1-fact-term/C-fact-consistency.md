## 🟡 轻微不一致

### C-01: ch04 表 4-3 中"重做日志"英文标注与全书约定冲突

**问题描述：**
- ch01 (L83, L212) 确立全书术语约定："重做日志（redo log）"，原则是"首次出现以'中文（英文）'标注，之后用中文。例如重做日志（redo log）"。
- ch04 表 4-3 (L253) 在 MySQL "持久化机制"一栏写道："重做日志（WAL）+ 脏页刷盘 + 双写缓冲"。
- 而 ch04 正文 (L151) 正确使用了"重做日志（redo log）"，并明确区分了 redo log 是具体实现、WAL 是通用原则。

**矛盾点：**
- 同章节正文与对比表使用了不同的英文标注：正文"(redo log)"、表"(WAL)"。
- "(WAL)"不是本书约定的注解方式——本书约定的是"重做日志（redo log）"。

**建议修复：**
将表 4-3 中 MySQL 列的"重做日志（WAL）"改为"重做日志（redo log）"，保持与全书约定一致。WAL 是通用原则，redo log 是具体实现，两者不应互换。

### C-02: ch04 与 ch08 对"重做日志"与"WAL"关系的精确度差异

**问题描述：**
- ch04 (L151) 写法："写一页之前，InnoDB 先把'这页要怎么改'写进**重做日志（redo log）**。重做日志是顺序追加写的。...这就是 **WAL（Write-Ahead Logging，预写日志）** 的核心：先写日志，后改数据页。" 
  -> 明确区分了"具体实现（redo log）"vs"通用原则（WAL）"。

- ch08 (L142) 写法："先写重做日志（这就是 WAL，预写日志），再改页；崩溃后用重做日志把页重做到一致状态。"
  -> 直接将"重做日志"等同于"WAL"。

- ch08 (L147) 写法："正常写路径上重做日志先落盘（WAL）"
  -> 再次直接等同。

**分析：** ch08 的"这就是 WAL"在狭义语境下（MySQL 内部）可以成立——因为对 MySQL 来说 redo log 就是它的 WAL。但与 ch04 的清晰分层（具体实现 ≠ 通用原则）相比，ch08 的表述精确度略低。这不是事实错误，但教学效果较 ch04 差，且两处读者看到的"重做日志与 WAL 的关系"微妙不同。

**建议修复（可选）：**
ch08 L142 可改为："先写重做日志（redo log，WAL 预写日志的具体实现），再改页；崩溃后用重做日志把页重做到一致状态。"——保持与 ch04 一致的"原理/实现"层次。

### C-03: 图 4-3 图注"7.x 默认的混合持久化"措辞边界模糊

**问题描述：**
图 4-3 正文图注 (L113)： "而 AOF（含 7.x 默认的混合持久化）走的是追加日志路径"

但本章前文 (L101) 明确指出："Redis 7.x 出厂默认并不开启 AOF（`appendonly no`，即 RDB-only）；只有显式打开 AOF 后，`appendfsync` 才以 `everysec` 为默认档"。

SVG 图标题写的是："写命令通往内存与 RDB 快照 / AOF（含混合持久化）两类持久化产物"——没有"7.x 默认的"字样。

**问题点：**
"7.x 默认的混合持久化"可能被读者理解为"7.x 默认持久化模式是混合持久化"，但实际默认是 RDB-only。更精确的语境是：当 AOF 开启时，7.x 默认使用混合持久化（`aof-use-rdb-preamble=yes`）。

**建议修复（二选一）：**
1. 与 SVG 保持一致，将"7.x 默认的"删除，改为"（含混合持久化）"——因为图注已经在"AOF"的上下文里，"AOF（含混合持久化）"自洽，无需加"7.x 默认的"。
2. 保留"默认"但加括号注明范围："AOF（开启时默认含混合持久化）"。

推荐方案 1，与 SVG 图标题一致，也避免歧义。
