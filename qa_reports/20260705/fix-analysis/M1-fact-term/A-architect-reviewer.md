## M1.1 ch04: Kafka PageCache 风险未充分展开

**原文定位**：`chapters/04-memory-disk/chapter.md` L181-L209，特别是"为什么不做自己的缓冲池"三个理由（L199-L203）。

**F02 报告说**：只讲了 PageCache 模式的好处，没展开延迟毛刺 / OOM 风险 / 冷读跃迁。

### 判断：必须修

**为什么是真问题**：这不是"多说一点细节"的锦上添花，而是 tradeoff 分析的必要对称性。Kafka 把缓存交给 OS 的决策有明确的生产代价，现在只列了三个好处（访问模式简单、避免双份缓存、重启后 PageCache 仍在），相当于只讲了半张 decision matrix。对于目标读者（架构师），代价缺位意味着读完后无法独立判断"我的场景能不能用这个模式"。

三个风险在生产中各自真实：

1. **延迟毛刺**：`vm.dirty_ratio`（20%）触发时写进程同步阻塞，Kafka 写延迟从亚毫秒跳到百毫秒级。64GB 机器，12.8GB 脏页到达阈值，刷盘期间的写延迟毛刺在生产监控里清晰可见。
2. **内存竞争**：PageCache 可以吞噬几乎全部空闲内存，同机其他进程（监控 agent、sidecar、日志采集）可能被 OOM killer 杀掉。这不是理论风险，是混合部署场景的常见事故。
3. **冷读跃迁**：消费者读的数据被挤出 PageCache 后，延迟从微秒级跳到十毫秒以上——生产里体现为"消费 lag 和磁盘读 I/O 同时上涨"的经典事故模式。

书中 L271 的"典型故障模式"里确实提了"消费者追不上（fetch 落到磁盘）"，但这是作为结论而非代价分析出现在横向对比段落的，读者读到这里已经过了"为什么不做缓冲池"的决策分析点。分析段（L199-L203）不该只列好处，应该在第三点之后对称增加一段代价。

### 建议修改方案

在 L203（"这对运维极友好。升级 Kafka 不等于让读性能掉回冷启动水平。"）之后新增一段：

```markdown
但把缓存交给 OS 也有三个明确的代价。

第一，延迟不可预测。PageCache 由内核全局管理，`vm.dirty_ratio`（默认 20%）达到时写进程同步阻塞，Kafka 的写延迟出现剧烈毛刺，从亚毫秒跳到数十到上百毫秒。自己管缓存的系统（Redis、MySQL）虽重启要预热，但运行中的延迟可预测——这是确定性换来的。

第二，内存竞争。PageCache 可以吞噬几乎全部空闲内存，同机部署监控 agent、日志采集等其他进程时，可能被 OOM killer 优先杀掉，或者因内存压力影响整机稳定性。Kafka 生产部署建议独占整机或通过 cgroup 严格隔离，正是为了避开这个风险。

第三，无冷热隔离。页缓存是全局 LRU，所有文件的访问按热度竞争缓存空间。Kafka 的冷分区数据可能把热分区的页缓存顶出去，造成热分区消费者突然落到磁盘读。自己管缓存则可以按业务优先级做精细淘汰。

Redis 和 MySQL 自己管内存，虽然重启后缓存要预热，但延迟可预测；Kafka 的 PageCache 模式把波动控制交给了内核——不是不付代价，只是换了一种代价的形式。
```

### 跨章联动

无。此修改仅限于 ch04 本节内部，不涉及其他章节。

## M1.2 ch07: Redis Cluster resharding 未提

**原文定位**：`chapters/07-cluster/chapter.md` L77-L92（7.2.3 Redis Cluster 节）。提到 MOVED/ASK 和槽迁移概念（L86），但未展开迁移过程的生产影响。

**F02 报告说**：槽迁移对生产的影响完全没提。

### 判断：必须修

**为什么是真问题**：Redis Cluster 的生产运维中，resharding（扩容 / 缩容 / 手动调整槽分布）是最高频的集群运维操作之一，也是初级架构师最容易低估代价的地方。书中 L84 提到"迁移粒度太粗"作为 16384 槽的折中，L86 给了 ASK 重定向语义——但这只回答了"技术上怎么回事"，没回答"做一次对系统有什么影响"。

对于架构师读者，缺少的具体信息包括：
- 迁移中大 key（百万成员的集合、大哈希）会导致该 key 访问的请求阻塞较长时间
- 迁移期间源节点和目标节点同时承受额外负载（读取 + 写入 + 网络传输，在大流量场景下可能导致延迟上升）
- 客户端 ASK 重定向处理不当可能引发大量重试（某些客户端库的缓存更新逻辑处理 ASK 不够优雅时，重试风暴确实发生过）

这些都是"生产里踩过才懂"的实战经验值。

### 建议修改方案

将 L86 的 ASK 描述之后增加一段（应在"MOVED 是状态变更后的最终路由，ASK 是迁移过程中的临时跳板"这段话之后）：

```markdown
ASK 重定向的存在本身就暗示了槽迁移是有代价的。实际迁移过程由 `redis-cli --cluster reshard` 驱动，大致流程是：在目标节点上导入槽的数据、在源节点上逐个发送 key 给目标节点（按 key 粒度或按 slot 粒度，取决于迁移工具版本和参数），一旦迁移完成源节点删除这些 key。

这个过程中的三个生产影响值得单独说明：

第一，**大 key 拖慢迁移**。如果一个集合或哈希里塞了几百万个元素，迁移这一个 key 就要花几秒甚至几十秒，期间该 key 的访问被阻塞。生产上建议在迁移前先处理大 key（拆小或提前迁出）。

第二，**双重负载**。迁移期间，源节点既要服务线上请求，又要读数据发给目标节点；目标节点既要服务线上请求，又要接收并写入迁移数据。CPU 和网络带宽在迁移期间都会上升一到两成。生产上建议在低峰期操作，并预留足够的资源余量。

第三，**客户端 ASK 风暴**。集群拓扑变更期间，不同客户端更新本地槽表的时机不同。如果客户端库的映射表更新逻辑存在问题（如重试时退避策略不当），ASK 重定向可能引发大量重试，进一步加重节点负担。成熟的客户端库（如 Redis StackExchange、lettuce、jedis）处理 ASK 已经稳定，但仍建议在大规模迁移前先测试客户端的重试行为。
```

### 跨章联动

无。此修改仅限于 ch07 本节内部。

## M1.3 ch07: acks=all 延迟模型不够精确

**原文定位**：`chapters/07-cluster/chapter.md` L168-L170（7.4.3 acks 节）。

原文说："代价是延迟：`acks` 越严，每条消息要等的副本确认越多，写延迟越高。"

**F02 报告说**：只说"更慢"，没讲跨机房 RTT 和 ISR 边界断崖。

### 判断：建议修

**为什么是真问题但优先级可调**：这句话本身没有错，但确实过于笼统。对于架构师读者，两个关键的生产边界是：
1. `acks=all` 下，写延迟取决于最慢 ISR 副本的网络 RTT —— 跨机房部署时，不是"稍慢"而是"单趟 RTT + 处理时间"，同城 2-5ms、异地 10-50ms，和 `acks=1` 的单机写延迟（亚毫秒）差了一到两个数量级。
2. 当 ISR 收缩到恰好等于 `min.insync.replicas` 时，任意一个副本 lag 触发写入失败，这是可用性断崖——不是"更慢"而是"可能不可用"。

但这两个点是"锦上添花的深度"而非"错误"。如果优先保证生产真实感，建议加；如果受篇幅限制，不修也不会误导读者（因为 `acks=all` 更慢本身是对的）。

### 建议修改方案

将 L170 的"代价是延迟"一句替换为：

```markdown
`acks=all` 的真实延迟模型取决于最慢的 ISR 副本。写延迟 ≈ max(所有 ISR 副本的确认时间)。跨机房部署时，这个值受限于最远端副本的网络单趟 RTT，同城 AZ 之间约 2–5ms，异地可达数十毫秒，和 `acks=1` 的单机写延迟（亚毫秒）差了一到两个数量级。

还有一个更隐蔽的边界：当 ISR 收缩到恰好等于 `min.insync.replicas` 时（例如 3 副本配置，ISR 里只剩 2 个，`min.insync.replicas=2`），任意一个副本因 lag 超时被踢出 ISR 都会导致写入全部失败——这是从"慢"到"不可用"的断崖。`acks=all` 配合极低的 `min.insync.replicas`（如 1），在 3.0 之前是默认组合，也是 Kafka 丢消息事故最常见的根因之一。
```

### 跨章联动

ch09 `acks` 与 `min.insync.replicas` 的精确交互有展开（书称在第 9 章）。ch07 此处做定性描述，ch09 做定量展开，合理。

## M1.4 ch04: LFU 衰减描述不精确

**原文定位**：`chapters/04-memory-disk/chapter.md` L77。原文："LFU 配合衰减机制，让长时间没访问的计数器自然回落。关键参数 `lfu-decay-time`（分钟）控制多久没访问降一级。"

**F02 报告说**：衰减是 lazy 的不是周期性的。

### 判断：必须修

**为什么是真问题**：原文说"多久没访问降一级"——这个措辞在字面上没错（确实按时间间隔降级），但缺失"什么时候降"这个关键信息。读者读到"自然回落"容易理解为后台有一个定时器周期性衰减所有计数器。实际实现是不同的：

- Redis 的 LFU 衰减不在后台线程做，而是在每次 `lookupKey()`（即每次访问键）时触发
- 衰减的具体实现是：用当前时间和键的上次访问时间算出差值，除以 `lfu-decay-time` 得到应衰减的级数，然后对计数器做右移（除以 2 的幂次）
- 所以一个长时间不被访问的冷 key，计数器在其被访问前*根本不会衰减*——这和"周期性衰减"有本质区别。对"冷 key 是否在内存中一直占位"这个问题的回答不同。

而且 `lfu-decay-time` 的单位是"分钟"，原文说"控制多久没访问降一级"——这也简化了：实际的衰减强度取决于 `lfu-decay-time` 和访问间隔的比值，不是"每过 N 分钟降一级"那么简单。

### 建议修改方案

将 L77 中"LFU 配合衰减机制"到"降一级"这一段改为：

```markdown
LFU 配合了衰减机制，但**不是后台周期性执行**的：衰减在每次访问键时 lazy 触发。具体来说，`lookupKey()` 函数在读取一个键时，会拿当前时间减该键的上次访问时间，除以 `lfu-decay-time`（分钟，默认 1）算出一个差值，把计数器右移这个差值次（近似除以 2^差值）。`lfu-decay-time` 控制的是一个衰减周期对应多少分钟，而不是"每分钟降一级"。由于衰减只在被访问的路径上触发，一个冷 key 在长时间不被访问期间，计数器保持原值不变——它必须在被淘汰扫描或被重新访问时才会计算衰减。
```

无。仅限于 ch04 本节。

## M1.5 ch02: 跳表选型遗漏范围扫描

**原文定位**：`chapters/02-data-structures-protocols/chapter.md` L37。原文："antirez 选跳表而非红黑树，就是出于实现简单和内存友好的考量。跳表的缓存局部性确实不如红黑树，但在内存里这点差异被 DRAM 的随机访问速度盖过了。"

**F05 报告说**：ZRANGEBYSCORE 在跳表底层链表前进 O(logN+M)，红黑树需中序遍历。

### 判断：必须修

**为什么是真问题**：这是一个事实上的遗漏，会导致读者对"为什么选跳表"的理解有重要缺口。antirez 本人多次公开解释选跳表的原因中，"范围扫描便利"是和"实现简单"并列的核心理由，甚至比"缓存局部性不是问题"更直接。对于 ZSET 这个 Redis 最常用的数据结构之一，skip list vs red-black tree 的选型分析缺了最直接的架构理由（范围操作），相当于只说了一半真相。

复杂度对比（跳表底层链表前进 O(log N + M)，红黑树需中序遍历 O(log N + M) 但实现复杂且缓存不友好）是对架构师读者有实际迁移价值的洞察。

### 建议修改方案

将 L37 中"antirez 选跳表而非红黑树"到"盖过了"这一段替换为：

```markdown
antirez 选跳表而非红黑树有三个原因。第一，实现简单：插入只改前后指针，不需旋转和染色。第二，**范围扫描便利**：有序集合上最常见的操作之一是 ZRANGEBYSCORE，跳表在底层链表上前进即可——找到起始节点后沿着底层双向链表扫 M 个节点，O(log N + M)；如果用红黑树，升序遍历要中序遍历加回溯（维护栈或 Morris 遍历），实现复杂且缓存不友好。第三，内存友好：跳表的平均指针数比红黑树略低（每个节点约 1.5 个额外指针 vs 2 个），在 ZSET 存千万级元素时能省下可观的内存。跳表的缓存局部性确实不如红黑树，但在内存里这点差异被 DRAM 的随机访问速度盖过了。
```

## M1.6 ch04: Kafka "全部交给 PageCache"忽略 mmap

**原文定位**：`chapters/04-memory-disk/chapter.md` L181-L183。原文："Kafka 不在进程里维护自己的缓冲池，把加速几乎全部交给操作系统的 PageCache。" 另有 L197-L203 进一步阐述"为什么不做自己的缓冲池"。

**F05 报告说**：Kafka 的索引文件通过 mmap 显式缓存。

### 判断：建议修

**为什么是建议修而非必须修**：这个判断在"大方向"上是对的——Kafka 的主要数据（`.log` 文件）确实不维护用户态缓存，靠 PageCache。索引文件通过 `java.nio.MappedByteBuffer` (mmap) 映射到进程地址空间，但这和"用户态另维护一份缓存"有本质区别：mmap 后索引文件的内容仍然在内核 PageCache 中，只是映射到了用户空间可以直接寻址。

换句话说，mmap 不是"副本"，是"映射"。Kafka 没有把索引数据复制一份到 JVM 堆里，索引的物理缓存仍然是 PageCache。所以"几乎全部"这个措辞在大方向上是成立的，但"全部"两字欠精确——因为索引文件的访问路径确实不同（mmap 允许进程通过指针直接读写缓存的页，不走 read/write 系统调用）。

### 建议修改方案

将 L182 的"把加速几乎全部交给操作系统的 PageCache"改为（注意原文 L64 也有"几乎全部"的类似措辞）：

```markdown
Kafka 不在进程里维护自己的缓冲池（数据部分的日志段文件），把加速主要交给操作系统的 PageCache。索引文件（`.index`、`.timeindex`）通过 Java NIO 的 MappedByteBuffer（mmap）映射到进程地址空间，但物理缓存仍在内核中——Kafka 没有把索引数据复制一份到用户态。
```

### 跨章联动

ch04 L64 也有"几乎全部"的类似措辞，需要同步修改：

L64 原文："Kafka 则干脆把'自己管内存'都省掉。把加速几乎全部交给操作系统的 PageCache"
建议改为："Kafka 则干脆把'自己管内存'都省掉。把数据文件的加速主要交给操作系统的 PageCache"

### M1.7b — ch01 MySQL "强一致"与 ch07 CAP 语境冲突

**原文定位**：
- ch01 L54："MySQL：用'B+ 树 + 预写日志 + MVCC'换强一致与通用查询"
- ch07 用 CAP 语境："MySQL MGR 多数派在最右（不丢、最慢）"，"偏 CP"

**F07 报告说**：ch01 的"强一致"是单节点 ACID 语境，ch07 是分布式 CAP 语境，可能混淆读者。

### 判断：不该修

**分析**：这两处"强一致"虽然字面相同，但在各自上下文中的指向是清晰且一致的：
- ch01 的"强一致"是指 MySQL 作为单节点 ACID 数据库的定位（事务隔离、不丢不错）。上下文是"三款软件的核心追求"，MySQL 的描述是"它优先保证一致性"，明显是 ACID 语义。
- ch07 的"偏 CP"是指分布式 CAP 语境，讨论的是 MGR 的多数派确认。

MySQL 产品定位本来就是 ACID 强一致 + 分布式偏 CP（MGR），两者本身就是一体的——它是因 ACID 强一致而"可靠"的产品，自然在 CAP 上倾向于 CP 方向。说"强一致"不矛盾。

而且 ch01 没有做分布式承诺，ch07 没有重新定义"强一致"——两章在不同层面说同一件事，不需要统一。强行统一反而可能制造表述混乱（例如 ch01 的"强一致"必须加上"ACID"前缀来刻意避嫌，读起来反而啰嗦）。

**结论**：不修改。这是一个"看起来很冲突但实际上不冲突"的假问题。

### M1.7c — ch04 表 4-3 Kafka "崩溃恢复代价=几乎无"过于简化

**原文定位**：`chapters/04-memory-disk/chapter.md` L256。表 4-3 行：`| 崩溃恢复代价 | 加载 RDB / 回放 AOF（大实例数分钟） | 重放重做日志 + 恢复脏页 | 几乎无（PageCache 热度取决于 OS） |`

以及 L269 的解读："Kafka 进程本身几乎不需要恢复，因为数据在磁盘、缓存归 OS 管，重启后立刻能服务。唯一受影响的是 PageCache 是否还热。"

**F07 报告说**："几乎无"过于简化。

### 判断：建议修

**分析**：说"几乎无"确实简化了，但上下文里已经加了括号解释"PageCache 热度取决于 OS"，正文段（L269）也说了"唯一受影响的是 PageCache 是否还热"。问题是这个括号放在表内太挤，容易被扫读时忽略。

而且严格来说，Kafka 崩溃恢复有两个维度：
1. 进程内恢复（加载元数据、分区重新分配）——确实快，几乎不需要"恢复"操作
2. PageCache 热度恢复——如果机器没重启、PageCache 还在，确实几乎无代价；但如果 PageCache 被完全刷新（例如机器重启），消费者读性能要等数据被重新加载进 PageCache 才能恢复

"几乎无"成立的前提是"进程级恢复"，如果读者误读为"从机器断电到完全恢复性能也几乎无"，就不对了。

### 建议修改方案

将表内"几乎无（PageCache 热度取决于 OS）"改为：

```markdown
| 几乎无（进程级；PageCache 冷时读性能需重新预热） |
```

对应正文 L269 段不做大改，可以微调为：

```markdown
Kafka 进程本身几乎不需要恢复（数据在磁盘、日志结构自带一致性，不需要重做或回滚），重启后立刻能服务。但"恢复"有两个层面：进程级确实快，但 PageCache 是否还热决定了读性能是不是也能立刻恢复——如果是整机重启，PageCache 空了，消费者读性能要等数据被重新加载才能回到内存速度。
```

### M1.8b — WAL（设计原则）与 redo log（具体实现）概念混用

**原文定位**：
- ch04 L149 标题为"### WAL 与崩溃恢复"，但正文 L151 说"写一页之前，InnoDB 先把'这页要怎么改'写进重做日志（redo log）。……这就是 WAL 的核心"
- ch04 L253 表 4-3 持久化机制行："重做日志（WAL）"
- ch08 L27/L111 也在用 WAL 指代 InnoDB 的 redo log

**F07 报告说**：WAL（设计原则）与 redo log（具体实现）概念混用。

### 判断：建议修

**分析**：这个问题在技术上成立——WAL 是一个通用的设计原则（任何"先写日志，后改数据"的做法都是 WAL），不单指 InnoDB 的 redo log。但 ch04 的上下文里，这个混用其实是有意为之的：该节要讲 MySQL 的 WAL 实现，以 redo log 为具体载体，用"InnoDB 把修改先写进重做日志——这就是 WAL 的核心"的递进句式来建立"原则→实现"的映射关系。在教科书式的写作里，把抽象原则和具体实现混用在同一个段落是常见的过渡手法。

但表 4-3 的"重做日志（WAL）"确实不妥——表里应该用精确的具体实现名称（redo log），WAL 作为设计原则应该注解而非并列。

另外 ch04 L253 表内"重做日志（WAL）"也有歧义：WAL 在 MySQL 里具体表现为 redo log 的预写机制，但它和 redo log 不是等号关系（WAL 是原则，redo log 是实现）。读者如果认为 WAL = redo log，会在理解其他系统时遇到困难（因为其他系统的 WAL 实现不同）。

### 建议修改方案

1. **ch04 L253 表 4-3**：将 MySQL 列"重做日志（WAL）"改为 "redo log（WAL 实现）"
2. **ch04 L149 标题**：保留"### WAL 与崩溃恢复"（引入 WAL 原则），但 L151 句尾可加一个更清晰的过渡："这就是 **WAL（预写日志）** 的核心思想在 InnoDB 中的具体体现。"
3. **ch08** 不修改，因为 ch08 L142 的表述"先写重做日志（这就是 WAL，预写日志），再改页"已经保持了"原则→实现→原则"的清晰递进，在上下文里不会混淆。

### 跨章联动

ch04 的修改可能影响 ch08 的引用（如果有 cross-ref）。ch04 L253 表 4-3 改为 "redo log（WAL 实现）" 后，如果其他地方引用此表，需同步确认。
