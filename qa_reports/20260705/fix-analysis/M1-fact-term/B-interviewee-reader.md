## 1. (F02) ch04: Kafka PageCache 只讲好处，未展开延迟毛刺/OOM 风险/冷读跃迁

**评分**：🟡 建议修

**精确原文**：
- 「为什么不做自己的缓冲池」三节（L197-209）：全部讲好处——访问模式顺序、避免双份缓存、进程重启后 PageCache 仍在。
- 关键数字框（L36）：提到 `vm.dirty_ratio=20%` 时写进程被同步阻塞强制刷盘。
- 典型故障模式（L271）：提到消费者追不上时读落到磁盘。

**读者感受**：
读前三节的时候会觉得「Kafka 永远赢麻了」。到了典型故障模式才看到一点代价，但那已经是全章末尾了。作为 2-3 年经验的读者，我在准备面试时如果被问到「Kafka 用 PageCache 有什么风险」，靠这章只能答出「消费 lag 会掉到磁盘读」，答不出「vm.dirty_ratio 触达时写延迟瞬间飙升」「冷读回源时的延迟跳变」「OS 内存压力下 PageCache 被回收后的性能悬崖」。关键数字框其实提到了 dirty_ratio 阻塞，但它在 4.1 节，读到 4.4 节时已经忘了前后关联。

**修复建议**：
在 4.4.3「为什么不做自己的缓冲池」末尾加一段边界，一句话点明三大风险即可：

```
把缓存交给 OS 也不是免费的。三个生产常见代价值得记住：第一，当写入过快触发 `vm.dirty_ratio`（默认 20%）时所有写进程同步阻塞等待刷盘，延迟瞬时飙升。第二，消费者读若长期落后于生产者，热数据被挤出 PageCache 后落到磁盘顺序读尚可接受，但突然跳到冷分区（消费者追赶历史数据或重启后 PageCache 已回收）时延迟从微秒级跌到毫秒级。第三，OS 在内存压力下可能回收 PageCache，Kafka 无法控制淘汰策略——这台机器上其他进程的大内存分配能直接冲掉 Kafka 的热缓存。
```

## 2. (F02) ch07: Redis Cluster resharding（槽迁移）对生产的影响完全没提

**评分**：🔴 必须修

**精确原文**：
L79：「加节点只是把部分槽从老节点迁移到新节点，大部分 key 不动。」

**读者感受**：
这一句让我以为迁移是轻量操作——「只是迁移」「大部分不动」。但真实情况是：每次槽迁移要逐 key 搬迁（`MIGRATE` 命令），网络带宽打满、CPU 飙升、客户端可能看到 ASK 多次重试。读到「只要 ISR 够用」这种级别的代价讨论时，对比之下 Redis 的迁移代价被严重弱化。

如果面试官问我「Redis Cluster 在线扩容有什么代价」，我靠这句话只能答出「大部分 key 不动」，会显得完全没有生产经验。这比「答得不准确」还糟糕——它给我一个虚假的轻松印象。

**修复建议**：
在 L79 末尾或 7.2.3 末尾加一段说明：

```
但槽迁移在生产中并不「轻量」。Redis 采用逐 key 搬迁：源节点遍历槽内所有 key，逐个执行 `MIGRATE` 命令将 key 序列化后发给目标节点。搬迁期间目标节点上该 key 的请求走 ASK 重定向（多一次 RTT），源节点会因序列化和网络 IO 占用额外 CPU。大量数据搬迁时可能出现带宽打满、延迟上升。因此生产实践中常配合限速（`CLUSTER SETSLOT` 的 `MIGRATING`/`IMPORTING` 状态）和业务低峰期执行。
```

## 3. (F02) ch07: acks=all 延迟模型只说「更慢」，未说跨机房 RTT 和 ISR 边界断崖

**评分**：🟡 建议修

**精确原文**：
L168-170：「代价是延迟：`acks` 越严，每条消息要等的副本确认越多，写延迟越高。」

**读者感受**：
话是对的，但太模糊了。如果面试官追问「acks=all 在跨机房部署下延迟会怎样」「ISR 里有 10 个副本和 2 个副本的延迟一样吗」，我根据书中内容只能笼统答「更慢」。关键缺失有两个：
1. acks=all 的等待时间是 ISR 中**最慢**的那个副本的确认时间——这天然受跨机房 RTT 支配。
2. 当 `min.insync.replicas` 接近 ISR 大小时，只要一个副本慢就拖慢全部写，有断崖效应。

**修复建议**：
在 L170 的「代价是延迟」后加一句展开：

```
具体来说，acks=all 等待 ISR 中全部副本确认，延迟取决于其中最慢的那个副本的网络往返时间。跨机房部署时，一个远端的 Follower 的 RTT 可能比本地多几毫秒到几十毫秒，这部分直接加到每次写入中。如果 `min.insync.replicas` 设得接近总副本数，任何一个副本抖动都会造成全局写延迟飙升。
```

## 4. (F02) ch09: MySQL 并行复制并行度受限于主库组提交并发度，未说清

**评分**：🟡 建议修

**精确原文**：
L167：「同一组提交（group commit）内的事务，本身就没冲突过。既然它们在主节点上是并行无冲突的，那在从节点上也可以并行回放。」
L169：「配合 `replica_parallel_workers` ... 复制延迟在现代硬件上可以降到很低。」

**读者感受**：
读完这一段我理解了「为什么可以并行」，但没理解「并行度的上限是什么」。主库的组提交并发度决定了从库的回放并行度——如果主库业务写冲突多或并发低，同一时刻能组提交的事务数很少，从库的并行线程大部分时间在空等。书中只说「可以并行」「降到很低」，却没说清楚这个上限。

面试场景：如果面试官问「MySQL 并行复制在什么场景下效果有限」，我靠书中内容只能答出「如果主库写冲突多就并行不起来」，但不理解具体为什么。

**修复建议**：
在 L167 末尾追加一句：

```
这意味着从库的并行度天生受主库的组提交并发度约束：如果主库同一时刻内组提交的事务数很少（例如写串行化、或并发度低），从库即便配置了更多并行工作线程也无事可做。并行复制解决的是主库高并发写场景下的回放瓶颈，对低并发写入场景帮助有限。
```

## 5. (F02) ch04: LFU 衰减机制描述不精确（衰减是 lazy 的，不是周期性）

**评分**：🟡 建议修

**精确原文**：
L77：「LFU 配合衰减机制，让长时间没访问的计数器自然回落。关键参数 `lfu-log-factor` 控制增长快慢，`lfu-decay-time`（分钟）控制多久没访问降一级。」

**读者感受**：
「多久没访问降一级」这句话让我误以为 Redis 有一个后台定时器线程周期性地衰减 LFU 计数器。实际上 Redis 的 LFU 衰减是 lazy（惰性）的——只在访问某个 key 时才计算它离上次访问过去了多久，按 `lfu-decay-time` 决定衰减多少级。不访问的 key 计数器永远不变。

这个区别不是钻牛角尖：如果你以为是周期性衰减，你会担心「所有键每隔 lfu-decay-time 分钟集体降一级」；知道是 lazy 衰减后，你会理解「一个冷 key 只要不被访问，它的计数器会永远保持原样——直到被访问时才触发衰减」。这对理解 LFU 的热度追踪行为很重要。

**修复建议**：
把「控制多久没访问降一级」改成：

```
衰减在 Redis 中是 lazy（惰性）的：每次访问一个 key 时，Redis 才检查它距离上次访问过了多久，按 `lfu-decay-time`（分钟）折算后衰减计数器。不访问的 key 计数器不变。
```

**精确原文**：

另：L17 已有「跳表偏范围扫描和有序操作」，L9 已有「跳表天然适合频繁更新 + 按序拉取」。

**读者感受**：
书中其他地方（L9、L17）已明确说了跳表支持范围扫描，问题出在**选型理由**这句——当书中明确比较跳表 vs 红黑树时只提了「实现简单和内存友好」，没提范围扫描。红黑树的中序遍历也能做范围扫描，但跳表的范围扫描实现更自然（顺着底层链表走就行），这是一个重要的工程选择理由。如果我是面试者，被问到「Redis 为什么用跳表不用红黑树」，只能答出「实现简单」——这个答案不够有深度。

**修复建议**：

```
另一层考量是范围扫描：跳表的最底层是一条完整的有序双向链表，做范围扫描只需定位起始节点后顺着链表往后走即可，不涉及中序遍历或父指针回溯，实现更自然。红黑树虽然也能做范围扫描，但需要中序遍历的递归或栈辅助。
```

## 11. (F05) ch02: SDS 惰性释放缺少 1MB 阈值说明

**评分**：🟡 建议修

**精确原文**：
L35：「惰性释放让缩短后多出来的空间先留着，下次增长时直接复用。」

**读者感受**：
「惰性释放」这个词让我以为 SDS 永远不释放缩短多出的空间。但实际 SDS v3.2+ 有阈值：如果缩短后剩余容量超过 1MB（`SDS_MAX_PREALLOC`），SDS 会主动释放到刚好容纳内容的大小。这个阈值解决了一个现实问题——对于大字符串（比如几 MB 的缓存），频繁缩短再增长时如果完全不释放，内存浪费会非常可观。

面试场景里，如果面试官问「Redis SDS 有什么缺陷」，我可以答出这个阈值相关的内存行为。书中目前让我以为 SDS 永远惰性、永远不释放，这是个重要的性能细节。

**修复建议**：
在「惰性释放」后添加阈值说明：

```
但惰性释放并非永远保留：SDS 内部有一个 1MB 的上限（`SDS_MAX_PREALLOC`），当缩短后剩余空闲容量超过 1MB 时，SDS 会主动释放到刚好容纳内容大小，防止大字符串的内存膨胀。
```

## 13. (F07) ch01: MySQL 用「强一致」与 ch07 CAP 语境冲突

**评分**：🟡 建议修

**精确原文**：
ch01 L54：「MySQL：用 B+ 树 + 预写日志 + MVCC 换强一致与通用查询」
ch01 L87：「典型场景：交易系统、订单与账户、任何需要强一致与复杂查询的主数据存储」
ch01 L115：「设计哲学/核心追求：不丢、不错（强一致与持久性）」

**读者感受**：
ch01 的「强一致」指的是 ACID 的事务一致性（单节点上事务的原子性和持久性保证），ch07 的「偏 CP」指的是分布式 CAP 中网络分区下的一致性行为。这两个「一致性」不是同一个概念。如果我把 ch01 记住的「MySQL 是强一致的」直接套到 ch07 的 CAP 语境里，会得出「MySQL 是 CP 系统」的印象——这本身没错，但推理链条不对。单节点 ACID 和分布式 CAP 是正交的，理解这个区别恰好是架构思维的关键一步。书里没有点明这层区别。

**修复建议**：
ch01 首次出现「强一致」描述 MySQL 时，加一个注释或脚注：

```
这里的「强一致」指 ACID 语义下的事务一致性（单节点），与第 7 章分布式 CAP 中的「一致性」是不同层面的概念。
```

或者更轻量的做法：把 ch01 中 MySQL 的「强一致」全部替换为「事务一致」（因为指的是 ACID），减少与 ch07 CAP 的混淆。

把表 4-3 Kafka 列从「几乎无」改为「较轻（日志段校验 + 索引重建，通常秒级），进程内无事务回滚」：

```
| 崩溃恢复代价 | 加载 RDB / 回放 AOF（大实例数分钟） | 重放重做日志 + 恢复脏页 | 较轻：日志段校验+索引重建（秒级），但 PageCache 冷启动可能影响读延迟 |
```

## 18. (T02) ch04/ch08: WAL（设计原则）与 redo log（具体实现）概念混用

**评分**：🟡 建议修

**精确原文**：
ch04 L151：「写一页之前，InnoDB 先把'这页要怎么改'写进**重做日志（redo log）**。...这就是 **WAL（Write-Ahead Logging，预写日志）** 的核心：先写日志，后改数据页。」
ch04 L253（表 4-3）：「重做日志（WAL）」——把 WAL 写在 redo log 的括号里，视作同义词。
ch08 L27（四个矛盾之一）：「校验和、双写、WAL、fsync 策略」——WAL 与具体机制并列，混淆了抽象层次。
ch08 L314：「定长页加索引...原地改加 WAL」——WAL 作为实现选择。

**读者感受**：
ch04 L151 的解释很好——先给具体实现（redo log），再上升到设计原则（WAL）。但后续使用时出现了层次混淆：ch04 表 4-3 把 WAL 当作 redo log 的同义词放在括号里；ch08 L27 在列举时把 WAL 和「双写」「fsync」并列，WAL 是设计原则，双写是具体机制，不该同级并列。

作为一个试图建立架构思维的读者，我关注「设计原则 vs 实现机制」的区别。WAL 可以是任何系统的设计原则（先写日志再改数据），但 redo log 是 InnoDB 的一个具体实现。书里这种混用会让读者失去区分抽象层次的机会——而这正是全书强调的架构视角。

**修复建议**：
- ch04 表 4-3：把「重做日志（WAL）」改为「重做日志（redo log）」——因为要的是具体实现，不是设计原则。
- ch08 L27：把「校验和、双写、WAL、fsync 策略」改为「校验和、双写、WAL（redo log 为其在 InnoDB 中的实现）、fsync 策略」，或者在列举时保持层次一致：把 WAL 改为 redo log 或在上下文解释 WAL 是先写日志的原则而 redo 是 InnoDB 的具体落地。
