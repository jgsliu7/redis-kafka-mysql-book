# G6-global-trademark 技术编辑审读意见

## 问题性质再定义

表面上这是一个"缺商标声明"的合规问题。从编辑视角看，底层的真正问题是：

**本书在产品名称的商标处理上处于"无决策状态"——既没有集中声明，也没有逐一标注，甚至没有一个明确的决策说"我们不标"。**

这不是一个写作遗漏（比如忘记写一段文字），而是一个出版流程缺失。C01 compliance 报告正确地将其标为 P0，但作为编辑，我更关心的是：**如果只是塞一段声明在序言里而不做任何决策记录，下次排版或转格式时它又会消失。**（前次 C01 报告就发生过"声称已修复、实际不存在"的情况，见 `C01-compliance.md` 第 13 行。）任何修改都需要附带一个显式的决策痕迹：这本书对商标处理采用什么策略，为什么。

## 插入位置判断

### 选项 A：序言末尾，"是为序。"之前

**结论：推荐。**

理由：

- **这本书不是纸质书**，没有"版权页 verso"这个物理位置。它是以 HTML 和 Markdown 分发的内容。序言是读者阅读正文前最后一站——版本声明（第 25 行）、AI 辅助披露（第 31-34 行）都在这里。商标声明放在同一段前材料末尾，逻辑上自洽。
- **"是为序。"是序言的自然终结符**。将商标声明安排在这个终结符之前，可以保持序言的完整性——读者读完序言的三个层次和创作说明，读到商标声明，然后"是为序"收尾。这个结构比"序言→商标声明→第一章"更干净。
- **标题层级**：建议用加粗段落（`**商标声明**`）而非 `##` 标题，避免商标声明在目录中占用条目。序言的"关于本书的创作过程"用的是 `##`，商标声明应该比它轻量。

### 选项 B：独立页面（例如 `chapters/00-trademark.md`）

利弊分析：

- 专业出版确实有"版权页独立"的惯例。但那是纸质书的产物——版权页必须印在扉页背面，和正文纸张不同。
- 对于这本 web-first 的书，多一个独立页面意味着 SUMMARY 多一条、链接多一个、构建脚本多一个文件。收益极低，管理成本上升。
- **否决**。

### 选项 C：后记末尾，版权声明附近

利弊分析：

- epilogue.md 第 49-55 行已经有版权与免责声明。把商标声明也塞过去确实能集中所有法律内容。
- 但读者是先读前言再读正文。商标声明需要在读者看到第一个"Redis"之前就完成告知——放在后记里不符合"事先告知"的原则。
- **否决**。

### 我的建议

插入 `chapters/00-preface.md`，具体位置在第 34 行（"关于本书的创作过程"段落结束后的空白行）和第 35 行（"是为序。"）之间。用一段分隔线 `---` 隔开，不新增目录条目，以简洁段落呈现。

## 中文技术书的惯例调查

基于经手过的十余本技术书和对行业的观察：

| 出版类型 | 书籍案例 | 商标处理方式 |
|---------|---------|-------------|
| 机械工业出版社 华章 | 各种技术译著 | 版权页集中声明，正文无标记 |
| 人民邮电出版社 图灵 | 《数据密集型应用系统设计》中文版 | 版权页集中声明，正文无标记 |
| 电子工业出版社 博文视点 | 《Redis 设计与实现》（第二版） | 版权页集中声明，正文无标记 |
| O'Reilly 原版书 | 各技术本 | 版权页集中声明，正文首次出现加 (R) 或 (TM) |
| 自出版技术书 | 社区作品 | 半数无声明，少数在前言末尾备注 |

**关键发现**：

1. **中文技术书压倒性地采用"集中声明"策略**，不在正文中标注 (R)。
2. **O'Reilly 原版书是严格标记的代表**，但即使是 O'Reilly 的中文翻译版，也通常去掉正文标记，只保留版权页声明。说明中文读者对 (R) 符号的接受度低。
3. **自出版的技术书普遍缺失商标声明**——这是一种风险，但也是事实。这本书既然把 publishable bar 作为质量标准，就应该补齐。

结论：**集中声明 + 正文不标** 是中文技术书的事实行业标准。

## 措辞的形式程度

本书的全文基调是现代技术散文：直接、清晰、没有法律腔。商标声明应该匹配这个基调。

### 不推荐的形式（过于法律化）

> 兹声明：本书所提及之"Redis"(R)系 Redis Ltd. 之注册商标，本书作者及出版方对该商标之使用纯属指代之目的，不得解释为与前述权利人之间存在任何关联关系、背书或赞助关系。其他产品名称亦同。

这是律师写的。读者读到会感觉"画风突变"。

### 不推荐的形式（过于随意）

> Redis 是 Redis 公司的商标。MySQL 是 Oracle 的商标。Kafka 是 Apache 的商标。有问题找他们。

虽然朴实但不够精确，并且"有问题找他们"有轻浮感。

### 推荐的形式（本书风格匹配）

```
---

**商标声明**

Redis 是 Redis Ltd. 的注册商标。MySQL 与 InnoDB 是 Oracle Corporation 的注册商标。Apache Kafka、Apache ZooKeeper 是 Apache Software Foundation 的注册商标或商标。本书在正文中使用上述名称时未逐一标注注册商标标记，仅用于指代对应的软件产品，不表示与权利人之间存在任何关联、背书或赞助关系。书中提及的其他产品与服务名称可能为其各自所有者的商标。
```

**设计理由**：

- 不用 (R) 符号而是用文字表述"注册商标"——更清晰，且与正文无符号的策略一致。
- "本书在正文中使用上述名称时未逐一标注注册商标标记"——这句话在法律上解释了为什么正文没有符号。这是一个实用的完整逻辑链。
- 覆盖了 InnoDB 和 ZooKeeper，因为它们在正文中出现频繁（InnoDB ~70 次，ZooKeeper ~24 次）。
- 不覆盖 Linux、Java、Docker 等通用第三方名称——它们在正文中出现极少且属于常识性品牌，不必在有限的声明空间里稀释重点。
- 用 `**商标声明**` 作为段落标题（加粗，不是 `##`），不与正文标题层级冲突，也不在目录中占位置。

## 正文中是否需要加 (R) 符号

**最终判断：不需要。**

量化分析：

| 维度 | 加符号 | 不加符号 |
|------|--------|---------|
| 修改量 | 全书约 600+ 处需修改 | 0 处 |
| 视觉影响 | 源代码中符号可能不显眼，但渲染后每个 Redis 都带尾巴，阅读被干扰 | 保持当前 clean 阅读体验 |
| 中文读者认知 | 多数读者不认识 (R) 或觉得突兀 | 无影响 |
| 法律充分性 | 声明 + 符号 = 超额满足 | 声明 = 已满足 |
| 出版社差异 | 如果将来转给出版社，出版社可能要求改 | 出版社也可能要求改 |

**关键决策逻辑**：加 (R) 不是"做对了不扣分"，而是"做了可能反扣分"——视觉干扰影响阅读体验。在已经提供集中声明的前提下，符号标记是一个纯粹的排版决策，不是法律决策。中文技术书的行业共识支持"不加"。

**但是如果出版社未来介入**，大概率会要求改成 O'Reilly 风格（正文首次出现加符号）。这是届时需要考虑的排版修改，现在不预留标记位置——等出版社提要求再批量处理，在 CI 上就是一次 grep 替换的事。
