# C-term-consistency: 商标标注的术语一致性分析

> 审查人: terminology consistency agent
> 审查日期: 2026-07-05
> 审查范围: 全书 chapters/*.md, docs/book-bible.md §7
> 关联发现: C01-compliance.md (P0: 商标声明完全缺失)

---

## 0. 背景与问题定义

全书缺失商标声明，正文中 Redis、MySQL、Kafka 等注册商标频繁出现（全书计数：Redis 368 次、MySQL 312 次、Kafka 341 次、InnoDB ~125 次、ZooKeeper ~25 次），但无一处标注 ® 或 ™，也无集中声明段。

**待决问题有两层:**
1. 插入位置的选址: 仅序言集中声明，还是正文首次出现也标注符号？
2. 若正文标注符号，与 book-bible.md §7 术语两档制是否冲突？

---

## 1. 术语两档制现状（book-bible.md §7）

book-bible.md §7 定义了两种术语使用策略:

| 策略 | 适用场景 |
|------|----------|
| **英文优先** | 工程界通用缩写/日常英文术语，首次"中文（英文）"之后以英文为主 |
| **中文优先** | 中文已充分沉淀的概念，继续用中文 |

商标归属标注与两种策略都**正交**:
- 英文优先的词（如 WAL、ISR、redo log）：加 ® 不会改变其"英文优先"属性，因为 ® 是法律标注符号，不是译名变体。
- 中文优先的词（如 缓冲池、副本、主节点）：不涉及商标符号，也不受影响。

**关键判断**: 商标符号 ® 不是术语译名变体，它不改变术语的"首选语种"。将 Redis 写作 Redis® 不会导致术语从"英文优先"漂移到别处。因此，理论上 body 加 ® 与 book-bible.md §7 的术语两档制**在语义层面不冲突**。

---

## 2. 两种方案的术语一致性分析

### 方案 A: 序言集中声明，正文不加符号（推荐）

```
在 chapters/00-preface.md 末尾、"是为序。"之前插入商标声明段落。
正文中所有名称保持原名不变（Redis 而非 Redis®）。
```

**术语一致性影响:**
- 全书术语表（book-bible §7）无需任何修改
- Redis / MySQL / Kafka 在正文中唯一形式不变，grep 能直接命中
- 术语两档制不受影响
- 与其他已统一的术语（ISR、WAL、offset 等）处于同一规范层次

**风险:** 无。

### 方案 B: 正文首次出现处加 ®，序言附加声明

```
在 chapters/01-introduction/chapter.md 中第一次出现 Redis/MySQL/Kafka 处加 ®。
后续所有次出现不再加。
序言保留集中声明段。
```

**术语一致性影响:**
- 需要更新 book-bible.md §7 术语表，增加"首次出现带®的记载"列
- 正文中 Redis 和 Redis® 同时存在，grep Redis 无法区分首次/非首次
- 需在全书范围确认第五个商标（InnoDB、ZooKeeper、Apache Kafka）的首次出现位置
- 每个商标的首次出现位置并非固定在同一章（InnoDB 首次在 references.md，ZooKeeper 首次在 references.md），可能导致某些章的第一次出现反而在 references 中
- 对 book-bible.md §7 的术语两档制有**操作层面的摩擦**:需要新增"首次带®形式"的追踪列

**具体冲突点:**
1. book-bible.md §7 术语表目前只有"英文 / 首次标注 / 后续用 / 说明"四列。若引入首次加 ® 规则，需新增第五列"商标符号规则"或统一在说明列标注
2. 术语表的 Redis 条目将被写成 Redis® 首次/Redis 后续，与其他无商标词（如 WAL、compaction）的格式不一致
3. 若出版社正式要求全文标注（即每次出现都加），则全书 ~1000+ 处需加符号，对构建脚本和 git diff 产生大量噪音

### 方案 C: 全文每处标注（不推荐）

传统法律极严格做法。不适用于技术图书——出版商界主流共识是序言集中声明 + 正文从简。本书正文 1000+ 次商标名出现，每处标注会严重破坏可读性。

## 3. 结论与推荐

### 3.1 正文商标符号与术语两档制的关系

| 问题 | 答案 |
|------|------|
| 正文加 ® 与 book-bible §7 术语两档制冲突吗？ | **不冲突**。® 是法律标注符号，不是译名变体，不影响"英文优先/中文优先"的分类 |
| 正文加 ® 是否需要更新术语表？ | **需要**。需在术语表增加"首次带®"追踪列，或明确"正文统一不带®，由序言声明覆盖" |
| 哪种做法对术语一致性维护更友好？ | **方案 A（序言-only）**。术语表无需新增列，grep 可全文唯一匹配，无首次/后续区分负担 |

### 3.2 推荐做法

**采用方案 A: 序言集中声明 + 正文不加符号**

理由:
1. **术语一致性成本最低**:book-bible.md §7 无需修改任何条目
2. **出版合规惯例**:O'Reilly、Manning、Pragmatic Bookshelf 等主流技术出版社均采用序言集中声明制，正文不加符号
3. **避免 grep/审校噪音**:正文所有 Redis 保持同一形式，审校 agent 无需区分"这是首次还是非首次"
4. **降低执行风险**:若未来更换出版社后对方有不同要求，调整范围仅限序言声明段一处
5. **符合项目历史判断**:round-5 合规审查已裁定"出版社惯例前言集中声明即可，正文沿用简称合规"（docs/审查-五轮交叉验证/round-5-合规结构.md L66）

### 3.3 需操作的唯一文件

chapters/00-preface.md

- 插入位置: 第 33 行"是为序。"之前
- 插入段落（from C01-compliance.md L49-55）:

> ## 商标声明
>
> Redis 是 Redis Ltd. 的注册商标。MySQL 与 InnoDB 是 Oracle Corporation 的注册商标。
> Apache Kafka、Apache ZooKeeper 是 Apache Software Foundation 在美国和/或其他国家的注册商标或商标。
> 本书中对上述商标的使用仅用于指代对应的软件产品，不表示与上述权利人之间存在任何关联、背书或赞助关系。
> 书中提及的其他产品和公司名称可能为其各自所有者的商标。

- 注意: 段落内保留 ® 符号（声明段是法律文书性质，必须带符号），但声明段中出现的名称 Redis® 与正文的 Redis 不同——这是声明段的固有属性，不构成术语漂移

### 3.4 可选增强（但非当前必要）

若未来出版社明确要求正文首次出现加 ®:
- 在 book-bible.md §7 术语表中增加一列"商标符号"（值: 有 / 无）
- 在 chapters/01-introduction/chapter.md 匹配每个商标名的第一次出现，改写为 Redis®、MySQL®、Kafka®
- 其他商标（InnoDB、ZooKeeper）在其首次出现的文件（均为 11-references.md）中标记
- 此操作延后至出版社明确要求时执行
