## 二、读者执行路径模拟

假设一个读者——工作 3 年的后端开发，有 Redis 使用经验但不是内核专家。

### 模拟路径 A：对照式读者（会实地验证书中命令）

1. 读到 ch04 "该配置项已移除"
2. 打开自己机器上 Redis 7.0+ 实例，想验证
3. 执行 `CONFIG GET aof-use-rdb-preamble`
4. 如果配置项仍存在且被识别：命令返回结果（非空），读者困惑：
   - "书说已移除，但我能查到。是书错了还是我的版本不是 7.0？"
   - 检查版本 -> `INFO server` -> redis_version:7.2.x
   - 结论：书和实际不一致。对全书的信任磨损一次。
5. 如果配置项虽已从 config 系统移除但被保留为兼容别名：
   - `CONFIG GET` 返回空，但 `CONFIG SET` 不报错
   - 仍然困惑："到底移没移？"

### 模拟路径 B：信任式读者（不验证，直接接受书中结论）

1. 读到 ch08 L92 "配置项已移除" 后
2. 不会去验证，因为两章都说同一件事
3. 如果需要调整混合持久化行为（例如想关闭混合格式），认为该选项不可用了
4. 查询 Redis 官方文档后发现选项还在——再次冲突，且附带烦躁感
5. 可能形成的错误认知：一种情况是"7.0 不再支持混合持久化"（最坏情况），另一种是"7.0 混合格式无法关闭"

### 模拟路径 C：扫读式读者（只扫描关键信息）

1. 看到 "已移除" 三个字，视线不会停留太久
2. 继续下文的 "base 文件始终为 RDB 格式，行为等效"
3. 扫读后的残留印象："7.x 没这配置了 + 效果一样"
4. 如果配置项实际上还可用于诊断（CONFIG GET 确认状态），扫读读者永远不会知道这个用法
5. 核心问题：扫读读者漏掉了"其实还可以查"这个信息

## 三、读者最可能产生的三个错误理解

### 错误理解 1："混合持久化不可用了"

- 原文实际说的是"配置项已移除"，强调的是配置项的管理面（interface），不是功能面（behavior）
- 但紧跟着的"base 文件始终为 RDB 格式，行为等效"在很大程度上消除了这个误解
- **读者可操作性风险**：中低。因为上下文紧接着解释了行为等效，不会被导向"功能没了"的纯错误结论。

### 错误理解 2："7.0 去掉了这个开关"

- 部分读者只扫到"已移除"三个字，没看清说的是配置项还是具体什么
- 扫读是技术阅读的客观行为规律，应该防范而非责备读者
- 形成印象："7.0 去掉了一个东西"
- **读者可操作性风险**：中。扫读读者容易抓住"已移除"三个字，留下机构变化印象。

### 错误理解 3："我不能再控制混合持久化的开关了"

- 如果 `aof-use-rdb-preamble` 作为一个配置项在 7.x 中仍然存在且有效（用户前提），这句话直接产生误导——读者会认为没法调整了
- 如果配置项已被降为 no-op（设 yes/no 不影响行为），则"已移除"描述的是管理面改动的后果，但措辞仍不精确——应该说"该选项已不再生效"
- **读者可操作性风险**：取决于实际行为：
    - 配置项仍生效可调 -> **高**。读者损失了一个配置途径
    - 配置项已 no-op -> **中**。需要说明其实际状态，而非用"移除"二字概括

### 综合风险评估

| 假设场景 | 读者错误理解概率 | 严重度 | 实际影响 |
|----------|-----------------|--------|---------|
| 配置项仍存在且生效 | 中 | P0 | 读者以为不可用，实际可用 |
| 配置项仍存在但 no-op | 中 | P1 | 读者以为不可用，实际是行为固化 |
| 配置项确实已移除（兼容名也无） | 低 | P2 | 结论正确，但"移除"措辞容易被扫读读者歪曲 |

**结论**：作者需在确认 7.x 实际行为后，决定修复方向和紧迫度。

## 五、修复措辞建议（读者视角）

核心原则：避免使用"已移除"这类容易被扫读歪曲的措辞。替换方案有两种，取决于 `aof-use-rdb-preamble` 在 7.x 中的实际行为。

### 方案 A：配置项仍存在可查（兼容性最高，推荐）

> `aof-use-rdb-preamble` 在 5.x–6.x 中默认即为 `yes`。7.0 引入 Multi-Part AOF 后
> base 文件**始终**为 RDB 格式，因此该配置项在 7.x 中虽然`CONFIG GET`可查、设值不报错，
> 但已无实际效果——混合格式行为强制定型，不可关闭。

优点：
- 精确：既说明配置项可查到，又说清行为不可调
- 读者可验证：`CONFIG GET` 不会返回空值，书和经验都对得上
- 扫读友好：第一眼看到"可查"而非"已移除"

### 方案 B：配置项已确实移除（如 7.0+ 不再识别该配置）

> `aof-use-rdb-preamble` 在 5.x–6.x 中默认即为 `yes`；7.0 多部 AOF 改造后该配置项已移除。
> **注意：** 这里的"已移除"指的是该配置项已从配置系统中删去（`CONFIG GET` 返回空），
> 但功能并未移除——7.0 起 base 文件始终为 RDB 格式，行为等同始终开启混合持久化。

优点：
- 明确区分"配置移除"与"功能保留"两个概念
- 加入读者提示，避免扫读误解

### 通用措辞改进点（两组方案都适用）

1. **扫读防护**：在"已移除"附近加一句读者提示："但功能未移除——行为等效于常开"
2. **版本防护**：补一句"该行为在 7.x 各版本中保持相同"（防止 2028 年读者见"7.0"字样而产生版本号焦虑）
3. **删去冗余信息**：两处原文都同时说了"已移除"和"行为等效"，这是正确的，但需要调整措辞顺序：先说功能没变（base 始终 RDB），再补配置面细节
4. **一致性**：ch04 和 ch08 两处改法完全一致

## 六、是否需要补充说明

**需要。** 建议在两处各补一句"为什么"说明，帮助读者理解架构变化而非死记事实：

> 混合格式在 7.0 中成为强制行为，是因为 Multi-Part AOF 的文件结构设计
> 天然要求 base 文件以 RDB 格式存在：manifest 文件通过 `type: B` 字段
> 标识基础文件类型，任何文件都可以独立确定格式，不再需要一个全局开关
> 来决定 base 文件的生成格式。行为固化后，配置接口自然收窄，功能保留。

这样读者理解的是"为什么"而不是背数字，能自己判断将来版本是否适用。
