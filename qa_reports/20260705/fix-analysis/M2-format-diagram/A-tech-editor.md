## P0: 必须修，读者会注意到

### M2.2 ch04: 缺失 5 张图的独立图注

**状态确认**：属实。

ch04 中的图 4-2 至 4-6 没有独立的图注行。说明文字直接以段落的首句跟在 `![]()` 之后。这与 ch04 自身的图 4-1（有独立图注 `*图 4-1　...*`）不一致，也与全书的图注习惯不一致。

**读者感知**：翻到这一页，图下面没有一句凝练的话总结"这张图在画什么"——第一段正文直接进了解读。问题不在于少了文字，而在于缺少了"图注"这个**定位锚点**。读者首先需要一个简洁的图注（类似"图 4-2　近似 LRU 采样淘汰与 LFU 计数器衰减机制"），再读下方的解释段落。

**改善程度**：9/10。不改的话，翻到图 4-2 至 4-6 的读者会感到困惑——"这张图叫什么名字？" 图注是快速导航工具，缺失它会破坏读者的阅读节奏。

**最小改动方案**：为图 4-2 至 4-6 各补一条独立图注行，使用全角空格（U+3000）分隔，使用正体（非斜体）。以下为建议文本：

```
图 4-2　Redis 近似 LRU 采样淘汰与 LFU 计数器衰减机制。LRU 侧用随机采样池加最近访问时间戳，LFU 侧用 Morris 计数器加衰减，共享同一个"采样 + 比较最差者"骨架。
```

```
图 4-3　Redis 写命令数据流通往内存与两类持久化产物：RDB 是 fork 子进程独立走全量快照，AOF（含混合持久化）走追加日志路径。
```

```
图 4-4　InnoDB 缓冲池三分链表：Free List 为空闲页来源、LRU List（新老分区）为缓存主体、Flush List 为脏页等待刷盘队列。
```

```
图 4-5　MySQL 一次写操作的全链路时序：WAL 先写重做日志落盘，脏页经双写缓冲到表空间，checkpoint 推进日志回收。
```

```
图 4-6　Kafka 生产-存储-消费零拷贝数据流：传统 4 次拷贝加 4 次上下文切换降为 2 次 DMA 拷贝，CPU 不参与数据搬运。
```

注意：当前正文首句与图注内容高度相似。补入图注后，正文首句需略作调整以避免重复（例如将"图 4-2 把..."改为"图中展示了..."）。

### M2.6 ch08: 全部 8 张图缺图前导语

**状态确认**：属实。

逐一检查 ch08 的 8 张图（8-1 至 8-8），**没有一张**有明确的引导句（如"如图 8-N 所示"）。部分图有弱上下文关联（如 fig 8-1 前文"这三句是后面它们格式差异的源头"——勉强算暗示读者要关注下图，但远不及其他章节的引导明确度）。对比其他章节：ch07 每张图前都有"图 7-N 把...画了出来"；ch09 也有"Redis 全量同步与部分重同步的完整分支见图 9-3"。ch08 全章空白的模式非常显眼。

**改善程度**：8/10。不改的话，有心的读者会注意到"其他章都有'如图'，这章怎么没有"。更重要的，这违反了全书契约中"图前一句导语引出"的硬性要求（见 book-bible.md 第 97 行）。

## P1: 建议修，对读者有明显影响

### M2.5 ch01: 图编号不连续

**状态确认**：属实。

ch01 中图出现顺序为：图 1-1（行 27，1.1.1 节）→ 图 1-3（行 126，1.2.4 节）→ 图 1-2（行 200，1.4.2 节）。图 1-3 出现在图 1-2 之前。

**原因**：这是编号约定（按章内出现序号）与写作逻辑（1.2.4 的雷达图需要紧跟在三款软件概览之后，1.4.2 的章节地图需要出现在介绍如何阅读时）之间的结构性冲突。不是笔误，而是编排上的两难。

**读者感知**：读者看到"图 1-3"出现在"图 1-2"之前，第一反应是"序号编错了"。虽然不影响理解，但会削弱读者对全书编排质量的信任。

**改善程度**：6/10。技术书读者通常对编号的连续性有预期。这不是致命问题，但破坏了一个隐性承诺（"编号=出现顺序"）。

**修复方案**（推荐方案 A——最小改动）：

将图 1-2 与图 1-3 的编号互换：
- 雷达图（现 1-3）→ 改为图 1-2
- 章节地图（现 1-2）→ 改为图 1-3

需同步更新：
- SVG 文件名（fig-1-2.svg → fig-1-3.svg 及反）
- `![]()` 中的文件名引用
- 图注行的编号
- 正文中对它们的引用

改动量约 6 处，全是替换式操作，不涉及内容重构，风险低。

### M2.4 ch01: 阅读路径 C 描述不准确

**状态确认**：属实。

ch01 行 222 原文为：
> **路径 C：对比驱动。** 每章只重点看 N.5 横向对比与 N.6 架构启示

但实际情况：
- ch02（数据结构）：横向对比在 **§2.4**，架构启示在 **§2.5**
- ch06（安全机制）：横向对比在 **§6.6**，架构启示在 **§6.7**

按本书模板（book-bible.md），标准结构为 N.5 横向对比 / N.6 架构启示 / N.7 小结。但 ch02 在 §2.1-§2.3 后接 §2.4 横向对比 → §2.5 架构启示 → §2.6 小结（多了 2.2 和 2.3 两个子节）。ch06 则因为多了 6.2-6.5 四个子节把横向对比推到 §6.6。

**读者感知**：路径 C 的读者通常是高级用户，大概率会快速浏览目录后自行定位。但 "N.6" 给了一个精确的位置承诺——在 ch06 里这个承诺不成立。承诺不精确，读一次就会让读者觉得本书不严谨。

**改善程度**：5/10。

**最小改动方案**：将原描述改写为：

> **路径 C：对比驱动。** 每章的重点在"横向对比"与"架构启示"两节（不同章因篇幅差异，节号可能不同；请以目录为准）。先看这两节吸收取舍精华，再按兴趣回看具体实现。

这保留了路径 C 的导航功能，同时为节号差异留了弹性空间。改动仅此一句话。

## P3: 格式统一类——标准定了才能修

### M2.1 ch01: 图注分隔符不统一 & M2.3 跨章三种变体

**完整调查结果**（已逐章逐图确认）：

| 章节 | 分隔符样式 | 备注 |
|------|-----------|------|
| 01-introduction | 全角冒号（图1-1/1-2）+ 全角空格（图1-3） | **同一章内不一致** |
| 02-data-structures | 全角空格（U+3000） | 内部一致 |
| 03-lifecycle | 全角空格（U+3000） | 内部一致 |
| 04-memory-disk | 全角空格（图4-1）+ 无独立图注（图4-2~4-6） | 内部不一致（涉及 M2.2） |
| 05-layered-arch | 全角空格（U+3000），斜体 | 内部一致 |
| 06-security | 全角冒号（U+FF1A），非斜体 | 内部一致 |
| 07-cluster | 全角空格（U+3000） | 内部一致 |
| 08-storage-format | 半角空格（U+0020），非斜体 | 内部一致 |
| 09-data-sync | 全角空格（U+3000） | 内部一致 |
| 10-summary | 半角空格（U+0020），非斜体 | 内部一致 |

附加差异：斜体 vs 非斜体——ch01（有斜体）、ch04 图 4-1（有斜体）、ch05（有斜体），其余均为非斜体。

**读者感知**：极低。读者不太可能注意到"冒号 vs 空格"或"全角空格 vs 半角空格"的差异。但如果图注与正文混排且缺少独立一行（如 M2.2 的情况），读者会注意到"有的图有独立图注有的没有"——这才是读者能感知到的差异。分隔符的种类不是。

**改善程度**：3/10（作为独立问题）；但如果作为"图注格式标准化"的一次性 pass 来解决，改善程度可达 7/10。

**推荐统一格式**：

```
图 N-M　说明文字
```

即：
1. 编号与文字之间使用**全角空格（U+3000）**——中文出版物图注的惯例做法，视觉留白自然
2. 使用**正体（非斜体）**——斜体在中文排版中非常规，只在书名/外文时才使用
3. 图注为独立行，不与正文段落嵌套

理由：ch03/ch07/ch09 已统一使用全角空格，覆盖率最高（7/10 章）；全角空格在中英文混排中视觉上最齐整；避免半角空格，因为中文混排时半角空格前后留白不均匀。

**统一涉及改动范围**（约 26 处）：

- ch01：将 fig 1-1/1-2 的 `：`（U+FF1A）改为 `　`（U+3000），取消斜体
- ch06：将 4 张图的 `：` 改为 `　`
- ch08：将 8 张图的 ` `（U+0020）改为 `　`（U+3000）
- ch10：将 3 张图的 ` `（U+0020）改为 `　`（U+3000）
- ch04：图 4-1 取消斜体
- ch05：7 张图取消斜体
