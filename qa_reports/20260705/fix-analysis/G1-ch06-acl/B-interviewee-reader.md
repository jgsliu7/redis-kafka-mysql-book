## 问题 1：Kafka `allow.everyone.if.no.acl.found` 版本历史错误

### 精确原文

第 135 行：

> Kafka 的默认策略偏保守：`allow.everyone.if.no.acl.found` 这个参数（对 `AclAuthorizer` 生效；KRaft 下的 `StandardAuthorizer` 同样遵循"默认拒绝"）**从 1.x 起默认就是 false**，即无 ACL 即拒绝，3.x 沿用这一默认。

### 事实核查

Kafka 的 `allow.everyone.if.no.acl.found` 默认值变更发生在 **2.0 版本**（KAFKA-6310 / KIP-126），具体 timeline：

| 版本 | 默认值 | 行为 |
|------|--------|------|
| 0.x | `true` | 无 ACL 则允许所有人 |
| 1.x | `true` | 无 ACL 则允许所有人 |
| 2.0+ | `false` | 无 ACL 则拒绝 |

书中"从 1.x 起默认就是 false"是错误的。**整个 1.x 生命周期内默认值一直是 `true`**，即默认不安全。

### 读者视角判断

**实际伤害：高。**

这里的关键不是"作者版本号写错了几个数字"——而是这本书对我的核心承诺是"建立比面经更深的理解"。面经的错误之一是"版本细节含混"，我正是因为讨厌面经的以讹传讹才来看这本书的。现在书中出现同样的错误，对我的信任打击很大。

具体场景推演：

1. **面试场景**：面试官问"你了解 Kafka 的 ACL 默认策略吗？" 我如果回答"从 1.x 起默认就是拒绝"，面试官会纠正我。我不仅答错了，还会留下"用过但没真正读过"的印象——这恰恰是我最想避免的。

2. **运维决策场景**：如果我在维护一个 1.x 集群（实际上我 2026 年不会碰到，但面试时会聊升级路径），读到这个说法会认为 1.x 已经默认安全，不会关注 ACL 的补齐工作。实际上 1.x 集群是需要显式设置参数的。

3. **学习路径依赖**：这个错误会让读者对 Kafka 的安全演进形成"一直很激进"的认知。而真实故事更有教育意义：Kafka 是 2.0 才把安全底线收紧的，这背后是整个社区对"默认安全"认知的演进——这个故事本身比正确的版本号更有价值。

### 修复建议

措辞上，建议改成类似于：

> `allow.everyone.if.no.acl.found` 这个参数在 1.x 及之前默认是 `true`（即无 ACL 则允许所有人，默认不安全），2.0 起才改为默认 `false`（KAFKA-6310 / KIP-126）。KRaft 下的 `StandardAuthorizer` 从引入起就遵循"默认拒绝"。换句话说，Kafka 的安全底线不是一开始就收紧的——2.0 这次默认值变更标志着社区"默认安全"意识的转折点。

保留原文中关于 `unclean.leader.election.enable` 的注释，那个对比是有价值的。

### 修复必要性

**必须修。** 不修的话，读者会：
- 面试答错（最直接的伤害）
- 对 Kafka 版本演进的理解建立错误基线
- 怀疑本书其他版本声称的准确性

## 问题 2：Redis ACL 密码"无加盐"陈述

### 精确原文

第 48 行：

> SHA-256 是哈希不是加密，且无加盐，对弱密码仍有字典攻击风险，因此 Redis 文档强调密码要足够长且随机。

### 事实核查

Redis 从 6.0 引入 ACL 时，密码是 **有加盐存储** 的。ACL 文件中密码的存储格式为：

```
#sha256::<16 字节随机 salt 的 Base64 编码>::<SHA-256(password + salt) 的 Base64 编码>
```

验证密码时，Redis 读出 salt，将其拼接到输入的密码上，再做 SHA-256 比对。salt 是每次 `ACL SETUSER >password` 时自动生成的随机数，用户无需手动干预。

换句话说，加盐是 Redis ACL 密码存储流程中不可分割的一步。书中说"无加盐"是错的。

### 需要澄清的重要细节

salt 本身确实以明文形式存储在 aclfile 中。这意味着：

- **加盐生效的部分**：防止预先计算的彩虹表攻击。即使攻击者拿到了完整 aclfile，也无法用预计算表同时破解所有用户的密码——每个用户的 salt 不同，必须逐个暴力。
- **加盐不防的地方**：如果攻击者有 aclfile 读权限，并且要针对某个特定弱密码做暴力破解，那么加盐只增加了每次计算 hash 的成本（拼接 salt 的开销忽略不计），没有本质防护增强。
- **本质风险**：正如原文所说，弱密码才是真正的问题。但原因不是"无加盐"，而是"加盐只防彩虹表，不防针对单个用户的暴力破解"。

### 读者视角判断

**实际伤害：中。**

对我这个读者来说，这个错误影响的是"对 Redis 密码存储机制的理解准确性"。具体来说：

1. **和我的既有知识冲突**：我之前知道 bcrypt、scrypt 这类密码哈希都是带盐的，Salt 明文存是标准做法。如果 Redis 真的不做加盐，我会觉得 Redis 很落后——但实际 Redis 做了，这个错误反而会让作者在我心中信用减分。读到"无加盐"时我没立刻怀疑，因为我在 Redis 文档里没见过这部分细节，面经也没人提。结果读到这个分析才意识到被误导了。

2. **不会直接造成安全操作事故**：和问题 1 不同，这个错误不会让我去错误地配置生产系统。即使我认为"无加盐"，我的应对方式（用足够长的随机密码）和 Redis 推荐的做法一样。所以实际影响小于问题 1。

3. **但会让章节的"信任感"打折扣**：本章的核心诉求是让读者建立对安全机制的正确理解。"无加盐"这个错误会让后面所有说理（比如对比 MySQL、Kafka 的认证设计）都蒙上阴影——读者会想，"连 Redis 密码有没有加盐都搞不清，MySQL 和 Kafka 的论述还能信吗？"

### 修复建议

修改第 48 行，保留"SHA-256 是哈希不是加密"和"弱密码仍有字典攻击风险"的正确论断，只修正加盐部分。建议措辞：

> SHA-256 是哈希不是加密。Redis 实际做了加盐：每次生成密码时随机产生 16 字节 salt，以明文形式一并存入 aclfile，验证时拼接待测密码再哈希比对。加盐使预计算的彩虹表失效。但 salt 明文存意味着：一旦 aclfile 泄露，针对单个用户的暴力破解不受影响。因此 Redis 文档强调密码要足够长且随机——加盐防的是"一次算表破解全库"，不防"拿到你的 hash 专门猜你的密码"。

这个修法有三个好处：
- 纠正了事实错误
- 顺便解释了"加盐但盐明文存"这个带争议的设计思路，比单纯删除"无加盐"三个字更能帮读者建立深度理解
- 最终结论（"密码要足够长且随机"）不受影响

### 修复必要性

**应该修。** 理由：
- 不修的话，读者对 Redis 密码存储形成错误认知（以为 Redis 连基础防护都没有）
- 修了之后，可以展开"加盐 vs 暴力破解"这个有深度的讨论——这恰恰是本书区别于面经的地方
- 改动极小，风险低
