# 文字校对报告：`sync_binlog=1` 措辞精确性

## 问题定位

- 章节：chapters/10-summary/chapter.md
- 行号：96
- 原文："MySQL 默认偏向持久性（`innodb_flush_log_at_trx_commit=1` 是默认），若再开启 `sync_binlog=1` 即"双 1"配置"

## 问题性质：事实性措辞错误（硬语病 / 文字级别错）

### 问题描述

"再开启"暗示 `sync_binlog=1` 是需要额外开启的选项，但 MySQL 8.0（本书目标版本，因书中提及 MGR 等 8.0 特性）中 `sync_binlog=1` 已是默认值。

| MySQL 版本 | sync_binlog 默认值 | 来源 |
|---|---|---|
| MySQL 5.7 | 0 | MySQL 5.7 官方文档 |
| MySQL 8.0（自 8.0.3） | 1 | MySQL 8.0 官方文档 |

因此，在 MySQL 8.0 语境下，"双 1"配置是**出厂默认组合**，而非需"额外开启"的选项。"再开启"一词给读者的印象是错误的——仿佛不开 `sync_binlog=1` 也能获得接近双 1 的持久性，实则 MySQL 8.0 中两者都是默认值，断电源自始至终就不丢提交数据。

（即使读者用的是 MySQL 5.7，该段也在 10.3.1 维度一框架下叙述，未注明版本限定；本书整体以 8.0 为参照，无版本降级提示。）

### 上下文（前后各 2 句）

```
[95 空行]
[96] **维度一：速度 vs 持久性**。数据写下去多久才算数，是这一维度要回答的问题。
[    ] Redis 默认偏向速度，`appendfsync=everysec` 把风险控制在约 1 秒量级的数据窗口内，因此写入开销更低。
[    ] MySQL 默认偏向持久性（`innodb_flush_log_at_trx_commit=1` 是默认），若再开启 `sync_binlog=1` 即"双 1"配置，

[    ] Kafka 偏向持久性，`acks=all` + 多副本让一条消息被 ISR 全部确认才算写入成功，付的是副本同步的 RTT。
[97] 三者各取所需：Redis 用秒级窗口换吞吐，MySQL 用每事务一次 fsync 换断电安全，Kafka 用副本同步换多副本下的数据不丢。
```

## 修改方案

### 最优修改

采用"已是默认值"，措辞最精确：陈述事实，无冗余解释，不改变句式节奏。

**old_string:**
```
MySQL 默认偏向持久性（`innodb_flush_log_at_trx_commit=1` 是默认），若再开启 `sync_binlog=1` 即"双 1"配置
```

**new_string:**
```
MySQL 默认偏向持久性（`innodb_flush_log_at_trx_commit=1` 与 `sync_binlog=1` 均为默认值，合称"双 1"配置）
```

**修改效果对比：**

| | 原文 | 修改后 |
|---|---|---|
| 对默认值的表述 | 只说明 `innodb_flush_log_at_trx_commit=1` 是默认，`sync_binlog=1` 暗示需额外开启 | 明确两者均为默认值 |
| "双 1"的定义方式 | 由"再开启"条件引导，像是到达某种高级配置 | 直接将双 1 定位为出厂组合 |
| 读者理解 | 可能认为 MySQL 默认只保证一半的持久性 | 清楚知道 MySQL 8.0 出厂就是双 1 |

### 备选方案

- "默认即开启"：口语化，但不如"已是默认值"精确，"开启"偏操作层面而非配置层面。
- "无须额外配置"：偏操作含义，不符合此段"解释参数含义"的说明性语境。
- "已是默认值"（已采纳）：精准说明该参数的值状态，与前半句"`innodb_flush_log_at_trx_commit=1` 是默认"形成平行结构，节奏一致。
