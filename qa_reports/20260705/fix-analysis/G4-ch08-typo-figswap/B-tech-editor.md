# B — 技术编辑视角：ch08 两个 P0 问题的修复分析

## 问题 1：标点冗余连用（第 40 行）

### 精确原文

chapters/08-storage-format/chapter.md 第 40 行：

```
权重的倒置，也是它们格式选择不同的原因。：为内存优化、为磁盘随机访问优化、为顺序追加优化的格式，自然就是三种样子。
```

段号（句号）与冒号连用：`原因。：`

### 编辑规范分析

中文标点规范中，句号是句末终止性标点，表示一个完整陈述句的结束。冒号是句中提示性标点，引出解释、列举或补充说明。二者连用属于标点冗余——句号截断了语气，冒号试图重新接上，语义上相互矛盾。

从句子结构看：

- 前半句 "权重的倒置，也是它们格式选择不同的原因" 是一个完整的因果判断。
- 后半句 "为内存优化、为磁盘随机访问优化、为顺序追加优化的格式，自然就是三种样子" 是对这个"原因"的具体展开——三个并列项加总括。
- 冒号在这里的作用是引出对"原因"的解释，是句内提示关系，不是句首另起。

因此正确的标点是：冒号，不是句号。去掉句号，保留冒号。

### 修复建议

删除句号，保留冒号：

```
权重的倒置，也是它们格式选择不同的原因：为内存优化、为磁盘随机访问优化、为顺序追加优化的格式，自然就是三种样子。
```

修改后读感：因果一气呵成。"原因"后略作停顿，冒号引出三个并列项，最后"自然就是三种样子"收束，节奏自然。

### 上下文衔接检验

修改前（36-40 行整段阅读）：

> 这张图把三款软件的内存与磁盘权重摆在同一坐标系下：Redis 内存占绝对主导，磁盘只是恢复副本；MySQL 磁盘是真相之源，内存缓冲池只是磁盘页的缓存副本；Kafka 磁盘就是数据本体，内核页缓存承担内存层角色。权重的倒置，也是它们格式选择不同的原因。：为内存优化、为磁盘随机访问优化、为顺序追加优化的格式，自然就是三种样子。

"也是"承接上文一整段对三款软件权重差异的描述，冒号后是对"原因"的补充展开。去掉句号后，"原因"和"为内存优化……"之间由冒号自然连接，节奏不被打断，比段号+冒号的顿挫更流畅。

## 问题 2：fig-8-7 与 fig-8-8 文件引用互换

### 现状排查

**文件系统中的 SVG 文件及其内容：**

| 文件名 | 内部标题文本 | 实际描绘内容 |
|--------|------------|------------|
| `fig-8-7.svg` | "图 8-7 存储格式设计决策树" | 决策树（三问流程图） |
| `fig-8-8.svg` | "图 8-8 Kafka Tiered Storage 日志段生命周期" | Tiered Storage 生命周期图 |

**chapter.md 中的引用（均未修改前）：**

| 所在章节 | 行号 | Markdown 引用 | 图文说明 |
|---------|------|--------------|---------|
| §8.4.6 Tiered Storage | 222 | `![图 8-7 ...](diagrams/fig-8-7.svg)` | "图 8-7 Kafka Tiered Storage 日志段生命周期：本地热段关闭后异步上传到远程对象存储……" |
| §8.6.6 给实践者的三问 | 316 | `![图 8-8 ...](diagrams/fig-8-8.svg)` | "图 8-8 存储格式设计决策树：从访问模式推到格式骨架。" |

**问题揭示：**

- chapter.md §8.4.6 说 fig-8-7 是 Tiered Storage，但引用的 `fig-8-7.svg` 文件中实际画的是决策树。
- chapter.md §8.6.6 说 fig-8-8 是决策树，但引用的 `fig-8-8.svg` 文件中实际画的是 Tiered Storage。
- 两张 SVG 自身的内部标题文本与它们的内容一致，但与 chapter.md 的图号分配相反。
- 结论：SVG 文件的图号命名与 chapter.md 的图号分配不一致——文件应属于对方的图号。

### 读者体验影响

当读者在 §8.4.6 读到"图 8-7 展示了 Kafka Tiered Storage 日志段生命周期"时，看到的却是一张决策树流程图。两种后果：

1. 如果读者粗读，只看图不看图注——图上大字标题写着"存储格式设计决策树"，与"Tiered Storage"完全无关。读者困惑。
2. 如果读者细读，连图连注一起看——图注说 Tiered Storage，图里画的是决策树，矛盾立刻暴露。信任受损。

同样，在 §8.6.6 看到决策树说明时，却看到 Tiered Storage 图，困惑加倍。

### 修复方案对比

#### 方案 A：只改 chapter.md 的引用（不推荐）

将第 222 行的 `fig-8-7.svg` 改为 `fig-8-8.svg`，第 316 行的 `fig-8-8.svg` 改为 `fig-8-7.svg`，同时互换图号。

问题：破坏了全书图号的顺序递进。§8.4.6 将出现 fig-8-8，§8.6.6 将出现 fig-8-7——读者会看到 fig-8-8 出现在 fig-8-7 之前。图号不再单调递增，对顺序阅读的读者是显眼的异常。

#### 方案 B：互换 SVG 文件名 + 修复 SVG 内部标题（推荐）

两步操作：

1. **互换文件名**：`fig-8-7.svg` ↔ `fig-8-8.svg`
2. **更新 SVG 内部标题**：
   - 原 `fig-8-7.svg`（决策树，将被重命名为 `fig-8-8.svg`）：内部标题 "图 8-7" → "图 8-8"
   - 原 `fig-8-8.svg`（Tiered Storage，将被重命名为 `fig-8-7.svg`）：内部标题 "图 8-8" → "图 8-7"

修复后状态：

| 文件名 | 内部标题 | 内容 | chapter.md 引用 |
|--------|---------|------|----------------|
| `fig-8-7.svg`（原 fig-8-8） | "图 8-7 Kafka Tiered Storage 日志段生命周期" | Tiered Storage | §8.4.6 引用 ✓ |
| `fig-8-8.svg`（原 fig-8-7） | "图 8-8 存储格式设计决策树" | 决策树 | §8.6.6 引用 ✓ |

读者体验：一切自洽。图号顺序递进（8-1 到 8-8），图片内部标题与章节正文的图注一致，内容与上下文完全吻合。

#### 方案 C：互换文件名 + 不改内部标题（不完整）

如果不改 SVG 内部标题，那么看到决策树图时，图内标题显示"图 8-7"但图注写"图 8-8"。细心的读者仍会困惑。必须内外一致。

#### 方案 D：只改 SVG 内部标题不改文件名 + 改 chapter.md 引用（不推荐）

相当于只修正 SVG 内部标题使其匹配文件命名，然后改 chapter.md 的引用。复杂度与方案 B 相当，但后果是所有引用都指向与内容不符的文件名，可维护性更差。

### 建议

**方案 B 是唯一满足"三个一致"的修复方案：**
- 文件名与 SVG 内容一致
- SVG 内部标题与文件名一致
- SVG 内部标题与 chapter.md 图号引用一致

操作步骤：

1. 将 `/Users/liu/dev/demos/redis-kafka-books/chapters/08-storage-format/diagrams/fig-8-7.svg` 重命名为 `fig-8-7.decision-tree.svg`（临时）
2. 将 `/Users/liu/dev/demos/redis-kafka-books/chapters/08-storage-format/diagrams/fig-8-8.svg` 重命名为 `fig-8-7.svg`
3. 将临时文件 `fig-8-7.decision-tree.svg` 重命名为 `fig-8-8.svg`
4. 在新 `fig-8-7.svg`（原 fig-8-8）中，将第 13 行 `<text ...>图 8-8 Kafka Tiered Storage 日志段生命周期</text>` 中的 "图 8-8" 改为 "图 8-7"
5. 在新 `fig-8-8.svg`（原 fig-8-7）中，将第 15 行 `<text ...>图 8-7 存储格式设计决策树</text>` 中的 "图 8-7" 改为 "图 8-8"

chapter.md 无需任何改动。
