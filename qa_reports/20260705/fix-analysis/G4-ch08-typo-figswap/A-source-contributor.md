## 问题 1：第 40 行标点冗余连用

### 精确原文与上下文

文件：`chapters/08-storage-format/chapter.md` 第 40 行

```
权重的倒置，也是它们格式选择不同的原因。：为内存优化、为磁盘随机访问优化、为顺序追加优化的格式，自然就是三种样子。
```

红色标记：`原因。：`——中文句号（。）紧接中文冒号（：），标点冗余。

### 判断：正确的中文标点

前文"权重的倒置，也是它们格式选择不同的原因"是一个完整的主谓宾结构（主语"权重的倒置" + 系动词"是" + 宾语"原因"），冒号在此用于引出对"原因"的具体展开——也就是"为内存优化、为磁盘随机访问优化、为顺序追加优化的格式"这三个并列项。

**正确的标点是只保留冒号，删除句号**：

- 改前：`不同的原因。：为内存优化`
- 改后：`不同的原因：为内存优化`

理由：冒号之前不应出现句号。冒号的语义是"提示下文"——前面的"不同的原因"恰好是需要被展开说明的对象，冒号直接承接即可。句号的存在强行切断了这一逻辑关系，使得"为内存优化……"变成一个悬空的片段。

### 可选的替代方案

如果作者认为"不同的原因"这句话在此地已经完句、需要重启一个句子来列举，也可以写成：

```
权重的倒置，也是它们格式选择不同的原因。为内存优化、为磁盘随机访问优化、为顺序追加优化的格式，自然就是三种样子。
```

但这样"为内存优化、为磁盘随机访问优化、为顺序追加优化的格式，自然就是三种样子"就变成了独立的陈述句，丢失了冒号那种"前面给原因、后面列展开"的紧密连接感。从原文的修辞节奏看，冒号是更好的选择。

**结论：删除句号，保留冒号。** 改后句子变为：

> 权重的倒置，也是它们格式选择不同的原因：为内存优化、为磁盘随机访问优化、为顺序追加优化的格式，自然就是三种样子。

### old_string / new_string

```
old_string: '不同的原因。：为内存优化'
new_string: '不同的原因：为内存优化'
```

### 风险评估

**低风险。** 单字符修改（删除一个句号），改变的是该句的逻辑衔接而非意义。不涉及任何其他引用、编号或结构。T01 typo-grammar agent 已确认为硬错误，修复只有明显收益、无副作用。

## 问题 2：fig-8-7 与 fig-8-8 文件引用互换

### 精确原文与上下文

**行 222（第 8.4.4 节 Kafka Tiered Storage）**：

```markdown
![图 8-7 Kafka Tiered Storage 日志段生命周期](diagrams/fig-8-7.svg)
图 8-7 Kafka Tiered Storage 日志段生命周期：本地热段关闭后异步上传到远程对象存储，消费者和 Follower 读取时先尝试本地、未命中则回源远程。
```

**行 316（第 8.6.6 节"给实践者的三问"）**：

```markdown
![图 8-8 存储格式设计决策树](diagrams/fig-8-8.svg)
图 8-8 存储格式设计决策树：从访问模式推到格式骨架。
```

### 问题本质

这是一个**三要素不匹配**的问题：

| 维度 | 行 222（§8.4.4） | 行 316（§8.6.6） |
|------|-----------------|-----------------|
| 正文所在节 | Kafka Tiered Storage | 三问决策树 |
| markdown 引用的文件名 | `fig-8-7.svg` | `fig-8-8.svg` |
| SVG 文件名 | `fig-8-7.svg` | `fig-8-8.svg` |
| SVG 内部标题 | "图 8-7 存储格式设计决策树" | "图 8-8 Kafka Tiered Storage 日志段生命周期" |
| SVG 实际展示内容 | 决策树（三问 → 三种格式范式） | Tiered Storage 生命周期 |
| markdown 图注描述 | "Kafka Tiered Storage 日志段生命周期" | "存储格式设计决策树" |

**根因**：两个 SVG 文件的内容在创建时被放反了。决策树图被渲染到了 `fig-8-7.svg`，Tiered Storage 生命周期图被渲染到了 `fig-8-8.svg`。SVG 的内部标题号与文件名一致（fig-8-7 写着"图 8-7"，fig-8-8 写着"图 8-8"），但内容归属与正文所在的节完全相反。

也就是说：行 222 所在的 Tiered Storage 节应展示 Tiered Storage 图（目前在 fig-8-8.svg），但它引用了 fig-8-7.svg（决策树图）。行 316 所在的决策树节应展示决策树图（目前在 fig-8-7.svg），但它引用了 fig-8-8.svg（Tiered Storage 图）。

### 修复方案选型

#### 方案 A（推荐）：交换文件名 + 修复 SVG 内标题号

**操作**：
1. 交换两个 SVG 文件的文件名（`fig-8-7.svg` <--> `fig-8-8.svg`，通过 3 步 mv 实现）
2. 修复交换后每个 SVG 文件内部标题中的图号

**修复后状态**：

| 验证维度 | fig-8-7.svg | fig-8-8.svg |
|----------|-------------|-------------|
| 文件名 | fig-8-7.svg | fig-8-8.svg |
| 图内容 | Kafka Tiered Storage 生命周期 | 存储格式设计决策树 |
| SVG 内标题 | "图 8-7 Kafka Tiered Storage 日志段生命周期" | "图 8-8 存储格式设计决策树" |
| 行 222 正文引用 | 正确指向 fig-8-7.svg + 描述匹配 | — |
| 行 316 正文引用 | — | 正确指向 fig-8-8.svg + 描述匹配 |
| 编号顺序 | 行 222（fig-8-7）< 行 316（fig-8-8），顺序正确 ||
| markdown 改动量 | 0 行（无需修改 chapter.md） |

#### 方案 B（不推荐）：改 markdown 正文引用

**操作**：
- 行 222：引用改为 `fig-8-8.svg`，图号改为"图 8-8"
- 行 316：引用改为 `fig-8-7.svg`，图号改为"图 8-7"
- 修改 4 处 markdown（alt text + caption 各 2 处）

**修复后问题**：
- 图 8-8（Tiered Storage）出现在第 222 行，图 8-7（决策树）出现在第 316 行——**图号顺序与物理出场顺序相反**
- SVG 内部标题号与正文图号不匹配（例如 fig-8-7.svg 内部写着"图 8-7"但正文引用为"图 8-8"）

这是"打补丁"式的修复，留下了内部标题与正文图号不一致的隐患。一旦未来有人手动重排 SVG 或执行批量工具检查，又会发现问题。

#### 方案 C（最差）：只改 SVG 内标题

**操作**：仅将 fig-8-7.svg 的"图 8-7"改为"图 8-8"，fig-8-8.svg 的"图 8-8"改为"图 8-7"

**修复后问题**：
- fig-8-7.svg 展示决策树但标题写着"图 8-8"——文件名与内部标题不一致
- fig-8-8.svg 展示 Tiered Storage 但标题写着"图 8-7"——文件名与内部标题不一致
- 正文 markdown 引用仍然指向错误的文件

最差的方案，症结完全没有触及。

### 推荐：方案 A 的详细操作步骤

#### 步骤 1：交换文件

```bash
cd chapters/08-storage-format/diagrams
mv fig-8-7.svg fig-8-7-temp.svg
mv fig-8-8.svg fig-8-7.svg
mv fig-8-7-temp.svg fig-8-8.svg
```

三个 mv 而非直接 `mv fig-8-7.svg fig-8-8.svg` 再补救——后者在同一个文件系统上执行 mv 会覆写第一个文件。三步 mv 是安全的原子操作。

#### 步骤 2：修复新 fig-8-7.svg 的内部标题

文件：`chapters/08-storage-format/diagrams/fig-8-7.svg`（现在包含 Tiered Storage 内容）

```
old_string: '图 8-8 Kafka Tiered Storage 日志段生命周期'
new_string: '图 8-7 Kafka Tiered Storage 日志段生命周期'
```

此行位于 SVG 的第 13 行（`<text>` 元素的标题文本），是该 SVG 文件顶部唯一自标的那一行。

#### 步骤 3：修复新 fig-8-8.svg 的内部标题

文件：`chapters/08-storage-format/diagrams/fig-8-8.svg`（现在包含决策树内容）

```
old_string: '图 8-7  存储格式设计决策树'
new_string: '图 8-8  存储格式设计决策树'
```

此行位于 SVG 的第 15 行。注意两个汉字的空格（全角空格或 Unicode 空格，随原 SVG 保留即可）。

#### 无需修改的内容

- `chapters/08-storage-format/chapter.md`：**零修改**。行 222 和行 316 的 markdown 引用与 alt text 描述均保持不变。
- 其他章节的 SVG：无影响。
- `dist/` 目录：构建时自动读取新的文件内容，无需手动干预。

### 风险评估

**中低风险。**

主要风险点及缓解措施：

1. **Git 跟踪重命名**：mv 操作会导致 git 将旧文件标记为 deleted、新文件标记为 added。这比单纯改 chapter.md（方案 B）的 git diff 更大。但是正确的做法——根因是 SVG 文件的内容放反了，不是 markdown 写错了。

2. **SVG 内部标题正确性**：标题字符串在每个 SVG 中只出现一次（首行的 `<text>` 元素）。替换前可 grep 确认唯一性：

   ```
   grep -n '图 8-7' diagrams/fig-8-7.svg   # 确认独一无二
   grep -n '图 8-8' diagrams/fig-8-8.svg   # 确认独一无二
   ```

   替换后可再次 grep 验证新文件的内容与文件名一致。

3. **构建系统**：如果 build 脚本通过文件路径引用 SVG（如 `diagrams/fig-8-N.svg`），则文件名不变、仅内容交换，构建不会报错。

4. **后续编辑风险**：若另一个分支已对 SVG 文件做了修改，重命名会引起合并冲突。这是通用 git 问题，不在本次修复范围内。如有多分支并行，建议优先合并此修复。

**收益**：修复后达到"文件名 = 内部标题 = 正文引用 = 图内容"四维一致，无遗留隐患。这是唯一能根除问题、不留技术债的方案。

### 从源码贡献者视角看 Why

我曾在维护一个开源基础设施项目时处理过类似的资源文件互换问题。当时有两个选择：改调用方代码或改资源文件。我们选了"改资源文件重新命名"——理由和这里的方案 A 一样：

**文件名是标识符，标识符应该准确描述它所标识的内容。** `fig-8-7.svg` 这个文件名承诺：打开这个文件，你会看到"图 8-7"的内容。如果内容不对，修复文件名（而不是修复调用方让它"反向引用"）才是正直的做法。

类比到代码：如果一个名为 `calculate_tax()` 的函数实际在计算运费，你不会在调用方把每个 `calculate_tax()` 的调用改成 `calculate_shipping()`，同时在函数定义里写一条注释"这个函数名实际上是计算运费但叫 tax"。你会直接重命名函数。同样的道理适用于 SVG 文件。
