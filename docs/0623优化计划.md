### 2.0 通用原则

每章需注入的内容分四类，按重要性排序：

| 类型 | 说明 | 优先级 |
|------|------|:---:|
| **A. 个人故事/踩坑** | "我"开头的真实经历，有具体场景、有情绪 | 必须 |
| **B. 可复现数字** | 性能声明配硬件+条件+数字，不可复现就不写 | 必须 |
| **D. 报错/命令输出** | 真实 ERROR 日志、命令输出、排查过程片段 | 至少 3 章 |
| **C. 自嘲/情绪/判断** | "我偏爱X因为Y""这玩意坑死我了""这个设计真漂亮" | 建议 |

**格式约定**：计划中用 `[需作者提供]` 标记需要你亲自写的内容——这些是 AI 编不出来的硬证据。其余句式/结构修改由 AI 执行。

### 3.4 跨章过渡句去模板化

当前每章末尾："下一章，我们要看……" → 改为：
- 有的章用钩子问题（"但数据存下来就安全了吗？下一章就是关于这道防线。"）

## 4. 第三轮：句式瘦身（批量修改 🟡）

### 4.1 "不是A而是B" 砍掉 25+ 处

**原则**：只保留有结构性映射的（如"复制不是抄数据，是用坐标对齐状态机"），其余用直接陈述代替。

**批量改法**：

| 模式 | 改法 |
|------|------|
| "不是 A，而是 B" | → 只说 B。如果 A 是常见误解，先说 B 再用括号补一句"（不是 A）" |
| "这/那 不是……，而是……" | → 去掉否定前半，"这是……" |
| "不在于 A，而在于 B" | → "关键是 B。A 不是重点。" |
| "没有 X，只有 Y" | → "Y，不存在 X。" 或直接 "Y" |
| "不是简单的 A，而是复杂的 B" | → "B"（简单/复杂是作者的评价，读者自己会判断） |
| **"这不是……"（压缩变体）** | → 去掉"这不是"，直接陈述事实。例："这不是巧合"→删；"这不是 bug"→"这是设计使然"或直接陈述原因。**注意**：这是"不是A而是B"的隐蔽变体——否定前置，隐含的"而是"不写出。单独出现、密度一高就是 AI 味。 |

**保留标准（12 处该留什么——4 条规则）**：

改到剩约 13 处时，按以下 4 条标准筛选保留项。不满足任何一条的，砍：

1. **技术事实对照**：A 是常见误解/表面现象，B 是深层机制。例：✅"复制不是抄数据，是用坐标对齐状态机"（有认知增量）。❌"不是简单的缓存，而是复杂的调度体系"（只是换说法，无认知增量）。
2. **B 比 A 有信息增量**：删掉 A 后，句子信息量不变 → 砍。删掉 A 后，句子信息量减少 → 留。
3. **非风格化修饰**：如果 A 只是为了制造"锐度"而人工设立的对立面，砍。例：❌"不是技术问题，而是思维问题"（空洞）。✅"Kafka 没有用内存缓存，而是把磁盘顺序读写和操作系统的 PageCache 组合在一起"（具体机制）。
4. **同章不超过 2 处**：即使某章有 3 处满足以上三条，也只保留最重要的 2 处。

**分章目标**：

| 章 | 当前 | 目标 | 必改位置提示 |
|------|:---:|:---:|------|
| Ch1 | 6 | ≤ 2 | L133, L150, L195, L157, L120 |
| Ch2 | 4 | ≤ 2 | L15, L213, L221, L232 |
| Ch3 | 3 | ≤ 1 | L7, L233 |
| Ch4 | 3 | ≤ 1 | L259, L75, L309 |
| Ch5 | 2 | ≤ 1 | L152, L87 |
| Ch6 | 3 | ≤ 1 | L13, L215, L196 |
| Ch7 | 3 | ≤ 1 | L7, L256, L291 |
| Ch8 | 4 | ≤ 2 | L276, L303, L83, L321 |
| Ch9 | 5 | ≤ 2 | L5, L37, L42, L86, L198 |
| Ch10 | 2 | ≤ 0 | L5, L16（后记字数最少，可全砍） |

### 4.2 "X是清晰的"前置判断 砍掉一半

**当前 ~12 处 → 目标 ≤ 6 处**

**改法**：去掉"是清晰的"，直接陈述内容。如果内容本身不清晰，加"是清晰的"也没用。
- ❌ "取舍很清楚：Kafka 用顺序 IO 换随机访问"  
- ✅ "Kafka 用顺序 IO 换随机访问。"

### 4.3 "值得X的"评价性引导 砍掉一半

**当前 ~15 处 → 目标 ≤ 7 处**

- ❌ "值得提一句的是" → 直接说
- ❌ "值得注意的细节是" → "注意："
- ❌ "这里有一个值得拆解的设计" → "拆开看："

### 4.4 第 6 章和第 8 章金句密度

- Ch6：5 句 → 2 句（保留"没有免费的一致性"和"把一致性做成旋钮"）
- Ch8：4 句 → 2 句（保留"同步的本质是重放历史"和"故障恢复是同步的反向应用"）
- Ch5：4 条原则选 2 条进引用块
- Ch7：3 条金句选 2 条进引用块

## 7. 执行建议

### 7.1 最优执行顺序

1. **AI 先跑 R3（句式瘦身）**：在作者填故事之前，先批量清理一遍"不是A而是B"等句式问题。这样作者填故事时看到的是已经去除了对仗模式的文本，新写的故事不容易沾染旧句式。
2. **你再写**：按 `[需作者提供]` 清单，分章节逐个填坑。每次填 1-2 章的故事和数字。
3. **AI 跟改**：每填完一章，AI 同步执行该章的 R2（结构）和 R4（规范）调整。
4. **最后统校**：全部章节改完后，R4 终校统一过一遍。

### 7.2 矫枉过正安全边界

| 风险 | 安全边界 |
|------|---------|
| **"我"注入过度** | 每章"我"段落 ≤ 3 处；机制段（N.2-N.4）禁用"我觉得""我认为"——"我"只出现于取舍段和个人故事段。不要破坏"你"作为全书第二人称的统一性 |
| **吐槽过度** | 吐槽对象是**技术设计/参数/场景**，不是团队/同事/工具。每条吐槽配一句"替代方案"或"后来怎么解决的" |
| **结构多样性破坏体系感** | 保留 X.5 横向对比和 X.6/7 启示/小结作为固定阅读路标。只改变三家讲解顺序和 X.1 的开头方式。每章 N.6 启示段保持统一位置 |
| **自我展示 > 技术服务** | 作者故事控制在 30-80 字/个。目的是让读者觉得"这是个真实的人在跟我说话"，而非"这人在吹自己的经历" |

### 7.3 填坑技巧

如果某个故事你一时想不起来，可以用以下方式快速激活记忆：

- 打开你公司的故障复盘文档/Wiki，找近两年的 P0/P1 事故
- 翻 git commit log 里带 "fix""hotfix""revert" 的提交
- 回想你最近一次对别人解释"这个参数为什么不能调大"的场景
- 翻你过去写的技术博客/内部分享文档
- 想想你面试候选人时最爱问的问题——那个问题背后通常藏着你自己的踩坑经历

### 7.4 写作声音速记卡

在写 `[需作者提供]` 的内容时，保持以下声音特质：

- **务实 > 华丽**："这个参数调了一下午发现单位是字节不是 KB"比"经过深入探索终于找到问题所在"好一百倍
- **有代价 > 全利好**：每个判断后面跟一句"代价是……"
- **敢吐槽 > 永远客气**："MySQL 查询缓存的设计，用今天的眼光看就是一个全局锁的定时炸弹"
- **自嘲 > 说教**："我犯过这个错"比"读者应该注意"有说服力得多
- **具体 > 概括**："4C8G 的 CVM 上跑了 3 天"比"经过长期测试"有用得多

### 7.5 Git 工作流建议

```
# 从 master 创建优化分支
git checkout master
git checkout -b opt-0623

# 每改完一类内容，按模块提交
git add -p chapters/           # R1: 作者故事（交互式暂存）
git commit -m "[R1/A] Ch2 补充mysqld关闭故事和排查命令"

git add -A chapters/           # R3: AI 批量句式替换
git commit -m "[R3] 批量替换'不是A而是B'句式 38→13"

# 每轮完成后打 checkpoint tag（方便回滚）
git tag checkpoint-r0           # 开始前
git tag checkpoint-r1           # R1 完成后
git tag checkpoint-r2           # R2 完成后

# 如有问题回退
git diff checkpoint-r1..checkpoint-r2  # 查看 R2 改了什么
git reset --hard checkpoint-r1         # 回退到 R1 完成状态

# 最终合并
git checkout master
git merge opt-0623
```
