# Mnemo：面向 Agent 的统一记忆存储 — 概念设计文档

> 工作名 **Mnemo**（Mnemosyne，希腊神话记忆之神；可改）。本文档是产品的"第 0 号设计文档"，定义 *为什么做、是什么、怎么长*。不含实现细节，实现细节留给后续设计文档。

---

## 0. TL;DR

**一句话**：Mnemo 是一个**单进程、可嵌入、零运维**的存储，用一条**追加写日志**作为唯一真相之源，在其上派生出 KV/文档、受限关系、向量检索三类视图，并原生提供**幂等、可重放、可恢复**的 Agent 运行时能力。它把今天 Agent 应用被迫缝合的 MySQL + Redis + Mongo + Kafka（+ 向量库）收敛进一个文件，定位是 **"Agent 的记忆层"**，而不是又一个通用数据库。

**Slogan**：*让 Agent 可靠的记忆层——幂等、可重放、可恢复、带语义召回。*

---

## 1. 背景：Agent 应用今天被四套基础设施缝合

一个 Agent 进程里，同一时刻发生着四类截然不同的访问模式，今天要用四套系统拼：

| 访问模式 | 典型用途 | 今天用什么 |
|---------|---------|-----------|
| 关系数据 | 业务实体、配置、权限 | Postgres / MySQL |
| 快速 KV / 文档 | 工作记忆、计数器、限流、TTL、会话草稿 | Redis / Mongo |
| 日志 / 流 | LLM 与工具调用的审计轨迹、Agent 间消息、会话历史 | Kafka |
| 语义召回 | RAG、长期记忆 | 向量库（pgvector / Qdrant） |

Agent 负载的特征是**突发、长会话、海量小读写、必须可审计可重放、状态要扛崩溃重试**。它不强求单库极致吞吐，但强求**正确、好部署、模式全覆盖**。

"四套专精系统缝合"恰恰在最痛的地方失血：

- **正确性**：一个 Agent run 跨 Redis/Kafka/Postgres，没有跨系统的原子性；重试的 LLM 调用可能重复触发工具调用，没有天然的幂等边界。
- **可观测性**："Agent 为什么做出这个决定？"——答案散落在 Kafka 的日志、Redis 的状态、Postgres 的表里，难以对齐重放。
- **运维**：Agent 应用 90% 是单进程或小规模，却要养活四套有状态系统的部署、备份、监控。

这就是机会窗口：**Agent 需要的不是更强的某个库，而是一个覆盖它全部访问模式、且原生理解它运行时语义的轻量存储。**

## 2. 定位与边界

**它是什么：**

- Agent 应用的**记忆层 / 状态层**——覆盖 Agent 90% 的存储需求。
- **单进程、可嵌入**（SQLite 式的单文件 / 进程内库），Agent 框架直接 in-process 调用。
- 日志为唯一真相之源，KV / 关系 / 向量都是其上的派生物化视图。

**它不是什么（边界，也是取舍）：**

- **不是分布式数据库**：先单机，坚决不碰集群、分片、多副本共识。Agent 应用大多不需要这些；需要的是正确与零运维。撑爆了，迁移到 Postgres + Kafka。
- **不是通用 OLTP 库**：不做跨 key / 跨实体 ACID 事务，不做复杂 SQL 优化器。
- **不是更好的向量库**：向量是*准入级差异点*（必须够用），不是护城河。

> 这个定位像极了 Redis 当年把自己叫 "data structure server" 而非通用库——**用一个窄而锋利的定位切市场**，而不是"又一个数据库"。

---

## 3. 核心论点：为什么"四合一"在理论上成立

合一是真的合一，不是硬凑。统一抽象是 **log（追加写日志）**。这一点有现成理论支撑——它就是《架构之道》提炼的**规律一："先定真相之源，其余皆为派生"**：

- 审计轨迹、会话历史、Agent 间消息 = **日志本身**（Kafka 形态）。
- 某 key 的最新状态（KV / 文档）= 日志**压实（compaction）**后的物化视图（Redis / Mongo 形态）。
- 关系视图 = 在物化状态上跑受限查询（MySQL 形态）。
- 向量索引 = 对日志里文本字段的**派生物化**（向量库形态）。
- Agent 的多步状态机、可恢复 run = 从日志**重放**重建。

Kafka 的 KTable 早就证明"log + 派生视图"能建模大量东西；Datomic / XTDB / EventStoreDB 证明"不可变日志 + 索引 + 事务"能做一个库。Mnemo 要做的，是把这套搬进**一个单二进制、嵌进程内、带 Agent 原语 API** 的轻量存储。

> 一句话：合一的合法性来自"日志为源"，而日志为源正是优秀存储软件反复浮现的共性。这本书替这个产品在理论上先做了论证。

## 4. 核心抽象（数据模型）

四类一等公民对象：

| 抽象 | 含义 | 例子 |
|------|------|------|
| **Event** | 不可变的事件记录，追加写入日志，全局有序（按 stream）。 | `tool_called`、`message_appended`、`state_changed` |
| **State** | 按 key 的最新值，由事件压实派生；支持 TTL。 | `session:42` 的当前工作记忆、`counter:ratelimit` |
| **Memory** | 长期语义记忆条目，带文本 + 元数据，自动建向量与词法索引。 | 一段被记住的用户偏好、一条事实 |
| **Run** | 一次 Agent 执行的边界，带幂等键、checkpoint、步骤记录。 | "总结这封邮件并回复"的一次 run |

底层只有两样东西：**事件日志（WAL）** 和 **派生索引**。State / Memory / Run 都是从日志派生 / 由日志支撑的视图与原语。

---

## 5. 写语义：最小一致模型（关键取舍）

派生视图把**读**统一得很漂亮，但四家的**写**语义不同——Redis 给原子原语（INCR/CAS/Lua）、MySQL 给 ACID+SQL、Kafka 给有序追加。Mnemo 必须选一个**最小一致写模型**。

**选择：单 key 线性一致状态转移 + 有序事件追加。**

- **per-key 原子性**：对单个 key 的 get / set / compare-and-set / 原子函数调用，线性一致（Redis 式）。
- **有序事件追加**：向某个 stream 追加事件，stream 内全序（Kafka 式）。
- **砍掉的**：**跨 key ACID 事务**。Agent 应用极少需要跨实体事务；真需要（如对账），外接 Postgres。

> 这个取舍比"不做查询优化器"更本质：Mnemo 不要的不是 SQL 引擎，而是 *多 key 事务*。它用"per-key 线性一致 + 事件有序"覆盖约 90% 的 Agent 写场景，换来一个远更简单、可嵌入、零运维的内核。

## 6. 架构（分层）

```
┌─────────────────────────────────────────────────────────┐
│  Agent 原语 API（state / event / memory / run）          │  ← 差异化护城河
├─────────────────────────────────────────────────────────┤
│  派生视图层                                              │
│   • 按 key 最新值（KV / 文档，含 TTL）                   │
│   • 受限关系投影（过滤/投影/简单 join，非优化器）        │
│   • 混合检索索引（向量 + BM25，后台按日志自动维护）      │
├─────────────────────────────────────────────────────────┤
│  事件日志（WAL）= 唯一真相之源                           │  ← 规律一
├─────────────────────────────────────────────────────────┤
│  嵌入式存储引擎底座（LSM，如 rocksdb / pebble）          │  ← 不自己造
└─────────────────────────────────────────────────────────┘
        单进程 / 单文件 / 进程内嵌入 / 零外部依赖
```

设计纪律：

- **不自己写存储引擎**：用成熟的 LSM（rocksdb / pebble / fury 等）做底座。
- **WAL 既是事件日志，也是备份 / 复制的基础**（未来要复制时，日志天然是 substrate）。
- **派生索引异步维护**：事件进日志后，后台 indexer 更新 KV 视图与向量索引（读最终一致；per-key 状态走强一致快路径）。

## 7. Agent 原生能力（真正的护城河）

向量谁都能往 SQLite 上 bolt。Mnemo 能防身的是**把 Agent 运行时的可靠性烧进存储 API**。这是现有所有方案最薄的地方。

### 7.1 幂等 run / 工具调用 exactly-once
Agent 最大的可靠性 bug：网络抖动导致 LLM 调用或 run 重试，工具被重复触发（重复发邮件、重复扣款）。Mnemo 用**幂等键 + 事件日志 + outbox** 原生解决：

```python
run = mnemo.run.start(idempotency_key="email-reply-<email_id>")
# 在 run 内，每一步带 step_id；重复执行同一步只生效一次
result = run.step("draft", step_id="draft-v1", fn=draft_reply)
run.step("send",  step_id="send-v1",  fn=send_email)   # 重试不会再发
```

幂等是*存储*问题（去重键、已提交日志），而不是应用层 if-else 能可靠解决的。

### 7.2 时间旅行 / 重放
"Agent 为什么这么做？"——按 run 或 stream **回放事件日志**即可重建任意时刻状态。日志为源时近乎免费；在四件套拼接里几乎做不到。

### 7.3 长 run 的 checkpoint / 可恢复
长任务（多轮、人在回路）原生 checkpoint，进程崩了从最近 checkpoint 恢复——LangGraph checkpointer 的能力，但是存储原生、零额外依赖。

### 7.4 记忆分层
工作记忆 / 会话记忆 / 长期记忆作为**一等公民的生命周期概念**，带自动晋升（热→温→冷）与淘汰（TTL / 容量策略），而不是让应用自己管三套存储。

## 8. 如何统一四家（映射表）

这是产品的核心卖点，要看得见摸得着：

| 传统系统 | 它解决的访问模式 | 在 Mnemo 里的形态 |
|---------|----------------|-----------------|
| **Redis** | 快速 KV、TTL、原子计数 | 按 key 最新值物化视图 + per-key 原子操作 + TTL |
| **MongoDB** | 文档、灵活 schema | 文档作为带 schema 的 State，索引自动维护 |
| **MySQL** | 关系、ACID、SQL | State 上的受限关系投影 + per-key 线性一致（**不做**跨 key ACID） |
| **Kafka** | 日志、流、审计、订阅 | WAL 事件日志本身（append / subscribe / replay） |
| **向量库** | 语义召回 | 日志文本字段的派生向量 + BM25 索引 |

不是把四个引擎塞进一个进程，而是**用一个抽象（日志）重新表达这四种访问模式**。

## 9. API 概览（三层原语）

接口设计原则：**不只是 SQL**。Agent 框架要的是语义化原语。

```python
import mnemo
db = mnemo.open("./agent.db")          # 嵌入式，单文件

# —— state 层（Redis / Mongo 形态）——
db.state.set("session:42", working_memory, ttl="2h")
v = db.state.get("session:42")
db.state.update("counter:rl", lambda n: n + 1)          # per-key 原子

# —— event 层（Kafka 形态）——
db.event.append("agent/messages", {"role":"assistant","text":"..."})
for ev in db.event.replay("agent/messages", since=cursor): ...
db.event.subscribe("agent/messages", on_event)           # 流式订阅

# —— memory 层（向量库形态）——
db.memory.remember("用户偏好简短回复", meta={"user":"u1"})
hits = db.memory.recall("怎么回复", k=5, hybrid=True)    # 向量+BM25 混合

# —— run 层（Agent 运行时，护城河）——
with db.run.start(idempotency_key=key) as run:
    run.checkpoint({"step": "research", "ctx": ...})
    run.step("act", step_id="act-1", fn=call_tool)
```

刻意省略的东西（取舍，不是遗漏）：跨 key 事务、复杂 SQL join、分布式事务。

## 10. 向量检索：准入级差异点，不是护城河

- **必须够用**：混合检索（向量 + BM25）、元数据过滤、量化压缩、可插拔 embedding（OpenAI / 本地模型）。
- **但不是别人选 Mnemo 的理由**：那是 Qdrant / LanceDB 的主场，且 Postgres+pgvector 在持续变强。
- 选 Mnemo 的理由是 **§7 的 Agent 原生可靠性 + 统一日志 + 嵌入零运维**，向量只是"配齐了，别让我再装一个库"。

> 把向量当护城河会输；把它当准入门槛 + 把护城河放在运行时语义上，才有存在理由。

## 11. 竞争地形

| 对手 | 形态 | Mnemo 的相对位 |
|------|------|---------------|
| **Mem0 / Zep / Letta(MemGPT)** | Postgres/Redis/向量库*之上*的薄记忆层 | 我们在它们**底下**的引擎，用日志统一；它们是潜在上层客户而非直接竞品 |
| **LangGraph checkpointer** | 仅 run 状态持久化 | 我们是超集（含 memory / event / 向量），且原生嵌入 |
| **Postgres + pgvector** | 通用库 + 向量扩展 | 我们显著**更轻、更好嵌入、Agent 原生**；它们更通用、更成熟。错位竞争 |
| **SQLite + 向量扩展** | 嵌入式 + 向量 | 最接近的形态参照；我们多了事件日志 / Agent 运行时原语 / 混合检索的统一 |
| **Qdrant / LanceDB** | 专精向量 | 不同赛道（它们只管召回，不管 Agent 状态/日志/幂等）|

**最大威胁不是这些，是 Postgres+pgvector 自己持续变好。** Mnemo 必须靠"显著更易用 + Agent 原生人体工学"立身，而不是"功能更多"。

## 12. 诚实的风险与取舍

1. **范围爆炸**：把四家引擎都做就是"做四个平庸库"。必须无情聚焦 Agent 负载——够用的关系 + 好的 KV + 一等公民的运行时原语 + 够用的向量。
2. **跨 key ACID 的缺失会挡掉一部分场景**：需要强跨实体事务的应用（对账、计费）不适合 Mnemo，应直接用 Postgres。这是主动放弃，不是缺陷。
3. **向量质量门槛在涨**：重排、多模态、量化——必须持续投入保持"够用"，否则被 pgvector 反超。
4. **生态采用**："为什么不直接 Postgres？"是每个用户都会问的。答案必须是无摩擦的嵌入 + 明确的 Agent 人体工学，而不是技术指标。
5. **单机的天花板**：扛不住时就该迁移，而不是给 Mnemo 加集群。纪律性很重要——一旦开始做分布式，复杂度会吞噬"轻量"这个核心价值。

## 13. 路线图

**MVP（锋利切片，2–3 周）**——证明手感：
- 嵌入式引擎底座 + WAL 事件日志
- 按 key 最新值（KV）+ per-key 原子操作 + TTL
- `event.append / replay / subscribe`
- 基础向量召回（单 Embedding + 余弦）

**v1**——立住护城河：
- `run` 原语：幂等键、step 幂等、checkpoint、崩溃恢复
- 时间旅行 / 重放 API
- 混合检索（向量 + BM25 + 元数据过滤）

**v1.x**——补齐记忆与运维：
- 记忆分层（工作 / 会话 / 长期）+ 自动晋升淘汰
- 备份 / 恢复（基于日志）
- 多语言绑定（Python 优先，Go / Node 次之）

**Beyond（仅当需要时）**：
- 只读副本 / 日志复制（解决读扩展，不做分片）
- 多模态 embedding
- 被需要时才考虑——**默认不做分布式**。

## 14. 附录：与《架构之道》规律的对应

这个产品不是凭空设计，它落地了书中提炼的几条共性规律：

| 书中规律 | 在 Mnemo 里的体现 |
|---------|------------------|
| 规律一：先定真相之源 | WAL 事件日志 = 唯一真相之源，KV/关系/向量皆派生 |
| 规律二：用层把可变性关进笼子 | 不可变日志 vs 可变派生视图，分层隔离 |
| 规律三：把随机变顺序、一次变一批 | 追加写日志 + 后台批量建索引 |
| 规律四：可靠性用性能买，先看价签 | per-key 强一致 vs 派生视图最终一致，分级付费 |
| 规律五：复杂状态显式建模，故障恢复可预测 | Run 的 checkpoint + 事件重放 = 可预测恢复 |

> 换句话说：这本书写完，相当于替这个产品先把"为什么这么设计"的理论稿写好了。差的就是有人把它做成一个轻量、好用、嵌入式的单文件。
