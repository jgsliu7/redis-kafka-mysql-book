# 写作标准：讲清楚一个设计

> 本文档定义了《架构之道》"讲清楚一个设计"的通用写作标准。适用于全书所有需要展开讲解 Redis/MySQL/Kafka 具体设计的章节。

---

## 0. 核心原则

**一个设计点必须同时满足"不熟悉的人能学会、熟悉的人能复习"两种用途。**

- 不熟悉的读者：跟着文中的具体问题和场景，从零建立起对这个设计的理解
- 熟悉的读者：扫标题、粗体、图和单句定位，30 秒内确认"这块我掌握了"

达不到这个标准的段落，就是"不够细"。

---

## 1. 文字的硬要求：因果链三问

每介绍一个设计点，必须闭环回答三个问题。缺任何一个，即判定为"不够细"。

| 问 | 含义 | 反面案例 |
|----|------|---------|
| **为什么需要？** | 要解决什么具体问题？不解决会怎样？ | "启动的本质是状态机重建"——读者不知道为什么要重建、问题在哪 |
| **怎么解决的？** | 具体步骤/字段/参数/阈值。精确到面经级别 | SDS 只用一句"长度记在头部"，没给出 len/free/buf 三字段布局 |
| **付出了什么代价？** | 这个方案在什么场景下吃亏、有什么限制 | 必须明确说出来，不能只讲好处 |

### 细则

**1.1 "为什么需要"必须给具体场景，不用抽象定义开头。**

- ✅ "凌晨三点发版，你敲下 `kill -9`，第二天发现丢了十几条订单。为什么？因为进程退出时没有把内存里的数据落盘。"
- ❌ "启动的本质是状态机重建。状态机重建是指……"

**1.2 "怎么解决"必须精确到参数名/默认值/阈值/字段名。不满足于"用了某种机制"这种泛泛的总结。**

- ✅ "近似 LRU 从全部键里随机采样 N 个（默认 5，由 `maxmemory-samples` 控制），淘汰其中最久未用的那个。"
- ❌ "Redis 采用了近似 LRU 算法来淘汰键。"

**1.3 "代价"不能停留在"各有优劣"，必须给出具体在什么场景下吃亏、什么程度。**

- ✅ "跳表的缓存局部性不如红黑树，在磁盘场景下这是致命的；但在纯内存里，DRAM 的随机访问速度盖过了这个代价。"
- ❌ "跳表也有它的不足。"

### 自检

读完一段技术描述后问自己：如果这是面试，我能用这段话回答面试官的追问吗？如果答不了，就是不够细。

## 2. 图的硬要求：机制图优先，概念图补充

### 2.1 图的分类

| 图类型 | 画什么 | 优先级 | 适用场景 |
|--------|--------|:------:|---------|
| **机制图** | 内部结构、字段布局、流程时序、数据流向 | 🔴 必须 | 讲数据结构/算法/流程/协议时 |
| **概念图** | 分类关系、适用场景、升级路径、三者对比 | 🟡 补充 | 总结/对比/导航时 |

机制图是教学工具——理解机制必须看图。概念图是导航工具——梳理关系时看图有用。

### 2.2 配图铁律

1. **同一层抽象的三家必须对等配图。** 如果 Kafka 的 RecordBatch 有字段布局图（图 2-2），MySQL 的 B+ 树就同样应该有一张结构图。不能一家有图、另一家没有。

2. **机制图必须能独立看懂。** 测试方法：关掉正文，光看图加图注，能不能把这个机制讲清楚？讲不清楚就是图不合格。

3. **图前有导语引出，图后有正文解读。** 图不能"裸奔"——前后必须有文字衔接。导语一句话说明"这张图在讲什么"，解读一段话说明"从图里能读出什么"。

4. **图中元素必须有标注。** 箭头标注方向，参数标注数值，框内列出关键字段名，角色标注职责。

### 2.3 机制图自检清单

- [ ] 不看正文，光看图＋图注，能讲清楚这个机制吗？
- [ ] 每个箭头/连线都有明确的含义标注吗？
- [ ] 关键参数名和值在图中标注了吗？
- [ ] 同类设计的三家都有对等的图吗？
- [ ] 图前有一句导语吗？图后有一段解读吗？

## 3. 细节密度标准：三家对等覆盖

同一章同一主题下，Redis/MySQL/Kafka 的细节密度必须拉平。不允许某家讲得深、另一家只有一句话。

### 3.1 每家的最小信息量

| 检查项 | 最低标准 | 例子 |
|--------|---------|------|
| **参数/配置名** | 至少 1-2 个，带默认值 | `maxmemory-samples 5`、`innodb_old_blocks_time 1000ms` |
| **版本标注** | 核心机制标注引入版本 | "Redis 4.0 引入 LFU""MySQL 8.0.20 起双写缓冲独立为文件" |
| **数据结构字段** | 核心字段名和含义 | SDS 的 len/free/buf，RecordBatch 的 baseOffset/batchLength/CRC |
| **流程步骤数** | 拆出 ≥3 步，每步有动词 | "第一步先处理子进程→第二步刷盘→第三步刷 AOF 缓冲→……" |

### 3.2 三家对等检查

写完一家后，立刻检查另外两家在同一主题下：
- 字数是否大致相当（允许 ±30%）？
- 参数密度是否大致相当？
- 是否每家都有对应的图？
- 是否每家都回答了因果链三问？

## 4. "复习＋学习"双用途标准

### 4.1 为复习者准备

| 元素 | 作用 | 要求 |
|------|------|------|
| **节标题** | 30 秒扫一遍就知道每段在讲什么 | 用动宾结构（"用顺序写替代随机写"），不用名词（"I/O 策略"） |
| **粗体关键句** | 每段最核心的一句话加粗 | 一句话就能说清这段的结论 |
| **图+图注** | 复习时看图回忆 | 图注必须包含核心结论，不光是"示意图"三个字 |
| **单句定位** | 每节末尾一句总结 | "Redis 是为快和简单设计的内存数据结构存储"这种——读完能直接引用 |

### 4.2 为学习者准备

| 元素 | 作用 | 要求 |
|------|------|------|
| **概念热身** | 在给定义之前先给场景 | 先问"你有没有遇到过 X 问题"或讲一个故事，再给抽象定义 |
| **术语自解释** | 关键术语在段内自封闭 | 不依赖读者翻回前几章；每个术语首次出现时在括号内给全称和解释 |
| **消化段** | 每 3 段高密度内容之后跟一段低密度过渡 | 单句总结、表格、项目符号列表、或一句"好，我们来总结下这一块" |

### 4.3 串联全文的技巧：贯穿式例子

每个主题如果可能，用一个具体例子从头串到尾。

- 第 1 章用"保存数据怎么保证不丢"串起了 WAL / 逻辑日志 vs 物理日志 / 落盘粒度
- 第 3 章用"凌晨三点 kill -9 丢订单"串起了启动关闭的全线

**效果**：熟悉的人看到例子就知道"这段在讲持久化"，不熟悉的人跟着例子一路走下来不会迷路。

## 5. 概念引入的五种方式

引入一个新概念时，用以下优先级排序：

| 优先级 | 方式 | 适用场景 | 例子 |
|:------:|------|---------|------|
| **1** | 问题先行（A） | 大多数技术概念 | "保存数据怎么保证不丢？→ 可靠存储的做法是先写日志（WAL）" |
| **2** | 故事/历史引入（E） | 讲一个软件的出身或设计动机 | "2009 年，Salvatore Sanfilippo 为自己的实时 Web 分析项目找存储方案……" |
| **3** | 对比引入（D） | 区分两个易混淆的概念 | "同样一句'Redis 是单线程的'，使用者和架构师看到的东西完全不同" |
| **4** | 类比/比喻（C） | 降低抽象度 | 雷达图对比三家定位、三把钥匙（约束/演化/抽象） |
| **5** | 定义先行（B） | 仅当概念极其简单或前 4 种不适合 | "InnoDB 的页大小默认是 16KB" |

**规则**：定义先行（B）占比不得超过全章的 40%。未润色章节的典型问题是 B 类占比超过 50%——概念空降，没有热身。

### 概念引入自检

- [ ] 读者读到这个定义之前，脑子里是否已经有了一个具体的问题或画面？
- [ ] 如果有，是通过哪种方式给的（问题/故事/对比/类比）？
- [ ] 如果没有，这个概念引入需要改造

## 6. 段落节奏的呼吸模型

### 6.1 模型

```
深吸（高密度技术） → 呼出（低密度总结/过渡） → 再深吸 → 再呼出
```

每 1-3 段高密度技术内容之后，必须跟一段低密度的过渡或总结。不允许连续 4 段以上高密度且无任何间歇。

### 6.2 段落长度分布目标

| 段落类型 | 占比目标 |
|----------|:------:|
| 短段落（1-2 句） | 45-55% |
| 中段落（3-5 句） | 30-40% |
| 长段落（6+ 句） | ≤15% |

- 短段落用于：单句定位、过渡、章节内部小结、图表前后引介
- 中段落用于：常规技术讲解
- 长段落是例外，只有当一个话题确实需要连续展开时才用；超过 8 句的段落应检查是否可以拆分

### 6.3 消化段的类型

长段落之后可以插入以下任意一种"消化段"：

- 单句总结（"一句话：B+ 树用 16KB 页对齐磁盘 I/O，一套结构覆盖点查+范围+排序"）
- 表格（把文字描述的对比用表格可视化）
- 图解引语（"下图把这个流程画出来"——然后放图）
- 向读者发问（"看到这里你可能想问……"——然后回答）
- 章节间链接（"理解了这个，第 X 章会从另一个角度再看这个设计"）

### 6.4 序号结构必须有包浆

使用"第一/第二/第三"或"1. 2. 3."的序号结构时，必须：

1. **开头有引导句**：预告为什么拆成这几个部分
2. **内部密度均衡**：每个序号项的长度和细节密度大致相当
3. **结尾有收束句**：总结这几个部分的共同逻辑

反面案例：八阶段启动直接开始、直接结束，没有任何引导和收束。
正面案例：第 1 章"三把钥匙"——开头有"架构视角落到实处，有三把可以反复使用的钥匙"，结尾有"三把钥匙合在一起，就是架构视角的完整方法论"。

## 7. 逻辑连接词使用标准

### 7.1 显式连接词的密度控制

以下连接词在全书中的密度应控制在合理范围内，优先让逻辑关系由上下文和句子结构自然承载：

| 连接词类别 | 示例 | 控制目标 |
|-----------|------|---------|
| 因果标记 | 因为/所以/因此/由于 | 能用句子先后顺序表达的因果不用"所以"硬连 |
| 使役标记 | 让/使/令 | 谨慎使用；"让"字作为使役动词容易让句子读起来像翻译体 |
| 文言转折 | 则 | 全书"则"字密度控制在合理水平；优先用"但/而"替代 |
| 假设标记 | 如果/假如 | 仅当假设条件确实需要显式标出时才用 |
| 被动标记 | 被 | 能用主动语态就用主动语态 |

### 7.2 禁止的机械连接短语

以下短语全书禁用（可作为正则扫描的黑名单）：

- "值得注意的是" / "需要说明的是" / "需要指出的是"
- "从 X 能读出 N 个 Y"（"从表里能读出四组结论"类元评述骨架）
- "综上所述" / "总而言之" / "由此可见"
- 段末抽象收口标签："……形成直接对照" / "……由此清晰" / "……是 X 和 Y 的分水岭"

## 8. 第 2 章特殊标准：数据结构章

第 2 章（数据结构与协议）是全书技术密度最高的章节之一，且是读者第一次系统接触三家的内部实现。除上述通用标准外，追加以下专项要求：

### 8.1 每个核心数据结构必须有字段布局

- Redis SDS：头部字段（len/free/buf/alloc）、预分配策略（<1MB 翻倍/≥1MB 每次加 1MB）、惰性释放阈值（1MB）
- Redis 字典：ht[0]/ht[1] 双表、rehashidx 游标、搬迁方向
- Redis 跳表：多层结构、前进指针/后退指针/跨度（span）、层高随机化
- Redis ziplist/listpack：字段排列顺序、连锁更新的精确机制
- Redis intset：编码升级（int16→int32→int64），非直接转 hashtable
- MySQL B+ 树：必须有至少一张结构图，展示内部节点（只存键）、叶子节点（存完整行数据）、叶子层双向链表
- MySQL 行格式 Dynamic：字段长度列表逆序、NULL 位图、溢出页阈值与指针
- Kafka RecordBatch V2：当前已有字段布局图和正文，检查是否与最新正文描述一致

### 8.2 每个协议的交互时序

- Redis RESP：五种前缀字符 + 完整交互示例 + Pipeline vs Batch 的区别
- MySQL 二进制协议：握手→命令→响应的三阶段时序 + 预编译语句的 COM_STMT_PREPARE→COM_STMT_EXECUTE 流程
- Kafka 请求/响应协议：统一头部四元组 + api_key/api_version 版本协商机制 + Produce/Fetch 批量交互

## 9. 自检流程

任何一个子主题写完后，按以下顺序自检：

1. **[因果链]** 读一遍，能找到"问题 → 方案 → 代价"各至少一句话吗？
2. **[图]** 有图吗？是机制图吗？关掉正文能看懂吗？三家对等吗？
3. **[参数]** 给出了至少一个具体的参数名/默认值/阈值吗？
4. **[版本]** 标注了核心机制的引入版本吗？
5. **[三家对等]** 另外两家在同一主题下的细节密度和本节一致吗？
6. **[消化段]** 本节末尾有没有让读者"停下来吸收一下"的段落？
7. **[一句话定位]** 熟悉的读者扫一眼节末，能在 30 秒内确认"这段的核心是 X"吗？
8. **[逻辑连接]** 本节中有无禁止的机械连接短语？显式因果标记的密度是否合理？
