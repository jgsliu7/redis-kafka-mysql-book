# 《架构之道》各章节内容检查方案

> 本文是**内容质量**的审查工具,定位于动笔后、交稿前的逐章体检。它管"内容硬伤"(技术准确性、事实校验、图表/引用一致性),与另外两份文档分工明确:
>
> - **`book-bible.md`**:写作契约(术语、版本基线、结构模板)——"写之前怎么定"
> - **`style-guide.md`**:文笔风格(机制段/取舍段分层)——"怎么写得好看"
> - **本文(检查方案)**:内容体检——"写完后怎么查出问题"
>
> 审校 agent 对每一章执行本方案时,先跑 §1 技术准确性(硬伤,最高优先级),再逐层往下。发现的问题按 §6 的分级标记,产出结构化报告交总调度决策。

---

## 0. 检查前准备

逐章检查前,先确认三份基线文档在手:

1. **版本基线**:Redis 7.x、MySQL 8.0.x、Kafka 3.x(book-bible §2)。本章所有参数/默认值以这组基线为准。
2. **术语对照表**(book-bible §7,line 105 起)。本章用到的术语必须与表一致。
3. **全书章节地图**(第 1 章图 1-3):确认本章在全书脉络里的位置,以及它前后承接的章。

**一条总原则**:凡涉及具体参数、版本行为、性能数字,不确定的**必须标注"以实际实现为准"或"待核实"**,**严禁编造**。这一条是技术书的生命线,违反即视为硬伤。

---

## 1. 技术准确性(硬伤,最高优先级)

这是最容易出"致命错误"的维度,也是读者最无法容忍的。逐项检查:

### 1.1 机制描述是否准确

| 检查项 | 怎么查 | 典型问题 |
|--------|--------|---------|
| 机制描述与版本基线是否一致 | 对照官方文档该版本章节 | "Redis 是单线程"(7.x 已有多线程 IO);"Kafka 依赖 ZooKeeper"(3.3+ 新集群 KRaft 生产可用) |
| 崩溃恢复、复制、选举等关键流程是否正确 | 画出流程图核对每一步 | 遗漏关键步骤、步骤顺序颠倒、因果倒置 |
| 算法描述是否正确 | 如 LFU 的 Morris 计数器、B+ 树分裂、Paxos/MGR 共识 | 把近似算法说成精确、把概率机制说成确定机制 |
| 是否有"想当然"的推断 | 标记所有"应该""大概""通常" | 没有出处的主观推断,可能误导读者 |

### 1.2 参数与默认值

- 本章出现的每个**具体参数名**(如 `innodb_fast_shutdown`、`min.insync.replicas`、`appendonly`)是否拼写正确、是否存在。
- 每个**默认值**是否准确(如 `rpl_semi_sync_source_timeout` 默认 10 秒)。默认值随版本变化,需核对基线版本。
- **版本敏感的参数**是否标注版本差异?如 8.0.26 的 master/slave → source/replica 重命名。
- 越界/越版参数:是否引用了基线版本之后才引入的参数?

### 1.3 性能与容量数字

- 本章每个**量化数字**(如"25 GB""200 万 TPS""延迟亚毫秒级")是否有出处?是实测、官方基准、还是社区文章?
- 出处是否标注?如"LinkedIn 公开基准测到约 200 万 TPS"。
- **数字是否有版本/条件前提?** 如"单实例建议 25 GB"——这个值在不同 Redis 版本、不同 OS、是否开 AOF 下是变的,要标注适用条件。
- 凡无法溯源的数字,标记为"待核实"或删除。

## 2. 事实校验(引用与出处)

### 2.1 内部引用(本书自指)

- **交叉引用是否正确**?如"详见第 6 章"——第 6 章是否真的讲了那个内容(已知历史问题:第 3 章曾误指第 2 章讲数据结构,实际在第 4 章)。
- **图/表引用是否对应**?正文的"见图 N-M"与实际图号是否一致、图是否在该位置附近。

### 2.2 外部引用(官方文档/KIP/论文)

- 引用的 **KIP 编号**是否真实存在、内容是否与描述一致?(本书引用 KIP-32/480/679/833/866 等)
- 引用的**官方文档**链接/说法是否对应正确版本?(不要把 5.x 文档的说法套到 7.x)
- 引用的**论文**是否准确?(如 ARIES、State Machine 复制、Morris 计数器)
- 引用的**企业实测**(IBM/Cloudflare/Confluent)是否标注了"社区实测"而非"官方结论"?

### 2.3 版本演化表述

- "X 版本起引入 / 移除 / 改为默认"这类表述,版本号是否准确?
- 是否有**过时表述**?如仍在用旧术语描述新行为。

## 3. 图表检查

### 3.1 图(SVG)

| 检查项 | 说明 |
|--------|------|
| **图号连续性** | 本章图号是否从 N-1 起连续、无跳号?(已知历史:曾有 7-9 跳过 7-7/7-8) |
| **文件名与图号一致** | `fig-N-M.svg` 的文件名编号 = 图内文字图号 = 正文引用图号,三者一致 |
| **内部图号文字** | SVG 内是否印了"图 N-M 标题"?第 8 章曾缺失,已补 |
| **正文咬合** | 每张图是否有导语(图前一句)+ 图注 + 解读段(图后一段)?有无"裸奔"的图? |
| **内容准确性** | 图里画的流程/结构是否与正文描述一致?有无箭头方向错、节点遗漏 |
| **viewBox/尺寸** | viewBox 格式正确、无坐标越界(可跑本书的 SVG 扫描脚本验证) |
| **marker id 冲突** | (已由构建脚本解决,但若手动嵌入需复查) |

### 3.2 表(横向对比表)

| 检查项 | 说明 |
|--------|------|
| **表号连续性** | 本章表号是否从 N-1 起连续 |
| **命名体例** | 表格是否命名为"表 N-M"而非误用"图 N-M"(历史问题已统一) |
| **三栏顺序** | 横向对比表的列序是否锁死为 Redis | MySQL | Kafka |
| **正文咬合** | 表前有导语(比什么)、表后有解读(只讲表外根因,长度 30-50% 表格信息量,禁止复述表格) |
| **内容准确性** | 表格每一格的描述是否与该章机制描述一致?有无"表格简化导致的失真" |

### 3.3 一致性交叉

- 同一事实在**正文、图、表**三处出现时,表述是否一致?(如 ISR 的定义,正文、图注、对比表里应是同一套说法)
- 术语在图/表里是否与术语表一致?

## 4. 横向对比一致性

本书核心特色是"三家同题作答",这一节专门检查对比的一致性。

### 4.1 跨章对比一致性

- 同一个机制/概念在**不同章**的描述是否一致?如 Redis 的 fork 阻塞,第 2 章(生命周期)、第 3 章(内存磁盘)、第 6 章(集群)都会提到,说法必须一致。
- 同一个**取舍判断**(如"Redis 走 AP")在全书是否稳定?不能第 6 章说 AP、第 8 章又说"强一致"。

### 4.2 对比维度的完整性

- 每个横向对比表是否覆盖了**三家**?有无只对比两家、漏掉一家的情况?
- 三家的**笔墨是否均衡**?有无某家明显着墨过少?(N.2-N.4 三家机制段篇幅应大致相当)

### 4.3 "共性"与"差异"的区分

- 本章提炼的"共性规律"是否真的三家共有?有无把某一家的特性误当成共性?
- 本章指出的"差异"是否有据可循?差异的根因是否讲清?

## 5. 结构与节奏(衔接 style-guide §1.5)

### 5.1 章节结构完整性

对照 book-bible §4 的结构模板,本章是否齐全:

- [ ] N.0 本章导读(150-250 字,立悬念)
- [ ] N.1 问题的本质(≤800 字,最尖锐)
- [ ] N.2 Redis 做法
- [ ] N.3 MySQL 做法(对照 Redis 讲)
- [ ] N.4 Kafka 做法(对照前两家讲)
- [ ] N.5 横向对比
- [ ] N.6 架构启示
- [ ] N.7 小结(呼应 N.0)

### 5.2 节奏与对照

- N.2→N.3→N.4 是否**层层对照**(讲 MySQL 时回扣 Redis 取舍),还是三家平行罗列各讲各的?
- N.5 横向对比的解读段是否只讲"表外根因",不复述表格?
- N.7 小结是否呼应 N.0 导读提出的问题(形成闭环)?

### 5.3 篇幅

- 本章总篇幅是否在 book-bible 规定的区间?
- 三家机制段篇幅是否大致均衡?有无某家过短或过长?

## 6. 问题分级与报告格式

审校 agent 发现的每个问题,按下面三级分类,并产出结构化报告。

### 分级

| 级别 | 定义 | 处理方式 | 示例 |
|------|------|---------|------|
| **P0 硬伤** | 技术错误、事实编造、参数错误、交叉引用指向错误 | **必须修**,总调度优先处理 | "Redis 是单线程";"详见第 2 章"实际在第 4 章 |
| **P1 缺陷** | 图表编号断裂、表格裸奔无解读、对比漏掉一家、机制段缺失关键步骤 | **应修**,影响可读性/完整性 | 图号跳号;横向对比表无解读段 |
| **P2 优化** | 表述可更准确、节奏可改善、比喻可收敛 | **建议改**,不强制 | "极致延迟"可改"低延迟";某段可拆长句 |

### 报告格式

审校完一章,产出如下结构的报告(交总调度):

```markdown
# 第 N 章 内容检查报告

检查人:[agent 身份]  检查日期:[日期]
基线版本:Redis 7.x / MySQL 8.0.x / Kafka 3.x

## P0 硬伤(共 X 处)
1. [位置 line X] [问题描述] [应为] [依据出处]
2. ...

## P1 缺陷(共 X 处)
1. [位置 line X] [问题描述] [建议修法]
2. ...

## P2 优化(共 X 处)
1. [位置 line X] [问题描述] [建议]
2. ...

## 汇总
- P0: X 处(必须修)
- P1: X 处(应修)
- P2: X 处(建议)
- 本章技术准确性: [通过/有硬伤]
- 本章图表完整性: [通过/有缺陷]
```

## 7. 检查顺序建议

为提高效率,按以下顺序逐章过(硬伤优先,优化最后):

1. **先跑技术准确性(§1)**——这是最致命的,一处硬伤可能让读者对全书失去信任。重点核对参数、默认值、版本行为。
2. **再跑事实校验(§2)**——交叉引用是否对、KIP/论文是否准确。
3. **图表检查(§3)**——图号/表号连续性、咬合、内容准确性。可用脚本辅助(本书已有 SVG 扫描脚本)。
4. **横向对比一致性(§4)**——跨章对照同一机制的说法。
5. **结构节奏(§5)**——对照模板查完整性。
6. **最后汇总报告(§6)**——分级产出。

**一个提效技巧**:§1.2 参数校验、§3 图表编号这两项可以用脚本批量初筛(全书扫描一遍),人工只复核脚本标红的疑点。其余维度目前需人工逐章读。
