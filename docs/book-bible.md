# 《架构之道：从 Redis、MySQL、Kafka 看软件设计的共性》写作圣经

> 本文件是全书的**唯一契约**。所有写作 / 审校 agent 在动笔前**必须完整阅读本文件**，并严格遵守其中的文风、术语、结构、篇幅、引用规范。本文件由总调度（主 agent）维护；如需修改，由总调度统一修订并通知所有角色。

---

## 1. 全书定位

- **一句话定位**：透过 Redis、MySQL、Kafka 三款软件的内部实现，提炼**可复用的软件架构设计共性**，帮读者从"会用"升级到"懂取舍"。
- **目标读者**：2 年以上经验的后端开发者；需要做技术选型的架构师；希望建立团队技术体系的技术负责人。
- **前置要求**：基本数据结构与算法、操作系统（进程 / 线程 / 内存 / IO）常识；至少用过三者之一。
- **不做的事**：不是使用手册，不是调优手册；不逐行贴源码；不重复市面上"怎么用"的内容。我们的差异化是"为什么这么设计"。

## 2. 版本基线

凡涉及具体行为 / 参数 / 默认值，一律以此基线为准；标注版本差异时显式说明。

| 软件 | 版本 | 关键特性锚点 |
|------|------|------|
| **Redis** | 7.x | 多线程 IO、ACL、Function、Sharded Pub/Sub、Cluster 改进 |
| **MySQL** | 8.0.x | 原子 DDL、新版 redo log、降序索引、直方图、组复制（MGR） |
| **Kafka** | 3.x | KRaft（去 ZooKeeper）、Tiered Storage、生产者幂等与事务 |

## 3. 文风规范（重要）

- **纯现代技术散文**。不使用古风 / 文言表述。README 中原有的"吾观……初心不忘"等将在终校阶段改写为现代散文，使全书文风统一。
- 句子有主语、有断句；避免长定语堆叠。能用短句就不用长句。
- 第二人称统一用"你"。
- **术语首次出现时以'中文（英文）'或'英文（中文）'标注**。后续使用遵循两档制：
  - **英文优先**（工程界通用缩写和日常口语化的英文术语）：ISR、GTID、MVCC、WAL、binlog、redo log、undo log、offset、compaction、fsync 等——首次标注全称后**以英文为主**，不强制转中文。
  - **中文优先**（中文已充分沉淀）：缓冲池、分区、快照、副本、主节点、从节点、多数派、刷盘、日志段——**继续用中文**。
- **解释优于堆术语**：每个概念先用一句大白话说清"它在解决什么问题"，再展开机制。
- 允许适度比喻，但每个比喻必须服务于理解，且不超过两句；比喻之后要落回技术事实。
- 禁止：空话套话、营销腔、过度感叹号、emoji、"众所周知""不言而喻"。

## 4. 章节统一结构模板

每章（第 1、10 章除外）遵循以下结构：

```

## N.1 问题的本质
（这一主题在软件设计中到底要解决什么；三家面临的共同问题与各自约束）

## N.2 Redis 的做法
（机制深讲：核心数据结构 / 算法 / 流程 → 关键取舍与代价）

## N.3 MySQL 的做法
（同上结构）

## N.4 Kafka 的做法
（同上结构）

## N.5 横向对比
（三栏对比表 + 解读：各家选了什么、代价是什么、为什么不同）

## N.6 架构启示
（本章精华：从具体实现提炼出的可复用设计原则与权衡规律。至少 1–2 句可被引用的箴言）

## N.7 小结
（3–5 句收束，呼应导读提出的问题）
```

**特例**：
- **第 1 章 引言**：不套三家逐节深讲。用「1.1 为什么是这三款软件 / 1.2 三家速写 / 1.3 架构视角 vs 使用视角 / 1.4 如何读本书」。结尾点出全书脉络。
- **第 10 章 总结**：是全书升华。用「10.1 九章一镜照 / 10.2 共性规律 / 10.3 取舍之道 / 10.4 给实践者的建议」。需呼应前 9 章每一章的核心启示。

## 6. 图示规范（铁律：必须真的画出来）

**每一张图都必须是真实可渲染的成品，绝不允许占位符、"图略"、TODO、或纯文字描述冒充图。** 这是硬性质量门槛，审校阶段逐张校验。

- 图一律产出为**自包含的 SVG 文件**，存于 `chapters/XX-slug/diagrams/fig-N-M.svg`：
  - 流程 / 时序 / 关系 / 分层架构 / 内存布局 / 文件格式 / 数据结构 —— **全部手写 SVG 绘制**（矩形、圆角框、箭头、连线、文字标签、克制配色）。
  - SVG 必须是**合法 XML、自包含**（不引用任何外部图片 / 字体 / CSS），可在浏览器、GitHub、GitBook、Typora 直接渲染。
  - 用 `viewBox` 控制尺寸；配色克制（1 主色 + 1–2 强调色）；字体用系统无衬线（`font-family="system-ui, sans-serif"`）。
  - 图中文字用**中文**，术语与正文 / 术语表一致。
- 仅当某图是极简单的流程 / 时序、且**比 SVG 更清晰**时，可用**内联 Mermaid 代码块**替代；其余一律 SVG。
- 每图编号 `图 N-M`（第 N 章第 M 张），在正文用 `![图 N-M 标题](diagrams/fig-N-M.svg)` 嵌入，下方配图注一行。
- 图前一句导语引出，图后一段正文解读；**一张图只讲一件事**。
- **插图师（Illustrator）角色**按大纲的图示清单绘制全部 SVG；**审校阶段**逐张校验：SVG 合法可渲染、与图注及正文描述一致、无错别字。

## 9. 质量标准（publishable bar）

1. **准确性**：机制描述与版本基线一致；不确定处标注"以实际实现为准"，**不得编造**。
2. **深度**：不止讲"是什么"，必须讲"为什么这么设计 / 换我会怎么取舍"。
3. **对比性**：每章必须有实质的三家横向对比，而非平铺三家文档。
4. **可读性**：导语先立问题再给方案；段落有论点句；避免大段未分层的文字墙。
5. **连贯性**：术语、译名、交叉引用全书一致。
6. **金句**：每章 N.6 架构启示至少 1–2 句可被独立引用的设计箴言。
