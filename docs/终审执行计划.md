### 阶段 1：只读审查（分 3 批并行，无 Write 工具，约 3-4h）

> **分批原因**：18 个 agent 不能一次全启动（API 限流 + 调度开销）。分 3 批，每批 4-7 个。
>
> **关键依赖**：T04 在 D01 完成后启动（D01 的图号发现 → T04 核验引用）；F01 发现流入 F02 prompt。
>
> **基线流入**：B04 断言清单 → 所有 agent 已知项不重报；B05 黑名单命中 → A01/T01 重点查上下文；B06 引用索引 → T04/D02 据此核验。

#### Agent 保护机制（所有 agent 通用）

遇到以下情况，标记 `[MAYBE_INTENTIONAL]` 而非直接报为"问题"：
- 作者第一人称叙述（"我踩过这个坑""我不推荐"）
- 工程圈口语化表达（"扛不住""就是这德行""这个优化收益不够大"）
- 个人判断锋芒（"这个设计很烂""看似优化实则挖坑"）
- "我们"作为读者邀请修辞（"我们看到 Redis 的做法"）
- 结构性锚点比喻（"真相之源""骨架""状态机"）

标记 `[MAYBE_INTENTIONAL]` 的条目在阶段 2 由人审最终判定，不给 fix agent 自动修复。

#### 严重度标注规范（所有 agent 通用）

每条 finding 必须标注严重度：
- 🔴 致命（事实错误/自相矛盾/命令不可运行/合规底线缺失）
- 🟠 高（术语不一致/数字无出处/导读承诺未兑现/图语义错误）
- 🟡 中（格式不一致/措辞可优化/标题可改进）
- ⚪ 低（锦上添花/个人偏好）

阶段 2 按严重度自动过滤：⚪ 直接跳过，不进入修复队列。

## 7. 使用说明

### 开工三步

1. 打开本文件，读 §0「快速恢复」——知道做到哪了
2. 在 §0「Agent 完成清单」中找到第一个 `[ ]`，或在任务总表中找到第一个 `[ ]`，执行
3. 执行完后立即回来更新状态：改 `[ ]` → `[x]` 或 `[!]`，并在 §0 勾选对应 agent（如适用）

### 关机前必做

1. 更新 §0 表格：当前阶段、上次会话做了什么、下一步、最新 git commit
2. 确认所有已完成 agent 在 §0 Agent 清单中已勾选
3. 确认发现已写入 `findings_master.md`，修改已写入 `fix-log/`
4. `git add docs/终审执行计划.md qa_reports/20260702/` + commit

### 中断恢复

- **阶段 0 中断**：脚本任务独立，重新按 `[ ]` 状态判断哪些没跑
- **阶段 1 中断**：查看 `findings/` 目录——文件存在 = agent 已完成；对照 §0 Agent 清单找出未完成的，重新派
- **阶段 2 中断**：`findings_master.md` 中的发现如果已标注"已处理至 #N"，从 N+1 继续
- **阶段 3 中断**：`git log` 查看 pre-fix 之后的 commit——最新的 commit 就是修到的位置。阶段 3a/3b 各自独立 commit，可精确恢复

### 状态更新规范

- 任务开始 → 改 `[ ]` 为 `[~]`
- 任务完成无发现 → 改 `[~]` 为 `[x]`，填结果摘要
- 任务完成有发现 → 改 `[~]` 为 `[!]`，在 `findings_master.md` 登记
- 任务跳过 → 改 `[ ]` 为 `[-]`，注明原因

### 发现登记规范

每条发现最少包含：来源 agent、文件:行号、类别、严重度（🔴🟠🟡⚪）、是否 `[MAYBE_INTENTIONAL]`、一句话描述、建议改法。

### 修改登记规范

每次 Edit 后登记：文件、大致行号、改前原文（截关键部分）、改后文字、原因、关联发现编号。阶段 3a（字词修复）和 3b（事实修复）的修改区分标注。
