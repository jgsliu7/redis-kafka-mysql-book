### Q1: CPU L1 缓存访问延迟 ~1 ns
**原文陈述**: CPU L1 缓存典型访问延迟约 1 ns
**核查结果**: ✅ 确认正确
**核查依据**: Intel/AMD 优化手册和 uops.info 实测数据：Skylake L1 延迟为 4-5 个周期（@3-4 GHz 约 1.0-1.5 ns），Zen 系列 L1 延迟为 4 个周期（约 1.0-1.3 ns）。"约 1 ns"作为数量级表述精确合理。
**建议**: 无需修正

### Q2: 内存（DRAM）访问延迟 ~100 ns
**原文陈述**: 内存（DRAM）典型访问延迟约 100 ns
**核查结果**: ✅ 确认正确
**核查依据**: DDR4-3200 系统级延迟约 62-73 ns，DDR5-4800 约 82.5 ns，DDR5-6000 约 70 ns。上述为 AIDA64 实测系统级延迟（含内存控制器、总线等开销）。"~100 ns"作为教学用数量级估算完全合理（实际主流配置在 60-85 ns 范围，100 ns 覆盖含 TLB miss 等更真实场景）。
**建议**: 无需修正。如果希望更精确，可标注"约 60-100 ns"，但当前数据在教学上足够准确。

### Q3: SSD 随机读延迟 ~100 μs
**原文陈述**: SSD（随机读）典型访问延迟约 100 μs
**核查结果**: ✅ 确认正确
**核查依据**: 企业级 NVMe SSD 规格典型延迟约 50-75 μs（低负载）; 消费级 NVMe SSD（如 Samsung 990 Pro）实测 4K QD1 延迟约 45 μs; 高队列深度下延迟升至 180-430 μs。取全部场景的中位数，"~100 μs"是合理的数量级表述，尤其考虑到 SATA SSD 和混合负载。
**建议**: 无需修正

### Q4: SSD 顺序写吞吐 ~300-500 MB/s
**原文陈述**: SSD（顺序写）吞吐约 300-500 MB/s
**核查结果**: ❌ 需要修正
**核查依据**: 现代 NVMe SSD 顺序写吞吐远高于此：
- Samsung 990 Pro (PCIe 4.0) 顺序写可达 6,900 MB/s（官方规格），实测 5,400-6,900 MB/s
- PCIe 5.0 SSD 可达 10,000+ MB/s
- SATA SSD 顺序写上限约 500-560 MB/s（SATA III 6 Gbps 理论上限 ~600 MB/s）
300-500 MB/s 仅对应 SATA SSD 或早期 NVMe SSD，已严重过时。
**建议**: 更新为"NVMe SSD 顺序写可达 2,000-7,000 MB/s（取决于代际和型号），SATA SSD 约 300-500 MB/s"。或简化表述为"现代 NVMe SSD 顺序写可达 GB/s 级"。

### Q7: 体感换算 "16 分钟" 与 "28 小时"
**原文陈述**: "如果内存访问是 1 秒，SSD 一次随机访问就是 17 分钟，机械盘一次随机寻道就是 28 小时"
**核查结果**: ✅ 确认正确
**核查依据**: 
- 100 ns 内存 → 100 μs SSD: 1000 倍。1 秒 × 1000 = 1000 秒 ≈ 16.7 分钟（书说 17 分钟，正确）。
- 100 ns 内存 → 10 ms 机械盘: 100000 倍。1 秒 × 100000 = 100000 秒 ≈ 27.8 小时（书说 28 小时，正确）。
**建议**: 无需修正

### Q8: SSD 顺序读比随机读快 100 倍

### Q11: 近似 LRU 从全部键随机采样 N 个
**原文陈述**: "淘汰时从全部键里随机采样 N 个（由 maxmemory-samples 控制，默认 5），从中淘汰最久未用的那一个"
**核查结果**: ⚠️ 需要澄清
**核查依据**: Redis 源码 evict.c 显示近似 LRU 实现比描述更复杂：
1. 首次采样 N 个键填充 eviction pool（EVPOOL_SIZE=16 的候选池）
2. 后续每次淘汰从池中选最佳（最久未用）的键
3. 若池中键已被删除，重新执行采样填充
4. 池始终按空闲时间升序排列，淘汰时取末尾（最大空闲时间）
书中描述过于简化——实际淘汰的是池中的最佳候选，而非单次采样的最佳。
**建议**: 建议补充："Redis 内部维护一个候选池（eviction pool），每次从全部键采样 N 个插入池中，最后从池里选最久未用的淘汰。这比单次采样更接近真实 LRU。"

### Q13: Morris 近似计数器：8 位对数计数器
**原文陈述**: "它用的是 Morris 近似计数器：一个 8 位的对数计数器"
**核查结果**: ⚠️ 需要澄清
**核查依据**: Redis 源码 evict.c 中 LFU 的 24 位 lru 字段拆分为：高 16 位（上次衰减时间戳，分钟精度）+ 低 8 位（LOG_C 对数计数器，范围 0-255）。8 位的描述准确。但严格来说，Redis 的 LFU 计数器使用线性概率 `p = 1/(baseval * lfu_log_factor + 1)` 而非经典 Morris 计数器的指数概率 `p = 1/(1+a)^X`。Redis 作者 antirez 本人在 GitHub Issue #7943 中承认了这一点。称其为"Morris 计数器"是技术上不精确的简化。
**建议**: 建议改为"近似对数计数器（类似 Morris 计数器思想，使用 8 位存储）"。

### Q19: AOF 重写把多次 INCR 合并为 SET

### Q28: 操作系统和磁盘扇区通常是 4 KB 或更小

**建议**: 无需修正

### Q30: AHI 把 3-4 次页访问压成 1 次

**建议**: 无需修正

### Q31: 缓冲池三分链表（Free / LRU / Flush）

**核查依据**: 此现象称为"缓冲池污染"（Buffer Pool Pollution）。大表扫描（全表扫描/报表查询）将大量冷数据页载入缓冲池的 Old Sublist，若未能及时提升（未在 innodb_old_blocks_time 内再次访问），会挤压热数据的生存空间，导致热数据被淘汰。此后需要重新从磁盘加载热数据，延迟显著上升。MySQL 的 midpoint insertion + innodb_old_blocks_time 正是为解决此问题而设计。

### Q60: MySQL 双写缓冲的描述准确性
**原文陈述**: "如果写一个 16 KB 页写到一半断电，这个页就损坏了，而且损坏的不是日志、是数据本身——重做日志也救不回来，因为它记录的是'怎么改这个页'，前提是这个页本来是好的"
**核查结果**: ✅ 确认正确
**核查依据**: MySQL 官方文档 Doublewrite Buffer 说明：部分页写入（torn page）问题指 16 KB InnoDB 页写入时发生崩溃，导致页部分内容为新数据、部分为旧数据。Double Write Buffer 通过先在双写缓冲区中写入完整页副本，再从双写区写入实际数据文件来解决此问题。重做日志（redo log）无法修复损坏的页，因为 redo log 记录的是对页的逻辑更改，依赖原页的完整性。描述的推理完全正确。
**建议**: 无需修正

## 修正优先级

### 高优先级（必须修正）
1. **Q4 / Q33: SSD 顺序写吞吐 300-500 MB/s** — 严重过时。现代 NVMe SSD 达 2,000-7,000 MB/s，300-500 MB/s 仅对应 SATA SSD。两处都需更新。
2. **Q35: 内存随机读吞吐 ~100 MB/s** — 缺少前提条件，可能严重误导读者对内存性能的理解。需注明"单线程随机指针追踪场景"的条件限定。

### 中优先级（建议修正）
1. **Q6 / Q34: 机械盘顺序写吞吐 100-200 MB/s** — 现代企业级 HDD 达 250-285 MB/s，范围偏窄。
2. **Q8: SSD 顺序读比随机读快 100 倍** — 教学有冲击力但未指明维度（吞吐 vs 延迟，QD1 vs QD32），可能被引申理解为始终 100 倍。
3. **Q11: 近似 LRU 采样描述** — 省略了 eviction pool 机制，描述过于简化。
4. **Q13: Morris 计数器命名** — 与经典 Morris 算法实现有出入，技术上说"Morris 计数器"不精确。
5. **Q39: 零拷贝 PageCache 依赖** — 需要说明 sendfile 也支持冷数据（先缺页读入）而非"只在 PageCache 命中时生效"。
6. **Q45: Kafka 段文件类型** — 遗漏 .txnindex 文件。
7. **Q49: Kafka 崩溃恢复"几乎无"** — 低估了索引加载和事务恢复的代价。
8. **Q51: Kafka 消费者落磁盘延迟** — 未区分 HDD 和 SSD 场景。
9. **Q59: Redis 7.x 默认持久化配置** — 默认是 RDB only 而非混合持久化，需澄清。

### 低优先级（可选优化）
1. **Q27: InnoDB 页大小** — 可提及 16KB 为默认值且可配置。
2. **Q35: 内存随机读吞吐** — 补充多线程场景的对比数据。
3. **Q52: 缓冲池命中率延迟影响** — 补充数学推导过程。
