## 一、❌ 必须修正的事实错误（12 项去重）

以下错误如不修正，会误导读者或损害本书技术公信力。

### Ch02 生命周期

#### Q10: SHUTDOWN SAVE 触发的是 SAVE 不是 BGSAVE
- **原文**："`SHUTDOWN SAVE` 会触发一次 `BGSAVE` 落盘（fork 子进程做 RDB）"
- **事实**：触发的是阻塞式 `SAVE`（`rdbSave()`），主进程直接写 RDB，不创建子进程
- **修正**：将 `BGSAVE` 改为阻塞式 `SAVE`

#### Q17: ibdata 文件存储数据字典（MySQL 8.0 已变更）
- **原文**："加载系统表空间（ibdata 文件），拿到数据字典"
- **事实**：MySQL 8.0 起数据字典已迁移到独立 `mysql.ibd`，`ibdata1` 只存 change buffer 和可选的 undo log
- **修正**：区分版本，说明 8.0 中数据字典在 `mysql.ibd`

#### Q27: 回滚是"redo 写入的逆过程"
- **原文**："回滚是 redo 写入的逆过程，按 undo 记录逐条撤"
- **事实**：回滚用 undo log（逻辑日志），redo 做物理重放，两者是不同的日志系统。ARIES 算法中 redo + undo 协同，undo 不是 redo 的"逆过程"
- **修正**："回滚使用 undo log 通过逻辑逆操作逐条撤销"

#### Q43: Kafka Broker 普通关闭不告知 Controller
- **原文**："它不主动告知 Controller 自己要走"
- **事实**：`controlled.shutdown.enable` 默认 `true`，Broker 收到 SIGTERM 会主动发 `ControlledShutdownRequest` 转移领导权
- **修正**：改为"强制关闭（非受控关闭）时不告知 Controller"或说明默认行为

### Ch03 内存与磁盘

#### Q4 / Q33: SSD 顺序写吞吐 ~300-500 MB/s（严重过时）
- **原文**：两处（表 3-1 和 3.4 节正文）都写 SSD 顺序写 300-500 MB/s
- **事实**：这是 SATA SSD 的上限。现代 NVMe SSD（PCIe 4.0）可达 6,900 MB/s，PCIe 5.0 可达 10,000+ MB/s
- **修正**：区分 NVMe 与 SATA，或改用 GB/s 级表述

### Ch05 安全

#### Q21: redo/undo log 加密版本号错误
- **原文**："8.0.15 起重做日志（redo log）、回滚日志（undo log）也支持加密"
- **事实**：MySQL 8.0.16 起 undo log 支持加密，8.0.17 起 redo log 支持加密
- **修正**：改为"8.0.16 起 undo log 支持加密，8.0.17 起 redo log 支持加密"

### Ch06 集群架构

#### Q19: Redis Cluster 归类为 AP 系统（争议，作者主张 CP）
- **原文**："Redis Cluster 是 AP 系统" / "Redis 走 AP 路线"
- **事实**：antirez 博客明确说分区时少数派**停止写**（牺牲 A 限制数据分歧），归类为 CP 更准确。但社区对此有争议，因为主从异步复制仍可能丢写
- **修正**：建议说明"分区时少数派停止写（CP 行为），但异步复制允许丢写——是介于 CP/AP 之间的混合体"或至少标注争议

#### Q45: KIP-866 迁移工具版本演进错误
- **原文**："迁移工具（KIP-866）在 3.4 起以 Early Access 引入、3.5–3.6 为 preview、3.8 起进入生产可用"
- **事实**：3.6 起已 GA（生产可用），非 3.8
- **修正**：将"3.8"改为"3.6"

#### Q48: 元数据内部 topic 名称错误
- **原文**："元数据本身也变成了 Kafka 的一个内部 topic（`@metadata`）"
- **事实**：KRaft 元数据 topic 名称是 `__cluster_metadata`，非 `@metadata`
- **修正**：将 `@metadata` 改为 `__cluster_metadata`

### Ch07 存储格式

#### Q21: innodb_page_size 32K"仅用于压缩表"
- **原文**："32K 仅用于压缩表"
- **事实**：32K 和 64K 都是通用页大小，可用于任何表。压缩（COMPRESSED）在页 > 16KB 时反而不被支持
- **修正**：删除"仅用于压缩表"的限定

### Ch08 数据同步

#### Q9: TIME 命令的"特殊改写"
- **原文**："把 `TIME` 在主节点取到的时间作为参数下发"
- **事实**：`TIME` 是只读命令，不出现在复制流中。Redis 3.2+ 的脚本效果复制（script effects replication）只复制写效果
- **修正**：删除 TIME 示例，改用脚本效果复制或 PEXPIREAT 改写说明

### Ch09 总结

#### Q7: `sync_binlog=1` 在 MySQL 5.5/5.6 中不是默认值
- **原文**："MySQL 默认押在持久性端，'双 1'配置"
- **事实**：`sync_binlog` 在 MySQL 5.5/5.6 的默认值是 0，5.7 才改为 1
- **修正**：改为"MySQL 5.7+ 默认'双 1'"，并加注 5.5/5.6 的差异

## 二、⚠️ 需要澄清的高频模式（133 项）

这部分并非错误，但在教学精确性与技术准确度之间存在可优化的空间。以下是高频出现的模式：

### 1. 版本边界不够精确（约 25 处）
- 未区分 MySQL 5.7 vs 8.0 的默认值/行为差异
- Redis 命令的废弃/引入版本未标注（如 `SLAVEOF→REPLICAOF` 在 5.0）
- Kafka 参数默认值变更（如 `acks=all` 在 3.0 成为默认）在跨版本语境下需要补充

### 2. 简化导致歧义（约 40 处）
- Sentinel 的"Gossip"实际是 Pub/Sub `__sentinel__:hello` 频道
- "Raft-like"是 Sentinel 官方用语但精确度有限
- InnoDB 缓冲池"3/8 vs 5/8"的分割比例在 8.0 中可通过 `innodb_old_blocks_pct` 调整
- jemalloc 是默认但非唯一内存分配器

### 3. 性能数据过时或偏保守（约 15 处）
- 除 SSD 吞吐外，HDD 吞吐 100-200 MB/s 偏保守（现代企业级可达 280+）
- Kafka 单分区吞吐"约 10MB/s"受硬件代际影响大
- 网络延迟数据未区分同机房/跨机房

### 4. 缺少边界条件说明（约 30 处）
- sendfile/零拷贝仅在 PageCache 命中时生效
- 部分重同步需要 master_replid 匹配（不仅仅是 offset）
- LRU 采样实际有候选池（eviction pool），非纯随机

### 5. 术语精确度（约 23 处）
- "CRC16"应说明是 CRC16-CCITT 变种
- "Paxos 变种"的精确性（XCom 是 Mencius 风格的多 Paxos）
- murmur2 是 Kafka 默认但非唯一分区器

## 四、总体评价

### 优点
1. **核心架构论述准确**：Redis"全内存"、MySQL"磁盘为主"、Kafka"顺序日志"三大范式的定性准确
2. **版本意识较好**：多数章节已区分 Redis 4.0/7.0、MySQL 5.7/8.0、Kafka 2.x/3.x 的行为差异
3. **取舍逻辑严谨**：一致性 vs 性能、容量 vs 延迟的 tradeoff 分析逻辑自洽
4. **没有系统性编造**：未发现为教学目的刻意编造技术事实的情况

### 需重点关注
1. **12 项明确的 ❌ 错误**需要修正（见第一部分）
2. **SSD 吞吐数据**是全书最严重的过时数据，在 Ch03 出现了两次
3. **RabbiMQ → Redis Cluster 的 CAP 归类**（AP vs CP）需要补充说明，这是技术圈内的真实争议
4. **版本依赖的默认值**（MySQL `sync_binlog`、Kafka `acks`）在说"默认"时应标注适用版本范围

### 建议
- 优先修正 12 项 ❌ 错误（工作量约 2-3 小时）
- 对 133 项 ⚠️ 按优先级分批处理：先处理性能数据过时和版本边界缺失，再处理术语精确度
- 在"序言"或"参考"中加一段关于版本时效性的声明："本书性能数据和默认值以各软件主流稳定版本为准（Redis 7.x、MySQL 8.0、Kafka 3.x），具体数值会随硬件代际和版本演进而变化"
