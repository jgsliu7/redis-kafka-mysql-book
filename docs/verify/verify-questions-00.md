# 第00章《序言》事实核查问题清单

### Q1：三大软件的生产环境打磨年限
**章节位置**: 第7段（"Redis、MySQL、Kafka 已经在生产环境里反复打磨了十几年"）
**原文陈述**: "Redis、MySQL、Kafka 已经在生产环境里反复打磨了十几年"
**待确认点**: 三个软件各自的首次公开发布年份，以及从首次发布到本书写作时间（约2024—2026年）是否均超过十年。
**建议验证来源**: 
- Redis: 官方发布历史 / Wikipedia — 首次发布约2009年
- MySQL: 官方发布历史 / Wikipedia — 首次发布约1995年
- Kafka: LinkedIn 博客或 Wikipedia — 首次发布约2011年
- 确认三者的"十几年"表述是否均成立

### Q2：本书版本基线
**章节位置**: 第15段（"本书基于 Redis 7.x、MySQL 8.0.x、Kafka 3.x"）
**原文陈述**: "版本会更迭（本书基于 Redis 7.x、MySQL 8.0.x、Kafka 3.x），但这些抉择的逻辑长期不变。"
**待确认点**: 
- Redis 7.x 的具体子版本（如 7.0, 7.2, 7.4）是否与书中引用的功能特性一致
- MySQL 8.0.x 的具体子版本（如 8.0.20, 8.0.30 等）是否与书中引用的功能特性一致
- Kafka 3.x 的具体子版本（如 3.0, 3.2, 3.5 等）是否与书中引用的功能特性一致
- 全书各章实际引用的版本号是否保持一致
**建议验证来源**: 逐一核对各章的版本引用，与实际发布版本及其引入的新特性对照

### Q3：Redis 核心抽象定位
**章节位置**: 第13段（"Redis 把'内存即数据'做到极致"）
**原文陈述**: "Redis 把'内存即数据'做到极致"
**待确认点**: 该定位是否准确概括了 Redis 架构设计的核心特征（全内存数据存储、持久化仅作为可选项等）
**建议验证来源**: Redis 官方文档中的设计哲学描述，antirez 的博客文章

### Q4：MySQL 核心设计原则
**章节位置**: 第13段（"MySQL 以'不丢不错'为第一原则"）
**原文陈述**: "MySQL 以'不丢不错'为第一原则"
**待确认点**: MySQL 的 ACID 事务保障、WAL（redo log/undo log）机制是否确实将"数据不丢失、不错乱"作为第一设计原则
**建议验证来源**: MySQL 官方手册关于 ACID 和事务持久化的说明

---

### Q5：Kafka 核心抽象定位
**章节位置**: 第13段（"Kafka 以'追加写日志'为核心抽象"）
**原文陈述**: "Kafka 以'追加写日志'为核心抽象"
**待确认点**: Kafka 的架构设计是否确实以 append-only log 作为最核心的抽象（而非消息队列、流处理等）
**建议验证来源**: Kafka 设计文档（Originally from LinkedIn）、Kafka: The Definitive Guide 书中对设计哲学的阐述

**原文陈述**: "因为这三款恰好覆盖了后端三种最基础的能力形态"
**待确认点**: 该分类是否涵盖了后端系统状态管理的核心范式，是否有其他重要的范式被遗漏
**建议验证来源**: 该分类属作者的分析框架，不属于可以"验证"的客观事实，但可标注为待与全书框架一致性核对

### Q7：架构抉择框架的恒常性
**章节位置**: 第15段（"这些抉择的逻辑长期不变"）
**原文陈述**: "版本会更迭（本书基于 Redis 7.x、MySQL 8.0.x、Kafka 3.x），但这些抉择的逻辑长期不变"
**待确认点**: 三款软件未来版本是否有过重大架构变更（如 Redis 放弃单线程模型、MySQL 转向分布式原生、Kafka 放弃 ZooKeeper）从而改变"抉择逻辑"
**建议验证来源**: 各软件版本发布说明（Changelog/Release Notes）中关于架构级变更的记录
