# 写书配图方案:手写 SVG

> 本文记录《架构之道》这本书的配图技术方案与实战经验,供后续维护配图、新增配图、或移植到其他技术书时参考。
>
> **一句话方案:全部 45 张图都是手写原生 SVG 源码,纯文本,进 git,构建时内联嵌入 HTML。** 没有用任何绘图软件(Visio/Draw.io/Figma),也没有用图表 DSL(Mermaid/PlantUML)。

---

## 1. 为什么选手写 SVG

这个选择不是随手做的,而是权衡了三种方案后的结论。

### 三种方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **绘图软件导出**(Visio/Draw.io/Figma → PNG/SVG) | 所见即所得,上手快 | 产生二进制或难读 XML,git diff 不可读;改一个坐标要重新打开软件;导出的 SVG 常带大量冗余元数据 |
| **图表 DSL**(Mermaid/PlantUML → SVG) | 纯文本,可 diff | 表达力受限,复杂布局难控;样式不精细;依赖运行时渲染或预编译 |
| **手写原生 SVG**(本书选择) | 纯文本可 diff;无依赖;矢量清晰;样式完全可控;可脚本批量处理 | 画起来费劲,坐标全靠手算 |

### 选手写 SVG 的根本理由

1. **可纯文本管理**:图就是代码。这次书稿修订,我用脚本批量重命名了 9 个 SVG 文件、批量改了图号文字、批量注入 width 属性——这在 PNG 上做不到,在 Draw.io 导出的 SVG 上也极痛苦。
2. **无依赖、可内联**:构建脚本(`build_html.py`)直接把 SVG 文本读进 HTML,不依赖任何图片服务器、字体文件或运行时渲染。整本书 45 张图,HTML 才 700 多 KB。
3. **矢量、双介质**:放大不糊,网页和印刷两种介质都清晰。
4. **样式完全可控**:色板、字号、对齐全部精确到像素,全书 45 张图视觉高度统一——这是绘图软件很难做到的(每张图手调一遍成本极高)。

### 代价(必须承认的)

- **画起来费劲**:每个坐标(x/y)、每个框的宽高都要手算,改一个框的位置要连带改所有连到它的箭头端点。
- **有学习曲线**:需要懂 SVG 的基本元素(`<rect>`/`<line>`/`<text>`/`<path>`/`<marker>`)。
- **不适合频繁改图的场景**:如果一张图还在剧烈变化,手写成本很高;适合"想清楚再画"。

## 2. 技术方案

### 2.1 文件组织

```
chapters/
  01-introduction/
    chapter.md
    diagrams/
      fig-1-1.svg    ← 图号与文件名绑定
      fig-1-2.svg
  02-lifecycle/
    diagrams/
      fig-2-1.svg
      ...
```

**命名铁律**:文件名 `fig-N-M.svg` 的编号 = 图内文字图号 = 正文引用图号,三者必须一致。这是全书 45 张图的硬约束,构建脚本和检查脚本都依赖它。

### 2.2 单张 SVG 的骨架

每张图都遵循同一个骨架(以 `fig-1-1.svg` 为例):

```xml
<?xml version="1.0" encoding="UTF-8"?>
<svg xmlns="http://www.w3.org/2000/svg"
     viewBox="0 0 880 560"
     font-family="system-ui, -apple-system, 'PingFang SC', 'Microsoft YaHei', sans-serif">
  <defs>
    <!-- 1. 箭头 marker:定义一次,marker-end 引用 -->
    <marker id="arrow" viewBox="0 0 10 10" refX="9" refY="5"
            markerWidth="7" markerHeight="7" orient="auto-start-reverse">
      <path d="M0,0 L10,5 L0,10 z" fill="#4B5563"/>
    </marker>
    <marker id="arrow-flow" viewBox="0 0 10 10" refX="9" refY="5"
            markerWidth="7" markerHeight="7" orient="auto-start-reverse">
      <path d="M0,0 L10,5 L0,10 z" fill="#EF4444"/>
    </marker>
  </defs>

  <!-- 2. 标题(图号 + 名称,印在图内顶部) -->
  <text x="440" y="32" text-anchor="middle" font-size="18" font-weight="700" fill="#111827">
    图1-1　后端基础设施的三种核心范式与三家代言
  </text>

  <!-- 3. 正文:用注释分区定位 -->
  <!-- ============ 第 1 行:三类核心能力 ============ -->
  <rect x="40" y="86" width="240" height="70" rx="6" fill="#F3F4F6" stroke="#6B7280" stroke-width="1.5"/>
  <text x="160" y="110" text-anchor="middle" font-size="15" font-weight="700" fill="#111827">快速读写</text>
  ...

  <!-- 4. 连线:引用 marker -->
  <line x1="160" y1="156" x2="160" y2="200" stroke="#4B5563" stroke-width="1.5" marker-end="url(#arrow)"/>
</svg>
```

**四个关键组成部分**:

1. **`<defs>` 里的 marker**:箭头头定义一次,多处 `marker-end="url(#arrow)"` 引用。本书有两种主箭头:灰色普通流(`#4B5563`)、红色数据流(`#EF4444`)。
2. **图内标题**:每张图顶部都印"图 N-M　标题",与正文 figcaption 双重对应。
3. **正文元素**:`<rect>` 画框、`<text>` 写字、`<line>`/`<path>` 画线。**用 `<!-- -->` 注释分区**,便于人眼定位("第 1 行""能力 1"这种),否则 SVG 改起来是噩梦。
4. **连线引用 marker**:`marker-end="url(#arrow-flow)"` 给线加上箭头。

### 2.3 色板体系(全书画一的关键)

本书有一套固定色板,**45 张图全部遵守**,这是视觉统一的根基:

| 角色 | 色值 | 用途 |
|------|------|------|
| **文本主色** | `#111827` | 标题、重要文字 |
| **文本次色** | `#374151` | 正文文字 |
| **灰边框/灰线** | `#6B7280` / `#4B5563` | 普通框边、普通箭头 |
| **浅灰背景** | `#F3F4F6` | 普通区块底色 |
| **Redis 主题** | `#4F46E5`(靛蓝)+ `#EEF2FF`(浅底) | Redis 相关框 |
| **MySQL 主题** | `#D97706`(琥珀)+ `#FEF3C7`(浅底) | MySQL 相关框 |
| **Kafka 主题** | `#059669`(翠绿)+ `#ECFDF5`(浅底) | Kafka 相关框 |
| **强调/数据流** | `#EF4444`(红) | 关键箭头、数据流、警示 |
| **背景** | `#FFFFFF` | 画布底 |

**经验**:三家的"主题色"贯穿全书——只要图中出现 Redis,就用靛蓝系;出现 MySQL 就用琥珀系。读者翻到任何一章,颜色本身就是导航线索。这套色板提取自 Tailwind CSS 的色阶,色相协调、对比度足够。

### 2.4 字体

全书统一字体栈,写在 `<svg>` 根标签上,所有子元素继承:

```
font-family="system-ui, -apple-system, 'PingFang SC', 'Microsoft YaHei', sans-serif"
```

- `system-ui` / `-apple-system`:Mac/iOS 上用苹方
- `PingFang SC`:中文兜底
- `Microsoft YaHei`:Windows 兜底
- 代码/参数文字另用 `ui-monospace, monospace`

**经验**:字体栈写在根 `<svg>` 上一次,子元素不重复写,保证全书统一。

### 2.5 画布尺寸(viewBox)

本书画布宽度集中在 **880~960**,高度按内容自适应(560~760):

| 宽 | 高 | 数量 | 典型用途 |
|----|----|------|---------|
| 880 | 560~580 | 7 | 标准横向图(如章节地图) |
| 940 | 600~660 | 7 | 中等复杂图(流程图) |
| 960 | 700~760 | 多数 | 复杂图(时序图、架构图) |

**经验**:
- 宽度别超过 1000,否则内联进网页后字号偏小。
- `viewBox` 的宽高比要适配内容:时序图偏高、对比图偏宽。
- **只写 `viewBox`,不写 `width`/`height`**——让构建脚本注入 `width="100%"`(见 §3)。

## 3. 构建流程:SVG 如何嵌入 HTML

这一节是踩坑最多的地方,记录三个关键处理。

### 3.1 嵌入方式:内联,不是 `<img src>`

`build_html.py` 的 `load_svg()` 把 SVG 源码**直接读进 HTML 文本**,而不是用 `<img src="fig-1-1.svg">`:

```python
def load_svg(path, prefix=''):
    s = open(path, encoding='utf-8').read()
    s = re.sub(r'<\?xml[^>]*\?>\s*', '', s)   # 去 XML 声明
    s = re.sub(r'<!DOCTYPE[^>]*>\s*', '', s)
    s = re.sub(r'(<svg\b)(?![^>]*\swidth=)', r'\1 width="100%"', s)  # 注入 width
    # ... id 去重(见 3.3)
    return s
```

**为什么内联而不是 `<img>`**:
- 内联后 SVG 是 DOM 的一部分,可以用 CSS 统一控制样式(宽度、居中、margin)。
- 不依赖 `file://` 协议能否正确加载图片(有些浏览器对本地图片有限制)。
- 整本书单文件,可离线分发。

### 3.2 坑一:SVG 塌缩为 0×0(必须注入 width)

**症状**:SVG 只有 `viewBox` 没有 `width`/`height`,内联进网页后**完全不显示**(塌缩成 0×0)。

**原因**:浏览器渲染内联 SVG 时,若没有显式 `width`,且 CSS 只设了 `max-width:100%`(只设上限不给确定值),SVG 会塌缩。

**解决**:`load_svg()` 用正则给每个 `<svg>` 标签注入 `width="100%"`:
```python
s = re.sub(r'(<svg\b)(?![^>]*\swidth=)', r'\1 width="100%"', s)
```
同时 CSS 兜底:
```css
figure.fig svg { width:100%; max-width:1140px; height:auto; }
```

**经验**:这是最隐蔽的坑——图明明在 HTML 里(`<svg>` 标签在),但就是看不见。遇到"图不显示"先查 width。

### 3.3 坑二:跨文件 marker id 冲突(必须去重)

**症状**:多个 SVG 内联到同一 HTML 后,箭头颜色/形状错乱——某些图的红箭头变成了灰箭头。

**原因**:每张图的 `<defs>` 里都定义了 `<marker id="arrow">`,内联进同一 HTML 文档后,**所有 `id="arrow"` 都冲突**(HTML 规范要求 id 唯一)。浏览器让所有 `url(#arrow)` 引用都指向**文档中第一个**出现的 `arrow` 定义,于是后续图的箭头全错。

**解决**:`load_svg()` 给每个图的 id 加上**唯一前缀**(基于文件名),同步替换引用:
```python
if prefix:  # prefix 如 "fig_1_1"
    ids = set(re.findall(r'\bid="([^"]+)"', s))
    for _id in ids:
        s = s.replace('id="%s"' % _id, 'id="%s_%s"' % (prefix, _id))
        s = s.replace('url(#%s)' % _id, 'url(#%s_%s)' % (prefix, _id))
        s = s.replace('href="#%s"' % _id, 'href="#%s_%s"' % (prefix, _id))
```
处理后,id 变成 `fig_1_1_arrow`、`fig_1_2_arrow`,全局唯一。

**经验**:这个坑在"单张 SVG 单独看"时完全不会暴露,只有内联到同一文档才发作。任何"多 SVG 内联"的项目都必须做 id 去重。

### 3.4 图区突破正文宽度(让图更大更清晰)

**问题**:内容区限宽 820px,图也被压在 820px 内,复杂图细节看不清。

**解决**:让 `<figure>` 用负 margin 突破正文宽度,在更宽区域(1140px)居中,正文文字仍保持 820px:
```css
figure.fig { margin: 2em calc((100% - 1140px)/2); text-align:center; }
figure.fig svg { width:100%; max-width:1140px; }
@media (max-width:1200px) { figure.fig { margin: 2em auto; } }  /* 窄屏回退 */
```

**经验**:正文用窄栏保证阅读舒适,图用宽栏保证清晰——两者解耦。

## 4. 实战经验与踩坑清单

### 4.1 画图流程(推荐步骤)

新增一张图时,按这个顺序:

1. **先想清楚图要讲什么**——一句话能说清再画,否则别画。图是论证工具不是装饰。
2. **在纸上/脑子里画草图**——确定有哪些框、怎么连线、哪个是重点。
3. **挑一张最接近的现有图作模板**——复制它的 `<defs>`、色板、字体栈,只改坐标和文字。
4. **从左到右、从上到下布局**——先摆框(`<rect>`),再填字(`<text>`),最后连线(`<line>` + `marker-end`)。
5. **加注释分区**——用 `<!-- 第 N 行 / 模块名 -->` 标注每个区块,方便后续改图定位。
6. **核对坐标不越界**——所有 `x+width ≤ viewBox 宽`,`y+height ≤ viewBox 高`。
7. **跑构建脚本 + 浏览器看效果**——`python3 scripts/build_html.py` 然后刷新。

### 4.2 坐标与对齐技巧

- **对齐靠坐标算,不靠肉眼**。三列均分 880 宽:左边距 40,每列宽 240,列间距 40 → 第一列 x=40,第二列 x=320,第三列 x=600。这种数学关系写进注释里。
- **`text-anchor="middle"` 让文字居中**——文字的 x 是中心点,不是左端。这是最常犯的错(以为 x 是左端,结果文字偏右)。
- **字号梯度**:标题 18px,区块标题 15-16px,正文 12.5-13px,标注 10.5-11.5px。保持全书一致。
- **`rx="6"` 给矩形圆角**——比直角柔和,全书统一用 6。

### 4.3 复用与一致性

- **marker 在每张图的 `<defs>` 里各自定义**(不抽成外部共享),因为内联时要各自做 id 去重(§3.3)。
- **色板写死在每张图里**(不用 CSS 变量),因为 SVG 内联后不一定继承外部 CSS。这也是为什么色板必须精简(§2.3),否则维护成本爆炸。
- **三家的主题色是全书最强的一致性线索**——Redis 永远靛蓝、MySQL 永远琥珀、Kafka 永远翠绿。读者靠颜色就能认出"这框说的是谁"。

### 4.4 已踩过的坑(供复查)

| 坑 | 症状 | 解决 | 检查方法 |
|----|------|------|---------|
| SVG 塌缩 0×0 | 网页上图看不见 | 注入 `width="100%"` | 浏览器看图区是否空白 |
| marker id 冲突 | 箭头颜色/形状错乱 | load_svg id 加前缀去重 | 检查 HTML 内 id 是否唯一 |
| 图号跳号 | 图 7-9 跳过 7-7/7-8 | 重命名修正 | 跑图号连续性扫描脚本 |
| 图/表命名混用 | 表格被命名成"图" | 全书统一图/表分编 | grep "图 N-M" 在表格上的误用 |
| 文件名≠内部图号 | 引用错位 | 三者绑定 | 跑 SVG 一致性扫描脚本 |
| 图内无图号文字 | 体例不统一(第8章曾缺) | 每张图顶部印图号 | grep 每张 SVG 是否有"图 N-M" |
| 坐标越界 | 文字/框溢出画布 | 调坐标 | 跑坐标越界扫描脚本 |

## 5. 配套脚本

本项目有两个关键脚本支持配图工作流:

### 5.1 `scripts/build_html.py`(构建)

- `load_svg()`:读取 SVG → 去 XML 声明 → 注入 width → id 去重 → 返回内联字符串。
- 正文 Markdown 里的 `![图 N-M 标题](diagrams/fig-N-M.svg)` 被转成 `<figure class="fig">`,SVG 内联进去,figcaption 取自 alt 文本。
- 表格(`|...|`)渲染为 HTML table。

### 5.2 SVG 一致性扫描(在文档审查时手动跑)

用于批量体检所有 SVG,检查项包括:
- XML 合法性
- viewBox 格式与尺寸
- 文件名图号 = 内部图号 = 正文引用图号(三者一致)
- 单文件内 id 重复
- 坐标/文字越界
- HTML 内联后跨文件 id 冲突

**经验**:这套扫描脚本在书稿修订时立了大功——45 张图的编号一致性、id 冲突、坐标越界,人工查几乎不可能,脚本几秒跑完。

## 6. 局限与未来改进

诚实记录这套方案的不足,供后续决策:

1. **marker id 命名不统一**:有的图叫 `arrow`、有的叫 `arrowRed`、有的叫 `arrow-red`、有的叫 `arrowR`。虽然 load_svg 的去重解决了冲突,但命名混乱。**改进**:全书统一一套 marker 命名(`arrow`/`arrow-red`/`arrow-green`),写进规范。
2. **色板写死在每张图里**:改主题色要改 45 个文件。**改进**:可考虑用构建脚本做色板变量替换(但会增加复杂度,当前 45 张图规模下还能接受)。
3. **手写成本高**:画一张复杂时序图要 1-2 小时。如果未来图量剧增,可考虑引入图表 DSL 做简单图、手写 SVG 做复杂图的混合方案。
4. **无版本化样式**:无法像 CSS 那样一次改完全书风格。当前靠"每张图都遵守同一骨架"来保证一致,靠人律。

---

## 7. 一句话总结

> **手写 SVG 的本质,是把"图"纳入和"正文"同等的第一公民地位:可 diff、可 grep、可脚本批处理、可内联。代价是画起来费劲,换来的是长期可维护性和视觉一致性——对一个要经年重读的技术书,这笔交易划算。**
