# 《架构之道》AI 味全维审校报告

> 2026-06-23 · 6 个审校 agent 并行审查 · 覆盖 11 章节 × 6 维度

---

## 总体判断

**这本书的 AI 味不在词汇表面，而在更深层——结构同构、作者消失、零真实数据、零踩坑故事。** 词汇层面已被前几轮编辑清理得很干净（B/F/G/H 四类黑名单全书命中数为 0），但"干净"本身恰恰制造了新的问题：一本完美到没有瑕疵的技术书，读起来就像 AI 写的。

一句话诊断：**它缺的不是内容，是作者。**

---

## 六个维度的核心发现

### 1. 词汇层（Agent 1）—— ✅ 基本健康

| 指标 | 结果 |
|------|------|
| 全书嫌疑词总次数 | ~14 处（对于约 8 万字，密度极低） |
| B 类互联网黑话 | **0** |
| F 类物理动作动词 | **0** |
| G 类后缀堆砌（性/化/度/力） | **0** |
| H 类"进行+动词" | **0** |
| D 类程度副词空泛量化 | 仅 4 处 |
| A 类框架词 | ~10 处，轻症 |

**唯一密度异常**：第 9 章"换你做会怎样"重复 5 次——同一模板在同一章内出现 ≥3 次即触发 AI 味。

**结论**：词汇层面不是问题。这本书没有"赋能/抓手/闭环/深入探讨"等常见 AI 套话。但如果只有表层词汇干净、深层全是模板，反而更可疑——因为人类写作的词汇本来就会有随机波动，过度干净本身就是信号。

### 2. 句式层（Agent 2）—— ⚠️ 中等

| 模式 | 全书次数 | 判定 |
|------|---------|------|
| "不是A，而是B" 对仗 | **38 次** | 🔴 严重超标 |
| "X是清晰的/很清楚" 前置判断 | ~12 次 | 🟡 偏多 |
| "值得X的" 评价性引导 | ~15 次 | 🟡 偏多 |
| "X很Y:" 起手式 | 0 | ✅ |
| "进行+动词" | 0 | ✅ |
| 翻译腔被动 | 1 次 | ✅ |

**最严重的 3 章**：第 9 章（5-6 次）、第 1 章（6 次）、第 2 章（4 次）。

"不是A而是B"是 AI 最爱的句式骨架——它用一个否定来制造"锐度"，再用一个肯定来建立"洞见"。单独一次没问题，但同一篇文章出现 38 次、某些章节连续两段使用同一模式，就是典型的 AI 纹理。

**建议**：全书 38 处至少砍掉 25 处，只保留真正有信息增量的（如"复制不是抄数据，是用坐标对齐状态机"这类结构性映射）。其余的用直接陈述代替。

### 3. 结构层（Agent 3）—— 🔴 全书最大问题

**核心发现：第 2-8 章的骨架完全一致。**

```
导读(总) → X.1 问题本质 → X.2 Redis → X.3 MySQL → X.4 Kafka
→ X.5 横向对比 → X.6 架构启示(5条) → X.7 小结
```

七个主章的模板相同到**可以互换章节编号而不违和**。人类写作天然有风格变异——有的章长、有的章短、有的打破常规——但本书的结构一致性已经到了可疑的程度。

| 结构问题 | 严重程度 | 涉及 |
|---------|---------|------|
| 2-8 章骨架完全相同 | 🔴 毁灭性 | 全部主章 |
| Redis→MySQL→Kafka 顺序从未改变 | 🔴 毁灭性 | 全部主章 |
| 第 9 章 35 个编号单元堆叠 | 🔴 重度 | Ch9 |
| 每节末尾"本节的结论/取舍总账"句式统一 | 🟡 中度 | Ch2-8 |
| 导读"读完本章，你会得到/学会/掌握..."连续 8 次 | 🟡 中度 | Ch1-8 |
| "第一/第二/第三"枚举被不节制使用 | 🟡 中度 | 多章 |
| 每章 N.6 启示段的"箴言"格式零变化 | 🟡 中度 | Ch2-8 |
| 章节间过渡句"下一章我们要看..."模板化 | 🟢 轻度 | Ch2-9 |

**最严重的 3 章**：第 9 章（35 个编号单元）> 第 7 章（四张力+五启示+三问+决策树，模板套模板）> 第 3 章（三段范式+三家取舍总账统一"用X换Y"句式）

### 5. 情感与声音层（Agent 5）—— 🔴 严重

**全书"人味"均分：4.9/10（哑铃形分布）**

```
序言 7/10 —————————————— 后记 9/10
          \
           第 1-9 章全部 4-5/10
```

**主体章全部位于 4-5 分区间的含义**：读者翻开序言觉得"这人有点意思"，进入正文后迅速感到"人味断崖"——正文读起来像一本没有作者的工具书。

| 维度 | 问题 |
|------|------|
| 第一人称 | 正文中**零个"我"字**。全程"你""本书""运维""DBA" |
| 情绪起伏 | 全程同一温度——讲到精妙设计无赞叹，讲到坑无咬牙，讲到取舍无纠结 |
| 作者偏好 | 读完全书不知道作者更欣赏三家中哪一家、有什么脾气 |
| 自嘲式坦诚 | 零——没有"这个参数我调了一下午最后发现是单位看错了" |
| 幽默/松弛感 | 零——全书从头到尾板着脸 |

**作者声音诊断**：可辨认，但很薄——"一个严谨、系统、爱画决策树的架构导师，擅长组织和提炼规律，但小心翼翼不暴露自己是谁。"

对比标杆：
- Kleppmann (DDIA)：审慎、学者气质、爱拆解 assumptions
- 黄健宏（黄书）：老实的工程匠人，每行代码拆给你看
- antirez：江湖气、简洁、有脾气
- **本书作者**：有工程视野但缺乏人味，有方法论但缺乏性情

**最缺人味的 3 章**：第 5 章（安全，本应最有戏剧张力）< 第 3 章（内存磁盘，速度与持久性的永恒矛盾本应最有激情）< 第 7 章（存储格式，最硬核也最冷）

## 修改路线图建议

### 第一轮：注入血肉（解决"缺作者"的根问题）
- 每章加 1-2 个"我"开头的真实故事/经验/感受
- 每个空泛量化补具体数字+测试条件
- 加 3-5 处自嘲式坦诚和情绪起伏
- 把后记的库存事故移到正文相关章节（作为案例引用），后记改为呼应而非首次出现

### 第二轮：打破模板（解决结构同构）
- 至少 2 章调换 Redis/MySQL/Kafka 的讲解顺序
- 至少 2 章不使用"X.1 问题本质"作为第一节（把问题融入叙事）
- 砍掉一半"读完本章你会…"的导读句
- 第 9 章编号砍半（5 条规律 → 3 条，去掉维度编号）

### 第三轮：句式瘦身（解决对仗过度）
- 砍掉 25+ 处"不是A而是B"
- 砍掉一半"X是清晰的"前置判断
- 砍掉一半"值得X的"评价性引导
- 第 6、8 章金句密度对齐规范

### 第四轮：终校润色
- 确认机制段无价值标签侵入
- 确认所有锚点比喻完好
- 确认组件咬合规范（导语+解读三件套）
- 全书术语一致性最终检查

## 一条总纲

> **这本书的技术质量和结构力极高——它缺的不是内容，是作者。** 目前的阅读体验是"一个优秀的 AI 写了一本完美的技术书"——而完美正是 AI 的致命伤。注入不完美、注入个性、注入情绪、注入真实的血与泪，这本书的"人味"会从 5 分跳升到 8 分，从"AI 能写出来的最好的书"变成"只有你才写得出来的书"。
