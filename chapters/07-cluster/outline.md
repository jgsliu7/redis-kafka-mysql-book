## 本章导读（150–250 字）

要解决的问题：单机在「扩展性 / 高可用 / 负载能力」三座大山下触顶，必须走向集群；但分布式同时带来一致性、分区容错、故障转移、元数据管理等一系列新问题。

为什么重要：集群架构是分布式系统最核心的命题，也是三家软件设计哲学分歧最大的地方——Redis 选 AP、MySQL 选 CP、Kafka 在 AP 与 CP 之间可调。读懂这一章就读懂了三者最根本的取舍。

读完能获得：从「单点 → 多副本 → 分片 → 自动容错」的完整演进脉络；理解共识协议（Raft / Paxos 变种）、Gossip、ISR 三种元数据/一致性手段的本质差异；提炼出可复用的集群设计原则。

> 节制提醒：本章聚焦「集群拓扑 / 角色 / 分片 / 容错 / 元数据管理」。复制的字节级细节（binlog 格式、RDB/AOF、副本同步流）归第 9 章，本章只在必要处一笔带过并交叉引用。

## 6.1 问题的本质（约 900 字）

**目标**：先把集群要解决的两类完全不同的问题立清楚——「水平扩展」与「高可用」，再点出分布式引入的四大新难题。为后面三家登场建立统一坐标系。

**核心要点**：
- 单机的三道天花板：CPU/内存/磁盘物理上限（扩展性）、单点故障（可用性）、单节点吞吐瓶颈（负载能力）。三者不是同一个问题，解法也不同。
- 两条独立的技术路线，常被混淆：
  - **水平扩展（scale-out）解决容量与吞吐**：把数据切分到多台机器（分片 / sharding / partitioning）。
  - **副本（replication）解决高可用**：同一份数据存多份，一台坏了另一台顶上。
  - 二者正交：可以只分片不复制（扛不住故障），也可以只复制不分片（扛不住容量）。真正成熟的集群两件事都做，但分工不同。
- 分布式引入的四大新难题（本章后面三家逐一回应）：
  1. **数据怎么切**——分片策略（范围分片 vs 哈希分片）与重分片的代价。
  2. **副本怎么一致**——同步/异步/多数派确认，以及一致性强度与延迟的取舍。
  3. **故障怎么发现与转移**——主观下线 vs 客观下线、Leader 选举、脑裂。
  4. **元数据谁说了算**——集中式（Controller/协调者）vs 去中心化（Gossip/共识协议）。
- 引出 CAP：网络分区（P）是分布式绕不开的现实，于是一致性（C）与可用性（A）必须取舍。这是贯穿全章的主轴。

**导语金句候选**：「单点是一切故障的根源，分布式是解决问题的手段，也是引入问题的来源。」

## 6.2 Redis 的做法（约 2400 字）

**目标**：讲清 Redis 从「主从 → Sentinel → Cluster」三段式演进，每一段解决了上一段的什么痛点、又留下什么新问题。重点落在 Redis Cluster 的 16384 槽分片、Gossip 元数据、客户端路由与 AP 取舍。

**核心要点（机制 + 取舍）**：

1. **主从复制：高可用的地基，但不是集群**
   - 机制：`replicaof` 建立复制；全量同步（fork + RDB + 积压缓冲区命令）vs 部分重同步（offset 落在复制积压缓冲区内则只补差量）。
   - 取舍：复制是**异步**的（性能优先，可丢数据）；写仍集中在单主（解决可用，不解决扩展）；故障转移需人工。→ 引出 Sentinel。
   - 复制流的字节级细节归第 9 章，这里只讲它在集群中的角色。

2. **Sentinel：把人工故障转移自动化**
   - 机制：Sentinel 集群监控 → SDOWN（主观下线，单点超时）→ ODOWN（客观下线，达 quorum）→ Raft 选 Leader Sentinel → 按优先级/offset/runid 选新主 → `SLAVEOF NO ONE` 提升。
   - 取舍：Sentinel 本身用 Raft quorum 防单点误判，但数据层仍是单主异步复制——**监控的强一致 ≠ 数据的强一致**。这是最容易被误解的点。
   - 留下的坑：还是不能水平扩展写、内存仍受单机限制、客户端要懂 Sentinel 协议。→ 引出 Cluster。

3. **Redis Cluster：分片 + 自动容错，去中心化元数据**
   - 分片：16384 槽，`CRC16(key) % 16384`。为什么是 16384 而不是更大？（消息压缩 vs 槽粒度，Gossip 携带槽位 bitmap 的折中——这是个值得点的设计决策）。
   - 元数据：**无中心**，纯 Gossip（PING/PONG/MEET/FAIL）传播，最终一致。取舍：无单点、自动发现，但收敛慢、可能出现短暂不一致视图。
   - 路由：MOVED（槽已迁移，客户端更新本地映射）vs ASK（槽迁移中，一次性重试）。哈希标签 `{...}` 保证多键落在同槽。
   - 容错：从节点检测主节点超时 → 发起选举请求 → 其他主节点按 epoch 投票（多数派）→ 升级新主 → Gossip 广播新拓扑。
   - 取舍 / 限制：**AP 系统**，分区时优先可用、可能丢写或脑裂；多键命令要求同槽；`SELECT` 禁用；故障转移期部分槽可能短暂不可用。
   - 最小拓扑 3 主 3 从是工程惯例而非硬约束，要讲清为什么。

**导语策略**：用「Redis 的集群演进是不断还债的过程」开场，每一段都对应还掉一笔具体的债。

## 6.3 MySQL 的做法（约 2400 字）

**目标**：讲清 MySQL「异步复制 → 半同步 → Group Replication（MGR）→ InnoDB Cluster」的演进，重点落在 MGR 的 Paxos 多数派确认、单主/多主模式、以及 MySQL 为何坚持 CP 路线。强调 MySQL **不做数据分片**这一根本取舍。

**核心要点（机制 + 取舍）**：

1. **基于 binlog 的复制：单向、异步、行流**
   - 机制：主写 binlog → Dump Thread 推送 → 从 I/O Thread 落 relay log → SQL Thread 回放。GTID（`source_id:transaction_id`）让从库自动定位、避免 file/position 脆弱性。
   - 取舍：异步 = 主不等等从，主挂了可能丢已提交事务；单 SQL 线程回放导致延迟。8.0 的多线程复制（`slave_parallel_type=LOGICAL_CLOCK`）按组提交并行化，缓解但不根治。
   - 字节级 binlog 格式（STATEMENT/ROW/MIXED）细节归第 8 章存储格式；本章只讲它在集群里承担「变更流的载体」角色。

2. **半同步复制：用一点延迟换不丢**
   - 机制：主在事务提交前，等至少一个从 ACK（`rpl_semi_sync_master_wait_for_slave_count`）才返回客户端。
   - 取舍：在「异步（快但可丢）」与「全同步（不丢但慢）」之间折中。超时降级回异步——这是个关键的降级语义，说明 MySQL 把可用性也看得很重。

3. **Group Replication：Paxos 多数派，CP 路线的正名**
   - 机制：XCom（Paxos 变种）在组成员间对事务做多数派确认后才提交；单主模式 Primary 读写、Secondary 只读；多主模式全员可写但有冲突检测。
   - 容错：最多容忍 (N-1)/2 节点故障；故障自动 View Change + 选新主 + GTID 对齐。
   - 取舍：**CP 系统**，少数派分区直接停止服务（宁可不可用也不让脑裂写脏数据）——与 Redis Cluster 的 AP 形成鲜明对照。代价是写延迟因多数派确认而上升。
   - 最关键的取舍：MGR **不做数据分片**，所有成员持有全量数据。扩容靠加副本（提可用）而非加分片（提容量）。容量扩展得靠上层方案（如 ShardingSphere / Vitess），这是 MySQL 生态的明确分工。

4. **InnoDB Cluster：把 MGR + Router + Shell 打包成可用方案**
   - 角色：MGR 提供强一致副本组；MySQL Router 做读写分离与故障切换代理；MySQL Shell 做运维。
   - 取舍：工程整合而非新机制，降低了 MGR 的使用门槛，但本质仍是 CP 单主全量复制。

**导语策略**：用「MySQL 的集群演进是在『不丢数据』这条不可动摇的底线之上，一步步挤可用性」开场。

## 6.4 Kafka 的做法（约 2400 字）

**目标**：讲清 Kafka 「Partition 分片 + Replica 副本 + ISR 一致性 + Controller/KRaft 元数据」四件套如何咬合。强调 Kafka 的特殊性：**分片和副本都做到极致，且天然分布式**，一致性强度可由 `acks` 在 AP/CP 之间滑动。

**核心要点（机制 + 取舍）**：

1. **Partition：分片即并行单位**
   - 机制：Topic 切成 N 个 Partition，每个 Partition 是有序追加日志；生产者按 key 的 `murmur2 % N` 路由，无 key 则粘性/轮询。
   - 取舍：**顺序性只在分区内成立**——这是 Kafka 一致性承诺的边界。分区数选择 = 吞吐（并行度）与运维开销（文件句柄、选举恢复时间）的权衡；经验值单分区 ~10MB/s 或 ~10k msg/s。

2. **Replica + ISR：副本的「够格」名单**
   - 机制：每个 Partition 有 1 个 Leader + N-1 个 Follower；Leader 读写，Follower 拉取同步。**ISR（同步副本集合）**是「在 `replica.lag.time.max.ms` 内跟上 Leader 的副本」，只有 ISR 里的副本才有资格当选 Leader。
   - HW（High Watermark，消费者可见 offset）与 LEO（Log End Offset，副本末尾 offset）的语义：HW 由 ISR 最小 LEO 决定，保证「已确认的消息不会因 Leader 切换而丢」。
   - 取舍：ISR 是个**动态门槛**——比 MGR 的「全员多数派」松，比 Redis 的「纯异步」紧。Follower 慢了就踢出 ISR，Leader 切换只在 ISR 内选，既保可用又保不丢。这是 Kafka 最精巧的设计之一。

3. **acks：一致性强度做成旋钮**
   - `acks=0/1/all` 三档对应「不等 / 等 Leader / 等 ISR 全部」。配合 `enable.idempotence` 与事务，可堆叠出 exactly-once 语义。
   - 取舍：把 CP/AP 的选择权交给使用者，而非写死在系统里——这是 Kafka 与 Redis/MySQL 最不同的产品哲学。要讲清这个旋钮的代价（延迟 vs 可靠）。

4. **Controller 与 KRaft：元数据管理的两次跃迁**
   - ZK 模式：元数据放 ZooKeeper，Kafka Broker 里选一个 Controller 做元数据操作。痛点：元数据与实际状态分离、脑裂风险、ZK 不适合存大量元数据（分区数受限）、运维两套系统。
   - KRaft（3.x 成熟）：用 Raft 在一组 Controller 节点内管理元数据，单进程、强一致、分区数可到百万级、恢复更快。
   - 取舍：从「外部依赖 + 临时独裁者（Controller）」收敛到「内嵌共识协议」。注意 Controller 的 Leader 选举（元数据层）与 Partition 的 Leader 选举（数据层）是两套独立机制，别混。

**导语策略**：用「Kafka 天生就是分布式——它的分片和副本不是后加的补丁，而是设计起点」开场。

## 6.5 横向对比（约 1200 字，不可缩水）

**目标**：把三家放进同一张坐标系，做有解读的对比。先给一句导语「比什么」，再上三栏表（顺序固定 Redis | MySQL | Kafka），表后跟一段「说明了什么」的解读，禁止表格裸奔（遵循圣经第 5 节）。

**核心对比维度（行）**：

1. **数据分片策略**
   - Redis：16384 槽哈希分片，客户端路由（MOVED/ASK）。
   - MySQL：**不分片**（MGR 全量复制），分片交给上层中间件。
   - Kafka：Partition 哈希/轮询分片，生产者端路由。
   - 解读：分片粒度（键 → 槽 → 节点）vs（键 → 分区）vs 无；MySQL 的「不分片」是有意为之的生态分工。

2. **副本一致性模型**
   - Redis：异步复制（AP）。
   - MySQL：异步 / 半同步 / MGR 多数派（CP）。
   - Kafka：ISR + acks 可调（AP↔CP 滑动）。
   - 解读：一致性强弱 = 写延迟高低，三者在这条光谱上的位置。

3. **故障检测与 Leader 选举**
   - Redis：Gossip + epoch 投票（去中心化）。
   - MySQL：XCom/Paxos View Change（共识驱动）。
   - Kafka：Controller 从 ISR 选 Leader（集中仲裁）。
   - 解读：三种完全不同的「谁来当 Leader」决策机制。

4. **元数据管理**
   - Redis：Gossip 去中心化、最终一致。
   - MySQL：组内 Paxos 强一致。
   - Kafka：Controller（KRaft Raft）集中强一致。
   - 解读：去中心化 vs 集中式的取舍——收敛速度 vs 一致性强度。

5. **CAP 立场**
   - Redis：AP（分区优先可用，可丢写/脑裂）。
   - MySQL：CP（少数派分区停服）。
   - Kafka：AP 默认、CP 可配（acks=all）。
   - 解读：三者 CAP 选择的根源 = 产品定位（缓存 / 交易账本 / 流管道）。

6. **横向扩展能力**
   - Redis：分片扩写、扩容量。
   - MySQL：加副本只提可用，扩容量靠上层分片。
   - Kafka：加 Partition + 加 Broker 同时扩吞吐与容量。
   - 解读：扩的是「可用性」还是「容量/吞吐」，三家解的能力不同。

## 6.6 架构启示（约 1100 字，不可缩水）

**目标**：从三家具体实现里提炼出**可复用的集群设计原则与权衡规律**。至少给出 1–2 句可被独立引用的箴言（圣经第 10.6 条质量标准）。

**核心启示**：

1. **分片与副本是正交的两件事，别混为一谈**。先问自己扩的是「容量/吞吐」还是「可用性」，再决定用分片还是用副本。Redis/Kafka 两者都做且分工清晰，MySQL 只做副本、把分片让给生态——没有对错，只有定位。

2. **一致性强度是写延迟的函数**。异步（Redis）= 快但可丢；多数派（MGR）= 不丢但每次写都要跨节点往返；ISR（Kafka）= 用「够格名单」在两者间滑动。没有免费的一致性，只有你愿意付多少延迟。

3. **元数据管理决定集群的天花板**。去中心化 Gossip 无单点但收敛慢；集中式共识强一致但协调者是瓶颈与单点。Kafka 从 ZK 走到 KRaft，本质上是在「把元数据的强一致和可用性同时拿到手」。

4. **把失败当常态，把一致性做成旋钮**。Kafka 的 `acks` 是教科书级的「把架构决策下放给使用者」。设计自己的系统时，问一句：哪些取舍应该写死，哪些应该交给配置？

5. **CAP 不是选择题，是定位题**。先确定业务落在 CAP 哪条边（缓存？账本？流？），再去选型。Redis 不该用来当账本，MySQL 不该用来当高吞吐缓存——选错 CAP 比选错参数严重得多。

**金句候选（至少 1–2 句可被引用）**：
- 「分片解决容量，副本解决可用——先把这两件事分开，再谈集群设计。」
- 「没有免费的一致性，只有你愿意付多少延迟。」
- 「把一致性做成旋钮，比把它写死更难，也更值钱。」

## 本章写作难点 / 不确定处

1. **与第 9 章（数据同步）的边界划分**：复制流程同时属于「集群」与「同步」两章。本章只讲复制在集群中扮演的角色（解决可用/扩展），字节级同步细节交第 9 章。动笔前需与总调度确认这条边界，避免两章重复或互相遗漏。
2. **共识协议的深讲分寸**：Raft（Redis Sentinel 选举、Kafka KRaft）与 Paxos 变种（MySQL MGR 的 XCom）若展开讲算法会冲掉全章字数预算且偏离「架构共性」定位。计划只讲「它保证什么、代价是什么」，不进入 term/epoch/log replication 细节。需总调度确认这个深度。
3. **Redis Cluster 「16384 槽为何是这个数」**：常见说法是「Gossip 消息里槽位 bitmap 占 2KB，16K 是 bitmap 大小与槽粒度的折中」。这个解释流传广但官方未严格论证，需标注「以社区主流解释为准」，不当作定论。
4. **MySQL MGR 的多主模式实战极少**：业界几乎只跑单主模式，多主冲突检测讲太深容易误导读者以为它可用。计划明确点出「多主是能力存在但生产罕用」，不展开冲突检测算法细节。
5. **Kafka KRaft 的成熟度口径**：圣经基线是 Kafka 3.x，KRaft 在 3.3+ 才生产可用、3.5 默认。写作时需明确「KRaft 生产可用始于 3.3+」，避免读者误以为任意 3.x 都稳。
6. **字数控制**：6.2/6.3/6.4 各 2400 字、6.5/6.6 各 1100–1200 字，加导读与小结约 10600 字，落在 8000–12000 区间偏上。若三家深讲小节超量，优先压缩 Sentinel/binlog 格式、半同步等「过渡方案」篇幅，保住 Cluster/MGR/ISR/KRaft 四个终态方案。
