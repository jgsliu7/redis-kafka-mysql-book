## 1.1 为什么是这三款软件（约 1800 字）

**这一节要讲清什么**：论证选 Redis、MySQL、Kafka 作为全书样本的正当性——不是随便选三个，而是它们恰好覆盖了后端基础设施的三种核心范式（内存型 / 持久化型 / 流式型），且都是各自领域的工业级标杆。落脚点是：把这三家读懂，就等于掌握了后端架构的"基本粒子"。

### 1.1.1 后端基础设施的三种核心范式

- 后端系统无论多复杂，拆到最底层都绕不开三类基础能力：**快速读写**（缓存、计数、会话）、**可靠存储**（事务、查询、持久化）、**异步解耦与流转**（消息、流、事件）。
- 用一句话给三类能力各配一个"代言人"：内存 → Redis；磁盘数据库 → MySQL；消息与流 → Kafka。
- 强调这三类能力是**互补而非替代**：一个真实的电商订单系统会同时用到三者（Redis 放购物车、MySQL 存订单、Kafka 通知下游）。

### 1.1.2 为什么是这三款，而不是别的

- **Redis**：内存数据结构存储的事实标准。同类（Memcached、Aerospike）要么功能单一要么生态薄弱，Redis 用"数据结构服务器"这一抽象把缓存、排行榜、消息、锁等多种场景统一掉。
- **MySQL**：全球使用最广的开源关系型数据库。InnoDB 引擎的事务、MVCC、B+ 树索引几乎成了关系型存储的教科书实现；它的演化史（MyISAM → InnoDB → MGR）本身就是一部架构演进史。
- **Kafka**：从 LinkedIn 内部日志系统演化为分布式流平台，单机百万级 TPS。"日志即核心"的设计哲学影响了后续一整代系统（Pulsar、EventStore、各种 Event Sourcing 方案）。
- 关键论点：选这三款不是因为它们"最新最酷"，而是因为它们**足够成熟、足够典型、足够有代表性**——每家都把某一种设计取向走到了极致，对照着看反而能看清取舍。

### 1.1.3 它们能教会我们什么（铺垫全书）

- 一句话点出三家的"教学价值"：Redis 教**极简与单点性能**，MySQL 教**严谨与一致性**，Kafka 教**吞吐与可扩展**。
- 强调：本书不是讲"三家各自多牛"，而是讲"面对同一个架构难题（如并发、持久化、复制、扩展），它们各自选了什么路、代价是什么"。这是后面 7 章主题式深讲的逻辑起点。
- **要点栏（核心取舍铺垫）**：
  - Redis：用内存 + 单线程换极简与低延迟，代价是数据量受内存约束、复杂计算让位给网络。
  - MySQL：用 B+ 树 + WAL + MVCC 换强一致与通用查询，代价是写放大与事务开销。
  - Kafka：用追加写日志 + 顺序 IO + 分区换极致吞吐，代价是延迟与灵活性。

## 1.2 三家速写（约 2800 字）

**这一节要讲清什么**：在深入任何主题之前，先给三家各画一张"设计身份证"——出身、核心抽象、设计哲学、典型场景、一句话定位。每个软件约 900 字，写成有信息密度的人物素描，不是罗列功能列表。每个子节都要落到"它最在乎什么 / 最不在乎什么"，为后文取舍分析定调。

### 1.2.1 Redis：内存即数据

- **出身**：2009 年由 Salvatore Sanfilippo（antirez）创建，起因是他想给自家实时日志系统找一个高性能的队列结构，市面工具都不顺手，于是自己写了一个。强调 Redis 是"为解决真实问题而生"，不是论文产物。
- **核心抽象**：数据结构服务器（data structure server）。一切对外暴露的都是结构化类型（字符串、哈希、列表、集合、有序集合、Stream…），命令直接在结构上做原子操作。这一抽象让它能同时充当缓存、队列、排行榜、锁服务等多种角色。
- **设计哲学（要点栏：核心机制 + 关键取舍）**：
  - 机制：全内存存储；单线程命令执行（7.x 引入多线程 IO 但执行仍单线程）；RDB 快照 + AOF 持久化；事件驱动网络模型。
  - 取舍：用"全部放内存 + 单线程"换来**低延迟、无锁、实现极简**；代价是数据量受物理内存限制、CPU 单核吞吐有上限、复杂聚合不是它的强项。Redis 宁可不做也不做半吊子。
- **典型场景**：缓存、分布式锁、计数器、排行榜、会话存储、轻量消息（Stream / Pub-Sub）。
- **一句话定位**：把"快"和"简单"做到极致的内存数据结构工具箱。

### 1.2.2 MySQL：把"不丢、不错"刻进骨子里的关系库

- **出身**：1995 年起源于瑞典 MySQL AB，先后经 Sun（2008）、Oracle（2010）之手。强调它经历了商业化收购、分支分叉（MariaDB）、存储引擎更迭（MyISAM → InnoDB 默认化），至今仍是 Web 应用的首选关系库。
- **核心抽象**：关系（表/行）+ SQL + 可插拔存储引擎。上层是统一的 SQL 接口与查询优化器，下层是 InnoDB 等引擎负责真正存取。InnoDB 把数据组织成**页（page，默认 16KB）**，以 B+ 树聚簇索引为中心。
- **设计哲学（要点栏：核心机制 + 关键取舍）**：
  - 机制：InnoDB 的 ACID 事务、MVCC 多版本并发控制、行级锁、重做日志（redo log）+ 回滚日志（undo log）+ 二进制日志（binlog）的三日志体系、基于成本的查询优化器、缓冲池（Buffer Pool）。
  - 取舍：把**数据正确性与持久性**放在第一位，几乎所有机制（WAL、doublewrite、两阶段提交、MVCC）都是为"不丢、不错、可并发"服务；代价是相对高的写开销、调优门槛、以及单机扩展的天花板。它不会为极致速度牺牲一致性。
- **典型场景**：交易系统、订单与账户、任何需要强一致与复杂查询的核心业务库。
- **一句话定位**：稳妥可靠的关系型数据持久化基石。

### 1.2.3 Kafka：把"日志"做成整个系统

- **出身**：2010 年起于 LinkedIn 内部，为解决海量活动日志的采集与流转而生，2011 年捐给 Apache。强调 Kafka 的"日志系统"基因决定了它今天的一切。
- **核心抽象**：**追加写日志（append-only log）**。Topic 被切成多个分区（partition），每个分区就是一条只追加、按位移（offset）寻址的有序日志。生产者写尾部，消费者各自维护位移读。这层抽象同时支撑了消息队列、事件流、数据管道、流处理。
- **设计哲学（要点栏：核心机制 + 关键取舍）**：
  - 机制：分区水平扩展；顺序磁盘 IO + 页缓存（page cache）+ 零拷贝（sendfile）榨取吞吐；副本（replica）+ ISR（同步副本集合）保障可靠；3.x KRaft 去掉 ZooKeeper 用共识协议自治理；生产者幂等与事务实现 exactly-once。
  - 取舍：用"只追加不修改 + 顺序 IO + 分区并行"换**单机百万级 TPS 与线性扩展**；代价是延迟高于内存型系统、不擅长随机查询、消息删除/更新需要靠压实（compaction）曲线救国。Kafka 把"吞吐与可扩展"摆在"灵活与低延迟"前面。
- **典型场景**：日志与事件采集、异步解耦、流处理管道、数据同步（CDC 下游）。
- **一句话定位**：建立在日志抽象之上的高吞吐分布式流平台。

## 1.3 架构视角 vs 使用视角（约 2000 字）

**这一节要讲清什么**：这是本章的思想核心。要让读者真切体会到"使用视角"和"架构视角"看的是同一个系统却得出完全不同的认识。核心方法是反复用"为什么这样设计 / 如果反过来会怎样"的追问，把读者从配置参数拉到设计决策层。这一节的好坏直接决定读者愿不愿意读完全书。

### 1.3.1 两种视角看到的同一个系统

- 用一个具体案例切入：同样一句"Redis 是单线程的"，使用者关心的是"所以我要避免在一个命令里跑 KEYS"；架构师关心的是"为什么单线程反而更快？多线程 IO 又是怎么嵌进来的？这套设计在什么规模下开始吃亏？"
- 列出两种视角的对照（配合 图1-3 表格）：
  - 使用视角：怎么做、配什么参数、踩过哪些坑、压到多少 QPS。
  - 架构视角：为什么这么做、换一种做法代价多大、这套取舍的边界在哪、这套思想能否迁移到我自己设计的系统。

### 1.3.2 架构视角的三把钥匙

- **第一把：看约束**。任何设计都是在特定约束下的取舍。要理解一个决策，先还原它面对的约束：目标场景、性能要求、可靠性要求、维护成本。例：Redis 选单线程，约束是"内存访问远快于网络，CPU 不是瓶颈"；MySQL 选 B+ 树，约束是"要支持范围查询和磁盘随机访问"。
- **第二把：看演化**。优秀的架构是被需求逼着演化出来的，不是一次设计到位。Redis 从单机到 Cluster、MySQL 从 MyISAM 到 InnoDB 再到 MGR、Kafka 从外部 ZooKeeper 到内置 KRaft。理解"为什么变"比记住"现在是什么"更有价值，因为约束会变，设计就得跟着变。
- **第三把：看抽象**。好的架构师善于抽象出问题的本质。Redis 把一切抽象成数据结构、MySQL 把数据抽象成页与索引、Kafka 把一切抽象成追加日志。**找到对的抽象，方案往往就自然浮现**——这是全书反复印证的一条主线。

### 1.3.3 用一个贯穿全书的例子把两种视角钉死

- 拿"数据写下去怎么保证不丢"这个后端最经典的问题，分别用两种视角回答：
  - 使用视角：Redis 开 AOF 并设 appendfsync always；MySQL 设 innodb_flush_log_at_trx_commit=1；Kafka 设 acks=all 且 min.insync.replicas=2。— 正确，但只是"配方"。
  - 架构视角：三家其实在回答同一个问题——**先写日志还是先改数据？日志落盘的粒度多细？宕机后怎么用日志恢复？** Redis 走 AOF 命令日志、MySQL 走 redo 物理日志 + 两阶段提交、Kafka 直接把日志当成数据本体。三种答案，一条共同主线（见第 4 章内存与磁盘、第 9 章数据同步）。
- 落脚金句（候选，待 N.6 / 第 10 章复用）：**"使用视角教你把车开起来，架构视角让你知道为什么雪天要换防滑链——也让你在设计自己的车时，知道该装哪条链。"**

## 1.4 如何读本书（约 1600 字）

**这一节要讲清什么**：给读者一份"使用说明书"——本书怎么组织、读者画像、版本基线、阅读路径、配套图表约定。让读者读完这一节就知道每一章会拿到什么、该按什么顺序读、遇到术语去哪查。

### 1.4.1 谁应该读，前置要求是什么

- **目标读者**：2 年以上后端开发者、需要做技术选型的架构师、想建立团队技术体系的技术负责人。
- **前置要求**：基本数据结构（哈希表、树、跳表）与算法、操作系统常识（进程/线程、内存、IO 模型）、至少实际用过 Redis/MySQL/Kafka 之一。不要求是内核专家。
- **不假设读者具备**：分布式一致性算法的完整推导、JVM/操作系统内核源码阅读经验——这些会在用到时补到"够用"。

### 1.4.2 本书的组织方式：主题式而非软件式

- 解释全书采用"一个架构主题 + 三家同题作答 + 横向对比 + 架构启示"的结构，而不是"先讲 Redis 整本、再讲 MySQL 整本"。
- 好处：避免知识孤岛、突出架构共性、训练对比分析能力（这正是技术选型的核心能力）。
- 配合 图1-4 给出全书章节地图（一张总览流程图，标出 8 个主题章与第 10 章总结）。

### 1.4.3 版本基线与图表约定

- **版本基线**：Redis 7.x、MySQL 8.0.x、Kafka 3.x。凡涉及具体行为/参数/默认值，一律以此为准；标注版本差异时显式说明。
- **图表约定**：图编号 `图 N-M`；优先 Mermaid（流程/时序）与 ASCII/表格（内存布局/文件格式）；每图必有正文解读。
- **术语约定**：全书统一译法（如重做日志、缓冲池、分区、副本），首次出现给英文原词。

### 1.4.4 推荐阅读路径

- **路径 A（系统建立）**：从第 1 章顺序读到第 10 章，建立完整架构认知。
- **路径 B（按需精读）**：先读第 1 章 + 第 10 章建立全局观，再按工作场景跳读主题章（如排查持久化问题直接看第 4 章；做高可用选型直接看第 7 章）。
- **路径 C（对比驱动）**：每章重点看 N.5 横向对比与 N.6 架构启示，再按兴趣回看具体实现。
- 通用建议：每章先读"本章导读"立问题、读完后看"小结"收束；把"架构启示"当金句反复品。

### 1.4.5 全书脉络一句话（结尾点题）

- 用一段话点出全书从"生与死（生命周期）"出发，经"内存与磁盘""分层架构""安全与权限""集群与高可用""存储格式""数据同步"，最终在第 10 章汇成"共性规律与取舍之道"。强调这条主线就是后端基础设施的核心骨架。
- 收束句：让读者带着"为什么这样设计"的追问，进入第 3 章。

## 本章三家横向对比的核心维度

> 引言章虽不做主题式深讲，但 1.2 节速写与 1.3 节贯穿例子仍需一组对比维度，为全书对比定下基调。本章对比落在"设计身份证"层面，不下沉到具体机制（那是各主题章的任务）。

1. **核心抽象**：数据结构服务器 / 关系 + SQL + 存储引擎 / 追加写日志。
2. **设计哲学 / 最在乎什么**：快与简单 / 不丢不错（强一致）/ 吞吐与可扩展。
3. **关键取舍 / 最不在乎什么**：内存容量与复杂计算 / 极致写性能 / 低延迟与灵活查询。
4. **性能取向**：低延迟（亚毫秒）/ 通用均衡（毫秒级、强一致）/ 高吞吐（百万 TPS）。
5. **持久化思路**：内存为主、RDB/AOF 补救 / 磁盘为主、WAL + 缓冲池 / 日志即数据、顺序写盘。
6. **演化主线**：单机 → 哨兵 → Cluster / MyISAM → InnoDB → MGR / 外部 ZK → 内置 KRaft。

## 写作难点与不确定处

1. **避免写成"三家功能介绍"**：1.2 节最容易滑成百度百科式罗列。必须紧扣"核心机制 + 关键取舍"，每家都要落到"它最在乎什么 / 最不在乎什么"，否则与使用手册无异。
2. **架构视角的论述要具体，不能空**：1.3 节是全章成败关键。"看约束 / 看演化 / 看抽象"三把钥匙必须每把都配一个三家的真实例子，否则变成正确的废话。贯穿例子（数据不丢）要在 200 字内让读者"啊原来如此"，难度较高。
3. **篇幅控制**：引言章不套三家深讲模板，但圣经仍要求全章 8000–12000 字。1.2 节三家速写各约 900 字、1.3 约 2000 字是主要承重墙，需保证信息密度而非注水。
4. **金句储备**：圣经要求每章有可被引用的箴言。本章作为开篇，金句需同时具备"立意高"与"可被后续章节复用"两点（候选已置于 1.3.3 与 1.4.5），定稿时需打磨。
5. **全书脉络预告的分寸**：1.4.5 / 图1-4 要预告后续 8 章，但不能剧透每章结论；只点"每章解决什么核心问题"，把悬念留给各章导读。需要与总调度确认最终章名与每章一句的核心问题（当前依据目录推断，**第 6 章"安全"与第 8 章"存储格式"的精确主题需总调度确认**）。
6. **术语首现规范**：本章会首次抛出 redo log / binlog / MVCC / WAL / ISR / KRaft 等大量术语，须严格按圣经"中文（英文）"标注，并在结构化摘要中回报新术语，由总调度补术语表。
