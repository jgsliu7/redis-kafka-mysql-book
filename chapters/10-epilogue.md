# 后记

九章走完，三款软件，八组问题，一种视角。

如果要在结尾留一句话，我会留一个问题：下次面对设计决策时，不妨先问：它在替谁解决什么约束、又替谁承担了什么代价？能把这两个问题答清楚，选型往往就不言自明。Redis 以简单为目标，MySQL 以严谨为目标，Kafka 以吞吐为目标，它们各自放弃了什么，比各自做到了什么更值得关注。

全书反复出现的几条线索，可以浓缩为五句话：先定真相之源，其余围绕它构建；用层把易变的东西隔离，让接口比实现活得更久；顺序写永远比随机写便宜，攒一批永远比一条条快；可靠性是要花钱买的，先看价签再决定买多少；故障是常态而非意外，把状态显式建模，恢复才可预测。这些规律不只属于这三款软件，它们在许多需要管理状态、权衡性能与可靠、并在分布式中维持一致性的系统中都适用。

从使用者到设计者，关键在养成一种习惯：遇到任何设计决策，先还原它面对的约束，再判断它的取舍是否值得。这本书如果能帮你养成这个习惯，就够了。

---

**一个真实的故事**

去年我在一个项目里犯了错。那个项目是一个活动库存系统，业务方说"要快"，我们就选了 Redis Cluster 做主存储。上线第一个月一切正常，响应时间在个位数毫秒，业务方很满意。

大促那天出了事。一个网络分区刚好把负责库存那个槽的主节点从集群里割了出去，Redis Cluster 自动选举了新主，但老主在被割出去的少数派分区里还在短暂地接受写入。网络恢复后，这部分写入被新主的数据覆盖了，库存多卖了。

事后复盘，团队所有人的第一反应是"Redis 丢了数据"。但 Redis 其实没出 bug，是我们把一个偏 AP（优先保证可用性）的系统用在了需要 CP（必须保证一致性）的场景上。库存是"账"，账本系统必须有一个在任何故障下都不被质疑的真相之源。Redis 不是那个真相之源，MySQL 才是。

我后来把库存系统改成了 MySQL 双 1 配置做真相之源，Redis 只做预扣减挡板（SETEX + INCR，挡掉大部分读流量）。架构是在你的约束下选你扛得住的代价。

这个故事在今天回头再看，全书九章其实都在说这一件事。

---

**下一步做什么**

这本书的价值不在读完。从自己维护的系统入手，先找出它的"真相之源"——是数据库、是日志、还是消息队列？然后问自己：如果这个真相之源丢了或坏了，哪些东西能重建、哪些东西就永久丢了？

接下来，挑一个天天在用但从来没动过的参数，把旋钮往反方向拧一次。`innodb_flush_log_at_trx_commit` 从 1 调到 2，吞吐能涨多少？`appendfsync` 从 everysec 调到 always，延迟会增加多少？读过十遍"可靠性是用性能买来的"，不如在自己的终端里看它发生一次。

在下一次技术评审时，试着先说约束再说方案。把"我们选 Kafka 因为它快"换成"我们的峰值写入是 XX MB/s、消费者可以接受 YY 秒延迟、数据允许丢不超过 ZZ 条，在这些约束下，Kafka 的追加日志+顺序写是合适的"。方案质量会在这些细节里体现出来。

花两周做这三件事，也许就能跨过使用者与设计者之间的分水岭。

剩下的，希望你在自己的系统里去印证。
