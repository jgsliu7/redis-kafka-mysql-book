# 后记

九章走完，三款软件，八组问题，一种视角。

如果要在结尾留一句话，大概不是一句结论，而是一个问题：下次面对一个设计决策，你能不能先放下"哪个更先进"，先问一句"它在替谁解决什么约束、又替谁背走了什么代价"？能把这两个问题答清楚，选型往往就不言自明。Redis 选了简单，MySQL 选了严谨，Kafka 选了吞吐——它们各自放弃了什么，比各自做到了什么更值得记住。

全书反复出现的几条线索，其实可以收得更短：

- 一切派生数据都有个"真相之源"，先定它，其余围绕它构建。
- 用层把容易变的东西关进笼子，让稳定的接口活得比实现更久。
- 顺序写永远比随机写便宜，攒一批永远比一条条快。
- 可靠性是要花钱买的，先看价签，再决定买多少。
- 故障不是意外，是常态；把状态显式建模，恢复才能可预测。

这些规律不只属于这三款软件。它们属于任何需要管理状态、权衡性能与可靠、并在分布式中维持一致性的系统。

从使用者到设计者，中间隔的不是更多的用法，而是一种习惯：遇到任何设计决策，先还原它面对的约束，再判断它的取舍是否值得。这本书如果能帮你养成这个习惯，它的使命就完成了。

**一个真实的故事**

去年我在一个项目里犯了错。那个项目是一个活动库存系统，业务方说"要快"，我们就选了 Redis Cluster 做主存储。上线第一个月一切正常——响应时间在个位数毫秒，业务方很满意。

大促那天出了事。一个网络分区刚好把负责库存那个槽的 master 从集群里割了出去，Redis Cluster 自动选举了新 master，但老 master 在被割出去的少数派分区里还在短暂地接受写入。网络恢复后，这部分写入被新 master 的数据覆盖了——库存多卖了。

事后复盘，团队所有人的第一反应是"Redis 丢了数据"。但实际上不是 Redis 的 bug——是我们把一个设计成 AP（优先保证可用性）的系统用在了需要 CP（必须保证一致性）的场景上。库存是"账"，账本系统必须有一个在任何故障下都不被质疑的真相之源。Redis 不是那个真相之源，MySQL 才是。

我后来把库存系统改成了 MySQL 双1配置做真相之源，Redis 只做预扣减挡板（SETEX + INCR，挡掉大部分流量读）。架构从来不是在最好的技术之间选，而是在你的约束下选一个代价你能承受的方案。

这个故事在今天回头再看，全书九章其实都在说这一件事。

**下一步做什么**

这本书的价值不在读完，而在读完之后的三个动作。

第一，回到你现在维护的系统，找出它的"真相之源"。是数据库、是日志、还是消息队列？然后问自己：如果这个真相之源丢了或坏了，哪些东西能重建、哪些东西就永久丢了？

第二，找出一个你一直默认使用的参数，把它调到另一端试一次——哪怕只是在测试环境。`innodb_flush_log_at_trx_commit` 从 1 调到 2，吞吐能涨多少？`appendfsync` 从 everysec 调到 always，延迟会增加多少？感受一次"可靠性是用性能买来的"这句话发生在自己的终端里，远比读十遍记得住。

第三，在下次技术评审时，试一次"先讲约束再讲方案"。不说"我们选 Kafka 因为它快"，而说"我们的峰值写入是 XX MB/s、消费者可以接受 YY 秒延迟、数据允许丢不超过 ZZ 条——在这些约束下，Kafka 的追加日志+顺序写是合适的"。你的评审通过率不会变低，但你的方案质量会明显提高。

这三件事花不了两周。但它们可能就是使用者与设计者之间的那道分水岭。

剩下的，留给你在自己的系统里去印证。
