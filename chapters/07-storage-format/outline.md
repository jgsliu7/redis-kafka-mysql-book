## 本章导读（约 200 字）

立问题：当数据从易失内存落到持久化磁盘，**文件结构本身就是一种架构决策**——它同时框定了软件的性能天花板、可靠性下限和演进自由度。本章不堆三种文件格式的字段表，而是追问同一个根问题：**给定不同的访问模式与可靠性诉求，磁盘上的字节该被组织成什么样？** 读完后，你能区分"快照 / 追加日志 / 定长页"三种范式各自服务的场景，能解释为什么 Redis 用变长紧凑编码、MySQL 死守 16KB 页、Kafka 把整批消息压进一个 Batch，并能在自己的系统里回答"我该选哪种格式骨架"。

## 7.1 问题的本质（约 900 字）

**这一节讲清什么**：把"磁盘存储格式"从一个模糊名词，拆成所有持久化系统都要面对的四个根本张力。不讲任何一家具体实现，先把问题场摊开，为 N.2–N.4 的三家深讲立靶子。

### 7.1.1 为什么磁盘值得单独一章
- 磁盘不是"慢一点的内存"：随机访问的寻道/页粒度、字节序、断电半写（partial write）、扇区原子性，都是内存世界没有的约束。
- 存储格式 = 把"内存里活的、可变的数据结构"翻译成"磁盘上死的、自描述的字节序列"的契约。一旦定型，向前/向后兼容就成了长期债务。

### 7.1.2 四个根本张力（每个张力一句话点破，作为后文反复回扣的主线）
1. **读写路径的不对称**：写要快（顺序追加）还是读要快（随机定位）？两者往往不能兼得。
2. **空间与时间的置换**：紧凑编码省 I/O 但吃 CPU；冗余/对齐省 CPU 但费空间。
3. **可靠性与性能的拉扯**：校验和、双写、落盘策略都在用性能换"断电后数据不坏"。
4. **定型与演进的矛盾**：格式一旦发布就无法收回，版本号、保留字段、升级路径是"未来的税"。

### 7.1.3 三家面对的共同问题与各自约束
- 共同问题：都要回答"基本 I/O 单元多大、写是追加还是原地改、怎么做崩溃恢复、怎么演进格式"。
- 各自约束（一句话，留钩子到 N.2–N.4）：Redis 内存才是家、磁盘是保险；MySQL 磁盘才是家、内存是缓存；Kafka 磁盘就是日志本体、读靠索引而非原地查找。

## 7.2 Redis 的做法（约 2400 字）

**这一节讲清什么**：Redis 把持久化当"内存数据的快照与回放工具"，而不是存储引擎。讲透 RDB 的紧凑二进制、AOF 的"协议即文件"、混合持久化的工程折中，重点抓"为什么 Redis 敢把磁盘格式做得这么简单"。

### 7.2.1 定位先行：磁盘服务于内存
- Redis 的真相数据始终在内存；RDB/AOF 的角色是"崩溃后把内存重建出来"，而非"线上查询的来源"。
- 这一定位直接决定了：格式可以紧凑、可以全量重写、不必支持随机改写——因为加载是整文件一次性灌入。

### 7.2.2 RDB：紧凑二进制快照
**核心机制**：
- 魔数 `REDIS` + 4 字节 ASCII 版本号（7.x 为 version 11）——文件自识别靠前缀，不靠扩展名。
- 长度字段用**变长整数编码**（6 位 / 14 位 / 32 位 / 64 位四档），按实际大小选最省字节的表示；小整数（0–127）直接塞进类型字节高两位，连长度字段都省了。
- 数据结构编码是**存储态编码**，可与内存态编码不同：内存里是 listpack 的 Hash，存盘时按紧凑二进制序列化，加载后按阈值重新决定编码。

**关键取舍**：
- 取"加载快、文件小"舍"可随机访问"——RDB 不支持按 key 定位，必须整体加载。
- listpack 取代 ziplist（7.x）：listpack 每个元素自记长度，消除了 ziplist 的连锁更新；这是"格式演进解决历史包袱"的范例（呼应 N.1.2 的演进张力）。

> 图 7-2（ascii-layout）：RDB 文件布局（magic / version / metadata / db-selector / entries / CRC64 footer）。

### 7.2.3 AOF：协议即文件
**核心机制**：
- AOF 记录的是"写命令本身"，文件内容就是 RESP 协议文本——**磁盘格式 = 线上协议**，这是 Redis 最聪明的设计决策之一。
- 收益：AOF 损坏可人工编辑修复、可用 redis-cli 直接回放、格式零额外定义成本。
- 代价：体积大、回放慢；为此引入 `fsync` 策略（always / everysec / no）在"最多少"与"多慢"之间切档。

**重写（rewrite）的关键取舍**：
- 重写**不解析旧 AOF**，而是 fork 子进程从当前内存直接生成新 AOF——绕开了"解析越积越大的日志"这个坑。
- 重写期间的新写命令进 **aof_rewrite_buffer**，重写完成后原子追加；这是"追加日志如何做在线 compaction"的经典手法，与第 8 章数据同步的日志压实遥相呼应。

### 7.2.4 混合持久化：工程实用主义的胜利
- 4.0+ 的混合格式：文件头是 RDB 全量快照，尾部接 AOF 增量命令。
- 设计意图：取 RDB 的"加载快"（大头走二进制快速灌入）+ AOF 的"少丢"（尾部增量精确回放）。
- 一句箴言味的话：**不追求理论纯粹，只追求组合后最优**——这是 Redis 全书反复出现的态度。

> 图 7-3（mermaid-flow）：AOF 重写 + 混合持久化的双缓冲协作时序。

## 7.3 MySQL 的做法（约 2500 字）

**这一节讲清什么**：InnoDB 把磁盘当数据的家，所有设计都围绕"定长页 + B+ 树 + 事务安全"。讲透表空间→段→页的三层结构、16KB 数据页内布局、Dynamic 行格式与溢出页、redo log 的循环日志与双写缓冲，重点抓"为什么死守 16KB 这个魔法数字"。

### 7.3.1 定位先行：磁盘是家，内存是缓存
- 与 Redis 相反，InnoDB 的真相数据在磁盘的表空间文件（共享 ibdata / 独立 .ibd）；缓冲池只是磁盘页的缓存副本。
- 这一倒置决定了：格式必须支持**页级随机读写**、必须**原地修改（in-place update）**、必须保证**页的原子性**。

### 7.3.2 三层骨架：表空间 → 段 → 页
**核心机制**：
- 表空间由连续的页（默认 16KB）组成，页号即文件内偏移的索引，O(1) 定位。
- 段（segment）是逻辑聚合：叶子段、非叶子段、回滚段——段的存在是为了给 B+ 树和回滚日志预留连续空间，减少随机分配的碎片。
- 同类型页用**前后指针串成双向链表**（在 FIL Header 里），空闲页、数据页各成一链——链表是页式存储做空间管理的通用工具。

**关键取舍**：
- 16KB 是预读效率（越大 I/O 越少）与缓冲池命中率（越小越少污染）的平衡点；选大还是选小没有标准答案，只有"对这台机器、这个负载"的答案。MySQL 允许配置但默认值 16KB 几乎不被改——这是"格式定型后难以撼动"的活教材。

### 7.3.3 数据页内布局：把一行行记录塞进 16KB
**核心机制**（自上而下七段）：
- FIL Header（38B，含页号、前后指针、LSN、页类型）→ Page Header（56B）→ Infimum/Supremum 虚拟记录（界定最小/最大）→ User Records（按主键有序）→ Free Space → Page Directory（稀疏槽）→ FIL Trailer（8B，LSN + 校验和）。
- **Page Directory 是稀疏索引**：每隔几条记录建一个槽，二分定位到槽再顺序扫——用很少空间把页内查找从 O(n) 降到 O(log n) + O(几条)。这是"在定长页内自己造个小索引"的微观范例。

**Dynamic 行格式的取舍**：
- 变长字段长度逆序存于记录头（从尾部倒着读，省一次正反向解析）。
- 大字段（VARCHAR/BLOB）超过页半时**溢出到独立溢出页**，行内只留指针；这是"页内紧凑"与"大对象不拖累整页"的折中。
- MySQL 8.0 对溢出页的版本/并发支持更高效，服务于在线 DDL——又一个"格式为演进留口子"的例子。

> 图 7-4（ascii-layout）：InnoDB 16KB 数据页内部七段布局。

### 7.3.4 redo log + 双写：让"原地改"不惧断电
**核心机制**：
- 原地改页的最大风险：断电导致一个 16KB 页只写了一半（partial write），页结构损坏且无法靠 redo 修复（redo 假设页本身完整）。
- **双写缓冲（doublewrite buffer）**：脏页先顺序写到共享表空间的连续双写区，再写到正式位置。恢复时若某页校验和不对，从双写区取完整副本——用"多写一份"换"页永远完整"。
- **redo log** 是固定大小的循环日志，512B 块对齐（匹配扇区原子写粒度）；先写 redo（WAL）再改页，崩溃后用 redo 把页重做到一致状态。
- 8.0 支持原子写设备时可关掉双写，减少写放大——"硬件能力上来了，软件冗余可以撤"。

**关键取舍**：redo 物理到页、逻辑到操作（物理逻辑混合日志），在"日志体积"与"恢复时能否幂等重做"之间取折中。

> 图 7-5（mermaid-flow）：脏页落盘流程——双写缓冲 → 正式页，配合 redo log 的崩溃恢复路径。

## 7.4 Kafka 的做法（约 2500 字）

**这一节讲清什么**：Kafka 把磁盘当作不可变的追加日志本体，所有设计服务于"顺序写 + 稀疏索引随机读"。讲透 Log Segment 三件套、Record Batch V2 的增量编码、偏移量/时间戳双索引、消息格式 V0→V1→V2 的演进，重点抓"为什么把一批消息当成一个存储单元是革命性的"。

### 7.4.1 定位先行：磁盘就是日志本体
- Kafka 的消息**只有一份真相**，就躺在磁盘的日志段里；不像 MySQL 还有个缓冲池当缓存层（页缓存由 OS 管，Kafka 不自己管）。
- 这意味着：格式必须支持**纯追加写**（永不改写旧字节）、**按位移随机读**（靠索引而非扫描）、**高效压缩整批**。

### 7.4.2 Log Segment：三件套
**核心机制**：
- 一个分区 = 一串 Log Segment；每个 Segment 三个文件：`.log`（消息数据）、`.index`（位移索引）、`.timeindex`（时间戳索引）。
- 文件名是该 Segment 的起始位移——文件名本身就是一个索引锚点，按位移定位时先按文件名二分找到 Segment，再进 `.index`。
- 这是"用文件系统目录结构免费做了一层稀疏索引"的妙手。

**关键取舍**：Segment 切分粒度（按大小或时间）影响"压实（compaction）成本"与"消费者按时间回溯的精度"——与第 8 章数据同步的 retention 直接相关。

> 图 7-6（ascii-layout）：分区 → Segment → 三件套文件的目录与内部结构。

### 7.4.3 Record Batch V2：把一批消息当成一个存储单元
**核心机制**：
- V2 格式以 **Batch** 为最小存储单元，而非单条消息。Batch 头（约 61B）存 BaseOffset、PartitionLeaderEpoch、ProducerId/Epoch（幂等与事务）、CRC、Attributes（压缩算法标记）等**整批共享的元数据**，避免每条消息重复。
- Batch 内每条 Record 只存**相对量**：时间戳差值（相对 Batch 的 FirstTimestamp）、位移差值（相对 BaseOffset）。绝大多数差值 1–2 字节就够，单条消息的元数据从 V1 的几十字节压到个位数。

**关键取舍**：
- 收益：压缩在 Batch 边界做，压缩比从 V1 的约 5:1 提升到 V2 的约 10:1（相近消息聚在一起更易压）。
- 代价：消费者要解整个 Batch 才能读单条——但消息消费本来就是成批的，这个代价被工作模式天然吸收。**让存储单元对齐访问单元**，是 Kafka 性能的根本之一。
- ProducerId/Epoch/BaseSequence 进了格式本身：**幂等与事务是存储格式问题，不只是协议问题**——这点常被忽略，是本章的洞察之一。

### 7.4.4 双索引：位移索引 + 时间戳索引
**核心机制**：
- 位移索引条目 8B（相对位移 4B + 物理位置 4B），**稀疏存储**（每写入 `log.index.interval.bytes` 默认 4KB 才落一个条目）。查找时二分定位到 ≤ 目标的最大条目，再进 `.log` 顺序扫几 KB——典型的"粗索引 + 顺序扫"组合。
- 时间戳索引条目 12B（时间戳 8B + 相对位移 4B），服务于按时间消费（如"从昨天 10 点开始消费"）。

**关键取舍**：稀疏而非稠密——用"少量顺序扫描"换"索引体积小到能常驻内存"。这与 MySQL 的 Page Directory（页内稀疏槽）是同一种思想在不同尺度上的复现。

### 7.4.5 格式演进 V0→V1→V2
- V0：原始单消息格式；V1：加时间戳（每条都带，冗余）；V2：Batch + 增量编码 + 幂等字段。
- 演进靠 **Magic 字段**版本号 + 向前/向后兼容策略——老 Broker 不认识新 Magic 就拒绝，消费者按 Magic 选解析路径。
- 一句话总结演进逻辑：**每代格式都在为上一代最贵的操作买单**（V1 为"想知道时间得扫全文件"买单，V2 为"每条消息元数据冗余 + 压缩低效"买单）。

## 7.5 横向对比（约 1800 字，不可缩水）

**比什么**：三家在同一个根问题——"磁盘上的字节怎么组织"——上给出的三种范式，以及每种范式背后的代价。

### 7.5.1 三栏对比表（维度作行，顺序 Redis | MySQL | Kafka）

| 维度 | Redis | MySQL | Kafka |
|------|-------|-------|-------|
| 数据的"家" | 内存 | 磁盘 | 磁盘（日志本体） |
| 基本 I/O 单元 | 单条命令/整个文件 | 16KB 定长页 | Record Batch（变长） |
| 写方式 | 全量重写（RDB）/ 追加（AOF） | 原地改页 + WAL | 纯追加，永不改写 |
| 读方式 | 加载到内存后随机访问 | B+ 树随机定位 | 稀疏索引 + 顺序扫描 |
| 索引形态 | 无（内存里现算） | B+ 树 + 页内 Page Directory | 稀疏位移/时间戳索引 |
| 崩溃恢复 | RDB 全量 + AOF 增量回放 | redo log 重做 + 双写修页 | 日志即真相，重放即可 |
| 演进机制 | RDB 版本号 + 编码升级 | 页类型/格式位 + 在线 DDL | Magic 版本号 |
| 校验 | RDB 尾部 CRC64 | 页校验和 + 双写 | Batch 级 CRC |

### 7.5.2 解读：三种范式各为谁优化
- **快照范式（Redis）**为"快速整体恢复"优化，代价是不能在线随机查询磁盘。
- **页式范式（MySQL）**为"事务安全的随机读写"优化，代价是双写/redo 的写放大与定长页的空间浪费。
- **追加日志范式（Kafka）**为"极高吞吐的顺序写 + 成批压缩"优化，代价是不能原地改写（要靠 compaction/retention 间接处理）。

### 7.5.3 为什么三家选择如此不同
- 不是技术品味差异，是**核心场景驱动**：Redis 的场景是"内存数据别丢"；MySQL 的场景是"线上事务随时读写"；Kafka 的场景是"海量消息流式进出"。
- 一条规律：**存储格式是访问模式的镜像**。先确定"数据怎么被读、怎么被写、读多还是写多、要不要随机定位"，格式骨架几乎是被这些约束推着长出来的。

## 7.6 架构启示（约 1800 字，重点不可缩水）

**这一节是本章精华**：从三家具体实现里提炼可复用的设计原则与权衡规律，每条都尽量给一句可被独立引用的箴言。

### 7.6.1 原则一：存储单元要对齐访问单元
- Kafka 的 Batch = 消费单元；MySQL 的页 = 预读/I/O 单元；Redis 的 AOF = 协议回放单元。
- 箴言候选：**让磁盘上的最小存储单元，等于业务上的最小访问单元，能省掉一个数量级的无效 I/O。**

### 7.6.2 原则二：定长 vs 变长是写读路径的根本选择
- 定长（MySQL 页）利于随机定位与原地改，代价是空间浪费与对齐开销。
- 变长（Redis 编码 / Kafka Batch）利于紧凑与压缩，代价是不能原地改、需 compaction/重写。
- 选择定长还是变长，本质上是在回答"我会不会原地改这块数据"。

### 7.6.3 原则三：索引越稀疏，越要靠顺序扫描兜底
- 三家都用了"稀疏索引 + 局部顺序扫描"：MySQL 页内 Page Directory、Kafka 位移索引、甚至 Redis 加载后内存里的渐进式结构。
- 箴言候选：**稀疏索引不是偷懒，是用"少量顺序扫描"换"索引小到能常驻内存"。**

### 7.6.4 原则四：可靠性要用性能去买，但要知道买的是什么
- 双写缓冲买的是"页完整性"，redo 买的是"改动能重放"，CRC 买的是"损坏可检测"。
- 每加一层可靠性都对应一类故障；撤掉它的前提是"那类故障已被下层（如原子写硬件）消灭"。

### 7.6.5 原则五：格式一旦发布就是长期债务
- 版本号、保留字段、Magic、编码切换（ziplist→listpack、V1→V2）都是为未来留的口子。
- 箴言候选：**存储格式要为"五年后还要读今天的文件"设计，而不是为"今天能写就行"设计。**

### 7.6.6 给实践者的三个判断题
面对自己的存储设计，先回答：
1. 我的数据会被原地改吗？（决定定长页还是追加日志）
2. 我最怕丢一小段还是怕全量重建慢？（决定快照 + 增量日志的组合）
3. 我的格式五年后还能被新代码读懂吗？（决定版本与兼容性投资）

> 图 7-9（mermaid-flow）：存储格式设计决策树——从访问模式推到格式骨架。

## 7.7 小结（约 300 字）

3–5 句收束，呼应导读。三种范式的差异不是技术品味，而是访问模式的镜像；Redis 的快照、MySQL 的页、Kafka 的日志，各自把"为谁优化"做到了极致。存储格式是长期债务也是性能杠杆，选对骨架等于给系统定了天花板。承接第 6 章集群视角，引出第 8 章数据同步——格式决定了"什么能被增量同步、什么必须全量重建"。
