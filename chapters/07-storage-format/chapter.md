# 第7章 磁盘存储格式

数据是软件的灵魂，而存储格式则是灵魂的载体。当我们将数据从易失性的内存写入持久化的磁盘时，如何设计存储格式决定了软件的性能、可靠性和可维护性。本章将深入剖析 Redis、MySQL 和 Kafka 三款软件的磁盘存储设计，探索不同场景下存储格式设计的艺术。

## 7.1 存储格式设计的核心考量

存储格式设计需要在多个维度之间寻求平衡。读写效率是首要考量，顺序写入能够充分利用磁盘的预读特性，而随机写入则受限于磁头寻道时间。合理的压缩可以减少磁盘 I/O，但会增加 CPU 开销；高效的索引结构能让数据查找从 O(n) 降到 O(log n) 甚至 O(1)；按页对齐可以减少磁盘读写次数，但可能造成空间浪费。

兼容性同样至关重要。软件在不断演进，存储格式需要具备版本标识能力，让新版本软件能够识别旧格式，旧版本软件能够安全处理未知格式。向前兼容保证新版本能读取旧版本的数据，向后兼容确保旧版本至少不会因新版本数据而崩溃。此外，还需要提供平滑的格式升级路径。

故障恢复能力则是存储系统的生命线。校验和用于检测数据损坏，事务日志支持崩溃恢复和点对点复制，检查点提供快速恢复的锚点，幂等性保证重复应用日志不会产生副作用。这三方面共同构成了存储格式设计的核心框架。

## 7.2 Redis：内存优先的存储艺术

Redis 的设计哲学是"内存优先"。作为内存数据库，它的持久化机制是服务于内存数据的保护和恢复的，而非核心存储引擎。这种定位决定了其存储格式的独特性。

### RDB 二进制快照格式

RDB（Redis Database）是 Redis 的快照持久化格式，它将某一时刻的内存数据以紧凑的二进制格式保存到磁盘。RDB 文件以固定的魔数字符串 "REDIS" 开头，后跟 4 字节的版本号（ASCII 编码），这种设计让文件类型识别变得简单可靠。Redis 7.x 使用版本 9 的格式。

Redis 根据数据类型和大小采用不同的编码策略。对于常用小整数（0-127），Redis 7.x 采用特殊编码，直接用类型字节的高两位标识，省去单独的长度字段。长度编码采用变长整数设计：6位长度直接编码，14位长度用两个字节，更大的长度则使用 32 或 64 位整数。

对于复杂数据结构，RDB 采用紧凑的二进制表示。Hash 表可以使用 ZIPLIST 或 HT 编码，ZIPLIST 结构包含总字节数、尾部偏移、节点数量和结束标记。当元素数量或大小超过阈值时，转换为标准 Hash Table。Redis 7.x 引入了 listpack 替代 ziplist，解决了 ziplist 的连锁更新问题。listpack 采用每个元素独立存储长度的设计，修改一个元素不会影响其他元素的偏移计算。

### AOF 日志格式

AOF（Append-Only File）以日志形式记录所有写操作，是 Redis 的另一种持久化方式。AOF 文件本质上是 Redis 协议的文本表示，这种设计的精妙之处在于 AOF 文件本身就是合法的 Redis 协议，可以用 redis-cli 直接加载。

随着运行时间增长，AOF 文件会膨胀。Redis 提供 BGREWRITEAOF 命令进行重写，重写过程不解析原 AOF，而是从内存数据直接生成。重写期间的新写入会被缓冲到内存，重写完成后追加到新文件，保证数据完整性。

### 混合持久化

Redis 4.0 引入的混合持久化结合了 RDB 的快速加载和 AOF 的细粒度恢复。文件以 RDB 格式前缀存储全量数据快照，后跟 AOF 格式的增量日志。加载时先以 RDB 方式加载全量数据（快速），再以 AOF 方式执行剩余命令（精确）。这种设计体现了工程上的实用主义：不追求理论的纯粹，而是取各家之长。

## 7.3 MySQL：事务安全的页式存储

MySQL 的 InnoDB 存储引擎采用完全不同的设计哲学：磁盘是数据的最终归宿，内存只是缓存。这种设计决定了其存储格式必须以磁盘为中心，同时保证事务安全。

### 表空间文件结构

InnoDB 使用表空间（Tablespace）管理存储，可以是共享的（ibdata1）或独立的（.ibd）。表空间文件包含空间头部、段和页面三个层次。空间头部记录空间 ID、页面大小（默认16KB）、空闲页面链表和碎片页面管理。段包括叶子节点段、非叶子节点段和回滚段。页面则分为数据页、索引页、Undo 页和系统页。

页面头结构共 38 字节，包含校验和、页号、前一页和后一页指针（将同类型页面组织成双向链表）、最后修改的 LSN、页面类型、刷盘 LSN 高位和表空间 ID。

### 数据页格式

数据页（Data Page）是 InnoDB 存储的基本单位，大小固定为 16KB。页面布局从上到下依次是：FIL Header（38 字节）、Page Header（56 字节）、Infimum Record、Supremum Record、User Records（行记录，按主键有序）、Free Space、Page Directory（稀疏索引）和 FIL Trailer（8 字节，包含 LSN 和校验和）。

MySQL 8.0 默认使用 Dynamic 行格式。行记录头信息 5 字节起步，包含标志位、字段数量、变长字段长度列表字节数和 NULL 值位图。对于 VARCHAR、BLOB 等变长字段，InnoDB 在记录头逆序存储长度列表。当字段超过页面一半大小时，数据存储到溢出页，溢出页指针包含表空间 ID、页号、页内偏移和版本号。MySQL 8.0 对溢出页的版本控制支持更高效的并发访问和在线 DDL。

### Redo Log 格式

Redo Log 是 InnoDB 实现崩溃恢复的核心，采用固定大小的循环写入设计。日志块 512 字节对齐，块头包含块编号、数据长度、首组偏移和检查点号。InnoDB 支持多种物理和逻辑日志类型，包括物理写入、逻辑插入、压缩表插入和 Undo 记录。标准记录插入的日志包含类型、SpaceID、PageNo、Offset 和 Data。

为了防止部分页面写入（Partial Write），InnoDB 引入双写机制。写入流程先将内存中脏页写入双写缓冲区（顺序写），再从双写缓冲区写入实际数据文件（随机写）。恢复时先检查页面校验和，若损坏则从双写缓冲区恢复，若双写也损坏则尝试从 redo log 恢复。MySQL 8.0 支持原子写的存储设备可以禁用双写缓冲，减少写放大。

## 7.4 Kafka：顺序写的日志之美

Kafka 的设计哲学是将顺序写的性能发挥到极致。作为消息队列，Kafka 的存储格式完全围绕日志追加这一核心操作设计。

### Log Segment 结构

Kafka 的每个分区（Partition）由一系列 Log Segment 组成，每个 Segment 包含三个文件：消息数据文件（.log）、偏移量索引（.index）和时间戳索引（.timeindex）。文件名的数字表示该 Segment 的起始偏移量。

Kafka 3.x 使用 Record Batch 格式，批量存储消息以提高效率。Batch Header 共 61 字节，包含 BaseOffset、BatchLength、PartitionLeaderEpoch、Magic、CRC、Attributes、LastOffsetDelta、FirstTimestamp、MaxTimestamp、ProducerId、ProducerEpoch、BaseSequence 和 RecordsCount。

单个 Record 的结构包括记录总长度（变长整数）、属性位、时间戳差值（相对 Batch）、偏移量差值、Key 长度、Key 数据、Value 长度、Value 数据、头数量及 Headers。Kafka 3.x 在 Batch 内采用增量编码，时间戳和偏移量可以用更少的字节表示，通常 1-2 字节即可。

### 索引文件设计

索引是 Kafka 支持随机读取的关键，同时保证顺序写的特性不被破坏。

偏移量索引条目每条约 8 字节，包含相对 Segment 起始的偏移量和对应记录在 .log 中的物理位置。索引采用稀疏索引设计，每隔 log.index.interval.bytes（默认 4KB）写入一个条目。查找偏移量时先二分查找索引，找到小于等于目标偏移量的最大条目，然后从对应位置开始顺序扫描 .log 文件。

Kafka 0.10.0 引入时间戳索引，支持按时间查询。时间戳索引条目每条约 12 字节，包含时间戳（毫秒）和相对偏移量，同样采用稀疏存储，支持按时间范围消费消息。

### 消息格式的演进

Kafka 消息格式经历了多次重大改进。V0 是原始格式，简单的消息存储；V1 添加时间戳，支持消息时间，但每条消息都有时间戳字段；V2 引入 Record Batch，支持批量压缩、增量编码和幂等性。V2 格式的优势在于批量压缩效率更高，典型压缩比可达 10:1，而 V1 仅约 5:1。

## 7.5 对比分析：为何选择不同的存储格式？

三款软件的存储格式差异反映了它们不同的设计目标和运行环境。

Redis 的主要用途是内存缓存和数据结构，追求最终一致性，访问模式为随机读写，数据规模在内存级（GB）。MySQL 是事务型数据库，要求强一致性（ACID），访问模式也是随机读写，数据规模在磁盘级（TB）。Kafka 是流式消息队列，追求顺序一致性，顺序写、随机读的访问模式，数据规模在磁盘级（PB）。

Redis 选择紧凑二进制格式的原因是内存数据库需要快速加载，RDB 的紧凑格式减少磁盘 I/O 时间，AOF 的文本协议保证可读性和可修复性。MySQL 选择固定页格式是因为事务安全要求页面原子写入，B+树索引需要固定大小的页面管理，行格式支持复杂的表结构和事务语义。Kafka 选择追加日志格式是因为消息队列的写操作天然是追加，顺序写性能是设计的核心，稀疏索引在顺序存储上非常高效。

尽管形式不同，三款软件的存储格式都体现了共同的设计智慧：校验和机制（Redis 的 RDB、MySQL 的页面、Kafka 的 Batch 都包含 CRC 校验）、版本控制（所有格式都包含版本信息，支持平滑升级）、紧凑编码（变长整数、差值编码等技术被广泛使用）、分层设计（存储格式与上层逻辑解耦，支持格式演进）。

## 7.6 架构启示：存储格式如何服务于软件特性

存储格式不是孤立的技术细节，而是软件架构的有机组成部分。存储格式决定了软件的性能天花板：Redis 的 RDB 全量快照让重启时间可控（分钟级），AOF 的增量日志让实时持久化成为可能；MySQL 的 16KB 页面大小是 I/O 效率（预读）和空间利用率（碎片）的平衡点；Kafka 的 Batch 设计让压缩和传输效率达到极致，支持百万级 TPS。

不同的可靠性需求催生不同的格式设计。Redis 的混合持久化在"快速恢复"和"数据不丢"之间找平衡；MySQL 的双写缓冲加 Redo Log 的组合，将数据损坏的概率降到极低；Kafka 的多副本加 ISR 机制，配合幂等性生产者，实现精确一次语义。

对于软件架构师而言，存储格式设计需要回答以下问题：访问模式是什么（顺序写适合日志格式，随机读写适合页式格式）？一致性要求多高（ACID 需要复杂的事务日志，最终一致可以简化设计）？数据规模多大（大数据量需要高效的压缩和索引，小数据量可以追求简单）？演进如何保障（版本号、兼容性字段是长远考虑的必要投资）？

存储格式的设计是一门权衡的艺术。Redis、MySQL 和 Kafka 给出了三种不同的答案，但它们的共同点在于：深刻理解自己的核心场景，不为通用性牺牲关键性能。这种"有所为，有所不为"的设计哲学，正是优秀软件的共同特质。
