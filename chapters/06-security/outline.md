## 本章导读（约 220 字）

**要解决的问题**：一个数据系统被攻击者拿到，损失的不是一台机器，而是全公司的数据。安全机制要回答四件事——"你是谁（认证）""你能做什么（授权）""数据在路上/落盘时是否被窃听或篡改（加密）""事后能不能查清谁干了什么（审计）"。

**为什么重要**：本章把这三个定位迥异的软件（Redis 重性能、MySQL 重企业级完整、Kafka 重分布式协作）放进同一张安全分析表里。它们对同一个问题的取舍截然不同：Redis 长期把安全让位给网络隔离；MySQL 一开始就把权限下沉到列级；Kafka 因为天生多跳（客户端 → Broker → 副本 → ZooKeeper/KRaft），必须在每一跳都重新谈一次身份。

**读完能获得什么**：建立"认证-授权-加密-审计"四维分析框架；理解三家的安全模型为何如此分叉；提炼出"默认安全策略要与部署环境匹配""最小权限不是口号而是可操作规则""纵深防御的代价是每层都收一点性能税"等可被引用的设计箴言。

## 5.1 问题的本质（约 550 字）

**本节目标**：先剥离三家差异，把"数据系统的安全"抽象成一个共性问题，再点出三家各自的约束，为后文深讲立靶。

**核心要点（要点栏）**：
- **安全的四个维度**：认证（Authentication，"你是谁"）、授权（Authorization，"你能做什么"）、加密（Encryption，传输 + 存储的机密性与完整性）、审计（Audit，事后可追溯）。这四个维度构成正文反复引用的分析骨架。
- **数据库/中间件特有的安全约束**（区别于 Web 应用）：
  1. **长连接 + 高频小操作**——每次操作都做完整认证/授权检查会吃掉性能，于是三家都在"会话级认证 + 每操作级授权"上做缓存与简化。
  2. **多租户共享存储**——不同应用读写同一实例的不同键/表/主题，需要细粒度的资源隔离。
  3. **复制与分布式协调**——主从之间、Broker 之间、Broker 与 KRaft 之间也要互相认证，身份不止"客户端 vs 服务端"一种。
- **三家的先天约束差异**（一句话铺垫，详细对比留到 5.5）：
  - Redis：单线程（6.0 前）/ 极简内核，安全特性长期被"性能优先 + 内网部署"的假设推迟。
  - MySQL：面向企业，权限模型从第一天就追求"列级 + 多层级 + 可审计"。
  - Kafka：天生分布式、多跳通信，安全设计必须可组合（SASL 框架 + 监听器分层）。

## 5.2 Redis 的做法（约 2000 字）

**本节目标**：讲清 Redis 如何从"裸奔"演进到 6.0 ACL，重点剖析 ACL 的规则语法为什么这么设计、TLS 的性能代价、审计为何长期是短板。

**核心要点（要点栏）**：

- **认证：从全局密码到 ACL 用户**
  - 6.0 前只有一个 `requirepass` 全局密码，所有人共享，无法区分身份——这是"安全让位给极简内核"的典型决策。
  - 6.0 引入 ACL：用户 + 多密码 + 命令白/黑名单 + 键模式。规则语法 `ACL SETUSER alice on >pwd ~keys:* +get +set` 的每一项都是"最小可表达单元"的设计取舍——为什么不搞 RBAC 角色？因为 Redis 内核要轻，把"组角色"这类复杂语义留给外部 IAM。
  - 密码用 SHA-256 存储（`ACL FILE` 持久化），不存明文；这是"凭证落盘"的最小安全要求。

- **授权：键模式 + 命令类别的二维控制**
  - 关键取舍：权限直接绑到"用户"，不做角色继承。理由——Redis 用户数通常很少（几台应用机器），角色层是过度设计。
  - 命令类别（`@read @write @admin @dangerous @fast @slow`）是"安全便利性"的关键抽象：把上百条命令折叠成几个语义组，运维一眼能配出"只读 + 不含危险命令"的用户。
  - 通道（channel，6.2+）权限单独控制，Pub/Sub 与键空间权限分离——避免一个客户端既能写又能订阅敏感事件。
  - 默认拒绝（`default` 用户默认 `nopass` + `off`，新版倾向 `resetpass`），保护模式（`protected-mode`）在监听到外网访问且无密码时拒绝连接。

- **加密：TLS 是 6.0 才补上的课**
  - 6.0 才原生支持 TLS，之前的"安全靠 stunnel 或 SSH 隧道"是性能/复杂度的妥协。
  - 关键取舍：Redis 追求单核十万级 QPS，TLS 握手开销与每包加解密直接拉低吞吐（实测单线程 TLS 1.2 约降至无加密的 60%）。Redis 选择"TLS 可选、按端口开启"，而不是强制——把决策权留给部署者。
  - 配置要点：关明文端口、限定 `tls-protocols TLSv1.2 TLSv1.3`、客户端证书可选。

- **审计：长期短板，靠 ACL LOG + SLOWLOG 凑合**
  - 关键取舍：审计要落盘、要写每条操作，与 Redis 的"内存即性能"哲学冲突，所以官方只提供轻量记录（`ACL LOG` 记认证失败/拒绝；`SLOWLOG` 记慢命令），不做完整操作审计。
  - `MONITOR` 命令能看全量命令流，但官方明确警告会显著降性能，仅限排障——这是"全量审计与性能不可兼得"的明证。

- **设计取舍小结**：Redis 的安全哲学是"内网优先 + 按需叠加"。它默认假设部署在可信网络，把安全能力做成可选模块，代价是公网部署时极易因配置疏漏被攻击（历史上未授权 Redis 写文件导致 RCE 的案例多到成梗）。

**预估字数**：2000 字

## 5.3 MySQL 的做法（约 2000 字）

**本节目标**：讲清 MySQL 如何把权限下沉到列级、角色系统为何 8.0 才补齐、TDE 与审计插件的架构，重点在"层级权限表 + 可插拔认证"这两个设计决策。

**核心要点（要点栏）**：

- **认证：可插拔认证插件（pluggable authentication）是核心抽象**
  - 用户身份 = `用户名@主机名`，主机名是身份的一部分——这让 MySQL 天生支持"同一个账号在不同来源有不同密码/权限"，是企业级多来源接入的现实需求。
  - 8.0 默认改用 `caching_sha2_password`：首次连接走完整挑战-响应（SHA-256），之后服务端缓存哈希走快路径。关键取舍——缓存换性能，但缓存失效（改密码/重启）要重新走慢路径，这是"长连接高频认证"场景的标准解法。
  - 可插拔意味着认证逻辑与协议解耦：`auth_socket`（本机免密）、`authentication_ldap_simple`/`authentication_pam`（对接企业目录）、`sha256_password`（高安全无缓存）。这是"把身份验证外包给最合适的子系统"的设计思想。
  - TLS 协商嵌在握手包里，可选 `REQUIRE SSL` / `REQUIRE X509` / 指定证书主题，做到"账号级强制加密"。

- **授权：多层级权限表 + 8.0 RBAC**
  - 权限按作用域分五层：全局 / 数据库 / 表 / 列 / 存储程序。这套层级不是装饰——它对应 `mysql.user / mysql.db / mysql.tables_priv / mysql.columns_priv` 多张表，授权检查时从全局到列逐层收窄，命中即停。关键取舍：层级越细检查越慢，所以 MySQL 把检查结果按"账号+库表"做内存缓存（`global_priv`/权限 bitmap）。
  - 8.0 才补齐 RBAC 角色系统（`CREATE ROLE` / `GRANT role TO user` / 会话级 `SET ROLE`）。为什么这么晚？因为 MySQL 用户量级大、权限矩阵复杂，角色引入要兼顾"不破坏已有 GRANT 语义"，所以采用"角色 = 一组权限的命名集合 + 会话激活"的轻量实现，而不是推翻权限表。
  - 权限语义分四类：数据操作（DML：SELECT/INSERT/UPDATE/DELETE）、结构变更（DDL：CREATE/ALTER/DROP，更危险）、管理（SUPER/PROCESS/FILE/RELOAD，高度敏感）、复制（REPLICATION SLAVE/CLIENT，专用于拓扑）。这种"按风险分桶"的命名约定本身就是安全设计。

- **加密：传输 TLS + 存储 TDE 两层**
  - 传输层 TLS 配置与服务端证书、CA、客户端证书验证同 Redis/Kafka，无特别处。
  - 存储加密（TDE，Transparent Data Encryption）：InnoDB 表空间级加密，密钥由密钥管理插件（如 HashiCorp Vault KMS 插件）托管。关键取舍——加密对应用透明（不改 SQL），但密钥管理变成新的单点（密钥丢失=数据不可读），所以 TDE 必须配密钥轮换与备份策略。
  - redo log / undo log / binlog 也可加密（8.0.15+），避免"表加密了但日志明文"的漏洞——这是纵深防御在存储层的体现。

- **审计：企业版插件 + 开源社区方案**
  - 关键取舍：官方企业审计是商业插件（按规则过滤 + 滚动文件），开源方案（McAfee mysql-audit、Percona Audit Plugin）靠拦截查询。审计必然有 I/O 与序列化开销，所以 MySQL 提供"按规则过滤"（只审计特定用户/特定 SQL）来降成本。
  - General Query Log 记录全部 SQL，但官方明确不建议生产常开——再次印证"全量审计与性能不可兼得"。

- **设计取舍小结**：MySQL 的安全哲学是"企业级默认安全 + 分层完整"。它默认装好就要求强 root 密码、可删匿名用户、可启用连接失败锁定（`connection_control` 插件），把安全作为一等公民，代价是配置项多、学习曲线陡。

**预估字数**：2000 字

## 5.4 Kafka 的做法（约 2000 字）

**本节目标**：讲清 Kafka 为何把认证拆成 SASL 框架、ACL 如何绑定资源类型、监听器分层如何让"内部 SASL_SSL + 外部不同端口"共存，重点在"多跳身份 + 可组合安全"这两个分布式特有问题。

**核心要点（要点栏）**：

- **认证：SASL 框架 + JAAS 配置，把认证机制变成可插拔模块**
  - SASL（Simple Authentication and Security Layer）是抽象层，下面挂 PLAIN / SCRAM-SHA-256|512 / GSSAPI（Kerberos）/ OAUTHBEARER（OAuth 2.0）多种机制。关键取舍——Kafka 不自造认证协议，复用成熟的 SASL/JAAS 生态，让它能无缝接入企业 Kerberos 与云原生 OAuth。
  - SCRAM 是推荐生产方案：挑战-响应、凭证只存服务端（不像 PLAIN 客户端要持有明文）。关键取舍——SCRAM 凭证存 ZooKeeper/KRaft（元数据存储），意味着元数据存储本身的安全成了新的信任根。
  - 委托令牌（Delegation Token）：短期令牌，避免长期凭证在客户端与 Broker 间频繁传递。这是分布式"身份传递"问题的标准解——后面架构启示要展开。
  - 监听器分层（`listeners` + `listener.security.protocol.map`）：同一 Broker 可同时开 SASL_SSL（内部，强制加密+认证）、SASL_PLAINTEXT（受信网络）、SSL（仅加密）多个监听器。这是"按网络域分级信任"的设计。

- **授权：ACL 绑定"Principal + Operation + Resource"三元组**
  - 资源类型：Topic / Group / Cluster / TransactionalId / DelegationToken。每类资源有自己的操作集（Read/Write/Create/Delete/Alter/Describe/ClusterAction/All）。
  - 关键取舍：Kafka 原生只有 ACL，没有 RBAC。理由——流平台的主语通常是"应用/服务"而非"人"，服务身份（Principal）相对稳定，角色层收益不大；企业要 RBAC 时通常外接 Ranger/Sentry。
  - 前缀授权（`--resource-pattern-type prefixed`）让"给 app-a 这个服务所有 app-a.* 主题的读写"一行配置完成，避免逐主题授权——这是为多租户场景设计的便利性。
  - `allow.everyone.if.no.acl.found` 默认应设 `false`：无 ACL 即拒绝，这是"默认拒绝"原则在 Kafka 上的落地，但默认值历史上是 true（向后兼容的代价），生产必须显式改。

- **加密：TLS 覆盖所有通信路径**
  - Kafka 的加密范围比单机系统广：客户端↔Broker、Broker↔Broker（副本同步）、Broker↔ZooKeeper/KRaft（元数据）都要 TLS，任何一段明文都是漏洞——这是分布式纵深防御的硬要求。
  - 性能取舍：副本同步是大流量长连接，TLS 加解密对吞吐影响明显（RSA 方案吞吐约降 37%，ECDSA 方案降约 25%）。优化手段：用 ECDSA 证书替代 RSA、启用 TLS 会话恢复、专用加密卸载硬件。
  - 内部通信与外部通信可用不同监听器+不同证书，做到"内网互信证书 + 外网 CA 签发证书"分层。

- **审计：Authorizer 日志 + 拦截器，官方能力有限**
  - 关键取舍：Kafka 原生审计能力弱（Authorizer 日志记授权决策、Request 日志记请求），完整审计靠 ProducerInterceptor/ConsumerInterceptor 自定义拦截器或外接 Ranger。原因——Kafka 定位是高吞吐流管道，每条消息都审计会反过来压垮自己，所以审计粒度交给使用者决定。

- **设计取舍小结**：Kafka 的安全哲学是"模块化可组合 + 按网络域分级"。它不强制任何一种安全机制，而是提供 SASL/TLS/ACL 三套可独立启用的积木，代价是配置矩阵复杂、容易配错，生产化通常需要 Ranger 这类统一管控层。

**预估字数**：2000 字

## 5.5 横向对比（约 1000 字，不可缩水）

**本节目标**：用三栏对比表把三家在四个维度上的选择并排放出来，然后解读"为什么不同"——不是平铺文档，要落到设计动机。

**导语**：下表按"认证 / 授权 / 加密 / 审计"四维，把 Redis、MySQL、Kafka 的安全机制并排，看同一问题上三家选了什么、代价是什么、为什么分叉。

**图 6-5：安全机制四维横向对比表**（type: table）

| 对比维度 | Redis（7.x） | MySQL（8.0） | Kafka（3.x） |
|---|---|---|---|
| 默认是否认证 | 否（仅可选密码） | 是（强 root 密码） | 否（明文监听器可选） |
| 认证抽象 | ACL 用户 + 多密码 | 可插拔认证插件 | SASL 框架 + JAAS |
| 推荐生产认证 | ACL + TLS 客户端证书 | caching_sha2_password | SCRAM-SHA-512 |
| 凭证存储 | 内存 + ACL FILE（SHA-256） | mysql.user（SHA-256 缓存） | ZK/KRaft（SCRAM 哈希） |
| 授权模型 | ACL（用户直绑） | RBAC + 五级层级 | ACL（资源三元组） |
| 最细粒度 | 键模式 + 命令 | 列级 | 资源类型级 |
| 角色继承 | 无 | 8.0 RBAC | 无（外接 Ranger） |
| 默认策略 | 默认拒绝（新版）/ 保护模式 | 默认安全（强密码、可删匿名） | 无 ACL 默认拒绝（需显式配） |
| 传输加密 | 6.0+ TLS，可选 | TLS（账号级 REQUIRE） | TLS 全路径（多监听器） |
| 存储加密 | 无原生 | TDE 表空间/日志加密 | 无原生（靠磁盘加密） |
| 审计能力 | ACL LOG + SLOWLOG（弱） | 企业审计插件 + General Log | Authorizer 日志 + 拦截器（弱） |
| 性能取向 | 性能优先，安全可选 | 安全与功能并重 | 模块化可组合 |
| 典型部署假设 | 可信内网 | 企业网络 | 混合/云原生多跳 |

**解读（要点栏）**：
- **认证抽象的分叉**：Redis 用最直白的"用户+密码"，因为它内核要轻；MySQL 用可插拔插件，因为企业要对接各种目录服务；Kafka 用 SASL 框架，因为它要在分布式多跳间复用标准协议。**抽象层的厚度与系统的协作复杂度成正比。**
- **授权粒度的取舍**：MySQL 做到列级，因为关系型数据天然有"列=字段权限"的需求；Redis 做到键模式+命令，因为键值模型没有"列"概念；Kafka 做到资源类型级，因为流的主语是服务而非人。**授权粒度跟随数据模型，不是越细越好。**
- **审计的普遍短板**：三家审计都不强（MySQL 企业版最好但要付费），共同原因——全量审计的 I/O 开销与"内存/吞吐即生命"的核心价值观冲突。**审计在数据系统里天生是二等公民，必须靠"规则过滤 + 外接 SIEM"补足。**

## 5.6 架构启示（约 1000 字，不可缩水）

**本节目标**：从三家具体实现提炼出可复用的安全设计原则，给出至少 2 句可被独立引用的箴言。

**核心要点（要点栏）**：

- **启示一：默认安全策略要与典型部署环境匹配**
  - Redis 默认开放（假设内网）、MySQL 默认收紧（假设企业网络暴露）、Kafka 默认中性（假设混合云）。三家的默认值不是随意选的，而是对"最可能被怎么部署"的预判。
  - **箴言**："默认安全不是一个布尔值，而是一个与部署环境对齐的函数。面向公网的服务应默认安全，内部工具可以更开放——把默认值配错，等于把攻击面留给了运维者的记性。"

- **启示二：纵深防御（Defense in Depth）= 每层都收一点性能税**
  - 网络隔离 → 传输加密 → 认证 → 授权 → 审计 → 存储加密，六层任何一层单独突破都不致命。代价是每层都有开销：TLS 握手 1-2 RTT、授权检查每次操作、审计 I/O。
  - **箴言**："安全的代价不是一次付清，而是每一层都收一点性能税；接受这笔税，比任何单点加固都便宜。"

- **启示三：最小权限原则是可操作规则，不是口号**
  - 三家都支持且都推荐：每个应用独立账号、只授必要命令/操作、限定键/表/主题范围、定期审查。落地手段：Redis 用键模式隔离、MySQL 用角色 + 列权限、Kafka 用前缀授权。
  - 反模式：用超级用户跑应用、`ALL` 权限"以防万一"、密码硬编码、长期不轮换。

- **启示四：分布式身份传递是单机安全没有的新问题**
  - Kafka 的委托令牌、服务账户、代理传递三种方案各有取舍：令牌短期有效但需要刷新机制；服务账户丢弃客户端身份；代理传递依赖服务间信任。
  - 单机系统的安全模型（认证一次、授权每操作）在分布式多跳下不够用，必须额外回答"身份如何在跳与跳之间安全传递且不被伪造"。

- **启示五：抽象层的厚度与协作复杂度成正比**
  - 回应 5.5 的解读：单点系统用直白认证（Redis），企业系统用可插拔插件（MySQL），分布式系统用标准框架（Kafka 的 SASL）。选型时先问"我要对接多少种身份源、跨多少跳"，再决定抽象层厚度。

**预估字数**：1000 字

## 5.7 小结（约 200 字，3–5 句）

呼应导读提出的四个问题：认证、授权、加密、审计在三家身上各有侧重。Redis 从裸奔到 ACL，体现的是"安全让位于性能、再按需补回"的演进；MySQL 一开始就把权限下沉到列级，体现企业级的完整追求；Kafka 用 SASL/TLS/ACL 三套积木应对多跳分布式，体现可组合哲学。三家差异背后是同一条规律——**安全设计的天花板，由数据模型（关系/键值/流）和部署形态（单机/企业/分布式）共同决定**。下一章将进入集群架构，看安全身份如何在多节点之间传递与校验。
