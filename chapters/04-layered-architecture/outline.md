## 本章导读（约 200 字）

要点（动笔时据此扩写）：
- **立问题**：任何一款"成熟"的系统软件，拆开看都长得不一样，但都做了一件事——把"对外收发消息、内部决定逻辑、底层落地数据"这三件事分开。这一章的问题就是：为什么一定要分开？分多深？分完之后三层之间拿什么传话？
- **为什么重要**：分层是软件设计里最普适的模式，但它有代价——每多一层就多一次上下文切换、多一份接口维护。三家软件在"分几层、层之间耦合多松、谁可以替换"上做出了截然不同的选择，恰好覆盖了分层设计最关键的取舍空间。
- **读完获得**：一套判断"自己的系统该分几层、什么时候分层过度、层间接口怎么定才稳"的决策框架；并能看懂 Redis / MySQL / Kafka 源码里的目录为什么这样组织。

预估字数：约 200 字。

## 4.1 问题的本质：为什么必须分层

**这一节要讲清什么**：先把"分层"从一个被喊滥的口号，还原成一个具体的工程问题——控制复杂度、控制修改的爆炸半径、让不同关注点（并发、逻辑、存储）可以独立演进。然后点明三家面临的共同约束和各自的特化约束，为后面三节深讲铺好共同的坐标系。

**核心要点**（要点栏）：
- 分层解决的三件事：**关注点分离**（关注点分离，Separation of Concerns）、**修改的局部性**（改一层不动另一层）、**可替换性**（接口不变，实现可换）。
- 共同约束：请求必须从"外部不可信的输入"走到"确定性的存储操作"，再原路返回；每穿过一层都要做一次抽象与代价的交换。
- 三家各自的特化约束：
  - Redis —— 追求单机极致延迟，**层数越少越好，跨层调用最好是函数调用**。
  - MySQL —— 要兼容 SQL 语义 + 支持多种存储后端，**必须有可插拔的接口**。
  - Kafka —— 天生分布式，**层与层之间天然有网络边界**，分层要为水平扩展服务。
- 概念校准：本章用"**交互层（协议/连接） / 逻辑层（命令/查询/请求处理） / 存储层（数据落地）**"三段式作为统一视角。这是本章主轴，与各家源码目录命名不必字面对应，但可以一一映射。
- 预告反模式：分层不是越多越好——镜像分层、空洞透传、循环依赖会在 4.6 展开。

预估字数：约 900 字。

## 4.2 Redis 的做法

**这一节要讲清什么**：深讲 Redis 如何用**极少的层 + 进程内直接函数调用**实现"该有的边界都有，但跨层几乎零开销"。重点抓三个机制：单线程 Reactor 把交互/逻辑压在一起、`redisCommandTable` 把逻辑层做成一张可查的表、`redisObject` 的类型/编码分离让逻辑层与存储层解耦。

### 4.2.1 交互层：RESP 与单线程多路复用
- RESP（Redis 序列化协议，Redis Serialization Protocol）解析：边读边解析，不等完整命令。
- 单线程 Reactor（6.0 前）：连接管理 + 命令执行同线程，**没有跨层线程切换**——这是 Redis"看起来分层薄"的根因。
- 多线程 IO（6.0+ / 7.x 基线）：只把协议读写拆出去，**命令执行依然单线程**。说明 Redis 的取舍是：可以分层，但绝不让逻辑层并发，以免破坏单线程内存模型。
- 客户端输出缓冲区（client output buffer）：固定区 + 动态区，防止慢客户端把内存拖垮。

**取舍点**：交互层和逻辑层在内存上是"贴着"的，唯一的解耦手段是输出缓冲区。Redis 用"薄分层"换延迟。

### 4.2.2 逻辑层：命令表 + robj 统一抽象
- `redisCommandTable`：每条命令一行，含处理函数指针、参数校验、命令标志（读/写、是否阻塞、键位置）。命令查找是 dict 一次哈希。
- 命令执行流水线：解析 → 查表 → ACL/参数校验 → 执行 → 慢查询/监控钩子。
- `redisObject`（robj）：**类型（type）与编码（encoding）分离**。type 决定语义（能执行哪些命令），encoding 决定内存表示（int / embstr / raw / ziplist / listpack / hashtable …）。
  - 关键决策：逻辑层只看 type，存储层只看 encoding；同一逻辑可挂多种编码，7.x 用 listpack 进一步省内存。
- 复用：`SET` 与 `SETEX` 共享底层存储操作；模块（Modules）可运行时扩展命令表。

**取舍点**：把"命令"做成数据（一张表），而不是做成代码分支——换来运行时可扩展，代价是表项的字段（标志位）一旦定下就难改。

### 4.2.3 存储层：内存管理 + 持久化双轨
- 内存：jemalloc/tcmalloc 分配；引用计数 + 惰性删除（lazy free，4.0+ 后台线程异步释放大对象）。
- 淘汰策略：noeviction / allkeys-lru / volatile-lru / allkeys-lfu …，LRU 用近似（采样），LFU 用计数衰减。
- 持久化：RDB（快照文件，RDB）全量 + AOF（仅追加文件，AOF）增量，4.0+ 混合重写。两者都属于"存储层的可选职责"，可关闭——Redis 把持久化当附加项而非内建强约束。

**取舍点**：存储层不抽象出"可插拔引擎"，因为内存数据库的存储介质只有一种（内存）。代价是它无法像 MySQL 那样换后端。

### 4.2.4 层间通信
- 交互→逻辑：`client` 结构体 + 直接函数调用。
- 逻辑→存储：`robj` 指针 + 直接内存访问。
- 全程无序列化、无虚函数（命令分派是函数指针，但不是多态 vtable）。

预估字数：约 2400 字。

## 4.3 MySQL 的做法

**这一节要讲清什么**：深讲 MySQL 如何用**标准化接口（Handler API）**把存储层做成可插拔，这是它与 Redis/Kafka 最根本的分层差异。重点抓三件事：连接层用 `THD` 贯穿全链路、服务层把 SQL 处理切成解析/优化/执行三段、存储引擎层用 Handler API 定义"行操作"契约。

### 4.3.1 交互层：连接、认证与线程模型
- 连接协议：TCP/IP、Unix Socket、共享内存、命名管道。
- 认证插件：`mysql_native_password` / `caching_sha2_password`（8.0 默认）；SSL/TLS 协商。
- 线程模型两种：thread-per-connection（默认，隔离好但高并发切换贵） vs 线程池（企业版/PerDB，队头阻塞风险换低切换）。
- **`THD`（线程句柄数据）**：每个连接一个 THD，贯穿连接→服务→引擎三层的会话上下文（当前库、事务、变量、权限）。
- 取舍：用"一个贯穿全程的对象"替代"层层复制参数"——边界靠对象，不靠拷贝。

**取舍点**：连接层与服务层之间没有真正的解耦点，THD 是它们共享的可变状态。这意味着 MySQL 的"分层"在顶上是"软"的。

### 4.3.2 逻辑层：SQL 全生命周期的三段切分
- 解析器：bison 生成（`sql_yacc.yy`），词法 → 解析树 → 预处理查询树（含语义检查）。
- 优化器：常量折叠、常量表检测、范围优化（OR→UNION）、基于直方图（8.0 新增）选索引、JOIN 顺序。
- 执行器：按计划调用 handler 接口，主循环"读一行 → 评估 WHERE → 聚合/返回"。
- 子组件：查询缓存（8.0 已移除——一个值得讲的"分层失败案例"：在服务层做缓存，与引擎层的数据失效难协调，最终被砍掉）。

**取舍点**：逻辑层内部其实是多层子层，但对外是一个"服务层"。子层之间用解析树/执行计划这类结构化中间表示传话，而不是函数调用——这是 SQL 系统的特征。

### 4.3.3 存储层：Handler API 与可插拔引擎
- **Handler API**：MySQL 分层的灵魂。定义表级（open/close）、行级（rnd_init/rnd_next/index_init/index_read）、事务（external_lock/start_stmt）等方法的契约。
- 引擎各显神通：InnoDB（事务+行锁+MVCC，默认）、MyISAM（无事务、全文索引好）、Memory、Archive。多引擎可同时在线。
- InnoDB 自身又是多层：事务管理 → 锁/MVCC → 缓冲池（Buffer Pool）→ 日志（重做日志 redo / 回滚日志 undo）→ 页/B 树。
- 文件组织：表空间 `.ibd`、`ib_logfile`、`undo_*`、双写缓冲。

**取舍点**：接口标准化带来生态（数十种引擎），代价是 handler 通过虚函数多态调用——每次行操作都过一次 vtable，这是 MySQL 相对"硬编码引擎"数据库的固有开销。MySQL 用这个代价换来了可插拔。

### 4.3.4 层间通信
- 连接→服务：THD 对象 + 函数调用。
- 服务→引擎：handler 虚函数（多态）。
- 引擎→OS：文件 API / Native AIO。

预估字数：约 2600 字。

## 4.4 Kafka 的做法

**这一节要讲清什么**：深讲 Kafka 如何用**日志为中心（Log-Centric）的分层**，让每一层都为水平扩展服务。重点抓三件事：网络层用 Reactor + RequestChannel 把"收发"和"处理"在多线程间解耦、API 层用统一的 `handle` 分派把"请求类型"做成可枚举的协议、Log 层把存储抽象成"追加写 + offset 寻址"使存储层与复制层共享同一份数据。

### 4.4.1 交互层：SocketServer 与 Reactor 多线程模型
- 受 Netty 启发：Acceptor（accept） → Processor（网络 IO + 协议编解码） → RequestChannel（有界队列） → KafkaRequestHandler（业务线程）。
- 解耦点：RequestChannel 是网络线程与业务线程之间的**有界队列**，天然带背压（慢消费会回压到上游）。
- 与 Redis 单线程的对照：Kafka 必须多线程，因为单机要扛多分区、多副本的并发 IO。

**取舍点**：Kafka 的交互层是三者里最"重"的（多层线程 + 队列），换的是单机能榨干多核、能水平扩展。延迟比 Redis 高，但吞吐量级不同。

### 4.4.2 逻辑层（API 层）：请求类型分派
- 统一请求类型：PRODUCE / FETCH / LIST_OFFSETS / METADATA / LEADER_AND_ISR / STOP_REPLICA …，每个有版本号。
- `KafkaApis.handle()` 按 ApiKeys 分派到各 handler 方法。
- 设计收益：新增 API 只扩 handle；统一做鉴权、配额（quota）、版本协商。
- 这一层不做业务逻辑，只做**协议适配 + 分派**——真正的"逻辑"在更下层（Log 层、ReplicaManager）。

**取舍点**：把"逻辑层"切成"协议分派"和"领域逻辑"两段，是因为分布式系统里协议版本演进和领域逻辑演进节奏不同，必须分开。

### 4.4.3 存储层：Log 抽象 + 零拷贝
- `Log` 是核心抽象：一组顺序追加的日志段（LogSegment），每段含消息文件 + 位移索引 + 时间戳索引。
- 写：顺序追加 → 落页缓存（page cache）→ 异步/同步刷盘（fsync）策略可配。
- 读：按 offset 二分定位日志段 + 稀疏索引；消费优先命中页缓存。
- **零拷贝（sendfile）**：消费者拉历史数据时，磁盘 → 页缓存 → 网卡，跳过用户态，省掉 2 次拷贝和 2 次上下文切换。
- 副本（replication）：Follower 用 FETCH 拉 Leader 的日志——**复用了消费者同一条读路径**。这是 Log 抽象的红利：存储层和复制层共享一份代码。

**取舍点**：Kafka 把"存储"做得异常薄（就是追加写日志），把复杂度推给上层（索引、复制、压实）。换来的是存储层几乎不可插拔，但极快。

### 4.4.4 协调层：在 Log 之上的额外一层
- ReplicaManager / Partition / Controller /（3.x 起）KRaft 元数据管理。
- 说明 Kafka 实际上是"交互 + API + 存储 + 协调"四段，协调层是分布式系统独有的、Redis/MySQL 没有的层。

### 4.4.5 层间通信
- 交互→API：RequestChannel 队列（跨线程）。
- API→存储/协调：对象方法调用（同 JVM 内）。
- 跨 Broker：永远是网络协议——这是 Kafka 分层的天然边界。

预估字数：约 2600 字。

## 4.5 横向对比

**这一节要讲清什么**：用三栏对比表把三家压在同一坐标系下，然后解读"为什么不同"——不是哪家更好，而是各自的性能目标和部署形态决定了分层形态。本节不可缩水。

**对比维度**（建议落为一张多行三栏表，顺序 Redis | MySQL | Kafka）：
1. **核心抽象** —— robj 键值对象 | 关系表 + 索引 | 追加日志（Log）
2. **主要存储介质** —— 内存（持久化为辅）| 磁盘（内存缓存）| 磁盘（靠页缓存）
3. **层数与粒度** —— 4 层、粗粒度 | 4 层、服务层内含多层子层 | 5 层、细粒度（含协调层）
4. **层间通信机制** —— 函数调用 + 对象指针（无序列化）| THD 对象 + handler 虚函数 | RequestChannel 队列 + 跨网络协议
5. **可插拔性** —— 无引擎可换（仅内存）| Handler API 支持多引擎并存 | 存储不可换，但有 Connect/Streams 插件生态
6. **分层的首要目标** —— 低延迟（少层）| 兼容性与生态（标准接口）| 水平扩展与高吞吐（网络边界）

**解读要点**：
- 存储介质决定能不能分层——内存只有一种介质，Redis 没必要做引擎抽象；磁盘有多种组织方式，MySQL 必须做；Kafka 的存储高度专用，做了反而碍事。
- 通信机制与部署形态绑定——单机进程内用函数调用，跨网络用协议，没有"高级低级"之分，只有匹不匹配。
- 可插拔不是越高越好——MySQL 的可插拔带来生态也带来 vtable 开销；Kafka 不可插拔反而让零拷贝路径极简。

预估字数：约 1300 字。

## 4.6 架构启示

**这一节要讲清什么**：从三家的具体实现提炼出可复用的分层设计原则与权衡规律。这是本章精华，至少产出 1–2 句可独立引用的箴言。本节不可缩水。

**核心启示**（要点栏）：
1. **分层深度匹配性能目标，而非追求"标准层数"**。
   - 箴言候选：「分层的厚度，应当等于你愿意为抽象支付的延迟预算。」
   - Redis 用薄分层换纳秒级延迟；Kafka 用厚分层换水平扩展。脱离性能目标谈"该分几层"是无意义的。
2. **接口稳定性 > 接口优雅性**。
   - Redis robj、MySQL Handler API、Kafka 日志格式——三者都多年稳定，是各自生态的根。
   - 箴言候选：「好的分层接口，是十年后还能不动的那一层。」
3. **让"可替换"出现在最值得替换的地方**。
   - 不是每层都要可插拔。MySQL 把可插拔放在存储层（变化最快的地方），Redis 干脆不做，Kafka 把扩展点放在边缘（Connect/Streams）。判断标准：这层的实现多样性是否真实存在。
4. **警惕分层的反模式**：镜像分层（每层一个同名对象，只是改个名）、空洞透传（一层只做转发不加价值）、循环依赖（破坏封闭性）。检验标准三问：能否独立测试？换实现会不会牵动别层？这层有没有引入新抽象？
5. **会"砍掉一层"也是分层能力**。
   - MySQL 8.0 移除查询缓存——当一层的价值不再抵得过它带来的协调成本，就该删。分层不是只增不减。

**演进规律收尾**：Redis 从缓存长成多结构数据库、MySQL 从单引擎长成多引擎体系、Kafka 从消息队列长成流平台，分层都在演进中逐步清晰。优秀的架构不是一次定稿，是保留扩展点让层次自然生长。

预估字数：约 1300 字。

## 4.7 小结（约 250 字）

**收束要点**（3–5 句，呼应导读）：
- 三家软件的分层形态截然不同，但都回答了同一个问题：怎么把交互、逻辑、存储分开，且分得值。
- Redis 教我们：性能敏感处，分层要薄，跨层要直接。MySQL 教我们：标准化接口能让生态长出来，但要为多态付开销。Kafka 教我们：分布式系统的层间边界天然是网络，分层要为水平扩展服务。
- 一句可引用的总纲：「分层不是把系统切碎，而是为每一次"该改哪里"画出边界。」
- 承上启下：分层画好了边界，下一章（第 5 章 安全机制）看这些边界如何被守护。

预估字数：约 250 字。
