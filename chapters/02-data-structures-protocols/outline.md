# 第 2 章 数据结构与协议 —— 写作大纲

## 核心论点
三个系统为了实现各自的核心目标（Redis 低延迟、MySQL 通用可靠、Kafka 高吞吐），选择了完全不同的内部数据结构（砖块）和通信协议（语言）。数据结构决定了能做什么、不能做什么；协议决定了客户端和服务端怎么互相理解。这条"核心目标 → 数据结构 → 协议形态"的因果链，是后面七章所有机制设计的起点。

## 章节结构

### 2.1 问题的本质（建立框架）
- 两个最基础的决定：内部怎么表示数据、客户端和服务端怎么对话
- 三个张力：简单 vs 效率、可读 vs 紧凑、固定 vs 可演进
- 三家核心目标一句话：Redis 低延迟、MySQL 通用可靠、Kafka 高吞吐
- 一句话预告：目标不同，砖块和语言不同

### 2.2 内部数据结构
#### 2.2.1 Redis
- SDS：二进制安全、O(1) 取长、预分配、惰性释放
- 跳表：多层有序链表，O(log N)，实现比红黑树简单
- 字典：链式哈希 + 渐进式 rehash（摊还集中任务）
- ziplist/listpack：极致内存密度，连锁更新→自记长度
- intset：有序整数数组，小数据紧凑
- 统一主题：全内存 + 低延迟 → 选最省的

#### 2.2.2 MySQL
- B+ 树：磁盘 16KB 页对齐，内部节点存键+指针，叶子层存数据+链表串联
- 行格式 Dynamic：变长字段逆序+NULL位图，大字段溢出
- 自适应哈希：B+ 树热页自动建内存哈希
- 统一主题：磁盘页 + 通用查询 → 每页利用率最大化

#### 2.2.3 Kafka
- RecordBatch V2 字段布局：baseOffset/batchLength/partitionLeaderEpoch/magic/CRC/attributes/lastOffsetDelta/baseTimestamp/producerId/baseSequence/recordsCount
- 差值编码：记录不存绝对 offset/timestamp
- Batch 级 CRC 替代逐条 CRC
- 统一主题：批量 + 顺序 + 演进 → 批量化写进格式

### 2.3 通信协议
#### 2.3.1 Redis RESP
- 5 种前缀字符（+、-、:、$、*）
- 文本、人眼可读、实现简单
- 代价：体积大、无版本协商

#### 2.3.2 MySQL 二进制协议
- 握手→命令→响应三阶段
- 预编译语句：解析一次+执行多次
- 二进制结果集：列元数据前置+行数据紧凑
- 代价：人不可读、实现复杂

#### 2.3.3 Kafka 请求/响应协议
- 统一头部：api_key + api_version + correlation_id + client_id
- 版本化：同一 API 多版本共存
- 批量优先：Produce/Fetch 以 Batch 为单位
- 代价：协议复杂性

### 2.5 架构启示（4 条）
1. 数据结构是核心目标的翻译器
2. 协议是数据结构的外部映射
3. 简单协议的粘性远超复杂系统
4. 版本化能力是协议的生命线
