# 第9章 数据同步机制 — 集群一致性的实现（大纲）

> 本章与第6章《集群架构》分工：**第6章讲集群拓扑、分片、选主、分区路由；本章只讲数据在节点间的字节级传播、位点、一致性达成与故障恢复。** 两章互为表里，多处交叉引用。

## 本章目标
- 把"数据同步"从运维概念提升为架构设计问题。
- 拆解 Redis / MySQL / Kafka 如何把一处状态变化可靠地传播到其他副本。
- 提炼可复用的"状态机复制 + 位点对齐"共性规律（呼应第9章规律五）。

## 篇幅目标
约 10000–11000 字（三家深讲各 2400–2600 字，横向对比 1200 字，架构启示 1200 字）。

## 本章导读（约 220 字）
- 立问题：数据放一份不够（容量、可用、读扩展都要多副本），可一旦放多份，就必须让多份"看起来像一份"。
- 同步的本质 = 把一处状态变化，可靠、有界延迟地传播到其他副本。
- 三家面对同一难题给出三种答案（命令流 / binlog event / 消息），背后却是同一条主线：状态机复制。
- 承接第6章：第6章回答"集群怎么组、谁来当主"，本章回答"主的数据怎么传到副本、传到什么程度算数、断了怎么续"。

## 8.1 问题的本质（约 1100 字）
- 单机的三道天花板 → 走向多副本；多副本立刻引入"状态扩散"问题。
- 同步的四个子问题（贯穿全章的分析骨架，对应图8-1）：
  1. **传播什么**——粒度：命令 / 事件 / 消息。
  2. **怎么传播**——推（主推送）vs 拉（副本拉取）。
  3. **传播到什么程度**——一致性：最终一致 / 强一致 / 可调。
  4. **传播断了或乱了怎么办**——断点续传 + 故障恢复。
- 一致性是延迟的函数：要等多副本确认才返回，延迟随确认范围上升。
- **状态机复制模型**：若每个副本按相同顺序执行相同输入序列，最终状态必然相同——三家都在用这个底层模型，差异只在"序列"的载体与"执行"的方式。
- 只立问题，不展开实现。

## 8.2 Redis 的做法（约 2400 字）
- **传播单元 = 命令本身**（协议即日志，呼应第7章"AOF 协议即文件"）。
- 复制是**单向异步**（AP）：主节点写完即返回，从节点异步追赶；从节点只能有一个主。
- **全量同步**：主节点 `BGSAVE` fork 出 RDB 传给从节点，传输期间新命令写入复制积压缓冲区（backlog），RDB 传完后补发积压。
- **增量同步（PSYNC2）**：靠 replication offset（主从各维护的字节进度坐标）+ 复制积压缓冲区；断线重连若 offset 仍在 backlog 窗口内 → 部分重同步（只补差额）；否则退回全量。
- **关键取舍**：backlog 默认约 1MB，是"断线容忍窗口"与"内存占用"的折中；窗口外的重连代价 = 全量同步（fork 阻塞 + 大实例网络传输）。
- **级联复制**（链式 tree）分担主节点出口带宽，代价是延迟随链路深度累积。
- `min-replicas-to-write` / `min-replicas-max-lag`：给"主可单点但至少 N 个从跟上、且延迟在阈值内"的弱保证，把同步强度做成可写门槛。
- **故障转移的数据语义**：Sentinel / Cluster 选新主后，新主可能没有老主的全部数据（异步复制 + 未同步的写会丢）——这是 Redis 选择 AP 的必然代价。
- 取舍总结：用"异步 + 可能丢少量写"换主节点的低延迟与高可用。

## 8.3 MySQL 的做法（约 2600 字）
- **传播单元 = binlog event**（行变更的逻辑 / 行级日志，ROW 格式为现代主流）。
- **传统复制链路**：主写 binlog → 从的 IO 线程拉取到 relay log → 从的 SQL 线程回放。本质是"网络拉取"与"本地回放"分离的两段式（推拉结合）。
- **异步复制**：主提交即返回（AP），主从切换时已提交事务可能丢失。
- **半同步复制**：主提交后等至少一个从 ACK（写日志）才返回给客户端；超时则**降级回异步**——降级语义是关键设计：一致性可以被关掉，但这一事实必须可观测、可告警。
- **GTID（全局事务标识 = source_id:transaction_id）**：让"从哪个位点继续"从脆弱的"文件名 + 字节偏移"变成全局唯一的事务序号，断点续传与故障切换大幅简化（从库自动从主库未同步的 GTID 开始追）。
- **8.0 多线程并行复制**：基于组提交的 LOGICAL_CLOCK，从库可按"同一组提交内"并行回放，缓解单 SQL 线程的复制延迟。
- **组复制（MGR）**：把 binlog event 当作共识协议（XCom，Paxos 变种）的 log entry，**多数派确认才提交**——这是状态机复制的强一致形态（CP）。
- **分布式恢复**：新成员加入时，用 clone 插件（8.0.2+）或 binlog 增量追平状态差。
- **一致性级别参数**（`group_replication_consistency`）：读写是否等待状态追平，把"一致性强度"做成可调旋钮。
- 取舍总结：从"异步可丢"到"多数派强一致"是一条连续谱，参数选择 = 延迟与安全的交易。

## 8.4 Kafka 的做法（约 2600 字）
- **传播单元 = 消息**（按分区有序追加的日志记录）；副本 = 分区日志的拷贝。
- **同步方式 = 纯拉模式**：Follower 主动 `FETCH` Leader 的日志（与消费者走**同一条读路径**，呼应第4章 Log 抽象的红利）。
- **LEO（日志末端位移）与 HW（高水位 = ISR 中最小 LEO）**：只有 HW 以下的消息算"已提交"，消费者只能读到 HW——HW 是"已被足够多副本确认"的进度线。
- **ISR（同步副本集合）**：跟上 Leader 的动态副本集合；只有 ISR 成员才有资格当选新 Leader。
- **leader epoch**：单调递增的任期号，解决老 Leader 复活导致的**日志分叉（脑裂）**——每个任期对应一段权威日志，跨任期用 epoch 界定，副本恢复时按 epoch 截断不一致的尾部。
- **acks 旋钮（0 / 1 / all）**：把"一致性强度"做成生产者级旋钮；`acks=all` + `min.insync.replicas` 约束最少确认副本数，二者配合才成立。
- **unclean leader 选举**：是否允许非 ISR 副本当选——允许（可用性优先，可能丢数据）vs 禁止（一致性优先，可能短时不可用），经典 CAP 取舍的显式开关。
- **KRaft 元数据同步**：元数据本身也是 Raft 复制日志（Controller quorum），与数据复制**同构**——同步思想从数据层延伸到元数据层。
- **幂等与事务**：ProducerId / Epoch / BaseSequence 跨 broker 维护状态，实现 exactly-once——状态机复制进一步延伸到"操作语义"层面（不止状态一致，连"这条算不算"也一致）。
- 取舍总结：把一致性、可靠性、可用性全部参数化，把权衡的选择权显式交给使用者。

## 8.5 横向对比（约 1200 字）
三栏对比表（**Redis | MySQL | Kafka**，圣经第5节固定列序），维度为行；表前导语点明"比什么"，表后解读"说明了什么"，禁止表格裸奔。

| 维度 | Redis | MySQL | Kafka |
|------|-------|-------|-------|
| 传播单元 | 命令流 | binlog event | 消息（按位移有序） |
| 传播方式 | 主推送命令 | 拉取 + 本地回放分离 | 纯拉取（FETCH） |
| 一致性模型 | 异步最终（AP） | 异步 / 半同步 / 多数派强一致（CP） | ISR + acks 可调（AP↔CP） |
| 位点语义 | replication offset | GTID（全局事务序号） | offset + leader epoch |
| 故障恢复粒度 | backlog 窗口内部分重同步，否则全量 | GTID 自动定位 + 分布式恢复 | leader epoch 截断 + ISR 仲裁 |
| 全量同步代价 | fork RDB 传输（主节点阻塞） | clone 插件或全量 binlog | 分区日志拷贝 |

- 解读一：**传播单元的选择决定一切下游设计**——命令流轻但不抗语义歧义，event 强语义但重，消息天然有序但只追加。
- 解读二：**位点（offset / GTID / 位移）是同步的通用语言**，形态不同本质都是"进度坐标"。

## 8.6 架构启示（约 1200 字）
- **启示一：数据同步的本质是状态机复制**——只要保证"相同输入序列 + 相同执行顺序"，副本终将一致。（箴言候选：*同步不是搬运数据，而是重放历史。*）
- **启示二：位点 / 位移是同步的通用货币**——offset、GTID、LEO/HW 形态不同，都是"进度坐标"。（箴言候选：*想让多份副本对齐，先给他们一把共同的尺子。*）
- **启示三：增量同步必须先解决断点续传**——全量同步是兜底手段，不是常态；能增量就别全量。
- **启示四：一致性的代价是延迟的函数**——没有免费的一致性，只有愿意付多少延迟。
- **启示五：故障恢复是同步的反向问题**——同步把状态推到副本，恢复把落后副本拉回一致；二者共用同一套位点与日志。
- 反模式：同步链路过深致延迟累积；位点丢失致全量重灌；忽视脑裂致数据分叉。

## 8.7 小结（约 250 字）
- 呼应导读：三家用不同单元（命令 / event / 消息）做同一件事——状态机复制。
- 共性骨架：传播单元 + 位点 + 一致性旋钮 + 故障恢复。
- 一致性是可标价的连续谱，不是非黑即白。
- 引出第9章：状态机复制 + 位点对齐，是全书共性规律之一（规律五的核心印证）。

## 难点与不确定处（risks，撰写时逐条核验）
- 复制积压缓冲区 backlog 默认大小（约 1MB）与 `repl-backlog-size` / `repl-backlog-ttl` 需核 Redis 7.x 默认值。
- PSYNC2 部分重同步窗口语义、replid 切换（故障转移后）细节需核。
- 半同步 `after_commit` / `after_sync` 语义、8.0 统一为 source/replica 术语、`rpl_semi_sync_source_wait_for_replica_count` / `timeout` 需核 8.0.x。
- GTID 生命周期、MGR 分布式恢复（clone 插件 8.0.2+）、`group_replication_consistency` 各档语义需核 8.0.x。
- leader epoch 解决老 Leader 日志分叉的精确机制、`unclean.leader.election.enable`（3.x 默认 false）、`min.insync.replicas` 与 `acks=all` 的交互需核 3.x。
- **与第6章边界严守**：集群拓扑 / 选主 / 分区路由归第6章，字节级传播 / 位点 / 一致性 / 恢复归本章，避免重复。
- 字数：三家深讲各 2400–2600 字 + 对比启示各 1200 字约 10500–11500 字，若超 12000 优先压缩级联复制 / 半同步降级等过渡内容，保住 PSYNC2 / GTID+MGR / ISR+epoch 三个终态方案。
