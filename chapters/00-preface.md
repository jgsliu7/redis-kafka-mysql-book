# 序言

我们今天开发中遇到的技术问题，绝大多数在软件行业几十年的历史里早就出现过。这些软件和工程师反复打磨过的方案，就是现成的答案。架构师提升架构思维最务实的方法，就是通过学习优秀软件的架构设计，这些沉淀下来的设计智慧，足以应对我们日常遇到的大多数问题。

这部书选择这三个软件：Redis、MySQL、Kafka，它们构成了整个后端基础设施，我们天天在用，却很少追问它们为什么长成现在这个样子。市面上的资料大多停在"怎么用"：Redis 的数据类型有几种，MySQL 的索引怎么设计，Kafka 的分区数设多少。架构决策者需要的不只是用法，还有背后的权衡。AI 可以辅助代码生成，但架构取舍涉及周期、成本、团队、现实约束——这些需要人来判断。所以架构师真正值钱的还是判断和取舍的能力，这一步往往直接决定系统的复杂度、能不能按期上线、好不好运维、够不够健壮。

会用的人很多，能把核心架构设计理解清楚的不多，能把这几个系统拉通、从更高视角统一来看的人更是寥寥无几。这本书适合有一定开发基础、喜欢钻研技术的中高级开发和架构师阅读。

我想先把这本书最核心的三层意思讲在最前面。

第一层：**这三个软件遇到的架构问题，不是它们独有的，我们自己设计系统的时候，也会遇到类似的问题。** 怎么启动才能不丢状态、怎么关闭才能不丢请求、内存和磁盘如何协同、怎么把系统分出清晰的层、怎么做权限与加密、单机扛不住时怎么分片和容错、文件该按什么格式落盘、副本之间怎么对齐。这些问题，Redis、MySQL、Kafka 已经在生产环境里反复打磨了十几年，交出过被千锤百炼的答案。把这些解法整理出来，就等于备下一套方法，下次遇到同类问题，至少知道有哪些选项。会用这些软件，是对开发者的要求；知道"这个问题有哪些解法"，是架构认知的第一步。

但这套方法只是起点。第二层是这本书真正想讲的：**同一个问题，三种解法都有人选了，他们为什么选这个、不选那个？选了之后付了什么代价、得到了什么？** 取舍，是架构设计里最难也最值钱的一步。能从这些软件的设计里，感受到"为什么这么取、为什么那么舍"。

还有第三层。一个系统牵涉很多技术问题，每个都有多种解法，排列组合下来就是几百种做法。架构师如果只知道"有哪些方案"，远远不够——难点在于根据"天时地利人和"去判断和选择，把各个环节的方案组合成一个能落地的整体。什么时候上线、跑在什么机器上、团队什么情况、遗留系统怎么处理、公司有什么硬性约束——这些条件没有一个能用公式量化，但每一个都在限制你能选什么。技术能力强的工程师能给出单个环节的优秀方案，但系统架构需要把各环节整合成可落地的整体。再说，什么叫"最优"？Redis 写盘不是最快的，Kafka 随机查询也不是最快的——但这本来就不是它们的核心目标。优秀软件并不在每个环节用最牛的方案，非核心的环节都在给核心目标让步，该让的让、该保的保。知道方案是"术"，在约束下做对取舍才是"道"。这本书就是带你走完这一步。

熟练使用这些软件，和能讲清它为什么这么设计，换一种做法要付出什么代价，是两回事。会用一个软件，和理解它的架构设计，中间有很大的认知差距。这本书的目的，就是缩小这个差距：把这三款软件里沉淀下来的架构经验，变成你设计系统时能用的参考。

回到这三款软件本身，它们都要回答一组类似的问题：怎么启动、怎么关闭、内存和磁盘如何协同、怎么分层、怎么保证安全、怎么组成集群、文件该长什么样、副本之间怎么对齐。问题是同一组，答案却各走各的路。这些答案的差异，就是取舍的差异。

为什么是这三款？因为后端最基础的三种能力，正好各有一款代表落在它们身上：Redis 把"内存即数据"推到底，MySQL 死守"不丢不错"，Kafka 把整件事建在"追加写日志"上。读懂它们如何各自回答同一组问题，等于掌握了后端架构的基本元素。

当然，优秀的软件远不止这三款。选它们，主要是因为它们足够典型，把这三款软件读透了，就获得了一套看多数软件的通用视角。以后遇到一个新系统，不只会问"它怎么用"，还会下意识地问"它为什么这么设计、是如何做取舍的"。这种眼光不绑定特定技术栈，可以跨系统复用，而且见过的系统越多越管用。这有助于你在面对新系统时多问一句'它为什么这么设计'。这套能力就在这种持续的追问中真正长出来。

这本书不讲最新的调优技巧，不讲基础知识，也不逐行贴源码。我会讲更稳定的东西：大多数存储系统都会反复遇到这些设计抉择：速度还是持久、一致还是可用、简单还是灵活、通用还是专用，以及做出这些抉择时必须支付的代价。版本会更迭（本书基于 Redis 7.x、MySQL 8.0.x、Kafka 3.x），但这些抉择的逻辑长期不变。

读这本书的方式，建议是带着比较去读。每一章都会从这三个软件架构的维度横向对比、提炼启示。读着读着就会发现，差异底下藏着同一套骨架：先定真相之源（也就是系统里那份数据，错了就出事）、用层包裹可变性（把容易变的部分关进一层，让接口比实现活得久）、把随机变顺序（顺序写、顺序读，让磁盘和 CPU 都不吃亏）、用性能买可靠性（多花一点 CPU 或内存，换故障后能恢复）、把复杂状态显式建模（把状态摆到明面上，而不是散落在代码里）。

这本书面向的，是有一定使用经验、想从"会用"进阶为真正可以设计决策的人。如果读完之后，再面对一个新系统时，能多问一句这类问题。更理想的是，某天我们自己设计系统时，能下意识地去考虑不同方案的取舍问题。到那时，这本书才算真正发挥了作用。
