# 序言

我们今天开发软件遇到的技术问题和设计问题，绝大多数在软件行业几十年的发展历程里，那些优秀的软件和顶尖的工程师早就遇到过了，并且留下了深思熟虑的方案。架构师提升自己架构思维最务实的方法，不是凭空想象，而是到这些已有的优秀软件里去找思路——这些沉淀下来的设计智慧，足以应对我们日常遇到的大多数问题。

这本书选了三个这样的软件：Redis、MySQL、Kafka。它们几乎撑起了整个后端基础设施，我们天天在用，却很少追问它们为什么长成现在这个样子。市面上的资料大多停在"怎么用"：Redis 的数据类型有几种，MySQL 的索引怎么设计，Kafka 的分区数设多少。这些当然重要，但对一个要做架构决策的人来说远远不够。就算 AI 能帮你写出整套系统，真正的架构取舍它也替不了你：周期、成本、团队、历史包袱这些现实约束，AI 看不见。所以架构师真正值钱的还是判断和取舍的能力，这一步往往直接决定系统的复杂度、能不能按期上线、好不好运维、够不够健壮。

懂用法的人多，懂取舍的人少。这本书适合有一定开发基础、喜欢钻研技术的中高级开发和架构师阅读。

我想先把这本书最核心的两层意思讲在最前面。

第一层：**这三个软件遇到的架构问题，不是它们独有的，我们自己设计系统的时候，也会遇到类似的问题。** 怎么启动才能不丢状态、怎么关闭才能不丢请求、内存和磁盘如何协同、怎么把系统分出清晰的层、怎么做权限与加密、单机扛不住时怎么分片和容错、文件该按什么格式落盘、副本之间怎么对齐。这些问题，Redis、MySQL、Kafka 已经在生产环境里反复打磨了十几年，交出过被千锤百炼的答案。把这些解法整理出来，就等于备下一套方法，下次遇到同类问题，至少知道有哪些选项。会用这些软件，是对开发者的要求；知道"这个问题有哪些解法"，是架构认知的第一步。

但这套方法只是起点。第二层才是这本书真正想讲的：**同一个问题，三种解法都有人选了，他们为什么选这个、不选那个？选了之后付了什么代价、得到了什么？** 这才是架构设计里最难、也最值钱的一步：做取舍。能从这些顶级软件的设计里，感受到"为什么这么取、为什么那么舍"，才是架构师的眼光。

还有第三层。一个系统由很多技术问题组成，每个问题都有多种解法，排列组合下来就是几百种做法。架构师如果只知道"有哪些方案"，远远不够——真正的挑战是根据"天时地利人和"去判断、去选择，把各个环节的方案组合成一个能落地的整体。什么时候上线？跑在什么机器上？团队什么情况？遗留系统怎么处理？公司有什么硬性约束？这些条件没有一个能用精确公式量化，但每一个都在限制你能选什么。身边有些技术很牛的人，方案信手拈来，但要设计一个系统的架构，并不是把每个环节的"最优方案"拼起来就行了。再说，什么叫"最优"？Redis 写盘不是最快的，Kafka 随机查询不是最快的——但它们并没有在这些点上追求最优，因为那不是它们的核心使命。一个优秀软件不是在每个环节都用最牛的方案，而是所有非核心的环节都在为核心目标服务，该让的让、该保的保。知道一个方案是"术"，能在约束下做出对的取舍组合才是"道"——这就是"知道""会用"和"取舍"之间那道巨大的鸿沟。这本书想做的，就是借这三个大家最熟悉的软件，和你一起跨过去。

这几年我面试过数百名架构师候选人，有一个现象反复出现：绝大多数人能熟练说出 Redis 的数据类型、MySQL 的索引原理、Kafka 的分区配置，却很少有人能讲清"它为什么这么设计，换一种做法要付出什么代价"。会用一个软件，和理解它的架构设计，中间隔着一条很深的沟。能从优秀软件里学到架构经验、变成自己设计能力的人少之又少。而这一步，正是普通开发者成长为架构师最难跨过去的地方。这本书想做的，就是帮你跨过去：把优秀软件里的架构经验，变成你自己设计系统的本事。

回到这三款软件本身，它们都要回答一组类似的问题：怎么启动、怎么关闭、内存和磁盘如何协同、怎么分层、怎么保证安全、怎么组成集群、文件该长什么样、副本之间怎么对齐。问题是同一组，答案却各走各的路。这些答案的差异，就是取舍的差异。

为什么是这三款？因为后端最基础的三种能力，正好各有一款代表落在它们身上：Redis 把"内存即数据"推到底，MySQL 死守"不丢不错"，Kafka 把整件事建在"追加写日志"上。读懂它们如何各自回答同一组问题，等于掌握了后端架构的基本元素。

当然，优秀的软件远不止这三款。选它们，主要是因为它们足够典型，把这三款软件读透了，就获得了一套看任何软件的架构视角。以后遇到一个新系统，不只会问"它怎么用"，还会下意识地问"它为什么这么设计、是如何做取舍的"。这种眼光换工作、换技术栈都带得走，而且见过的系统越多越管用。学完这三款软件不是终点，从此看任何新软件都能带着这套视角去问一句“它为什么这么设计”，那才是这套能力真正长出来的时候。

这本书不讲最新的调优技巧，不讲基础知识，也不逐行贴源码。我会讲更稳定的东西：不管什么系统，只要它要记东西、不能丢，那些反复出现的设计抉择：速度还是持久、一致还是可用、简单还是灵活、通用还是专用，以及做出这些抉择时必须支付的代价。版本会更迭（本书基于 Redis 7.x、MySQL 8.0.x、Kafka 3.x），但这些抉择的逻辑长期不变。

读这本书的方式，建议是带着比较去读。每一章都会从这三个软件架构的维度横向对比、提炼启示。读着读着就会发现，差异底下藏着同一套骨架：先定真相之源（也就是系统里那份数据，错了就出事）、用层包裹可变性（把容易变的部分关进一层，让接口比实现活得久）、把随机变顺序（顺序写、顺序读，让磁盘和 CPU 都不吃亏）、用性能买可靠性（多花一点 CPU 或内存，换故障后能恢复）、把复杂状态显式建模（把状态摆到明面上，而不是散落在代码里）。

这本书面向的，是有一定使用经验、想跨过"会用"这道坎、走到"懂取舍"那一侧的人。如果读完之后，再面对任何一个新系统，第一反应能从"它怎么用"变成"它为什么这么设计、反过来会怎样、这套取舍的边界在哪"，那么这本书就达到了它的目的。更理想的是，某天我们自己设计系统时，能下意识地去考虑不同方案的取舍问题，那才是这本书最希望发生的事。
