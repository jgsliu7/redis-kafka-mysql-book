# 序言

懂用法的人多，懂取舍的人少。

本书不是技术入门和基础教程，适合有一定开发基础且喜欢钻研技术的中高级开发和架构师阅读。

这本书的起点，是一个再普通不过的疑问：Redis、MySQL、Kafka——这三款几乎撑起整个后端基础设施的软件——我们天天在用，却很少追问它们为什么长成现在这个样子。市面上的资料大多停在"怎么用"：Redis 的数据类型有几种，MySQL的索引怎么设计，Kafka的分区数设多少。这些当然重要，但对一个真正做架构决策的人来说，远远不够，即使现在ai可以帮你写出完整的系统，但是大家都知道：真正的架构取舍需要考虑很多因素（项目周期/成本/基础设施情况/团队情况/历史背景等等），很难把所有技术之外的因素都提供给ai来做决策，所以架构师的判断和取舍能力非常重要，甚至在某些场景下架构师的判断和决策对系统的复杂度/按期达到业务目标/可运维/可维护/健壮/性能等起着决定性的作用。

我想先把这本书最核心的两层意思讲在最前面。

第一层：**这三个软件遇到的架构问题，不是它们独有的——我们自己设计系统的时候，也会遇到类似的问题。** 怎么启动才能不丢状态、怎么关闭才能不丢请求、内存和磁盘如何协同、怎么把系统分出清晰的层、怎么做权限与加密、单机扛不住时怎么分片和容错、文件该按什么格式落盘、副本之间怎么对齐。这些问题，Redis、MySQL、Kafka 已经在生产环境里反复打磨了十几年，交出过被千锤百炼的答案。把它们的解法整理出来，就是一套弹药库，下次若遇到同类问题，至少知道有哪些选项。会用这些软件，是对开发者的要求；知道"这个问题有哪些解法"，是架构认知的第一步。

但弹药库只是起点。第二层才是这本书真正想讲的：**同一个问题，三种解法都有人选了——他们为什么选这个、不选那个？选了之后付了什么代价、得到了什么？** 这才是架构设计里最难也最值钱的东西：取舍之道。能从这些顶级软件的设计里，感受到"为什么这么取、为什么那么舍"，才是架构师的眼光。读完这本书，脑子里留下的不该只是"Redis 快、MySQL 稳、Kafka 能扛"，而是一张能自己推演的取舍地图——下次设计系统遇到同类问题，信手拈来，恰到好处。这才是读这本书真正的回报。

这几年我面试过数百名架构师候选人，有一个现象反复出现：绝大多数人能熟练说出 Redis 的数据类型、MySQL 的索引原理、Kafka 的分区配置，却很少有人能讲清"它为什么这么设计，换一种做法要付出什么代价"。会用一个软件，和理解它的架构设计，中间隔着一条很深的沟。能从优秀软件里汲取架构经验、沉淀成自己设计能力的人少之又少——而这恰恰是普通开发者成长为架构师的关键一跃。这本书想做的，就是在这条沟上架一座桥：把优秀软件的架构智慧，转化成我们个人的架构设计能力。

回到这三款软件本身,它们都要回答一组类似的问题：怎么启动、怎么关闭、内存和磁盘如何协同、怎么分层、怎么保证安全、怎么组成集群、文件该长什么样、副本之间怎么对齐。问题是同一组，答案却截然不同——而正是这些不同，暴露了软件设计中最有价值的东西：取舍。

为什么是这三款？因为它们恰好覆盖了后端三种最基础的能力形态：Redis 把"内存即数据"做到极致，MySQL 以"不丢不错"为第一原则，Kafka 以"追加写日志"为核心抽象。读懂它们如何各自回答同一组问题，等于掌握了后端架构的基本元素。

当然，优秀的软件远不止这三款。选它们，不是因为只有它们值得学，是因为它们足够典型，把这三款软件读透了，就获得了一套看任何软件的架构视角。以后遇到一个新系统，不只会问"它怎么用"，还会下意识地问"它为什么这么设计、是如何做取舍的"。这种能力不会因为换了一份工作、换了一套技术栈而过时，这种能力会随着你见过的系统越来越多而持续上升。这才是架构设计能力的真正增长方式：不是学完三款软件就结束了，而是从此看任何软件，都可以从架构视角去审视和分析。

这本书不讲最新的调优技巧，不讲基础知识，也不逐行贴源码。我会讲更稳定的东西：不管什么系统，只要它要记东西、不能丢，那些反复出现的设计抉择：速度还是持久、一致还是可用、简单还是灵活、通用还是专用——以及做出这些抉择时必须支付的代价。版本会更迭（本书基于 Redis 7.x、MySQL 8.0.x、Kafka 3.x），但这些抉择的逻辑长期不变。

读这本书的方式，建议是带着比较去读。每一章都会从这三个软件架构的维度横向对比、提炼启示。读着读着就会发现，差异底下藏着同一套骨架：先定真相之源、用层包裹可变性、把随机变顺序、用性能买可靠性、把复杂状态显式建模。

这本书面向的，是有一定使用经验、想跨过"会用"这道坎、走到"懂取舍"那一侧的人。如果读完之后，再面对任何一个新系统，第一反应不再是"它怎么用"，而是"它为什么这么设计、反过来会怎样、这套取舍的边界在哪"——那么这本书就达到了它的目的。更理想的是，某天我们自己设计系统时，能下意识地去考虑不同方案的取舍问题——那才是这本书最希望发生的事。

是为序。
