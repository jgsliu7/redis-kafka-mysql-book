《架构之道：从 Redis、MySQL、Kafka 看软件设计的共性》——中文技术书，通过横向对比 Redis / MySQL / Kafka 提炼可复用的软件架构设计规律。

- **构建**：`scripts/build_html.py`（单页）、`scripts/build_pages.py`（分章）
- **规范**：见 §3 参考文档索引

## 4. 审查执行流水线

### 阶段 0：确定性基线（快，先跑，10 分钟）

```bash
python3 scripts/svg_audit.py          # 插图几何审计
python3 scripts/build_html.py          # 构建验证
# 运行去 AI 味黑名单扫描脚本         # 见去ai味道自查方法.md §6
# 交叉引用脚本（可选，如需自动化）    # 见出版审查清单与盲区.md §4
```

### 阶段 1：多维度只读审查（并行 agent，零冲突）

按 §1 矩阵派 agent，至少覆盖 C2-C7+T1-T5+T8+S1-S6+R1+R4+P1。每个 agent 负责一个维度全书通读。

**铁律**：语义类问题（句式/逻辑/导读兑现/启示含金量/声音分布）**必须 agent 逐段通读**，**严禁用 grep 关键词自证"已清零"**。

### 阶段 2：分区修复（并行 agent，按文件独占）

阶段 1 产出的 findings 按文件路由，每个文件一个 fix agent（独占），带 FIX/SKIP 清单。

### 阶段 3：回归 + 对抗复检

重跑阶段 0 断言 + 派 agent 复检改动最重处 → 查回退和改坏。

### 阶段 4：飞轮沉淀

新发现的病理 → 命名 → 补进 `去ai味道自查方法.md` 或本文 §5。

## 6. 强制规则（铁律）

### 规则 0：总调度只做调度，不做具体工作

总调度（你）的职责是**编排、派发、综合、决策**。任何需要"读文件→分析→产出结论"的实质性工作，必须派子 agent 完成，不得自己 inline 做。

- ✅ 正确：发现一个疑点 → 派 2-3 个不同视角 agent 独立判断 → 综合他们的结论 → 决策
- ❌ 错误：自己读完一个章节后直接判断"这段话有问题"

### 规则 1：只读分析——永远多 agent，永远交叉验证

**适用于**：策划分析、检查、审查、评估——任何不产生文件改动的场景。

**做法**：对同一个问题，派 ≥2 个不同视角的 agent 独立分析，看结论是否一致。

- 一致 → 结论可信，继续
- 不一致 → 综合分歧点，总调度决定是否需要第三个 agent 打破僵局
- ≥2 同意是采信阈值，1/3 同意不算通过

**为什么**：单个 agent（或单个人、单个 AI）有盲区。交叉验证是唯一降低误判率的手段。

### 规则 2：改动文件——改前多 agent，改时单 agent，改后多 agent

**适用于**：任何对文件的修改。

**三阶段**：

1. **改前分析（多 agent）**：派 ≥2 个 agent 独立判断"是否该改"+"怎么改"。≥2 同意才进入修改。
2. **执行修改（单 agent 或总调度直接 Edit）**：因为文件冲突风险，只有一个人动手。改动必须精确（用 Edit，不整段重写）。
3. **改后验证（多 agent）**：派 ≥2 个 agent 独立检查——改动正确吗？引入新问题了吗？上下文衔接断了吗？

**为什么**：分析阶段多 agent 是为了"做对决策"；修改阶段单 agent 是为了"避免冲突"；验证阶段多 agent 是为了"确认没改坏"。

### 规则 3：语义类不用 grep，agent 通读是唯一手段

AI 味句式/逻辑类是语义问题。grep 既漏报（换措辞就看不见）又给虚假"0 残漏"信心。词汇类（黑名单词）才能 grep。

### 规则 4：发现问题 → 命名病理 → 全扫同类 → 统一修

修完一处即止 = 同类残留。每发现一个新问题：(1) 命名病理（像语言学家）；(2) 全扫确定全书面貌；(3) 逐处按规则 2 流程修复；(4) 补进 `去ai味道自查方法.md` 或本文 §5。

### 规则 5：checklist 写了 ≠ agent 跑了

本书审查维度覆盖很全，但历史执行严重偏斜。总调度必须在每轮审查前列出"本轮要跑的维度清单"，确保之前只存在于 checklist 上的维度真正被执行。

### 规则 6：grep 断言只锁字串类回归

语义属性（"CAP 归类是否 hedged""AI 味是否消除"）不能用 grep 断言锁死。权威回归检查 = agent 通读。grep 断言只是"已知坏字串没回弹"的快速兜底。

### 规则 7：裸数字三选一（补出处 / 补前提 / 改定性）

技术书的量化数字（TPS、延迟、容量、百分比）必须有可追溯的依据。每个裸数字按以下优先级处理：

1. **补出处**：如果能找到官方基准/论文/社区实测，直接引用 —— `（据 redis-benchmark 基线，……）`
2. **补前提**：如果数字本身合理但缺上下文，补限定条件 —— `（假定每条消息约 1KB，NVMe SSD 环境）`
3. **改定性**：如果出处和前提都给不了，把精确数字改成定性表述 —— `吞吐显著下降` 替代 `降至六到八成`

**执行方式**：`number-provenance` agent 全书扫荡 → WebSearch 核实关键数字 → 对每个裸数字选方案并给出 old_string → new_string。

**已修复的典型**：Kafka 单分区吞吐严重低估（1 万条/秒 → 1-5 万条/秒 + 小消息可达 10 万+）、zstd 压缩比虚高（3-10x → 2-6x 补前提）、NVMe 吞吐过时（2-7GB/s → 2-14GB/s 补 PCIe 代际）。

## 7. 约束

- **改字词不改意思**：去 AI 味只改表达方式，不增删信息、不动事实/数字/命令。
- **绝对保留**：作者第一人称真实案例、取舍段判断锋芒、结构性锚点比喻（真相之源/骨架/状态机）、优雅(graceful)/同题作答/章节副标题。
- **术语两档制**：英文优先（redo log/binlog/WAL/MVCC/ISR/GTID/offset/compaction/fsync）首次标注后继续用英文；中文优先（缓冲池/分区/快照/副本/刷盘/日志段）用中文。详见 `docs/book-bible.md` §7 术语对照表。
- **offset ≠ 位移**："位移"是物理学术语（displacement）。全书"位移"应改为"偏移量（offset）"。
- **破折号已收敛**：全书"——"已压到个位数。
- **不碰 outline.md**：草稿，build 不读。
- **改了章节就要重建 HTML**：每次修改 `chapters/` 下任何 md 文件后，必须运行：
  ```bash
  python3 scripts/build_html.py    # 单页 → 架构之道.html（git tracked）
  python3 scripts/build_pages.py   # 分章 → dist/（gitignored）
  ```
  然后 `git add 架构之道.html` 随修改一起提交。
- **dist/ 不在 git 里**，但 `架构之道.html` 是 tracked。
- **本环境无法看图**：视觉验证靠像素渲染 + 源码坐标推理。
